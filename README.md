# learning-video-recommendation-system

# 基于视频内容的学习推荐系统 MVP 整体设计文档

## 1. 文档目的

本文档用于描述当前 MVP 阶段的整体系统设计。
系统的核心目标是：**围绕用户当前学习目标，结合学习历史与视频内容，生成适合当前阶段的学习任务。**

该产品不是传统的信息流推荐系统，也不是纯粹的单词记忆工具，而是一个面向“**通过视频学习学习内容**”场景的学习调度与内容匹配系统。这里的“学习内容”不是永久限定为单词，而是统一落在 `semantic.coarse_unit` 这一学习内容层之上；当前系统底层已经支持至少三类学习单元：

- `word`
- `phrase`
- `grammar`

本文档重点解决四个问题：

1. 当前用户最应该学习哪些学习内容
2. 哪些视频内容最适合承载这些学习目标
3. 如何将学习目标与视频内容组织成最终可消费的学习任务
4. 推荐系统模块如何以**一个完整后端能力模块**的形式，对上层业务 API 提供结果，而不是把内部三层分别暴露出去

---

## 2. 产品目标

### 2.1 核心目标

系统希望帮助用户在视频语境中高效学习学习内容，并同时兼顾以下目标：

- 记忆保留
- 新内容持续推进
- 学习负担控制
- 内容消费体验
- 推荐结果可解释性
- 系统行为可调试性

### 2.2 MVP 阶段的设计原则

#### 1. 先保证学习闭环成立

优先打通“目标学习内容生成 -> 视频匹配 -> 学习任务输出 -> 学习反馈回流”的完整流程，而不是一开始追求复杂模型。

#### 2. 先做可控系统，不做黑盒系统

当前重点是构建一个规则清晰、行为可解释、易于调试的推荐框架，而不是直接引入复杂机器学习推荐模型。

#### 3. 先围绕统一学习单元建模

虽然当前产品叙事经常从“单词学习”出发，但底层调度对象不应写死为单词。
MVP 阶段的稳定抽象是 `semantic.coarse_unit`，而不是未来需要被推翻的“词表”概念。

#### 4. 最终输出应是学习任务，而不是单纯视频列表

系统推荐的终点不是“你可能想看什么视频”，而是“你现在适合完成什么学习任务”。

#### 5. 推荐系统应作为一个完整后端模块对外提供能力

对上层业务 API 来说，推荐系统是**一个整体模块**。
上层业务 API 不分别调用“学习内容调度器”“视频召回层”“学习任务层”，而是只调用推荐系统模块；该模块内部自行完成三层处理，并返回最终学习任务结果。

---

## 3. 系统定位

从系统本质上看，本产品内部可以拆成三层能力，但这三层只用于**模块内部组织**，不构成对上层暴露的三个独立服务。

### 3.1 学习目标生成

回答：**现在最该学什么。**

这一层基于：

- 用户目标学习内容集合
- 用户-学习内容状态
- 学习历史
- 复习节奏模型
- 配额与负载控制规则

输出当前时刻最适合学习的一组目标学习内容。

### 3.2 内容匹配与组织

回答：**这些学习目标应该通过什么内容来学。**

这一层基于：

- 目标学习内容集合
- 视频 transcript
- 视频与学习内容映射关系
- 视频覆盖质量
- 语境清晰度
- 信息负担与学习收益

输出覆盖这些学习目标的候选视频集合。

### 3.3 学习任务生成

回答：**最终应该交付什么。**

这一层将目标学习内容与候选视频组织成最终可消费的学习任务。
它输出的不是抽象学习内容列表，而是前端最终需要消费的任务实体，例如：

- 推荐学习哪个视频
- 本次重点学习哪些学习内容
- 哪些是复习内容，哪些是新内容
- 推荐原因是什么

---

## 4. 系统边界与调用关系

这是本次修订最关键的部分。

### 4.1 外部边界

推荐系统模块对外只表现为**一个后端内部能力模块**。
上层业务 API 调用它，请求某个用户当前的学习任务推荐结果；推荐系统模块内部完成全部三层处理后，返回最终学习任务结果给上层业务 API，再由上层业务 API 返回前端。

### 4.2 内部边界

推荐系统模块内部包含三个子层：

1. 学习内容调度器
2. 视频召回层
3. 学习任务层

但这三层只是**模块内部流水线**，不直接暴露给上层业务 API，也不直接面向前端。

### 4.3 正确的调用关系

推荐生成的正确调用链路如下：

```text
前端
  ↓
[上层业务 API]
  ↓
[推荐系统模块]
  ├── 学习内容调度器
  ├── 视频召回层
  └── 学习任务层
  ↓
最终学习任务结果
  ↓
[上层业务 API 返回前端]
```

### 4.4 为什么要这样设计

这样设计有四个直接好处：

第一，推荐系统可以与上层业务接口协议解耦。
第二，推荐链路内部可以独立迭代，而不影响前端接口形态。
第三，内部三层之间的数据结构可以保持面向推荐系统本身优化，而不是被前端展示协议绑死。
第四，推荐系统的职责边界更清晰：它输出的是完整学习任务结果，而不是让上层 API 再自己拼装三层结果。

## 5. 整体系统架构

当前 MVP 建议拆分为 4 个文档对应的 4 个核心部分：

1. **整体设计文档**
2. **学习内容调度器**
3. **视频召回层**
4. **学习任务层**

其中本文档是总览文档，其他三个文档分别描述推荐系统模块内部各子层的设计。

### 5.1 模块级视角

```text
用户学习状态 + 学习历史 + 视频内容索引
                ↓
          [推荐系统模块]
                ├── 学习内容调度器
                ├── 视频召回层
                └── 学习任务层
                ↓
         输出：最终学习任务列表
```

### 5.2 内部数据流视角

```text
用户目标学习内容 + 学习状态 + 学习历史
                ↓
          [学习内容调度器]
                ↓
      输出：目标学习内容列表
                ↓
            [视频召回层]
                ↓
        输出：候选视频列表
                ↓
            [学习任务层]
                ↓
      输出：最终学习任务列表
```

从内部数据流上看，系统不是“直接推荐视频”，而是：

> **先生成学习目标，再为学习目标找到最适合的视频内容，最后包装成学习任务。**

## 6. 核心概念定义

为了避免模块边界不清，先统一几个核心概念。

### 6.1 目标学习内容表

指用户当前计划学习的内容集合。
它的底层载体是 `semantic.coarse_unit`，不是某个只属于单词的专用表。当前系统中，学习内容至少包含：

- word
- phrase
- grammar

每个学习内容可以带有：

- 优先级
- 当前进度
- 当前状态
- 来源
- 所属课程或主题

这部分定义了用户“要学什么”。

### 6.2 学习事件

指用户一次真实学习行为的标准化记录，例如：

- 学习了哪个学习内容
- 学习时间
- 学习类型（新学 / 复习 / 测验 / 曝光）
- 是否答对
- 本次质量评分
- 来源任务或来源视频

这部分是系统最底层的行为事实来源。

### 6.3 学习内容状态

指系统对每个“用户-学习内容”当前状态的聚合结果，例如：

- 是否学过
- 当前进度
- 当前属于 `new / learning / reviewing / mastered / suspended` 哪个阶段
- 下次推荐复习时间
- 最近表现是否稳定

这部分是学习内容调度器的主要输入。

### 6.4 目标学习内容列表

指由学习内容调度器输出的、本轮应优先学习的一组学习内容。
它不是最终用户看到的结果，而是推荐系统模块内部中间层结果。

每个目标学习内容通常应带有：

- `coarse_unit_id`
- 内容本身
- 内容类型：word / phrase / grammar
- 推荐类型：new / review
- 当前权重或优先级
- 推荐原因

### 6.5 候选视频列表

指视频召回层基于目标学习内容集合召回的一批视频内容。
当前视频召回的基本单位是**视频片段**，而不是长视频整段，因为现有视频片段已经 transcript，并且已经映射到学习内容层。

### 6.6 学习任务

学习任务是推荐系统模块最终输出给上层业务 API 的结果实体。
它不是纯视频实体，而是一条带有学习语义的任务记录，例如：

- 推荐学习哪个视频
- 本视频覆盖哪些重点学习内容
- 哪些是复习内容
- 哪些是新内容
- 为什么推荐这条任务

## 7. 模块职责划分

## 7.1 学习内容调度器

### 目标

从用户全部目标学习内容中，筛选出当前最值得学习的一组目标内容。

### 主要职责

- 维护用户学习内容状态
- 接收并处理标准化学习事件
- 根据学习历史更新状态
- 根据复习模型决定旧内容何时应复习
- 根据配额规则控制新内容引入
- 输出本轮目标学习内容列表

### 输出

一个带有推荐类型、优先级、权重和原因码的目标学习内容列表。

### 不负责

- 不直接处理视频内容
- 不直接生成前端任务
- 不直接对上层业务 API 暴露独立接口

---

## 7.2 视频召回层

### 目标

基于目标学习内容列表，召回适合承载这些学习目标的视频片段。

### 主要职责

- 使用视频 transcript 和学习内容映射关系建立检索基础
- 根据目标学习内容覆盖情况召回候选视频
- 评估视频对当前学习目标的承载能力
- 输出候选视频列表给学习任务层

### 输出

候选视频列表，以及每个候选视频对目标学习内容的覆盖关系与匹配信号。

### 不负责

- 不独立决定用户当前该学什么
- 不直接负责最终任务包装
- 不直接面向上层业务 API 暴露接口

---

## 7.3 学习任务层

### 目标

将目标学习内容和候选视频组织成最终学习任务。

### 主要职责

- 将“视频内容”与“学习目标”绑定
- 生成任务级解释信息
- 控制最终输出结果的多样性与冗余
- 构造最终学习任务实体供推荐系统模块返回

### 输出

最终学习任务列表。

### 不负责

- 不负责底层复习调度
- 不负责底层 transcript 解析与视频索引构建
- 不直接处理前端原始行为

---

## 8. 核心数据流

下面描述一次典型推荐请求的内部主链路。

### 8.1 输入阶段

推荐系统模块接收上层业务 API 提供的用户上下文，包括：

- 用户标识
- 用户目标学习内容集合
- 用户学习内容状态
- 最近学习历史
- 当前请求上下文
- 当前 session 请求限制

### 8.2 学习目标生成阶段

学习内容调度器根据用户状态与学习历史，输出一组当前最适合学习的目标学习内容。
这些目标内容通常分为两类：

- 复习内容
- 新内容

同时还应附带每个内容的权重、推荐原因和调度属性。

### 8.3 视频召回阶段

视频召回层接收目标学习内容列表，然后基于视频 transcript 与学习内容映射数据，找出：

- 覆盖这些目标内容的视频
- 对当前学习目标匹配度更高的视频
- 信息负担相对合适的视频
- 语境质量相对更清晰的视频

输出候选视频集合。

### 8.4 学习任务生成阶段

学习任务层将候选视频与目标学习内容重新组织，构造出最终学习任务。
每条任务至少包含：

- 视频实体
- 覆盖的目标学习内容
- 重点复习内容 / 新内容
- 推荐原因

### 8.5 返回阶段

推荐系统模块把最终学习任务结果返回给上层业务 API。
上层业务 API 再把前端真正需要的响应返回给前端。

---

## 9. 学习反馈与状态回流

推荐系统不仅负责生成结果，也要参与学习闭环。

### 9.1 前端行为不直接写入调度器

前端产生的原始行为，例如：

- 看完一个视频
- 点开某个学习内容的解释
- 完成一次回忆题
- 完成一次测验
- 结束一轮学习任务

都不直接进入调度器。

### 9.2 上层业务 API 负责行为标准化

上层业务 API / 学习事件结算服务负责把前端原始行为转换成标准化学习事件，例如：

- `exposure`
- `lookup`
- `new_learn`
- `review`
- `quiz`

### 9.3 调度器负责状态更新

学习内容调度器接收这些标准化学习事件，更新用户-学习内容状态，并影响下一轮推荐。

## 10. MVP 的核心推荐逻辑

### 10.1 推荐的真实终点是学习任务

虽然系统内部存在“学习内容调度”和“视频召回”，但从产品视角看，最终推荐对象应是学习任务，而不是单独的内容项，也不是纯视频内容。

### 10.2 学习内容列表只是中间表示层

系统不应把学习内容列表误认为最终结果。
学习内容列表的作用是作为：

- 学习内容调度层的输出
- 视频召回层的输入
- 学习任务层的学习目标依据

它是推荐系统模块内部的中间语义层。

### 10.3 视频内容是学习载体，不是推荐目标本身

视频在这个系统里不是纯娱乐内容，而是学习媒介。
因此视频召回不能只考虑“用户想看什么”，还必须考虑：

- 是否覆盖当前目标学习内容
- 是否适合当前学习负担
- 是否有合理学习收益

### 10.4 推荐系统对外返回的是任务结果，不是内部中间结果

推荐系统模块对上层业务 API 的主返回结果应是最终学习任务。
内部中间结果如：

- 目标学习内容列表
- 候选视频列表
- 中间分数
- 原因码

主要用于模块内部流转、调试与解释，而不是作为对外主接口结果。

## 11. 当前数据库映射

为了让文档和当前数据结构一致，这里补充说明几个已经落地的核心表。

### 11.1 `auth.users`

该表是 Supabase Auth 的标准用户表，保存登录身份、邮箱、密码哈希、确认状态等认证信息。
在当前系统里，它的主要作用是：

- 提供学习行为的用户主体
- 作为推荐系统内部状态表与事件表的用户外键目标

### 11.2 `catalog.videos`

该表是视频主表，保存视频上传者、标题、描述、原始文件引用、处理状态、AI 分析状态、摘要、标签等信息。
它不是一个“纯媒体文件表”，而是一个带有学习语义的内容载体表。对推荐系统来说，它承担两层责任：

- 作为视频内容实体
- 作为学习任务最终承载物

### 11.3 `catalog.video_user_states`

该表是用户对视频互动状态的聚合表，当前包含：

- `has_liked`
- `has_bookmarked`
- `has_watched`

它更像行为投影表，而不是原始事件表。未来任务排序、去重、重复曝光控制，都可以基于这张表继续扩展。

### 11.4 `semantic.coarse_unit`

这张表非常关键。
它不是“将来扩展时才会用到的抽象层”，而是**当前就已经在承载具体学习内容**的主表。根据现有库说明，`semantic.coarse_unit.kind` 当前至少包括：

- `word`
- `phrase`
- `grammar`

这意味着：

- 单词只是 `semantic.coarse_unit` 的一种内容类型
- phrase 和 grammar 不是未来概念，而是当前模型已经支持的学习内容
- 推荐系统在整体设计上，应直接把“学习目标”理解为统一学习内容，而不是永远写死为单词

### 11.5 当前数据模型对系统设计的直接影响

由于 `semantic.coarse_unit` 已经是统一学习内容层，本文档里很多旧的“目标词”“单词状态”“目标单词列表”表述，在实现上都应理解为：

- 目标学习内容
- 学习内容状态
- 目标学习内容列表

未来演进时，优先扩展规则、匹配逻辑和任务表达，而不是重做底层表结构。

## 12. MVP 范围定义

### 12.1 当前包含的能力

在当前 MVP 中，系统应至少包含以下能力。

#### 学习目标侧

- 维护目标学习内容表
- 维护用户学习内容状态
- 基于复习策略生成新旧内容混合的目标学习内容列表

#### 内容侧

- 视频片段已完成 transcript
- 视频片段已映射到学习内容层
- 能基于目标学习内容召回候选视频

#### 任务侧

- 能将候选视频与目标学习内容组织成学习任务
- 能输出最终学习任务列表供上层业务 API 返回前端

#### 闭环侧

- 能接收标准化学习事件
- 能根据学习反馈更新学习内容状态
- 能影响下一轮推荐结果

### 12.2 当前不包含的能力

MVP 阶段暂不重点处理以下问题：

- 复杂个性化深度学习排序
- 用户兴趣建模的重度优化
- 跨模态 embedding 召回
- 复杂 session 内多轮交互编排
- 长期课程路径优化

这些能力应在 MVP 验证通过后再逐步增强。

## 13. 系统设计上的关键判断

### 13.1 为什么先以学习内容为核心，而不是直接以视频为核心

因为学习产品首先需要确定“学什么”，再决定“通过什么内容学”。
如果直接从视频出发，系统会更像内容消费系统，而不是学习系统。

### 13.2 为什么推荐系统要作为一个整体模块对外提供能力

如果上层业务 API 需要分别调用调度器、视频召回层、任务层，就会把推荐系统的内部组织暴露到外部边界上，导致：

- 上层 API 与内部实现强耦合
- 内部数据结构不稳定
- 后续重构困难
- 责任边界模糊

把推荐系统收敛成一个整体模块，对外只返回最终学习任务结果，可以更好地解耦上层业务 API 与内部实现。

### 13.3 为什么学习任务层必须存在

如果只有“学习内容调度 + 视频召回”，系统很容易停留在“给用户一个视频列表”。
但教育产品需要的是任务表达，而不是纯内容流表达。

把学习任务层单独抽出来，有助于：

- 明确最终推荐对象
- 增强结果解释性
- 支持后续扩展出练习、测试、复盘等能力

### 13.4 为什么视频召回层应和调度层分离

学习内容调度解决的是学习节奏问题，视频召回解决的是内容匹配问题。
这两者属于不同空间的问题，混在一起会让系统很难调试和演进。

## 14. 潜在风险与设计注意点

### 14.1 不要把“视频包含目标学习内容”误当成“视频适合学习目标”

一个视频即使覆盖了目标学习内容，也未必适合学习，因为还会受以下因素影响：

- 目标内容出现频次
- 出现语境是否清晰
- 语言难度是否过高
- 非目标陌生内容是否过多

因此视频召回层不能只做简单包含匹配。

### 14.2 不要把“学习内容列表”误当成最终产品结果

学习内容列表是推荐系统内部中间层，不是最终用户价值交付层。
用户最终消费的是学习任务，而不是抽象学习内容集合。

### 14.3 不要过早做复杂推荐模型

当前系统最重要的是链路成立和逻辑清晰，而不是模型先进。
过早引入复杂排序会掩盖核心问题，并增加调试成本。

### 14.4 不要把模块内部边界暴露给上层 API

上层业务 API 只应依赖推荐系统模块整体返回结果，而不应依赖内部三层的中间结构。否则会破坏解耦。

### 14.5 需要为未来扩展保留空间

虽然当前产品叙事仍经常以“单词学习”为核心，但数据层已经提前保留了扩展空间。
未来仍可能继续扩展到：

- sentence pattern
- topic
- 其他 coarse unit 类型

因此整体设计上应把“学习目标”理解为可扩展抽象，而不是永远写死为单词。

## 16. 总结

当前 MVP 产品的整体逻辑应理解为：

> **用户并不是被直接推荐“视频”，而是被推荐“通过某些视频完成某些学习目标的学习任务”。**

同时，推荐系统的系统边界应明确为：

> **上层业务 API 调用推荐系统模块；推荐系统模块内部再依次完成学习内容调度、视频召回、学习任务生成，并把最终学习任务结果返回给上层业务 API。**

因此系统整体设计应分为三个内部子层：

1. **学习内容调度层**：决定现在该学什么
2. **视频召回层**：决定通过什么内容来学
3. **学习任务层**：决定最终以什么任务形式交付

而在对外边界上，它们共同构成一个完整的推荐系统模块。

这套设计在 MVP 阶段具备以下优点：

- 逻辑清晰
- 模块边界明确
- 对上层业务 API 解耦
- 推荐结果可解释
- 容易调试和迭代
- 后续可自然扩展到更复杂的学习目标和排序能力
