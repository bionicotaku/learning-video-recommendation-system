# 学习内容调度系统工程实现稿（Go / Supabase / AI Agent 友好版）

## 1. 工程目标

本模块的目标是在推荐系统模块内部落地一个**可被上层推荐编排逻辑调用的学习内容调度子系统**，完成以下核心能力：

一，维护 `user × coarse_unit` 的状态快照。
二，接收标准化学习事件并更新状态。
三，根据当前状态生成一轮目标学习内容列表。
四，支持事件回放重建状态，保证以后修改规则后可修复。
五，把结果以内部中间结构返回给推荐系统模块后续子层，而不是直接给前端。

这次实现**不包含**视频召回层本身和学习任务层本身，只要求把调度系统做成一个可集成、可测试、可验收的内部模块，供推荐系统模块后续使用。

---

## 2. 最终技术方案

### 2.1 唯一采用的方案

**Go + `sqlc` + PostgreSQL 直连 + `pgx/v5` 驱动 + 纯领域规则 + 应用层编排**

这个方案的核心优点不是“最灵活”，而是**最适合 AI agent**：

- SQL 被固定在 query 文件中，字段映射错误率显著下降
- 领域规则与 SQL 分离，agent 不容易把业务逻辑塞进仓储层
- 事务边界容易统一管理
- integration test 更容易写
- replay 与在线状态更新能共用同一套领域引擎
- 后期你人工 review 也更容易，因为结构非常固定

### 2.2 为什么不用手写 repository + 原生 SQL

不采用这个方案，不是因为它不强，而是因为它**不适合当前目标**。
你现在不是追求极限灵活性，而是追求：

- AI agent 实现时不乱写
- 每一层职责清楚
- 规则能验收
- 回放能重建
- 改规则后不崩

手写 repository 的问题是，AI agent 很容易把下面这些混在一起：

- SQL
- Scan
- 状态判断
- 配额规则
- 排序逻辑
- 事务控制

最后你会得到一堆“能工作但不可维护”的 Go 文件。
所以这次工程稿把方案收紧为：**所有主查询和写入都由 `sqlc` 生成，业务逻辑全部上移到 application / domain 层。**

## 3. Supabase 数据库访问方式

这里需要非常明确、客观地说明。

### 3.1 数据库平台采用 Supabase

数据库平台确定为 **Supabase**。
当前已有的核心表结构语义已经基于 Supabase/PostgreSQL 体系，包括 `auth.users`、`catalog.videos`、`catalog.video_user_states`、`semantic.coarse_unit` 等。

### 3.2 本模块的访问方式必须是 PostgreSQL 直连

具体访问信息在 `.env` 里的 `DATABASE_URL`，但从工程实现角度必须明确：

> **学习内容调度系统不能通过 Supabase 的 HTTP API / REST 接口作为核心访问方式。**
> **本模块必须使用 PostgreSQL 直连。**

原因很简单，这个模块高度依赖：

- 事务一致性
- 批量查询
- 状态与事件同事务提交
- replay 顺序回放
- 高强度读写与可控 SQL

这些都不适合用 Supabase 的 HTTP 封装方式去做。

## 4. 项目分层与目录结构

这部分是整个工程稿最重要的地方之一。
你后面让 AI agent 写代码时，最容易出问题的不是“规则写错”，而是**分层写乱**。所以这里必须写得非常死。

推荐目录如下：

```text id="g886al"
internal/
  recommendation/
    scheduler/
      domain/
        enum/
        model/
        policy/
        rule/
        service/
      application/
        command/
        query/
        dto/
        usecase/
      infrastructure/
        persistence/
          query/
          repository/
          tx/
          mapper/
        migration/
      integration/
        contract/
        mapper/
      test/
        fixture/
        integration/
        replay/
```

### 4.1 各层职责

#### `domain/`

只放纯领域规则与对象。
不能依赖数据库、不能依赖 HTTP、不能依赖 Supabase SDK。

#### `application/`

只放用例编排。
它负责把 repository、事务、领域规则组合起来，形成完整用例。

#### `infrastructure/`

放 `sqlc` 生成代码、PostgreSQL 持久化实现、事务适配器、migration。

#### `integration/`

放调度系统输出给推荐系统模块后续子层的契约和映射器。

#### `test/`

放夹具、integration 测试、replay 测试。

### 4.2 一条硬性约束

**不允许把业务规则写进 repository。**
这一条必须作为 AI agent 的硬性约束。任何以下行为都算结构失败：

- 在 repository 里做状态迁移
- 在 SQL 查询里偷做复杂排序决策
- 在 persistence mapper 里偷偷改业务字段
- 在事务层里塞 scoring 逻辑

## 5.4 `domain/rule` 与 `domain/service`

这一层最重要。
AI agent 最容易在这里写烂，所以这里必须按职责拆细。

### 模块 1：弱事件处理器

输入：

- 当前状态 `UserUnitState | nil`
- 事件 `LearningEvent`

输出：

- 更新后的 `UserUnitState`

规则：

#### `exposure`

更新：

- `SeenCount += 1`
- `LastSeenAt = OccurredAt`

不更新：

- `Repetition`
- `IntervalDays`
- `EaseFactor`
- `NextReviewAt`
- `Status`

#### `lookup`

更新：

- `SeenCount += 1`
- `LastSeenAt = OccurredAt`

仍然不推进正式调度。

### 模块 2：强事件基础更新器

所有强事件统一更新：

- `SeenCount += 1`
- `StrongEventCount += 1`
- `LastSeenAt = OccurredAt`

若 `EventType in ('review','quiz')`：

- `ReviewCount += 1`
- `LastReviewedAt = OccurredAt`

若 `IsCorrect = true`：

- `CorrectCount += 1`
- `ConsecutiveCorrect += 1`
- `ConsecutiveWrong = 0`

若 `IsCorrect = false`：

- `WrongCount += 1`
- `ConsecutiveWrong += 1`
- `ConsecutiveCorrect = 0`

若 `Quality != nil`：

- 更新 `LastQuality`

### 模块 3：SM-2 更新器

建议单独做纯领域服务。

```go id="t2yu23"
type SM2Updater interface {
    Apply(state *UserUnitState, quality int, occurredAt time.Time, policy SchedulerPolicy)
}
```

#### 成功分支：`quality >= 3`

- `Repetition += 1`
- `IntervalDays = 1 / 3 / 6 / interval * EF`
- `ConsecutiveCorrect += 1`
- `ConsecutiveWrong = 0`
- 更新 EF
- 更新 `NextReviewAt`

#### 失败分支：`quality < 3`

- `Repetition = 0`
- `IntervalDays = 1`
- `NextReviewAt = occurredAt + 1 day`
- **EF 不变**

#### EF 更新公式

[
EF_{new} = EF_{old} + (0.1 - (5-q)\times(0.08 + (5-q)\times0.02))
]

并保证：

[
EF_{new} \ge 1.3
]

### 模块 4：状态迁移器

```go id="l0p40u"
type StatusTransitioner interface {
    Recompute(state *UserUnitState, recentQualities []int, policy SchedulerPolicy)
}
```

规则：

- `new -> learning`：第一次强事件
- `learning -> reviewing`：`StrongEventCount >= 2` 且最近两次 `quality >= 3`
- `reviewing -> mastered`：`IntervalDays >= 21` 且近期稳定且 `ConsecutiveWrong = 0`
- `mastered -> reviewing`：近期明显失败或不稳定

### 模块 5：Progress 计算器

```go id="wvzjlwm"
type ProgressCalculator interface {
    Compute(intervalDays float64, policy SchedulerPolicy) float64
}
```

公式：

[
progress_percent =
\min\left(
100,
\frac{\ln(interval_days + 1)}{\ln(22)} \times 100
\right)
]

### 模块 6：Mastery 计算器

```go id="eb097h"
type MasteryScoreCalculator interface {
    Compute(state *UserUnitState, recentAccuracy float64, policy SchedulerPolicy) float64
}
```

公式：

[
stability_score = \min(1, interval_days / 21)
]

[
mastery_score =
0.45 \times (progress_percent / 100)

- 0.35 \times recent_accuracy
- 0.20 \times stability_score
  ]

### 模块 7：状态更新引擎

最后把上面几块组合成一个纯领域服务：

```go id="6lahy1"
type StateUpdater interface {
    Apply(current *UserUnitState, event LearningEvent, ctx UpdateContext) (*UserUnitState, StateTransitionResult, error)
}
```

`UpdateContext` 里包含：

- `SchedulerPolicy`
- 最近事件摘要
- 当前时间

### 验收要求

这一层必须通过纯单元测试覆盖以下场景：

1. `exposure` 不推进 interval
2. `lookup` 不推进 status
3. 第一次强事件使 `new -> learning`
4. 两次有效强事件使 `learning -> reviewing`
5. `quality >= 3` 正确更新 `repetition / intervalDays / EF`
6. `quality < 3` 时 `Repetition=0`，`Interval=1`，EF 不变
7. `IntervalDays >= 21` 且稳定时进入 `mastered`
8. 已掌握内容再次失败会降回 `reviewing`

如果这一层不过，后面所有开发都不要继续。

## 7. `sqlc` 方案设计

这是本工程稿的确定性方案核心。

## 7.1 为什么必须采用 `sqlc`

原因总结为四个字：

> **减少自由度**

你这个模块最怕的是：

- SQL 写对了，Scan 错了
- 状态机规则混进 repository
- agent 自己发明新的 persistence struct
- Go struct 和 SQL 字段不一致
- 后期 migration 改了，repository 没同步

`sqlc` 正好把这些问题压下去：

- SQL 是显式文件
- 生成的 Go 结构是确定的
- query 层职责更清楚
- AI agent 只能在上层编排和领域规则里做事

## 7.2 `sqlc` 配置建议

建议配置文件：

```yaml id="rsx67b"
version: "2"
sql:
  - engine: "postgresql"
    queries: "internal/recommendation/scheduler/infrastructure/persistence/query"
    schema: "internal/recommendation/scheduler/infrastructure/migration"
    gen:
      go:
        package: "sqlcgen"
        out: "internal/recommendation/scheduler/infrastructure/persistence/sqlcgen"
        sql_package: "pgx/v5"
        emit_json_tags: true
        emit_interface: false
        emit_exact_table_names: false
        overrides:
          - db_type: "uuid"
            go_type: "github.com/google/uuid.UUID"
```

### 原则

- `sqlc` 只生成 SQL 对应的 struct 和 query 方法
- 领域对象永远不要直接复用 `sqlc` struct
- 必须加 mapper 层，把 `sqlc` 输出转成领域对象

这是为了避免领域层被 persistence 层反向污染。

## 8.2 事务抽象

状态写入与事件写入必须在同一个事务里，所以需要一个统一的 TxManager。

```go id="k7hncv"
type TxManager interface {
    WithinTx(ctx context.Context, fn func(ctx context.Context, q db.Querier) error) error
}
```

实现方式：

- 底层用 `pgxpool.Pool`
- `WithinTx` 内部开 `BeginTx`
- 用 `sqlcgen.New(tx)` 构造事务 querier

### 为什么要这样抽象

因为这样 application 层只关心“需要事务”，不关心底层是 pool 还是 tx。

## 10. 配额与排序引擎设计

这部分建议全部写在 `domain/service` 与 `application/usecase` 之间，不要进 repository。

## 10.1 BacklogCalculator

输入：

- `reviewBacklog`
- `requestedLimit`
- `UserSchedulerSettings`

输出：

- `ReviewQuota`
- `NewQuota`
- `BacklogProtection`

规则保持与你设计稿一致：

当 `R = 0`：

- `reviewQuota = 0`
- `newQuota = min(N, dailyNewUnitQuota)`

当 `1 <= R <= 5`：

- `reviewQuota = ceil(0.5N)`
- `newQuota = min(N-reviewQuota, dailyNewUnitQuota)`

当 `6 <= R <= 20`：

- `reviewQuota = ceil(0.7N)`
- `newQuota = min(N-reviewQuota, dailyNewUnitQuota)`

当 `21 <= R <= softLimit`：

- `reviewQuota = ceil(0.85N)`
- `newQuota = min(3, N-reviewQuota)`

当 `R > softLimit`：

- `reviewQuota = N`
- `newQuota = 0`

当 `R > hardLimit`：

- `reviewQuota = min(N, hardLimit)`
- `newQuota = 0`
- `backlogProtection = true`

公式：

0.45 \times overdue_score

- 0.25 \times target_priority
- 0.20 \times weak_memory_score
- 0.10 \times recency_adjustment
  ]

其中：

[
overdue_score = \min\left(1,\frac{now-next_review_at}{72h}\right)
]

[
weak_memory_score =
\min\left(
1,
0.5 \times (1-mastery_score)

- 0.3 \times \frac{consecutive_wrong}{3}
- 0.2 \times quality_penalty
  \right)
  ]

## 12. 事务边界与一致性要求

这套系统最重要的一条工程约束是：

> **事件写入与状态更新必须在同一个数据库事务中完成。**

因为如果你先写事件、状态异步补，就会出现：

- 推荐读取旧状态
- 回放与在线状态不一致
- 某些事件写入成功但状态丢失

所以 `RecordLearningEventsAndUpdateState` 必须是完整事务。

## 13. 测试与验收设计

### 13.1 领域层单元测试

必须覆盖：

1. `exposure` 不推进 interval
2. `lookup` 不推进 status
3. 第一次强事件 `new -> learning`
4. 两次有效强事件 `learning -> reviewing`
5. `quality >= 3` 正确更新 `repetition / intervalDays / EF`
6. `quality < 3` 时 `repetition=0`、`interval=1`、EF 不变
7. `intervalDays >= 21` 时进入 `mastered`
8. 已掌握内容再次失败会降回 `reviewing`

这一层不过，后面所有集成都没有意义。

### 13.2 集成测试

至少要有这些场景：

#### 场景 A：新用户初次学习

- 一批 target units 初始全是 `new`
- 调度结果以 new 为主
- 写入一次 `new_learn`
- 状态进入 `learning`

#### 场景 B：正常复习推进

- 连续成功强事件
- 间隔按 `1 / 3 / 6 / ...` 增长
- 达到 21 天后进入 `mastered`

#### 场景 C：失败回退

- 已 reviewing 内容失败一次
- interval 重置为 1
- nextReviewAt 提前
- EF 不衰减

#### 场景 D：backlog 保护

- 制造大量 due 内容
- 调度结果不引入 new
- `backlogProtection = true`

#### 场景 E：统一支持 kind

- 同时插入 word / phrase / grammar
- 都能被调度
- 不因为 kind 不同而崩

### 13.3 回放验收

必须验证：

- 在线逐步写入事件得到的最终状态
- 与离线 replay 同样一批事件得到的最终状态

两者完全一致。
如果做不到这点，后面你改规则几乎没法修系统。

## 16. 最终结论

这版工程稿的唯一实现方案已经确定：

> **Go + `sqlc` + PostgreSQL 直连（Supabase 承载）+ 纯领域规则 + 应用层编排 + 回放可重建。**

这是当前最适合 AI agent 的实现方式，因为它：

- 把 SQL 层固定住，减少 agent 自由发挥
- 把业务规则集中在领域层，便于验收
- 把事务和用例集中在应用层，便于控制
- 把状态与事件分层，便于 replay 和修复
- 与你已经确定的系统边界完全一致：调度系统只是推荐系统模块内部子系统，不是前端 API，不是独立对外服务。
