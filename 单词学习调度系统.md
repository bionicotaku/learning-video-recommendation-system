# 单词学习调度系统设计文档

## 1. 文档目标

本文档定义 MVP 阶段的单词学习调度系统设计。该系统位于整个产品链路的上游，其职责是基于用户的目标词表、单词当前学习状态以及最近学习历史，生成当前一轮最值得学习的一组目标单词列表。这个列表将作为后续视频召回层的输入，而不是直接面向用户展示。

本系统解决的核心问题不是“用户喜欢什么内容”，而是“用户现在最应该学什么词”。因此它本质上是一个**学习调度系统**，而不是传统内容推荐系统。

在当前产品中，用户最终消费的是视频学习任务，但在视频推荐之前，必须先确定本轮学习目标。单词学习调度器正是负责生成这一层中间语义表示。

---

## 2. 系统职责与边界

单词学习调度系统负责三件事。

第一，维护每个“用户-单词”的当前学习状态。  
第二，根据记忆模型和业务规则决定哪些旧词应进入复习。  
第三，根据新词引入策略和当前复习压力决定哪些新词应被加入本轮学习目标。

它不处理视频 transcript，不做视频召回，不直接生成前端展示任务，也不负责用户兴趣内容排序。这些属于后续模块的职责。

系统的输出是**目标单词学习列表**。列表中的每个单词会带有类型、优先级、推荐原因和必要的调度上下文，以便视频召回层继续使用。

---

## 3. 总体架构

单词学习调度系统内部可拆分为五个逻辑模块：

1. 学习事件写入模块
    
2. 用户-单词状态更新模块
    
3. 复习候选生成模块
    
4. 新词候选生成模块
    
5. 调度排序与配额模块
    

这五个模块共享同一套数据库表，并共同产出最终的目标单词学习列表。

整体链路如下：

```text
用户完成一次学习
   ↓
写入 learning_event
   ↓
更新 user_word
   ↓
用户请求新一轮学习推荐
   ↓
生成 review 候选
   + 生成 new 候选
   ↓
按动态配额排序和混合
   ↓
输出 target_word_list
```

---

## 4. 数据模型设计

当前阶段采用一个明确的设计决定：

> **用户-单词表同时包含学习目标和学习状态。**

也就是说，不单独拆出 `user_word_plan` 和 `user_word_state`，而是合并到一张 `user_word` 表中。这样做的前提是当前 MVP 默认一个用户主要围绕一套目标词表学习，不需要复杂的多计划复用能力。这个设计可以显著降低实现复杂度，同时仍然足以支撑后续调度逻辑。

数据库至少包含四张核心表：

- `word`
    
- `video`
    
- `user_word`
    
- `learning_event`
    

其中 `video` 不是本调度器核心表，但由于单词表通常要和视频词表映射一致，所以这里会顺带定义它与调度器相关的最小字段。

---

## 5. 数据表详细设计

## 5.1 word 表

`word` 表保存单词基础信息，是全局静态词典表。

### 作用

用于提供单词本体、难度、频率等公共属性，供调度器与后续召回层使用。

### 推荐字段

```sql
CREATE TABLE word (
  id BIGSERIAL PRIMARY KEY,
  lemma VARCHAR(128) NOT NULL,
  surface VARCHAR(128) NOT NULL,
  pos VARCHAR(32),
  language VARCHAR(16) DEFAULT 'en',

  difficulty_score NUMERIC(5,4) DEFAULT 0.5,
  frequency_rank INT,
  cefr_level VARCHAR(8),

  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

### 字段说明

`lemma` 表示词元，例如 `run`。  
`surface` 表示实际展示词形，MVP 中可与 lemma 一致。  
`difficulty_score` 是单词难度分数，范围可以约定在 0 到 1。  
`frequency_rank` 可选，用于表达词频高低。  
`cefr_level` 可选，用于后续做难度控制。

---

## 5.2 user_word 表

`user_word` 是本系统最核心的表。它同时承担“学习目标集合”和“学习状态存储”的职责。

### 作用

表示某个用户是否需要学习某个词，以及这个词目前学到了什么状态、下次何时应复习、当前优先级多高。

### 推荐字段

```sql
CREATE TABLE user_word (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT NOT NULL,
  word_id BIGINT NOT NULL REFERENCES word(id),

  -- 学习目标属性
  is_active BOOLEAN NOT NULL DEFAULT TRUE,
  target_progress INT NOT NULL DEFAULT 100,
  plan_priority NUMERIC(5,4) NOT NULL DEFAULT 0.5,
  source_type VARCHAR(32),              -- 如 system/import/manual/video
  source_ref_id BIGINT,                 -- 可选，表示来源对象
  added_at TIMESTAMP NOT NULL DEFAULT NOW(),

  -- 当前学习状态
  status VARCHAR(32) NOT NULL DEFAULT 'new',  -- new / learning / reviewing / mastered / suspended
  progress INT NOT NULL DEFAULT 0,            -- 0~100
  mastery_score NUMERIC(5,4) NOT NULL DEFAULT 0.0,

  seen_count INT NOT NULL DEFAULT 0,
  review_count INT NOT NULL DEFAULT 0,
  correct_count INT NOT NULL DEFAULT 0,
  wrong_count INT NOT NULL DEFAULT 0,
  consecutive_correct INT NOT NULL DEFAULT 0,
  consecutive_wrong INT NOT NULL DEFAULT 0,

  first_seen_at TIMESTAMP,
  last_seen_at TIMESTAMP,
  last_reviewed_at TIMESTAMP,
  last_result_quality INT,

  -- SM-2 调度字段
  repetition INT NOT NULL DEFAULT 0,
  interval_days NUMERIC(8,2) NOT NULL DEFAULT 0,
  ease_factor NUMERIC(6,4) NOT NULL DEFAULT 2.5,
  next_review_at TIMESTAMP,

  -- 派生辅助字段
  overdue_score NUMERIC(6,4) NOT NULL DEFAULT 0,
  retrievability_score NUMERIC(6,4) NOT NULL DEFAULT 0,
  last_recommendation_at TIMESTAMP,

  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),

  CONSTRAINT uq_user_word UNIQUE (user_id, word_id)
);
```

### 为什么合并目标和状态

因为当前系统中，“这个词是用户目标的一部分”和“这个词当前状态如何”高度耦合。  
如果拆成两张表，查询推荐时必须频繁 join，而且多数时候两张表是一一对应关系。MVP 阶段拆开没有明显收益，反而增加实现和维护成本。

后续只有在这些场景真正出现时才值得拆分：

- 一个词属于多个独立学习计划
    
- 同一个用户维护多套课程/词书
    
- 想把长期记忆状态跨计划复用
    
- 需要统计 plan completion 和 memory state 两类独立概念
    

在当前阶段，合并更合理。

### 核心字段解释

`is_active` 表示该词当前是否处于目标集合中。  
`plan_priority` 表示该词在当前学习目标中的重要程度。  
`status` 表示学习阶段。  
`progress` 是面向业务层的掌握进度，范围 0 到 100。  
`mastery_score` 是连续数值型掌握度，用于内部排序辅助。  
`repetition / interval_days / ease_factor / next_review_at` 构成简化 SM-2 的核心调度状态。  
`overdue_score` 和 `retrievability_score` 是可预计算字段，用于提高查询效率与排序速度。

## 5.3 learning_event 表

`learning_event` 保存每一次学习行为，是系统的事实日志。

### 作用

用于记录原始学习行为，支持状态更新、分析、回放与后续重算。

### 推荐字段

```sql
CREATE TABLE learning_event (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT NOT NULL,
  word_id BIGINT NOT NULL REFERENCES word(id),
  user_word_id BIGINT REFERENCES user_word(id),

  event_type VARCHAR(32) NOT NULL,      -- new_learn / review / test / exposure
  source_type VARCHAR(32),              -- video_task / quiz / flashcard
  source_ref_id BIGINT,                 -- 对应任务或视频记录

  is_correct BOOLEAN,
  quality INT,                          -- 0~5
  response_time_ms INT,
  hint_used BOOLEAN DEFAULT FALSE,

  occurred_at TIMESTAMP NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

### 说明

这里 `quality` 是复习调度更新时最重要的输入之一。  
`source_type` 和 `source_ref_id` 可以帮助后面分析不同学习载体的效果，比如某类视频是否更适合记忆某些词。

## 5.4 video 表（与调度相关的最小定义）

虽然视频召回层会单独设计，但调度系统通常仍会和视频源保持关联，因此给出最小字段定义。

```sql
CREATE TABLE video (
  id BIGSERIAL PRIMARY KEY,
  title VARCHAR(512),
  duration_sec INT NOT NULL,
  difficulty_score NUMERIC(5,4),
  transcript_status VARCHAR(32),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

## 6. 索引设计

为了支撑调度查询，`user_word` 表必须有比较强的索引设计。

### 推荐索引

```sql
CREATE INDEX idx_user_word_user_active
ON user_word(user_id, is_active);

CREATE INDEX idx_user_word_user_next_review
ON user_word(user_id, next_review_at);

CREATE INDEX idx_user_word_user_status
ON user_word(user_id, status);

CREATE INDEX idx_user_word_user_progress
ON user_word(user_id, progress);

CREATE INDEX idx_learning_event_user_word_time
ON learning_event(user_id, word_id, occurred_at DESC);
```

### 说明

推荐请求最常见的查询包括：

- 找到当前用户所有活跃单词
    
- 找到当前应复习的词
    
- 找到当前未学的新词
    
- 查询最近学习历史
    

所以上面这些索引基本是必须的。

## 7. 模块设计与输入输出

### 作用

接收一次学习行为结果，并写入 `learning_event`。

### 输入

```json
{
  "user_id": 1001,
  "word_id": 2002,
  "event_type": "review",
  "source_type": "video_task",
  "source_ref_id": 3003,
  "is_correct": true,
  "quality": 4,
  "response_time_ms": 1800,
  "hint_used": false,
  "occurred_at": "2026-04-02T10:00:00Z"
}
```

### 输出

```json
{
  "event_id": 90001,
  "user_word_id": 50001,
  "accepted": true
}
```

### 说明

这个模块只负责写事实，不负责复杂调度。  
写完事件后，应立即触发用户-单词状态更新模块。

---

## 7.2 用户-单词状态更新模块

### 作用

根据新写入的学习事件，更新 `user_word` 当前状态。

### 输入

- `learning_event`
    
- 对应的 `user_word`
    
- 可选：最近若干条相关历史事件
    

### 输出

更新后的 `user_word` 记录，重点变化包括：

- `seen_count`
    
- `correct_count / wrong_count`
    
- `progress`
    
- `status`
    
- `repetition / interval_days / ease_factor / next_review_at`
    
- `mastery_score`
    

### 更新算法

这一步分两部分：

#### 第一部分：行为统计更新

这是纯计数逻辑。

- `seen_count += 1`
    
- 如果是 review，则 `review_count += 1`
    
- `is_correct = true` 时 `correct_count += 1`
    
- 否则 `wrong_count += 1`
    
- 更新 `last_seen_at`, `last_reviewed_at`, `last_result_quality`
    

#### 第二部分：调度状态更新

采用**简化版 SM-2**。

如果 `quality < 3`：

- `repetition = 0`
    
- `interval_days = 1`
    
- `ease_factor = max(1.3, ease_factor - penalty)`
    
- `consecutive_wrong += 1`
    
- `consecutive_correct = 0`
    

如果 `quality >= 3`：

- `repetition += 1`
    
- `consecutive_correct += 1`
    
- `consecutive_wrong = 0`
    

间隔更新规则：

- `repetition == 1` 时，`interval_days = 1`
    
- `repetition == 2` 时，`interval_days = 6`
    
- `repetition >= 3` 时，`interval_days = round(interval_days * ease_factor)`
    

ease factor 更新公式采用 SM-2 标准公式：

[  
EF' = EF + (0.1 - (5-q)\times(0.08 + (5-q)\times0.02))  
]

并限制：

[  
EF' \ge 1.3  
]

然后：

- `next_review_at = occurred_at + interval_days`
    

### 对新词的特殊处理

对于 `status = new` 的词，不能直接无脑进入长期复习周期。  
因此在首次接触后，先进入 `learning` 状态。

规则建议如下：

- 第一次学习完成后：`status = learning`
    
- `progress += 10 ~ 20`
    
- `next_review_at = occurred_at + 0.5 day` 或次日
    
- 当连续 2~3 次质量评分 `>= 3` 后，转入 `reviewing`
    

这样能避免新词第一次看到后直接被放入过长周期。

## 7.3 复习候选生成模块

### 作用

从 `user_word` 中筛出当前应进入复习候选集的词。

### 输入

- `user_id`
    
- 当前时间 `now`
    
- 可选：请求数量 `limit`
    
- 可选：学习会话参数
    

### 查询条件

候选复习词一般满足以下任一条件：

- `status IN ('learning', 'reviewing', 'mastered')`
    
- `is_active = true`
    
- `next_review_at <= now`
    
- 或者 `retrievability_score` 低于某个阈值
    

MVP 阶段最直接做法是以 `next_review_at <= now` 为主条件。

### 输出

复习候选列表，每项包含：

```json
{
  "word_id": 2002,
  "candidate_type": "review",
  "status": "reviewing",
  "plan_priority": 0.8,
  "next_review_at": "2026-04-01T09:00:00Z",
  "interval_days": 6,
  "ease_factor": 2.36
}
```

## 8. 排序算法设计

## 8.1 复习词排序算法

复习词不是只按 `next_review_at` 排，而应综合多因素。

### 推荐评分公式

[  
review_score =  
0.45 \cdot overdue_score +  
0.20 \cdot priority_score +  
0.15 \cdot difficulty_score +  
0.10 \cdot weak_memory_score +  
0.10 \cdot recency_penalty_adjustment  
]

### 各项定义

`overdue_score`：  
表示距离应复习时间超出多少。可以定义为：

[  
overdue_score = \min(1, \frac{now - next_review_at}{3\ days})  
]

未 overdue 则为 0。

`priority_score`：  
直接取 `plan_priority`。

`difficulty_score`：  
来自 `word.difficulty_score`。

`weak_memory_score`：  
用于强调近期表现差的词。可以根据 `consecutive_wrong`、最近质量评分、正确率等计算。

一个简单版本：

[  
weak_memory_score = \min(1,\ 0.3 \cdot consecutive_wrong + 0.7 \cdot (1 - mastery_score))  
]

`recency_penalty_adjustment`：  
如果这个词刚刚被推荐过但用户尚未真正学习，可以轻微降权，避免同一个词短时间重复霸榜。

---

## 8.2 新词排序算法

新词排序应强调目标价值和学习负担控制。

### 推荐评分公式

[  
new_score =  
0.50 \cdot priority_score +  
0.20 \cdot curriculum_score +  
0.15 \cdot difficulty_fit_score +  
0.15 \cdot freshness_score  
]

### 各项定义

`priority_score`：  
来自 `plan_priority`。

`curriculum_score`：  
如果词表有课程顺序，则越靠当前应推进位置越高；如果没有顺序，可先默认为常数。

`difficulty_fit_score`：  
偏向选择与当前用户阶段更匹配的新词，而不是过难或过易的词。

`freshness_score`：  
用于优先完全未学过的新词。MVP 中几乎可直接设为常数 1。

---

## 8.3 动态配额算法

这是系统很关键的一部分。

### 输入

- 本轮目标数量 `N`
    
- 当前 review backlog 数量 `R`
    

### 配额规则建议

如果 `R <= 5`：

- `review_quota = ceil(0.5 * N)`
    
- `new_quota = N - review_quota`
    

如果 `5 < R <= 20`：

- `review_quota = ceil(0.7 * N)`
    
- `new_quota = N - review_quota`
    

如果 `20 < R <= 40`：

- `review_quota = ceil(0.85 * N)`
    
- `new_quota = N - review_quota`
    

如果 `R > 40`：

- `review_quota = N`
    
- `new_quota = 0`
    

### 说明

这个规则的本质是：  
复习积压越大，越少引入新词。  
否则系统会越来越不可控。

## 9. progress 与 mastery_score 设计

`progress` 和 `mastery_score` 不应混为一谈。

`progress` 是业务层展示指标，适合表达“这个词整体学到了多少”，范围 0 到 100。  
`mastery_score` 是内部调度辅助指标，适合表达“当前掌握强度”，范围 0 到 1。

### progress 更新建议

可以采用一个简化规则：

- 首次接触成功后，`progress += 15`
    
- 复习成功且质量 >= 4，`progress += 10`
    
- 复习质量 = 3，`progress += 6`
    
- 复习失败，`progress -= 5`
    
- 最终限制在 0 到 100
    

### mastery_score 更新建议

可以由最近表现和调度状态共同决定，例如：

[  
mastery_score =  
0.35 \cdot normalized(progress) +  
0.35 \cdot recent_accuracy +  
0.30 \cdot stability_score  
]

其中：

- `normalized(progress) = progress / 100`
    
- `recent_accuracy` 可取最近 5 次正确率
    
- `stability_score` 可由 `interval_days` 归一化得到
    

这样 `progress` 不至于完全等同于“看过几次”，而 `mastery_score` 也更接近调度需要。

---

## 10. 状态流转规则

系统建议采用以下状态机：

### `new -> learning`

当一个词第一次被学习事件触达后，进入 `learning`。

### `learning -> reviewing`

当该词累计学习次数达到 2~3 次，且最近质量评分多数 `>= 3`，转入 `reviewing`。

### `reviewing -> mastered`

当满足以下条件时可进入 `mastered`：

- `progress >= 85`
    
- `interval_days >= 14`
    
- 最近 5 次正确率较高
    
- 连续错误数很低
    

### `mastered -> reviewing`

当近期再次连续错误，或质量评分显著下滑时，降级回 `reviewing`。

## 12. 一次完整链路示例

假设用户当前有 500 个目标词，其中：

- `status = new` 的词有 320 个
    
- 已进入学习或复习状态的词有 180 个
    
- 当前 overdue 的复习词有 18 个
    

用户请求本轮推荐 20 个词。

### 第一步

系统查询 `user_word`，识别出 18 个复习候选。

### 第二步

系统查询 `user_word`，识别出所有 `status = new` 的新词候选。

### 第三步

根据 backlog = 18，动态配额策略决定：

- `review_quota = 14`
    
- `new_quota = 6`
    

### 第四步

复习候选按 `review_score` 排序，新词候选按 `new_score` 排序。

### 第五步

分别取前 14 个 review 和前 6 个 new，组成最终目标单词列表。

### 第六步

将目标单词列表传给视频召回层。

这就是一个完整的调度周期。

## 13. MVP 阶段的技术取舍

当前阶段不建议做以下事情：

- 不引入复杂深度学习调度模型
    
- 不做用户级参数学习
    
- 不在调度层混入视频兴趣排序
    
- 不把“单词调度”和“视频召回”混成一个黑盒流程
    

当前阶段最重要的是保证这几点：

- 单词状态可持续更新
    
- 复习节奏有基本记忆模型支撑
    
- 新词引入受控
    
- 输出结构化、稳定、可解释
    

---

## 14. 后续演进方向

未来调度器可以沿三个方向升级。

第一，从简化 SM-2 升级到更现代的记忆模型，例如 FSRS。  
第二，将 `mastery_score` 和 `retrievability_score` 做得更真实，加入个性化参数。  
第三，把学习目标从“单词”扩展到“短语、句型、语法点”，但这不影响当前这版架构。

## 15. 总结

这版单词学习调度系统的核心设计是：

- 使用 `user_word` 统一存储学习目标和学习状态
    
- 使用 `learning_event` 记录原始学习行为
    
- 用简化 SM-2 管理旧词复习节奏
    
- 用规则策略控制新词引入
    
- 用动态配额平衡复习与推进
    
- 输出结构化目标单词学习列表供后续视频召回层使用
    

这套设计不追求花哨，但已经具备清晰的数据模型、明确的模块边界、可实现的算法和完整的推荐闭环，足够作为你当前 MVP 的正式技术方案。
