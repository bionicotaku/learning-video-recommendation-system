# 学习调度系统模块实施说明

本文档是对 [模块 README](/Users/evan/Downloads/learning-video-recommendation-system/internal/recommendation/scheduler/README.md) 的补充，重点不是快速启动，而是帮助新人理解：

1. 当前代码为什么这样分层
2. 三条核心链路实际怎么走
3. 每个目录里最先应该读哪些文件
4. 以后改规则、改表、加字段、补测试时，应该从哪里下手

如果你刚接手这个模块，建议先读 README，再读本文。

## 1. 当前实现目标

当前模块已经落地了学习调度子系统的 3 条核心用例：

1. 生成一轮学习推荐批次
2. 记录学习事件并更新状态
3. 根据历史事件 replay 重建状态

这里有一个很容易理解错的点：

这个模块输出的是“该学哪些 coarse unit”，不是“该看哪些视频”。
视频召回层和学习任务层还在推荐系统其他子层，不在这里。

## 2. 为什么要这样分层

### 2.1 先看最容易写乱的几种错误

如果没有明确分层，AI 或人工在实现这种模块时很容易出现下面几种坏味道：

- repository 同时做 SQL、状态迁移、打分、事务控制
- SQL 文件里混进复杂排序和业务判断
- replay 另写一套状态机，导致在线更新和回放不一致
- application 直接依赖 `pgx.Tx` 或 `sqlcgen`，后续几乎无法收口
- 测试偷用真实用户数据，结果环境一变就不稳定

当前实现的核心目标就是避免这些问题。

### 2.2 当前分层的一句话理解

- `domain` 负责“规则”
- `application` 负责“编排”
- `infrastructure` 负责“落库”
- `test` 负责“证明这套规则在真实库上也成立”

## 3. 目录逐层说明

### 3.1 `domain/enum`

定义统一枚举，解决字符串字面量散落的问题。

这里的意义不只是“好看”，而是避免：

- 某个 SQL mapper 用 `"review"`，另一个地方写 `"reviewing"`
- 某个测试写 `"phrase"`，业务代码写 `"phrases"`

所以所有业务层一律引用枚举，不写裸字符串。

### 3.2 `domain/model`

这里的 model 是领域模型，不是数据库 row model。

它们代表的是调度语义，而不是表结构的逐列镜像。举例：

- `UserUnitState` 表示用户对某学习对象的调度状态
- `LearningEvent` 表示一个标准化学习事件
- `RecommendationBatch` 表示调度输出

换句话说，数据库是这些模型的持久化载体，不是这些模型的定义来源。

### 3.3 `domain/rule`

这里只做“事件进入状态机前的基础字段更新”。

为什么要拆成 `WeakEventHandler` 和 `StrongEventHandler`：

- 弱事件和强事件是设计文档里的一级分界
- 弱事件不推进调度，强事件才推进调度
- 把这层拆开之后，后续排查“为什么这次 exposure 没有推进 next_review_at”会非常直观

### 3.4 `domain/service`

这里是真正规则核心区，主要分成 4 类：

1. 状态推进
   - `SM2Updater`
   - `StatusTransitioner`
2. 学习质量度量
   - `ProgressCalculator`
   - `MasteryScoreCalculator`
3. 调度分配
   - `BacklogCalculator`
   - `QuotaAllocator`
4. 推荐排序和输出
   - `ReviewScorer`
   - `NewScorer`
   - `PriorityZeroExtractor`
   - `RecommendationAssembler`

真正需要记住的只有一句：

所有调度规则都应该在 `domain/service` 找，而不是去 repository 或 SQL 里找。

补一条当前代码里更关键的结构点：

- `domain/aggregate/UserUnitReducer`

现在真正把“当前状态 + 当前事件 -> 新投影状态”串起来的是 reducer。
`domain/service` 提供的是它内部组合用到的规则组件。

### 3.5 `application/usecase`

这是本模块的“服务入口层”，但不是 HTTP service。

每个 usecase 都只做三件事：

1. 组织调用顺序
2. 管理事务边界
3. 把 repository 和 domain service 串起来

它不应该重新定义规则，更不应该做 SQL 细节处理。

### 3.6 `application/repository`

这是 port interface 层，目的是把 application 和 infrastructure 断开。

如果以后你在 usecase 里直接 import：

- `sqlcgen`
- `pgx.Tx`
- `pgxpool.Pool`

那通常就意味着边界又被打穿了。

### 3.7 `infrastructure/persistence`

这里再按职责切一层：

- `query/`
  SQL 输入文件
- `sqlcgen/`
  自动生成，不手改
- `mapper/`
  参数和领域模型互转
- `repository/`
  `application/repository` 的具体实现
- `queryctx/`
  事务 querier 注入到上下文
- `tx/`
  `TxManager` 的 `pgx` 实现

这一层的核心原则是：

只负责“怎么从库里拿”和“怎么往库里写”，不负责“为什么这样算”。

## 4. 三条核心链路怎么读

## 4.1 生成推荐批次

最应该先读的文件：

- `application/usecase/generate_recommendations.go`

它的流程可以按 8 步理解：

1. 读取用户调度设置
2. 读取 due review 候选
3. 读取 new 候选
4. 根据 due review 数量算 backlog
5. 根据 backlog 和用户设置分配 quota
6. 对 review/new 分别打分
7. 提取 priority-0
8. 组装推荐批次，并在需要时保存 run 快照

这里的代码特征很值得保留：

- 它没有手写 SQL
- 它没有直接操作 `pgx`
- 它没有把评分公式硬塞在 if/else 里
- 它只是把几个明确的组件按顺序连起来

这正是 application 层应该有的样子。

## 4.2 记录事件并更新状态

入口：

- `application/usecase/record_events_and_update_state.go`

这条链路最关键的不是“写了几张表”，而是事务顺序和上下文构造：

1. 进入 `TxManager.WithinTx`
2. 查当前快照
3. 组装领域 `LearningEvent`
4. 先写事件表
5. 调 `UserUnitReducer.Reduce`
6. 再 upsert 状态表

为什么先写事件再写状态：

- 事件表是事实源
- 状态表是快照
- 如果快照写失败但事件已经存在，只要事务回滚，事实和快照都不会残留半成品
- reducer 不再依赖“每次先查整段历史再拼 recent context”
- `recent_quality_window` 和 `recent_correctness_window` 已经内聚到 `user_unit_states`

## 4.3 replay 重建状态

入口：

- `application/usecase/replay_user_unit_states.go`

流程是：

1. 在事务里先删目标用户的旧快照
2. 查出该用户完整事件历史
3. `UserStateRebuilder` 按 `coarse_unit_id` 聚合重建
4. 每条事件都走 `UserUnitReducer.Reduce`
5. 最后批量 upsert 状态

最重要的一点是：

replay 没有自己的状态规则实现。
只要你发现 replay 和在线更新的逻辑开始分叉，这就是结构性 bug。

当前 MVP 已经明确收敛：

- replay 只支持 `FullUserReplay`
- 不支持 `FromTime`
- 不支持 scoped replay
- 不支持增量 replay

原因很简单：MVP 阶段只保留最容易保证正确的一条重建链。

## 5. `UserUnitReducer` 为什么是整个模块的核心

`UserUnitReducer` 把多个原子规则串成一个可复用入口。

它内部大致是这样工作的：

1. 看事件是弱事件还是强事件
2. 更新基础统计字段
3. 更新最近质量 / 正确性窗口
4. 如果是带质量分的强事件，再更新 SM-2
5. 根据最近质量窗口重算状态迁移
6. 最后重算 progress 和 mastery

因此它同时服务两个场景：

- 在线事件写入
- 历史 replay

这带来的好处是：

- 规则只有一份
- 测试可以针对一个入口覆盖多条链
- 线上状态和 replay 状态更容易保持一致

## 6. 事务设计为什么这样做

### 6.1 为什么 application 只依赖 `TxManager`

因为 transaction boundary 是应用编排问题，不是领域问题，也不该变成某个具体数据库类型问题。

所以 application 只知道：

```go
type TxManager interface {
    WithinTx(ctx context.Context, fn func(ctx context.Context) error) error
}
```

这样 usecase 不需要知道：

- `pgx.Tx` 是什么
- `sqlcgen.New(tx)` 怎么建
- repository 在事务里怎么切换 querier

### 6.2 repository 怎么同时支持事务内和事务外

这块如果不理解，后面很容易写出一堆重复方法。

当前做法是：

1. repository 构造时持有一个 fallback querier，通常来自 `sqlcgen.New(pool)`
2. 事务开启后，`TxManager` 把 `sqlcgen.New(tx)` 放进 `context`
3. repository 每次执行前先调用 `resolveQuerier`
4. 如果 `context` 里有事务 querier，就用事务版
5. 如果没有，就用 fallback querier

这个设计的好处是：

- repository 接口不需要额外传 `tx`
- 用例层也不需要把事务对象一路往下传
- 只读用例可以直接用 pool
- 写用例进入事务时能自动切换

## 7. 如何判断一个改动放错层了

这里给新人一个很实用的判断法。

如果你的改动回答的是下面这些问题，它大概率该放在 `domain/service`：

- 什么时候从 `learning` 进入 `reviewing`
- `quality=2` 应该怎么回退
- backlog 多大时压制 new
- review/new 分数怎么算
- P0 怎么提取

如果你的改动回答的是下面这些问题，它大概率该放在 `infrastructure`：

- 这条 SQL 应该查哪些列
- `pgtype.Numeric` 怎么转 `float64`
- `uuid` 怎么映射 PG 参数
- 事务 querier 怎么从 `context` 取

如果你的改动回答的是下面这些问题，它大概率该放在 `application/usecase`：

- 先查 settings 还是先查 candidates
- 这个流程是否需要事务
- 写完状态后是否还要写 run snapshot

## 8. 文件级阅读建议

### 8.1 第一天上手

建议顺序：

1. `internal/recommendation/scheduler/README.md`
2. `application/usecase/generate_recommendations.go`
3. `application/usecase/record_events_and_update_state.go`
4. `application/usecase/replay_user_unit_states.go`
5. `domain/service/state_updater.go`

读完这 5 个文件，你就已经掌握模块主干。

### 8.2 第二天开始动手改规则

建议顺序：

1. 看相关 domain service 的单测
2. 再看对应实现
3. 改完先跑领域单测
4. 再跑集成测试

不要先改 SQL，再倒推规则。这种顺序通常会把结构写反。

### 8.3 想排查线上异常

建议顺序：

1. 先确认是“生成推荐不对”还是“状态推进不对”
2. 如果是推荐不对，先看 quota/scorer/assembler
3. 如果是状态推进不对，先看 state updater 及其上下游
4. 最后再看 repository/query 是否拿错数据

## 9. 数据库改动的标准流程

凡是涉及新字段、新表或索引，按下面顺序走：

1. 改 migration
2. 改 `persistence/query/*.sql` 或 `persistence/schema/external.sql`
3. 执行 `make sqlc-generate`
4. 改 `mapper/`
5. 改 domain model / application query
6. 补单测和集成测试
7. 执行 `make check`

绝对不要直接改 `sqlcgen/`。

## 10. 测试策略说明

当前测试大致分两层：

### 10.1 领域单测

位置：

- `internal/recommendation/scheduler/domain/...`

职责：

- 证明规则在纯内存条件下成立
- 不依赖真实数据库
- 快速验证状态机、SM-2、progress/mastery、quota/scorer 等核心逻辑

### 10.2 集成测试

位置：

- `internal/recommendation/scheduler/test/integration/...`

职责：

- 证明 `sqlc + repository + tx + usecase + real DB` 能一起跑通
- 验证回滚、快照落库、replay 一致性
- 验证多场景调度闭环

重要限制：

- 不使用已有用户数据
- 全部自己创建临时测试用户
- 事务测试靠 rollback 清理
- 提交型测试靠 teardown 删除用户，连带清理学习表数据

这条规则很重要，因为一旦测试依赖现有用户，环境就不再可重复。

## 11. 后续扩展建议

当前模块已经能独立完成调度，但还有几个自然扩展点：

- 在 `integration/contract` 中定义更稳定的下游契约
- 给 `fixture/` 补更系统的测试数据构造器
- 给 replay 增加更丰富的局部重建场景
- 为上层推荐模块提供单独的模块 wiring 包

这些扩展应该在不破坏当前分层的前提下做。

## 12. 一句话总结

如果你只能记住一句话，那就是：

这个模块的本质不是“查几张表然后排个序”，而是“用统一的领域规则维护学习状态，并把当前该学什么稳定地交给下游系统”。

只要你在改动时始终守住这条主线，代码通常就不会写歪。
