## 4. 分层与依赖审查

### 4.1 通过项

- `domain` 不依赖 `infrastructure`
- `application` 不依赖 HTTP
- `application` 不依赖 `sqlcgen`
- repository 不包含状态迁移、SM-2、配额、评分逻辑
- SQL 只做查询和写入，不承载业务策略

### 4.2 本次核对方式

- 全局搜索 `http / gin / controller / handler / ServeHTTP`
- 全局搜索 `sqlcgen / pgx / pgxpool / database/sql / supabase` 在 `application + domain` 的出现情况
- 抽查 `query/*.sql` 与 `repository/*.go`

### 4.3 结构优化结论

当前结构相比最初任务描述有一个合理收口：

- `TxManager` interface 现在位于 `application/repository/tx_manager.go`

这是刻意优化，不是偏离设计。
原因是事务边界属于 application port，而不是 infrastructure 细节。具体 `pgx` 实现在：

- `infrastructure/persistence/tx/pgx_tx_manager.go`

## 5. 查询与规则边界审查

结论：通过。

观察结果：

- `unit_states.sql` 只负责状态查询与 upsert
- `unit_events.sql` 只负责事件写入与按用户顺序读取
- `scheduler_runs.sql` 只负责快照落库
- 配额、评分、状态迁移、SM-2、progress/mastery 全部在 `domain/service` 或 `domain/rule`

本次顺手做了一个结构优化：

- 把 `GetUserSchedulerSettings` 从 `scheduler_runs.sql` 拆到了单独的 `user_scheduler_settings.sql`

这让 query 文件主题更清晰，也更符合前面“按主题拆 SQL 文件”的要求。

## 6. replay 一致性审查

结论：通过，但本次验收中修正了一个边界实现。

### 6.1 已验证通过的场景

- 全量 replay 可重建在线状态
- replay 顺序为 `occurred_at asc, event_id asc`
- replay 与在线更新共用 `UserUnitReducer`

### 6.2 当前 MVP 边界

当前代码已经明确收敛为：

1. 只支持 `FullUserReplay`
2. 先按用户删除全部状态投影
3. 再按完整事件历史重建全部 unit 状态

`FromTime / scoped replay / 增量 replay` 都不再属于 MVP 实现。
