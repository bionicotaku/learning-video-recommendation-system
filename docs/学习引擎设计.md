# 学习引擎设计

本文是 Learning Engine 的当前权威设计。Learning Engine 在代码上是一个边界容器，当前包含两个平级子模块：

- `internal/learningengine/reducer`：维护 `learning.*` 表、normalized event ledger、状态归约、replay、target/control 和状态查询。
- `internal/learningengine/normalizer`：只读 `analytics.*` raw fact，把 quiz / interaction 解释成 reducer 可消费的 normalized event。

其中 reducer 子模块的核心边界是把已经归一化的学习事件 `learning.unit_learning_events` reduce 成用户在学习单元上的状态投影
`learning.user_unit_states`。

reducer 不负责解释 `analytics.*` 原始流水。lookup、exposure、quiz、self mark 等 raw fact 如何被解释为
normalized Learning Engine event，由 normalizer 决定；reducer 只消费已经持久化的 normalized event。

## 设计目标

1. 保持学习事件为可 replay 的 append-only ledger。
2. 让 reducer 只基于明确的 `reducer_effect` 分发，而不是把 `event_type` 隐含解释成 strong / weak。
3. 用 progress / schedule 语义替代旧的 review / legacy progress-driving event 命名。
4. 让 Recommendation 只消费稳定的状态输出字段，不依赖 Learning Engine 内部事件解释。
5. MVP 阶段保持 schema 和 reducer 简单，后续可以扩展更多 effect，但不提前引入复杂抽象。

## 模块边界

### Learning Engine 根模块负责

- 定义 Learning Engine 的整体边界。
- 固定 reducer 与 normalizer 的依赖方向。
- 说明未来 `internal/api -> analytics -> normalizer -> reducer` 的组合链路。

根模块不拥有业务实现骨架，不再放置 `application/domain/infrastructure/test`。

### Reducer 子模块负责

- 持久化 normalized Learning Engine event。
- 校验 normalized event 的 reducer 输入契约。
- 将 event reduce 到 `learning.user_unit_states`。
- 对同一批 event 支持在线写入和 replay 得到一致结果。
- 维护 progress、schedule、status、mastery 等学习状态字段。
- 维护 target/control 命令和状态查询。
- 维护 Learning Engine owner 的 `learning.*` migrations。

### Normalizer 子模块负责

- read-only 读取 `analytics.quiz_events` 和 `analytics.learning_interaction_events`。
- 执行 raw fact 到 normalized event 的学习语义解释。
- 调用 reducer 的 `RecordLearningEvents` 写入 normalized ledger 并触发状态归约。
- 提供 by-ID 主路径和 pending repair/backfill 路径。

### Learning Engine 不负责

- reducer 不读取或解释 `analytics.*` raw log。
- analytics 不判断某次 lookup、exposure、quiz 在学习语义上是否应该推进进度。
- 不在 reducer 内重新计算 raw event 的语义上下文。
- 不维护 Recommendation 的候选召回、排序、去重策略。

## 数据流

```text
analytics raw fact
        ↓
learningengine/normalizer
        ↓
learning.unit_learning_events
        ↓
learningengine/reducer
        ↓
learning.user_unit_states
        ↓
recommendation
```

`normalizer` 是 raw fact 到 normalized event 的解释层。它属于 Learning Engine，但不是 reducer 的一部分。它负责决定：

- `event_type`
- `reducer_effect`
- `progress_quality`
- `source_type`
- `source_ref_id`
- `metadata`

`reducer` 是 `learning.*` 表 owner。它只根据 `reducer_effect` 分发：

- `observe_only`：只更新 observation 字段。
- `affects_progress`：先更新 observation 字段，再进入 progress / schedule path。
- `set_mastered`：先更新 observation 字段，再直接把该学习单元标记为已掌握并移出目标学习范围。

Replay 只重放 `learning.unit_learning_events` 中已经持久化的 normalized event，不重新解释
`analytics.*` raw fact。因此 replay 结果只受 normalized ledger 影响。

## `learning.unit_learning_events`

`unit_learning_events` 是 Learning Engine 的 normalized event ledger，不是 analytics raw log。

它表达的是：上游 normalizer 已经把某条学习交互解释成一个针对 `coarse_unit_id` 的 reducer 输入。

### 字段

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `event_id` | `uuid` | Learning Engine event 主键，默认 `gen_random_uuid()`。 |
| `user_id` | `uuid` | 用户 id。 |
| `coarse_unit_id` | `bigint` | 被影响的学习单元。 |
| `video_id` | `uuid` | 事件关联视频，可为空。用于追溯上下文，不参与 reducer 决策。 |
| `event_type` | `text` | normalized event 类型。MVP 支持 `exposure`、`lookup`、`quiz`、`self_mark_mastered`。 |
| `reducer_effect` | `text` | reducer 分发依据。MVP 支持 `observe_only`、`affects_progress`、`set_mastered`。 |
| `progress_quality` | `smallint` | progress 质量分。仅 `affects_progress` 必填，范围 `0..5`；`observe_only` 和 `set_mastered` 必须为空。 |
| `is_correct` | `boolean` | 上游提供的正确性结果，可为空。MVP 中 reducer 的 pass/fail 以 `progress_quality >= 3` 为准。 |
| `source_type` | `text` | normalized event 的来源类型，例如 `learning_interaction_event`、`quiz_event`。 |
| `source_ref_id` | `text` | 来源 raw fact 的稳定引用 id。 |
| `metadata` | `jsonb` | 上游补充上下文，默认 `{}`，必须为 object。Reducer 不依赖其中字段做核心分发。 |
| `occurred_at` | `timestamptz` | 事件实际发生时间。 |
| `created_at` | `timestamptz` | normalized event 入库时间，默认 `now()`。 |

### 约束

```sql
event_type in ('exposure', 'lookup', 'quiz', 'self_mark_mastered')
reducer_effect in ('observe_only', 'affects_progress', 'set_mastered')
metadata must be a json object

(
  reducer_effect = 'affects_progress'
  and progress_quality between 0 and 5
)
or
(
  reducer_effect = 'observe_only'
  and progress_quality is null
)
or
(
  reducer_effect = 'set_mastered'
  and event_type = 'self_mark_mastered'
  and progress_quality is null
)
```

幂等约束：

```sql
unique (user_id, source_type, source_ref_id, coarse_unit_id)
```

这条约束用于 normalizer 重试。相同 raw fact 被重复解释时，只会生成同一条 normalized event。

### `event_type` 和 `reducer_effect`

`event_type` 描述事件来自哪类业务交互。它不是 reducer 是否推进进度的依据。

示例：

- `lookup` 可以是 `observe_only`，只记录用户见过该单元。
- `self_mark_mastered` 必须被上游解释为 `set_mastered`，用于用户主动声明已经掌握。
- `quiz` 可以是 `affects_progress`，例如一次完成的答题结果。
- `exposure` 可以是 `observe_only`，也可以在上游满足明确条件后变成 `affects_progress`。

`reducer_effect` 是 reducer 的唯一分发字段。旧设计中的 `legacy progress-driving event` / `legacy observe-only event` 不再作为业务语义存在。

## `learning.user_unit_states`

`user_unit_states` 是用户在学习单元上的当前状态投影。它不是事件流水；同一用户和同一
`coarse_unit_id` 只有一行。

字段按职责分组。

### Identity

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `user_id` | `uuid` | 用户 id。 |
| `coarse_unit_id` | `bigint` | 学习单元 id。 |
| `created_at` | `timestamptz` | 状态行创建时间。 |
| `updated_at` | `timestamptz` | 状态行最后更新时间。 |

### Control

Control 字段决定这行状态是否进入推荐目标范围，以及它为何被纳入目标范围。

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `is_target` | `boolean` | 是否是推荐系统可消费的目标学习单元。默认 `false`。普通 observation 不会自动进入推荐目标范围。 |
| `target_source` | `text` | target 来源，例如用户主动添加、课程计划、系统任务。 |
| `target_source_ref_id` | `text` | target 来源引用。 |
| `target_priority` | `numeric(8,4)` | target 优先级，默认 `0`。Recommendation 可用于候选排序或 tie-break。 |
| `suspended_reason` | `text` | 暂停推荐原因。为空表示未暂停。 |

### Recommendation Output

这些字段是 Recommendation 的稳定消费面。

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `status` | `text` | 当前学习状态。MVP 支持 `new`、`learning`、`reviewing`、`mastered`、`suspended`。 |
| `progress_percent` | `numeric(5,2)` | 面向产品展示的进度百分比，范围 `0..100`。 |
| `mastery_score` | `numeric(5,4)` | 掌握度分数，范围 `0..1`。 |
| `next_review_at` | `timestamptz` | 下次建议复习时间。为空表示尚未进入复习调度或不需要复习。 |

Recommendation MVP 实际读取：

- `status`
- `target_priority`
- `mastery_score`
- `next_review_at`
- `last_progress_quality`

### Observation

Observation 字段记录用户是否接触过该学习单元。`observe_only` 和 `affects_progress` 都会更新这些字段。

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `first_observed_at` | `timestamptz` | 第一次观察到该学习单元的时间。 |
| `last_observed_at` | `timestamptz` | 最近一次观察到该学习单元的时间。 |
| `observation_count` | `integer` | 累计 observation 次数。 |

### Progress

Progress 字段只由 `reducer_effect = 'affects_progress'` 的事件更新。

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `progress_event_count` | `integer` | 累计推进 progress 的事件数。替代旧 `strong_event_count`。 |
| `last_progress_at` | `timestamptz` | 最近一次推进 progress 的时间。 |
| `last_progress_quality` | `smallint` | 最近一次 progress quality。替代旧 `last_quality`。 |
| `recent_progress_qualities` | `smallint[]` | 最近若干次 progress quality，用于状态和 mastery 计算。 |
| `recent_progress_passes` | `boolean[]` | 最近若干次 progress pass/fail，`progress_quality >= 3` 为 pass。 |
| `progress_success_count` | `integer` | progress pass 累计次数。 |
| `progress_failure_count` | `integer` | progress fail 累计次数。 |
| `consecutive_success_count` | `integer` | 连续 pass 次数。 |
| `consecutive_failure_count` | `integer` | 连续 fail 次数。 |

旧 `review_count` 不再保留。它既不是清晰的业务概念，也不是 Recommendation 的实际依赖字段。

### Schedule

Schedule 字段保留 SM-2-style 算法所需的最小状态，但命名改为 schedule 语义。

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `schedule_repetition` | `integer` | 当前调度重复次数。替代旧 `repetition`。 |
| `schedule_interval_days` | `numeric(8,2)` | 当前调度间隔天数。替代旧 `interval_days`。 |
| `schedule_ease_factor` | `numeric(6,4)` | 当前调度 ease factor。替代旧 `ease_factor`。 |

## Reducer 规则

### 公共规则

每个 event 都会先更新 observation：

1. 如果状态行不存在，创建默认状态行。
2. `first_observed_at` 取第一次 event 的 `occurred_at`。
3. `last_observed_at` 取目前已处理 event 中最大的 `occurred_at`。
4. `observation_count += 1`。

如果状态行处于 suspended overlay：

- `suspended_reason` 不为空时，状态对外显示为 `suspended`。
- 内部 progress / schedule 仍可继续更新，方便解除暂停后恢复推荐。

### `observe_only`

`observe_only` 只更新 observation 字段，不改变：

- `progress_event_count`
- `last_progress_at`
- `last_progress_quality`
- `recent_progress_qualities`
- `recent_progress_passes`
- `progress_success_count`
- `progress_failure_count`
- `consecutive_success_count`
- `consecutive_failure_count`
- `schedule_repetition`
- `schedule_interval_days`
- `schedule_ease_factor`
- `next_review_at`
- `mastery_score`
- `progress_percent`
- `status`

### `affects_progress`

`affects_progress` 代表该 normalized event 被上游解释为应该推进学习进度。

处理顺序：

1. 校验 `progress_quality` 必填且在 `0..5`。
2. 拒绝早于当前 `last_progress_at` 的乱序 progress event，返回 `ErrLateProgressEvent`。
3. 更新 progress 计数字段和 recent window。
4. 根据 `progress_quality >= 3` 计算 pass/fail。
5. 进入 schedule path，更新 repetition、interval、ease、next review。
6. 重新计算 mastery、progress percent、status。

质量分语义：

| `progress_quality` | 语义 |
| --- | --- |
| `0` | 完全失败或明确遗忘。 |
| `1` | 大部分失败。 |
| `2` | 未达到可推进掌握的质量。 |
| `3` | 勉强通过。 |
| `4` | 正常通过。 |
| `5` | 非常稳定掌握。 |

MVP 规则：`progress_quality >= 3` 视为 pass。

### `set_mastered`

`set_mastered` 代表用户主动声明某个学习单元已经掌握。它不是 quiz quality，不进入 SM-2 schedule path，也不使用 `progress_quality`。

处理顺序：

1. 校验 `event_type = 'self_mark_mastered'` 且 `progress_quality is null`。
2. 如果状态行不存在，创建默认状态行。
3. 更新 observation 字段。
4. 直接把状态收敛到 completed mastered：
   - `status = 'mastered'`
   - `progress_percent = 100`
   - `mastery_score = 1`
   - `next_review_at = null`
   - `is_target = false`
   - `suspended_reason = ''`

自然学习路径达到 `mastered` 时，也应用同一 completed mastered invariant。也就是说，`mastered` 表示该 unit 已经离开当前目标学习范围；Recommendation 不应继续把它作为 active target 消费。

当状态已经是 `status = 'mastered'` 且 `is_target = false` 时，它是 terminal mastered。后续普通 `observe_only` / `affects_progress` event 不改变状态。只有显式 target/control 命令重新把该 unit 加入目标范围后，新的学习事件才可以再次影响状态。

### Status 推导

`status` 是 Learning Engine 给 Recommendation 和产品层的简化状态。

基础推导：

- 没有 progress event：`new`
- progress event 后未达到稳定复习：`learning`
- 已进入间隔复习：`reviewing`
- 多次稳定 pass 且 schedule 足够长：`mastered`
- `suspended_reason` 不为空：对外显示 `suspended`

普通学习中的 `progress_quality < 3` 会把状态拉回需要复习的路径；但 terminal mastered 后续普通学习事件会被忽略，不再回落。

## Replay

Replay 用于从 event ledger 重新构建状态投影。

规则：

1. 清空或替换指定用户范围内的 `user_unit_states`。
2. 按 `occurred_at, event_id` 顺序读取 `unit_learning_events`。
3. 对每条 event 调用与在线写入相同的 reducer。
4. Replay 不读取 `analytics.*`，不重新生成 `reducer_effect`。
5. Replay 后的状态必须与在线写入同一批 event 得到的状态一致。

## Recommendation 消费契约

Recommendation 只读 `learning.user_unit_states`，不读 `learning.unit_learning_events`。

候选输入范围：

- `is_target = true`
- `status <> 'suspended'`

MVP planner 使用字段：

- `status`
- `target_priority`
- `mastery_score`
- `next_review_at`
- `last_progress_quality`

判断规则保持简单：

- `last_progress_quality < 3`：hard review。
- `last_progress_quality < 4`：soft review。
- `next_review_at <= now`：hard review。
- `next_review_at <= now + 72h`：soft review。
- `mastery_score < 0.6`：soft review。

## 与旧设计的差异

旧概念到新概念的替换：

| 旧概念 | 新概念 |
| --- | --- |
| legacy progress-driving event | `reducer_effect = 'affects_progress'` |
| legacy observe-only event | `reducer_effect = 'observe_only'` |
| `legacy-new-learn` / `review` | 不再作为 Learning Engine event type |
| `quality` | `progress_quality` |
| `first_seen_at` | `first_observed_at` |
| `last_seen_at` | `last_observed_at` |
| `seen_count` | `observation_count` |
| `strong_event_count` | `progress_event_count` |
| `last_reviewed_at` | `last_progress_at` |
| `last_quality` | `last_progress_quality` |
| `recent_quality_window` | `recent_progress_qualities` |
| `recent_correctness_window` | `recent_progress_passes` |
| `correct_count` | `progress_success_count` |
| `wrong_count` | `progress_failure_count` |
| `consecutive_correct` | `consecutive_success_count` |
| `consecutive_wrong` | `consecutive_failure_count` |
| `repetition` | `schedule_repetition` |
| `interval_days` | `schedule_interval_days` |
| `ease_factor` | `schedule_ease_factor` |
| `review_count` | 删除 |

重要语义变化：

- `event_type` 不再决定是否推进 progress。
- 所有 progress 推进都必须显式标记 `reducer_effect = 'affects_progress'`。
- 用户主动声明掌握必须显式标记 `reducer_effect = 'set_mastered'`。
- 所有 observation 都可以落状态，但默认不会进入 Recommendation 目标范围。
- 只有 target/control command 才把 `is_target` 置为 `true`。

## MVP 取舍

当前阶段暂不引入：

- `manual_mastery` / `difficulty_signal` 等额外 event type。
- event 权重模型。
- raw fact 到 event 的通用 DSL。
- Recommendation 直接读取 event ledger。

这些能力后续可以在 normalizer 或新增 reducer effect 中扩展，但不应该污染当前最小闭环。
