# Learning engine 详细设计文档

## 0. 文档信息

文档名称：Learning engine 详细设计文档
适用阶段：MVP
目标读者：后端、数据、测试、Recommendation 对接方
文档目标：定义 Learning engine 的职责边界、领域模型、数据结构、核心规则、接口与工程约束，使其可以**独立实施**，并与 Recommendation 并行开发。

---

## 1. 模块定位

### 1.1 Learning engine 是什么

Learning engine 是学习状态引擎。
它回答的问题只有一类：

- 用户对某个学习单元当前学到什么程度
- 这个学习单元下一次什么时候该复习
- 最近表现是否稳定
- 这些状态如何由学习行为事件稳定归约出来

一句话：

> **Learning engine 负责维护“学得怎么样”，不负责“这次推什么”。**

### 1.2 Learning engine 不是什么

Learning engine 不是：

- 视频推荐系统
- 视频召回系统
- 视频排序系统
- 推荐投放系统
- 推荐审计系统
- 学习任务编排系统

这些都属于 Recommendation。

### 1.3 与 Recommendation 的关系

二者是**平级模块**，但依赖是**单向**的：

- Learning engine 产出学习状态
- Recommendation 消费学习状态
- Recommendation 不回写 Learning engine 的学习事件表和状态表

正确关系如下：

```text
业务学习行为
   ↓
Learning engine
   ├── learning.unit_learning_events
   └── learning.user_unit_states
            ↓
      Recommendation 读取
```

这条边界是本系统最重要的架构前提之一。

## 3. 设计目标

Learning engine 在 MVP 阶段的设计目标只有六个。

### 3.1 状态必须可重建

学习状态不是独立真相层。
一旦状态错了，必须能从事件流重新计算。

### 3.2 输出必须稳定可读

Recommendation 已经确定会围绕 `learning.user_unit_states` 做需求规划，因此 Learning engine 必须输出**稳定、明确、语义清楚**的状态字段，而不是模糊或依赖上下文解释的临时字段。

### 3.3 规则必须在线 / Replay 一致

不能有：

- 一套在线写入逻辑
- 一套 Replay 逻辑

两者必须共用同一个 reducer。

### 3.4 MVP 必须可实施

MVP 先不做：

- 增量 Replay
- scoped replay
- 复杂记忆模型
- 多种学习策略共存
- 用户级可配置记忆模型

### 3.5 与 Recommendation 并行实施时不能互相阻塞

Recommendation 只需要知道：

- 表结构
- 字段语义
- 读取契约

Learning engine 不需要等待 Recommendation 内部细节稳定后再开发。

### 3.6 目标单元必须可在“零学习事件”前提下存在

Recommendation 的 `new_now` 规划需要读取那些：

- 还没有发生强学习事件
- 但已经进入当前 target scope 的学习单元

因此 Learning engine 不能只靠事件被动长出状态行，必须支持显式 target upsert。

## 4. 核心设计原则

### 4.1 事件是真相，状态是归约结果

`learning.unit_learning_events` 是事实层。
`learning.user_unit_states` 是事件归约后的状态层。

如果状态有问题，修复方式是：

- 修正事件源
- 做 Replay

而不是直接人工改状态表当成真相层。

这是模块设计的根原则。

### 4.2 状态以“用户-学习单元”为聚合根

Learning engine 的核心聚合是：

> `user_id + coarse_unit_id`

即“某用户对某学习单元的当前状态”。

所有 reducer 逻辑都围绕这个聚合展开。

### 4.3 弱事件不推进记忆状态，强事件才推进

MVP 明确区分两类事件：

弱事件：

- `exposure`
- `lookup`

强事件：

- `new_learn`
- `review`
- `quiz`

弱事件只更新接触/曝光信息。
强事件才推进：

- 记忆状态
- 间隔复习参数
- 掌握程度
- 状态迁移

这条边界必须写死。

### 4.4 在线写入与 Replay 共用同一个 reducer

在线写入：

> 当前状态 + 新事件 -> 新状态

Replay：

> 空状态 + 历史事件流 -> 重建状态

必须使用同一个 reducer，保证行为一致。

### 4.5 MVP 只支持 Full Replay

MVP 只支持：

> 对某个用户按完整事件历史重建全部学习状态

不支持：

- partial replay
- scoped replay
- patch replay
- 增量修补 Replay

这条范围控制也是既定约束。

### 4.6 Learning engine 只拥有学习域

Learning engine 拥有：

- 学习事件
- 学习状态
- target/control 字段
- Replay

它不拥有：

- 推荐投放状态
- `last_recommended_at`
- video serving state
- recommendation run / items

这些归 Recommendation 所有。

### 4.7 状态表内部必须区分“控制字段”和“学习进展字段”

这是当前 MVP 最重要的实现细节之一。

原因很简单：

- 学习进展字段来自学习事件
- `is_target`、`target_priority` 这类控制字段不来自学习事件

因此 Replay 不能简单理解为“只靠事件重建整行所有字段”，而应理解为：

> **用事件重建学习进展字段，并在重建后重新注入控制字段。**

## 5. 领域模型

### 5.1 学习单元

统一使用：

- `semantic.coarse_unit`

表示学习目标对象。

当前至少支持：

- `word`
- `phrase`
- `grammar`

Learning engine 不关心它是单词、短语还是语法点；只要它是统一学习单元即可。

### 5.2 用户-学习单元状态

表示：

- 这个用户对这个学习单元当前学到什么程度
- 是否仍在当前 target 范围内
- 下次何时该复习
- 最近是否稳定

### 5.3 学习事件

表示：

- 用户围绕某个学习单元发生的一次标准化学习行为

事件由上游业务 API / 事件标准化层产生，Learning engine 只接受**标准化事件**，不直接处理原始前端行为。

### 5.4 Target 控制信息

表示：

- 这个学习单元是否在当前学习目标范围内
- 来源是什么
- 优先级多少
- 是否被挂起

这部分信息属于 Learning engine 聚合的一部分，但**不由学习事件归约**。

---

## 6. 状态聚合的字段所有权

`learning.user_unit_states` 虽然是一张表，但其中字段应分成两类。

### 6.1 控制字段（非事件归约）

这些字段不是从学习事件直接推出来的，而是由上游目标管理或 Learning engine 控制命令维护：

- `is_target`
- `target_source`
- `target_source_ref_id`
- `target_priority`
- `suspended_reason`

### 6.2 学习进展字段（由事件归约）

这些字段由事件 reducer 推导：

- `status`（除 `suspended` 外）
- `progress_percent`
- `mastery_score`
- `first_seen_at`
- `last_seen_at`
- `last_reviewed_at`
- 各类计数器
- `last_quality`
- `recent_quality_window`
- `recent_correctness_window`
- `repetition`
- `interval_days`
- `ease_factor`
- `next_review_at`

### 6.3 为什么必须区分这两类字段

因为这直接影响 Replay 设计。

事件是真相，适用于**学习进展字段**。
但 `is_target`、`target_priority` 这类字段并不来自学习事件。

因此 Replay 的正确含义不是：

> 只靠事件重建整行所有字段

而是：

> **用事件重建学习进展字段；同时保留或重新注入控制字段。**

## 7. 状态模型

### 7.1 状态枚举

Learning engine 使用以下状态：

- `new`
- `learning`
- `reviewing`
- `mastered`
- `suspended`

### 7.2 各状态语义

#### `new`

已进入 target 范围，但还没有形成有效学习闭环。
典型特征：

- 可能只有 target 记录，没有强事件
- 或只有少量弱事件
- `next_review_at` 为空
- `mastery_score` 接近 0

#### `learning`

已经发生正式学习，但仍处于密集重复阶段。
典型特征：

- 至少 1 次强事件
- 还没有建立稳定间隔复习节奏
- `interval_days` 很小
- `next_review_at` 已开始有意义

#### `reviewing`

已经离开新手阶段，进入间隔复习阶段。
典型特征：

- 至少经历多次通过的强事件
- 间隔开始增长
- `next_review_at` 是核心信号

#### `mastered`

在当前 MVP 标准下已达到“阶段性掌握”。
典型特征：

- 间隔较长
- 最近表现稳定
- 没有明显失败

#### `suspended`

被业务层暂时停用，不参与正常推荐与学习推进。
这不是自然学习迁移结果，而是一个控制态。

### 7.3 状态迁移规则

#### `new -> learning`

满足任一条件：

- 至少发生一次强事件

#### `learning -> reviewing`

满足全部条件：

- 至少两次通过的强事件
- 最近两次质量分都 `>= 3`

#### `reviewing -> mastered`

满足全部条件：

- `interval_days >= 21`
- 最近表现稳定
- 没有近期明显失败

#### `mastered -> reviewing`

满足任一条件：

- 最近一次强事件 `quality < 3`
- 或最近窗口表现明显下滑

#### `* -> suspended`

仅通过显式控制命令或外部 target 管理操作触发。
**不会**因为普通学习事件自动进入 `suspended`。

#### `suspended -> active`

通过显式恢复命令触发。
恢复时不直接写死成某个状态，而是根据当前学习进展字段**重新计算**为 `new / learning / reviewing / mastered` 中之一。

## 8. 事件模型

### 8.1 事件类型

Learning engine MVP 统一支持 5 类事件：

- `exposure`
- `lookup`
- `new_learn`
- `review`
- `quiz`

### 8.2 弱事件

#### `exposure`

表示用户看到了该学习单元。
例如：

- 视频中出现该单元
- 字幕中经过该单元
- 卡片中曝光该单元

作用：

- 增加 `seen_count`
- 更新 `first_seen_at` / `last_seen_at`

不作用于：

- `repetition`
- `interval_days`
- `ease_factor`
- `next_review_at`
- 状态迁移

#### `lookup`

表示用户显式查看该单元的释义、解释或翻译。
作用同属弱事件：

- 增加接触信息
- 不推进记忆状态

### 8.3 强事件

#### `new_learn`

表示用户第一次正式学习该单元。
通常来自：

- 新内容学习卡
- 显式学习任务
- 首次带反馈的学习动作

#### `review`

表示用户对已进入学习过程的单元进行复习。
通常来自：

- 回忆题
- 复习任务
- 复习型互动

#### `quiz`

表示用户针对该单元进行了测验型验证。
它本质也是强事件，但通常信号更接近结果确认。

### 8.4 强事件的必填 / 选填字段

强事件建议具备：

- `quality`：必填，范围 `0..5`
- `is_correct`：选填
- `response_time_ms`：选填
- `video_id`：选填
- `source_type`：必填
- `source_ref_id`：选填
- `occurred_at`：必填

MVP 建议要求：

> **强事件必须提供 `quality`。**

如果上游只有“对 / 错”，应在标准化阶段映射为质量分后再写入 Learning engine。

### 8.5 `source_type` 建议约束

`source_type` 应代表业务来源，例如：

- `video_clip`
- `learning_task`
- `quiz_session`
- `dictionary_lookup`
- `manual_import`
- `admin_repair`

Learning engine 不要求 DB 层强枚举，但应用层必须校验。`source_type` 表示业务事件来源类别，不要求与 Catalog 的物理表名一一对应；当前阶段不建议继续使用容易被误解为物理表实体的旧 `segment` 类命名。

### 8.6 `metadata` 的定位

`metadata` 是审计和溯源字段，不参与核心 reducer 计算。
典型内容可包括：

- `video_id`
- `sentence_index`
- `span_index`
- `evidence_ref`
- `task_id`
- `session_id`
- `replay_job_id`
- `device_type`

## 9. 质量评分体系

Learning engine 统一使用 `0..5` 质量分。

推荐语义：

- `5`：轻松正确
- `4`：正确但有轻微犹豫
- `3`：勉强正确
- `2`：错误但经提示可想起
- `1`：明显错误
- `0`：完全不会

MVP 的关键分界线：

- `quality >= 3`：通过
- `quality < 3`：失败

这条线同时用于：

- 窗口正确性统计
- SM-2 成功 / 失败分支
- 状态迁移判断

## 10. 数据模型设计

## 10.1 `learning.user_unit_states`

这是 Learning engine 的核心状态表。

建议结构：

```sql
create table if not exists learning.user_unit_states (
  user_id uuid not null references auth.users(id) on delete cascade,
  coarse_unit_id bigint not null references semantic.coarse_unit(id) on delete cascade,

  is_target boolean not null default true,
  target_source text,
  target_source_ref_id text,
  target_priority numeric(5,4) not null default 0.5,

  status text not null default 'new'
    check (status in ('new', 'learning', 'reviewing', 'mastered', 'suspended')),

  progress_percent numeric(5,2) not null default 0
    check (progress_percent between 0 and 100),

  mastery_score numeric(5,4) not null default 0
    check (mastery_score between 0 and 1),

  first_seen_at timestamptz,
  last_seen_at timestamptz,
  last_reviewed_at timestamptz,

  seen_count int not null default 0,
  strong_event_count int not null default 0,
  review_count int not null default 0,
  correct_count int not null default 0,
  wrong_count int not null default 0,
  consecutive_correct int not null default 0,
  consecutive_wrong int not null default 0,

  last_quality smallint check (last_quality between 0 and 5),
  recent_quality_window smallint[] not null default '{}',
  recent_correctness_window boolean[] not null default '{}',

  repetition int not null default 0,
  interval_days numeric(8,2) not null default 0,
  ease_factor numeric(6,4) not null default 2.5,
  next_review_at timestamptz,

  suspended_reason text,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  primary key (user_id, coarse_unit_id)
);
```

这个结构与当前既定方向一致：target 属性和学习状态属性共存在同一聚合上，但 Recommendation 的 serving state 不进入这张表。

### 10.1.1 字段语义说明

#### target 相关字段

`is_target`
是否仍在当前学习目标范围内。Recommendation 默认只读取 `is_target = true` 的状态。

`target_source`
该目标单元来自哪里，例如课程、学习计划、用户显式收藏。

`target_source_ref_id`
目标来源的引用 ID。

`target_priority`
目标优先级，范围 `0..1`。该字段不由事件归约，而由 target 控制命令更新。

#### 状态相关字段

`status`
当前学习状态。`suspended` 是控制态，其余由 reducer 计算。

`progress_percent`
面向产品与对外消费的阶段性进度百分比。它是**可解释的 UI 信号**，不是记忆模型真相字段。

`mastery_score`
`0..1` 的掌握分。它是综合信号，不等同于单一 SM-2 参数。

#### 时间字段

`first_seen_at`
第一次接触该单元的时间。

`last_seen_at`
最近一次接触该单元的时间。弱事件和强事件都会更新。

`last_reviewed_at`
最近一次强事件时间。

#### 计数字段

`seen_count`
总接触次数。弱 / 强事件都加 1。

`strong_event_count`
强事件总数。

`review_count`
复习类强事件数量。建议只对 `review` 和 `quiz` 增加，不对 `new_learn` 增加。

`correct_count` / `wrong_count`
按 `quality >= 3` / `< 3` 统计。

`consecutive_correct` / `consecutive_wrong`
连续通过 / 失败计数。

#### 记忆调度字段

`last_quality`
最近一次强事件质量分。

`recent_quality_window`
最近 N 次强事件质量窗口。MVP 建议 `N = 5`。

`recent_correctness_window`
最近 N 次强事件通过 / 失败窗口。MVP 建议 `N = 5`。

`repetition`
SM-2 风格复习成功次数计数。

`interval_days`
当前间隔天数。

`ease_factor`
SM-2 风格 `easiness factor`，默认 `2.5`，下限 `1.3`。

`next_review_at`
下一次推荐复习时间。Recommendation 会把它当成重要输入。

#### 挂起字段

`suspended_reason`
挂起原因。为空时不代表一定 active，因为 `is_target=false` 也可能导致 Recommendation 不读取该行。

### 10.1.2 建议索引

```sql
create index if not exists idx_learning_states_user_target_status_due
on learning.user_unit_states (user_id, is_target, status, next_review_at);

create index if not exists idx_learning_states_user_updated_at
on learning.user_unit_states (user_id, updated_at desc);
```

用途：

- Recommendation 读当前用户 active states
- 后台检查某用户最近更新的状态

## 10.2 `learning.unit_learning_events`

这是 Learning engine 的事实表。

建议结构：

```sql
create table if not exists learning.unit_learning_events (
  event_id bigint generated always as identity primary key,
  user_id uuid not null references auth.users(id) on delete cascade,
  coarse_unit_id bigint not null references semantic.coarse_unit(id) on delete cascade,
  video_id uuid references catalog.videos(video_id) on delete set null,

  event_type text not null,
  source_type text not null,
  source_ref_id text,

  is_correct boolean,
  quality smallint check (quality between 0 and 5),
  response_time_ms int,

  metadata jsonb not null default '{}'::jsonb,
  occurred_at timestamptz not null,
  created_at timestamptz not null default now()
);
```

### 10.2.1 事件表的关键约束

#### append-only

Learning engine 将这张表视为 append-only。
MVP 不支持直接更新历史事件来“修状态”。

#### 事件排序

Replay 和在线 reducer 的稳定顺序统一采用：

1. `occurred_at ASC`
2. `event_id ASC`

即使用 `event_id` 作为同时间戳下的最终 tie-breaker。

### 10.2.2 建议索引

```sql
create index if not exists idx_learning_events_user_time
on learning.unit_learning_events (user_id, occurred_at, event_id);

create index if not exists idx_learning_events_user_unit_time
on learning.unit_learning_events (user_id, coarse_unit_id, occurred_at, event_id);
```

## 11. Target 控制命令设计

这个部分在很多初版设计里容易漏掉，但对当前 MVP 非常关键。

因为 Recommendation 需要能读取：

- 还没有发生任何强事件，但已经进入学习目标范围的 unit
- 这些 unit 的 `status='new'`
- 它们的 `target_priority`

这意味着 Learning engine 不能只靠学习事件“被动长出状态行”，还需要有显式的 target 管理命令。

### 11.1 `EnsureTargetUnits`

作用：

- 为用户的一组目标单元建立或更新状态行
- 如果状态行不存在，则创建默认 `new` 行
- 如果状态行存在，则只更新 control slice，不改 learning progression slice

输入建议：

```json
{
  "user_id": "uuid",
  "targets": [
    {
      "coarse_unit_id": 101,
      "target_source": "curriculum",
      "target_source_ref_id": "lesson_12",
      "target_priority": 0.9
    }
  ]
}
```

行为规则：

- 不写 `learning.unit_learning_events`
- 若行不存在，创建默认状态：
  - `is_target=true`
  - `status='new'`
  - 其他学习进展字段为默认值

- 若行存在：
  - 仅更新 `is_target` / `target_*`
  - 不重置学习进展字段

### 11.2 `SetTargetInactive`

作用：

- 将某个状态行标记为不再参与当前 target 范围

推荐行为：

- 设置 `is_target=false`
- 不建议物理删除行

原因：

- 保留历史学习状态
- Replay 时仍可保留控制字段快照
- Recommendation 只需过滤 `is_target=true`

### 11.3 `SuspendTargetUnit`

作用：

- 暂时挂起某个学习单元

行为：

- 设置 `status='suspended'`
- 写 `suspended_reason`

说明：

- 挂起是控制态，不由 reducer 自动进入
- 恢复时需要显式命令

### 11.4 `ResumeTargetUnit`

作用：

- 取消挂起

行为：

- 清空 `suspended_reason`
- 按当前学习进展字段重新计算 `status`

### 11.5 为什么需要控制命令，而不是都走事件

因为 target 控制信息不是学习行为事实。
Learning engine 的 event log 记录的是“学了什么”，而不是“当前课程计划怎么定”。这两类信息必须分开。

## 12. Reducer 设计

Reducer 是 Learning engine 的核心。

### 12.1 Reducer 输入 / 输出

输入：

- 当前 `UserUnitState`
- 一条 `LearningEvent`

输出：

- 新的 `UserUnitState`

MVP 中 reducer 以**单事件、单聚合**为最小处理单位。

### 12.2 Reducer 处理顺序

建议固定为以下顺序：

1. 校验事件合法性
2. 初始化空状态（若当前无状态）
3. 识别弱 / 强事件
4. 更新基础时间与计数字段
5. 如果是强事件：
   - 更新质量窗口与正确性窗口
   - 更新 SM-2 字段
   - 更新通过 / 失败计数

6. 重算状态迁移
7. 重算 `progress_percent`
8. 重算 `mastery_score`
9. 如果当前处于挂起控制态，则覆盖 `status='suspended'`
10. 更新 `updated_at`

### 12.3 空状态初始化

如果某个 `user_id + coarse_unit_id` 没有状态行，初始化为：

- `is_target=true`（若由强事件首次触发）
- `status='new'`
- 所有计数为 0
- `ease_factor=2.5`
- `progress_percent=0`
- `mastery_score=0`

说明：

- 如果该 unit 本应通过 `EnsureTargetUnits` 预先建行，则强事件只是落在已有新行上
- 如果某条学习事件先到，也允许惰性创建状态行

---

## 13. 弱事件更新规则

弱事件包括 `exposure` 和 `lookup`。

对弱事件，Reducer 只做这些事：

- `first_seen_at = min(first_seen_at, occurred_at)`，若为空则填当前时间
- `last_seen_at = max(last_seen_at, occurred_at)`
- `seen_count += 1`

不更新：

- `strong_event_count`
- `review_count`
- `correct_count`
- `wrong_count`
- `last_quality`
- `repetition`
- `interval_days`
- `ease_factor`
- `next_review_at`

也不触发：

- `new -> learning`
- `learning -> reviewing`
- `reviewing -> mastered`

---

## 14. 强事件更新规则

强事件包括 `new_learn`、`review`、`quiz`。

### 14.1 基础计数

所有强事件：

- `seen_count += 1`
- `strong_event_count += 1`
- `last_seen_at = occurred_at`
- `last_reviewed_at = occurred_at`
- `last_quality = quality`

仅 `review` / `quiz`：

- `review_count += 1`

### 14.2 通过 / 失败判定

统一按 `quality >= 3` 判为通过。

通过时：

- `correct_count += 1`
- `consecutive_correct += 1`
- `consecutive_wrong = 0`

失败时：

- `wrong_count += 1`
- `consecutive_wrong += 1`
- `consecutive_correct = 0`

### 14.3 质量窗口

建议窗口长度：

- `recent_quality_window`：最多保留最近 5 个值
- `recent_correctness_window`：最多保留最近 5 个布尔值

写法：

- 新值 append 到尾部
- 超过长度则丢弃最早值

---

## 15. SM-2 简化规则

MVP 使用简化 SM-2。

### 15.1 成功分支（`quality >= 3`）

#### repetition 更新

```text
repetition = repetition + 1
```

#### interval 更新

建议逻辑：

- 第 1 次成功：`interval_days = 1`
- 第 2 次成功：`interval_days = 3`
- 第 3 次成功：`interval_days = 6`
- 第 4 次及以后：`interval_days = round(interval_days * ease_factor)`

#### ease_factor 更新

采用标准 SM-2 风格更新：

```text
EF' = EF + (0.1 - (5-q) * (0.08 + (5-q) * 0.02))
EF' = max(1.3, EF')
```

其中 `q = quality`。

#### next_review_at 更新

```text
next_review_at = occurred_at + interval_days
```

### 15.2 失败分支（`quality < 3`）

建议逻辑：

- `repetition = 0`
- `interval_days = 1`
- `next_review_at = occurred_at + 1 day`
- `ease_factor` 保持不变

说明：

- MVP 不做更复杂的 failure EF 惩罚
- 失败后快速回到短周期复习即可

### 15.3 为什么 MVP 采用简化 SM-2

因为当前 Recommendation 需要的是：

- 稳定的 due 信号
- 稳定的 mastery 信号
- 足以让 Recommendation 自行派生 `fragility` / `instability` 判断的底层状态输入

而不是一套非常复杂的记忆建模。
简化 SM-2 足够支撑当前视频学习场景。

---

## 16. 状态迁移判定

Reducer 在每次强事件后都重算状态。

### 16.1 基本判定顺序

建议顺序：

1. 如果被显式挂起，则最终状态强制为 `suspended`
2. 如果 `strong_event_count = 0`，则 `new`
3. 如果尚未达到 reviewing 条件，则 `learning`
4. 如果达到 mastered 条件，则 `mastered`
5. 否则 `reviewing`

### 16.2 `learning` 判定

满足以下任一条件时保持 `learning`：

- 强事件次数不足 2
- 最近两次强事件不都通过
- 当前 `interval_days < 3`

### 16.3 `reviewing` 判定

满足全部条件：

- 至少 2 次通过的强事件
- 最近两次质量分都 `>= 3`
- 当前不满足 `mastered`

### 16.4 `mastered` 判定

建议条件：

- `interval_days >= 21`
- 最近 3 次强事件中没有失败
- `recent_correctness_window` 最近 3 项全部为 `true`，或不存在失败
- `mastery_score >= 0.8` 可作为增强条件

MVP 的掌握阈值继续使用 **21 天**。

### 16.5 `mastered` 回退

若最近强事件失败：

- 状态回退到 `reviewing`
- `repetition` / `interval_days` 按失败分支重置

## 17. `progress_percent` 计算

`progress_percent` 的用途是：

- 面向产品和调试展示
- 给 Recommendation 一个较柔和的进度参考

它**不是**唯一真相字段，不能取代 `status + next_review_at + last_quality`。

为了便于实施，MVP 建议明确公式。

### 17.1 设计原则

- 范围 `0..100`
- 可以回退，不要求单调不减
- 主要反映“从新学到阶段性掌握”的推进程度
- 对失败敏感，但不应剧烈抖动

### 17.2 推荐公式

定义：

```text
interval_component   = min(interval_days / 21, 1.0)
accuracy_component   = avg(recent_correctness_window)      // 空窗口记为 0
repetition_component = min(repetition / 4, 1.0)
quality_component    = avg(recent_quality_window) / 5.0   // 空窗口记为 0
```

推荐：

```text
progress_percent =
  100 * clamp01(
    0.45 * interval_component +
    0.20 * accuracy_component +
    0.20 * repetition_component +
    0.15 * quality_component
  )
```

附加规则：

- 若 `strong_event_count = 0`，直接为 `0`
- 若 `status = mastered`，直接封顶为 `100`
- 若 `status = suspended`，仍按当前学习进展计算，但 Recommendation 不消费该值

### 17.3 为什么这样定义

这能保证：

- 刚学过但还没建立间隔时，进度不会虚高
- 间隔逐步增长时，进度稳步增加
- 失败会导致 interval / accuracy 下滑，进度合理回落

---

## 18. `mastery_score` 计算

`mastery_score` 是 `0..1` 的综合掌握分。
它是 Recommendation 进行轻量 fragility / maturity 判断的重要输入之一。

### 18.1 设计原则

- 比 `progress_percent` 更偏“当前掌握程度”
- 对最近失败更敏感
- 不完全等同于 interval
- 应与 `status` 大致一致，但允许更细粒度

### 18.2 推荐公式

定义：

```text
interval_component   = min(interval_days / 21, 1.0)
accuracy_component   = avg(recent_correctness_window)      // 空窗口记为 0
stability_component  = min(consecutive_correct / 3, 1.0)
repetition_component = min(repetition / 4, 1.0)
failure_penalty      = 0.20 if last_quality is not null and last_quality < 3 else 0
```

推荐：

```text
mastery_score =
  clamp01(
    0.45 * interval_component +
    0.25 * accuracy_component +
    0.15 * stability_component +
    0.15 * repetition_component -
    failure_penalty
  )
```

附加规则：

- `strong_event_count = 0` 时为 `0`
- `status = mastered` 但最近失败时，`mastery_score` 会明显回落

### 18.3 使用建议

Recommendation 应把它视为**辅助强度信号**，而不是硬 due 判定来源。
真正的 due 判定仍应优先看：

- `status`
- `next_review_at`
- `last_quality`

---

## 19. Reducer 伪代码

```text
Reduce(currentState, event):

  state = initIfNil(currentState)

  validate(event)

  updateSeenFields(state, event)

  if isWeakEvent(event):
      state.progress_percent = recomputeProgress(state)
      state.mastery_score    = recomputeMastery(state)
      state.status           = applySuspensionOverlay(
                                 recomputeStatusWithoutSuspension(state)
                               )
      state.updated_at = now()
      return state

  // strong event
  pass = (event.quality >= 3)

  state.strong_event_count += 1
  state.last_reviewed_at = event.occurred_at
  state.last_quality = event.quality

  if event.event_type in (review, quiz):
      state.review_count += 1

  appendRecentQuality(state, event.quality)
  appendRecentCorrectness(state, pass)

  if pass:
      state.correct_count += 1
      state.consecutive_correct += 1
      state.consecutive_wrong = 0
      applySm2Success(state, event.quality, event.occurred_at)
  else:
      state.wrong_count += 1
      state.consecutive_wrong += 1
      state.consecutive_correct = 0
      applySm2Failure(state, event.occurred_at)

  state.progress_percent = recomputeProgress(state)
  state.mastery_score    = recomputeMastery(state)
  state.status           = applySuspensionOverlay(
                             recomputeStatusWithoutSuspension(state)
                           )
  state.updated_at = now()

  return state
```

## 20. 在线写入主流程

Learning engine 的主写入用例是：

> `RecordLearningEvents`

### 20.1 输入

```json
{
  "user_id": "uuid",
  "events": [
    {
      "coarse_unit_id": 101,
      "event_type": "review",
      "source_type": "quiz_session",
      "source_ref_id": "quiz_attempt_123",
      "quality": 4,
      "is_correct": true,
      "occurred_at": "2026-04-16T10:00:00Z",
      "metadata": {}
    }
  ]
}
```

### 20.2 处理流程

```text
RecordLearningEvents
  ↓
参数校验 / 事件标准化校验
  ↓
按 (user_id, coarse_unit_id) 分组并按 occurred_at 排序
  ↓
开启事务
  ↓
批量 append 到 learning.unit_learning_events
  ↓
逐个聚合读取 current state（FOR UPDATE）
  ↓
按顺序调用 reducer
  ↓
upsert learning.user_unit_states
  ↓
提交事务
```

### 20.3 事务边界

**写事件 + 写状态必须在同一事务中完成。**

否则会出现：

- 事件写入成功，状态没更新
- 状态更新成功，事件没落表

两者都会破坏 Replay 与在线一致性。

---

## 21. Replay 设计

Learning engine 的第二个核心用例是：

> `ReplayUserStates`

### 21.1 MVP 范围

MVP 只做：

- `FullUserReplay(user_id)`

即：

- 对某个用户读取全部学习事件
- 完整重建该用户所有 Learning engine 状态

### 21.2 Replay 的关键语义

Replay 重建的是：

- 学习进展字段

不是直接从事件表重建所有控制字段。
因此 Replay 必须先拿到**控制字段快照**。

### 21.3 Replay 流程

建议流程：

```text
ReplayUserStates(user_id)
  ↓
开启事务
  ↓
读取该用户当前 user_unit_states 的 control slice 快照
  （is_target / target_* / suspended_reason）
  ↓
读取该用户全部 unit_learning_events
  按 occurred_at, event_id 排序
  ↓
删除该用户全部 user_unit_states
  ↓
对每个 unit 从空状态开始应用 reducer，得到重建后的学习进展字段
  ↓
将控制字段快照与重建结果 merge
  ↓
若某行当前被挂起，则最终 status 覆盖为 suspended
  ↓
batch upsert 新状态
  ↓
提交事务
```

### 21.4 为什么 Replay 不能直接“只靠事件重建整行”

因为当前表里有：

- `is_target`
- `target_priority`
- `target_source`

这些字段并不来自学习事件。
如果 Replay 不先保留控制字段，重建后会丢失 target 信息，Recommendation 也就无法读取那些仍应属于 target scope 的 `new` 单元。

### 21.5 Replay 输出一致性要求

同一批事件历史，在以下两条路径上必须得到一致结果：

- 在线逐条写入后的最终状态
- full Replay 重建状态

这是必须通过自动化测试保证的。

---

## 22. 晚到事件 / out-of-order 事件处理

### 22.1 正常假设

在线写入路径默认假设：

- 上游已经基本按时间顺序提交事件
- 单个用户 / 单个 unit 的事件不存在大规模历史倒灌

### 22.2 如果出现晚到历史强事件

如果某强事件满足：

- `occurred_at` 明显早于该 unit 当前已处理的最后强事件时间

则单纯在当前状态上继续 reduce 可能不再精确。

MVP 推荐策略：

#### 策略 A：在线主路径拒绝晚到强事件

适用于高实时链路。
由上游改走修复通道。

#### 策略 B：允许 append，但随后触发该用户 full Replay

适用于导入、纠错、补录场景。

二选一即可。
**不建议**在在线路径里尝试“局部回滚 + 局部重算”。

推荐默认：

- 实时链路用 A
- 导入 / 修复链路用 B

### 22.3 弱事件晚到

弱事件只影响 seen 信息，影响较小。
MVP 可直接接受并更新：

- `first_seen_at`
- `last_seen_at`
- `seen_count`

如果需要绝对一致，也可统一走 Replay。

## 23. 与 Recommendation 的读取契约

Recommendation 不应该理解 Learning engine 内部 reducer，只需要理解读取语义。

### 23.1 Recommendation 读取的主表

Recommendation 读取：

- `learning.user_unit_states`

### 23.2 Recommendation 主要使用字段

至少包括：

- `is_target`
- `target_priority`
- `status`
- `progress_percent`
- `mastery_score`
- `last_quality`
- `next_review_at`
- `recent_quality_window`
- `recent_correctness_window`
- `strong_event_count`
- `review_count`

这些字段是当前视频需求规划的稳定输入。

Learning engine 不单独持久化 `fragility` / `fragility_score` 字段。若 Recommendation 需要使用“当前脆弱度”这类判断，应基于以下稳定输出字段自行派生：

- `status`
- `next_review_at`
- `mastery_score`
- `last_quality`
- `recent_quality_window`
- `recent_correctness_window`
- `strong_event_count`
- `review_count`

### 23.3 字段语义约定

`status`
粗粒度学习阶段。`status='new'` 不表示 due；只表示尚未正式建立学习闭环。

`next_review_at`
若为空，通常表示：

- 尚未发生足够强事件
- 或还未形成可调度复习节奏

Recommendation 不应把空值视为“早该复习”。

`last_quality`
最近一次强事件的结果。若为空，表示还没有强事件。

`progress_percent`
可解释进度，不应作为唯一调度依据。

`mastery_score`
综合掌握分，可用于软判断，也可作为 Recommendation 派生 `fragility` / `instability` 判断的底层输入之一，但不替代 `next_review_at`。

### 23.4 Recommendation 不得依赖的内容

Recommendation 不应从 Learning engine 读取或期待以下字段：

- `last_recommended_at`
- 推荐 run 信息
- 视频 serving state
- 推荐冷却信息

这些都不属于 Learning engine。

## 26. 事务与并发控制

### 26.1 单事务原则

以下操作必须单事务：

- append events
- load current state
- reduce
- upsert state

### 26.2 行级锁建议

对在线写入路径，建议对当前状态行执行：

- `SELECT ... FOR UPDATE`

场景：

- 同一用户、同一 unit 的事件可能并发进入
- 如果不锁，可能发生 lost update

### 26.3 批量事件建议

若同一次请求包含多个 unit 的事件：

- 先按 `(user_id, coarse_unit_id)` 分组
- 每组内部按 `occurred_at` 排序
- 每组各自 reduce
- 最后统一 commit

### 26.4 Replay 与在线写入冲突

MVP 建议：

- Replay 期间禁止同一用户的在线写入，或
- Replay 放在低频后台维护通道中执行

最简单的策略是：

- 同一用户 Replay 时加用户级互斥

## 30. 测试设计

### 30.1 领域单测

必须覆盖：

#### 弱事件

- exposure 只更新 seen，不推进状态

#### 强事件成功

- `new -> learning`
- repetition / interval / next_review_at 正确更新

#### 强事件失败

- repetition 重置
- `interval_days=1`
- mastered 回退

#### 窗口维护

- recent windows 长度截断正确

#### 状态迁移

- `learning -> reviewing`
- `reviewing -> mastered`
- `mastered -> reviewing`

#### progress / mastery

- 成功推进时增长
- 失败时回落

### 30.2 集成测试

至少覆盖以下场景。

#### 场景 A：目标单元预建

- 调用 `EnsureTargetUnits`
- 状态行创建为 `new`
- 无事件时 `next_review_at` 为空

#### 场景 B：新用户首次正式学习

- 首次强事件后 `new -> learning`

#### 场景 C：正常推进

- 连续成功后进入 `reviewing`
- 间隔增长
- 21 天阈值后进入 `mastered`

#### 场景 D：失败回退

- `mastered` 失败后回到 `reviewing`
- 间隔重置为 1 天

#### 场景 E：Full Replay 一致性

- 在线多次写入后的最终状态
- 与 Replay 重建状态完全一致

#### 场景 F：事务回滚

- 若状态 upsert 失败，事件 append 必须一并回滚

#### 场景 G：目标控制字段保留

- target 行先存在
- Full Replay 后 `is_target / target_priority / target_source` 仍被保留

### 30.3 Golden tests 建议

建议维护若干固定事件序列作为 golden cases，例如：

1. `new -> learning -> reviewing`
2. `reviewing -> mastered`
3. `mastered -> review failure -> reviewing`
4. `只发生弱事件`
5. `先 EnsureTargetUnits，再发生强事件`
6. `同 timestamp 多事件`

## 31. 示例：一个完整状态演化

假设某用户对 unit 101 的学习过程如下。

### 初始

通过 `EnsureTargetUnits` 建立状态行：

```text
status = new
progress_percent = 0
mastery_score = 0
next_review_at = null
```

### 事件 1：exposure

```text
event_type = exposure
occurred_at = T1
```

结果：

```text
seen_count = 1
first_seen_at = T1
last_seen_at = T1
status 仍为 new
```

### 事件 2：new_learn, quality=4

```text
event_type = new_learn
quality = 4
occurred_at = T2
```

结果：

```text
status = learning
strong_event_count = 1
repetition = 1
interval_days = 1
next_review_at = T2 + 1d
```

### 事件 3：review, quality=4

```text
event_type = review
quality = 4
occurred_at = T3
```

结果：

```text
status = reviewing
strong_event_count = 2
review_count = 1
repetition = 2
interval_days = 3
next_review_at = T3 + 3d
```

### 事件 4：review, quality=5

```text
event_type = review
quality = 5
occurred_at = T4
```

结果：

```text
status = reviewing
repetition = 3
interval_days = 6
next_review_at = T4 + 6d
```

### 事件 5：review, quality=5，且 interval 扩大到 21+

若后续成功使间隔达到 21 天以上，且最近窗口稳定：

```text
status = mastered
progress_percent = 100
mastery_score ≈ 0.8+
```

### 事件 6：review, quality=1

结果：

```text
status = reviewing
repetition = 0
interval_days = 1
next_review_at = T6 + 1d
mastery_score 明显下降
```

这个例子展示了：

- 弱事件只加 seen
- 强事件推进状态
- mastered 可以回退
- 状态始终可由事件重建

- `learning.user_unit_states`

### 32.2 可稳定读取的字段

- `is_target`
- `target_priority`
- `status`
- `progress_percent`
- `mastery_score`
- `last_quality`
- `next_review_at`
- `recent_quality_window`
- `recent_correctness_window`
- `strong_event_count`
- `review_count`
- `updated_at`

### 32.3 Recommendation 读取过滤建议

```sql
where user_id = $1
  and is_target = true
  and status <> 'suspended'
```

### 32.4 Learning engine 对 Recommendation 的 SLA 语义

- `status` 是粗粒度阶段，不是推荐策略本身
- `next_review_at` 是硬 due 相关字段
- `mastery_score` 是辅助连续值
- `progress_percent` 是展示 / 软判断字段
- 控制字段更新与事件归约解耦，不会因为 Replay 丢失

## 33. 本期明确不做的内容

为了保持范围稳定，本期 Learning engine 明确不做：

- partial / scoped replay
- 自动处理复杂乱序历史事件
- 用户自定义记忆策略
- 多种 spaced repetition 模型并存
- 事件编辑式修复工具
- Recommendation serving state
- 视频级投放冷却
- task 层反馈建模

## 34. 最终结论

Learning engine 的最终职责可以概括成一句话：

> **基于统一学习事件流，维护用户对目标学习单元的稳定状态总表，并通过 Full Replay 保证状态可重建、可验证、可修复。**

因此在最终版本里：

- `learning.unit_learning_events` 是事实真相层
- `learning.user_unit_states` 是 Learning engine 维护的状态总表
- Learning engine 不负责 Recommendation 的投放状态
- Recommendation 只消费 Learning engine 的结果，不反向维护其业务表
- Target 控制信息通过显式控制命令维护，学习进展字段通过事件归约维护
- Replay 重建学习进展字段，同时保留控制字段快照
