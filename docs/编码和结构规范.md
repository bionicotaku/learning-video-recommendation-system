# 编码和结构规范

## 1. 目的

本文档定义当前项目的统一编码与目录规范，供人和 Codex agent 共同遵守。目标只有三个：先确定 owner 和边界，再落目录与代码；同类职责只有一个标准落点；任何实现都优先服从当前三域设计，而不是临时发明结构。当前系统的顶层业务域固定为 `catalog`、`learningengine`、`recommendation`，三者平级，owner 清晰，依赖方向稳定。

## 2. 最高优先级规则

先定边界，再写代码。新增能力前必须先回答四个问题：它属于哪个顶层模块；它是模块根职责还是某个子能力；它读哪些外部契约；它写哪些 owner 对象。不要先写文件，再回头解释为什么放在这里。模块之间必须低耦合、高内聚；业务规则只能有一个 owner；依赖必须面向抽象并通过注入获得，不能在业务代码里偷偷 `new` 出外部依赖。

任何跨模块共享契约变更，必须在同一提交中同时更新对应设计文档。不要让代码先跑到文档前面。

## 3. 顶层结构

仓库内业务代码统一放在 `internal/` 下。当前标准根结构如下：

```text
internal/
  catalog/
  learningengine/
  recommendation/
  test/
    e2e/
```

`internal/test/e2e` 只放跨顶层模块端到端测试或跨模块真实数据库链路测试，不放任何单一模块自己的单测或集成测试。模块根必须优先有 `README.md`，建议有 `doc.go`。如果模块根只是边界与子模块容器，就不要在根目录再复制一套 `application/domain/infrastructure/test`。只有真正承载实现的层级，才使用完整骨架。

## 4. 模块骨架

任何承载具体实现的模块或子模块，统一采用这一套骨架：

```text
<module-or-submodule>/
  README.md
  doc.go
  application/
  domain/
  infrastructure/
  test/
```

这是一种克制的 `DDD-lite + Clean Architecture` 结构：`application` 负责用例编排，`domain` 负责业务规则，`infrastructure` 负责技术落地，`test` 负责验证。不要为不同模块再发明第二套骨架。

### application

`application` 负责“怎么串起来执行”。标准落点如下：

```text
application/
  command/
  dto/
  query/
  repository/
  service/
  usecase/
```

`command` 放输入对象，`dto` 放输出对象，`query` 放面向用例的读模型或候选结构，`repository` 只定义 port interface，`service` 放应用层辅助服务，`usecase` 放主业务入口。不要在 `application` 写纯领域规则。

### domain

`domain` 负责“规则本身是什么”。标准落点如下：

```text
domain/
  aggregate/
  enum/
  model/
  policy/
  rule/
  service/
```

`aggregate` 放聚合根或 reducer；`enum` 放稳定枚举；`model` 放领域对象；`policy` 放固定策略、默认参数、阈值；`rule` 放原子规则处理器；`service` 放可复用的领域计算器。`domain` 不得依赖数据库实现，不得依赖 `sqlc` 生成代码，不得直接 import 持久化包。

### infrastructure

`infrastructure` 负责技术实现。标准落点如下：

```text
infrastructure/
  config.go
  db.go
  migration/
  persistence/
    mapper/
    query/
    queryctx/
    repository/
    schema/
    sqlcgen/
    tx/
```

规则是：SQL 放 `persistence/query`，`sqlc` 生成代码只放 `sqlcgen`，不要手改；repository 的具体实现只放 `persistence/repository`；事务实现只放 `persistence/tx`；migration 只定义本模块 owner 的数据库对象，不跨模块定义别人的表或视图。

### test

`test` 负责“如何验证这个模块”。标准落点如下：

```text
test/
  fixture/
  unit/
  integration/
```

扩展需要时再加：

```text
test/
  golden/
  scenario/
```

`fixture` 放共享测试基座、seed helper、fake builder、golden loader 等公共测试能力；`unit` 尽量镜像源码路径；`integration` 放真实数据库、真实事务、真实跨层编排测试；`scenario` 放单模块内部跨多个用例的长链路业务场景；`golden` 放 golden 输入输出数据文件。所有**单模块内部测试**必须统一放在模块自己的 `test/` 目录下，不允许把 `*_test.go` 散落在 `application/`、`domain/`、`infrastructure/` 等业务实现目录中。

推荐的单模块测试结构如下：

```text
internal/<module>/
  application/
  domain/
  infrastructure/
  test/
    fixture/
    unit/
      application/
      domain/
      infrastructure/
    integration/
      application/
      infrastructure/
    golden/      # 按需
    scenario/    # 按需
```

这是当前项目内**单模块测试结构的唯一标准**。如果模块需要测试，就按这套目录补；不要再为某个模块单独发明第二套测试树。

补充规则：

- `fixture/` 只放测试公共能力，例如共享 embedded Postgres、template database、schema apply、seed helper、builder、golden loader；不承载业务规则，也不再新增散落的 `testutil/`、`dbtest/`、`mockdata/` 目录。
- `unit/` 只放不依赖真实数据库、真实事务、真实基础设施的测试；`integration/` 只放真实数据库、真实事务、真实 repository/sqlc/tx 实现测试；`golden/` 只放数据文件；`scenario/` 只在单模块内部长链路场景确有必要时使用。
- `unit/` 与 `integration/` 的目录层级尽量镜像源码路径。例如 `domain/aggregate/reducer.go -> test/unit/domain/aggregate/reducer_test.go`。
- 模块内真实数据库测试基座统一收敛到 `test/fixture/`，推荐模式是“模块级共享测试基座 + 测试级隔离数据”，而不是“每个测试各自起库”或“所有测试共用一份脏数据库”。
- `internal/<module>/test/...` 只放该模块 owner 范围内的测试；跨顶层模块链路测试继续固定在 `internal/test/e2e`，职责不变。
- 测试文件命名必须表达被测对象或场景，禁止新增 `common_test.go`、`misc_test.go`、`sample_test.go` 这类低信息文件名。
- 对已有模块做测试结构迁移时，顺序固定为：先建 `test/` 骨架，再迁 `integration`，再迁 `unit`，最后把旧 `testutil/` 和散落 `*_test.go` 收口到 `test/fixture/` 并删除旧路径，不保留双轨结构。

## 5. 当前项目的 owner 约束

当前系统三域 owner 必须严格遵守。

`catalog` 只拥有内容事实、结构化 transcript 读模型、Recall-ready 索引、入库审计、用户视频互动投影。不要在这里新增 Recommendation serving state，不要重新引入 `catalog.video_segments` 或 `catalog.segment_unit_mappings` 这类已被淘汰的结构。内容最细事实层是 `catalog.video_semantic_spans`，视频级索引是 `catalog.video_unit_index`。

`learningengine` 只拥有学习事件、学习状态、target/control 字段和 replay，不拥有推荐投放状态。`last_recommended_at`、video serving state、recommendation run/item 均不属于 Learning engine。

`recommendation` 只拥有推荐主链路、serving state、预计算读模型和推荐审计。当前 Recommendation 必须建立在 `videos -> video_transcripts -> video_transcript_sentences -> video_semantic_spans -> video_unit_index` 这条内容链之上，不允许重新假设独立的 `video_segments` / `segment_unit_mappings` 物理表存在。

## 6. 文件与命名规则

文件名按职责命名，优先使用 `generate_xxx.go`、`record_xxx.go`、`quota_allocator.go`、`review_scorer.go` 这类“职责名”文件名。不要默认发明 `utils.go`、`common.go`、`helper.go`、`manager.go`、`misc.go`。只有当文件真的只承载极少量共享辅助逻辑时，才允许类似名称。

同一种职责只允许一个标准落点。已经有 `domain/service`，就不要再出现 `domain/services`、`domain/helper`、`domain/manager`。已经有 `application/usecase`，就不要再出现一套平行的 `handlers`、`coordinators`、`executors`。不要让同一份业务规则散落在多个位置。

命名必须显式表达单位与语义。时间字段用 `_ms` 或 `_at`；比例字段用 `_ratio`；计数字段用 `_count`；布尔字段使用 `is_` / `has_` 前缀。不要出现不带单位的时间数值。当前内容域统一使用毫秒时间单位，这一点必须延续到 Recommendation 的内容相关代码中。

## 7. 代码规则

业务规则只写一遍。能放在 `domain/policy`、`domain/rule`、`domain/service` 的规则，不要重复出现在 usecase、repository、handler 或 test fixture 中。上层模块只依赖抽象接口，下层负责实现；对象依赖必须通过构造器或注入获得，不允许在核心业务逻辑里直接构建数据库、事务、外部客户端。这样才能保持可替换和可测试。

非平凡 SQL 不要写在 Go 字符串里。统一落到 `infrastructure/persistence/query`，通过 `sqlc` 生成访问代码。Repository interface 定义在 `application/repository`，实现写在 `infrastructure/persistence/repository`。Query 结构与领域对象之间的转换，统一放在 `infrastructure/persistence/mapper`。

不要制造包循环，不要把模块根与子模块实现混在一起，不要跨模块直接读写非 owner 表的内部细节。跨域读取只能依赖设计文档已经稳定定义的共享表、视图和字段。

## 8. 测试规则

单元测试默认验证导出行为和稳定边界；如果未导出逻辑复杂到必须单测，应优先提炼成可命名的领域对象或服务，而不是把测试绑在包内私有细节上。集成测试使用真实数据库、真实事务、真实 repository 实现。跨模块链路测试统一上收至 `internal/test/e2e`，不允许藏在某个模块自己的 `test/` 目录里。设计必须服务于可测试性：状态可控制、结果可观察、依赖可隔离。

单模块测试组织必须额外满足以下规则：

- 模块内部测试统一集中在 `internal/<module>/test/` 下，不允许新的 `*_test.go` 直接放回业务实现目录。
- 模块内部共享测试基座统一放在 `internal/<module>/test/fixture/`，不再新增散落的 `testutil/` 目录。
- `unit/` 只放不依赖真实数据库的测试；`integration/` 只放依赖真实数据库、真实事务或真实基础设施的测试。
- `golden/` 只放 golden 数据文件，不放测试代码。
- 若需要共享 embedded Postgres、template database、seed helper，应优先设计成模块级 `test/fixture` 能力，由整个模块的 integration 测试复用，而不是让每个业务包各自起库。
- 跨模块 E2E 与单模块 integration 是两层不同验证，不允许混放；`internal/test/e2e` 的职责保持不变。
- 推荐的共享模式是“模块级共享测试基座 + 测试级隔离数据”，而不是“每个测试各自起库”或“所有测试共用同一份脏数据库状态”。
- 模块内测试结构一旦迁入 `test/`，旧路径必须删除，不保留长期兼容壳。

## 9. Codex agent 执行流程

Codex agent 在新增模块、子模块或大改结构时，统一按以下顺序执行：

1. 先确认 owner：`catalog / learningengine / recommendation / internal/test/e2e`。
2. 再确认层级：这是模块根、子模块，还是现有子模块内的一个职责文件。
3. 若是承载实现的层级，按标准骨架创建当前真实需要的目录。
4. 同时补 `README.md`；建议补 `doc.go`。
5. 如涉及数据库对象，再补 `migration / query / sqlcgen / tx`。
6. 如涉及跨模块共享契约，必须同步修改对应设计文档。
7. 不允许新增非标准目录名，不允许绕过 owner 边界直接实现。

## 10. 禁止事项

不要每个模块自己发明目录命名。不要同类职责出现多个落点。不要在模块根和子模块同时维护两套完整骨架。不要为了整齐创建大量空目录。不要在没有标准说明的情况下临时堆 `utils / misc / impl / helper / manager` 目录。不要把 Recommendation 的投放状态塞进 Learning engine 或 Catalog。不要重新引入已废弃的旧内容结构。

## 11. 最短执行准则

如果 Codex agent 只能记住几条，就记住下面这些：

- 先定 owner，再写代码。
- 同一职责只允许一个标准落点。
- 承载实现的层级统一用 `application / domain / infrastructure / test`。
- SQL 进 `query/`，接口进 `application/repository/`，实现进 `infrastructure/persistence/repository/`。
- 业务规则写在 `domain`，不要散落。
- 跨模块测试只进 `internal/test/e2e`。
- 不要发明新目录名，不要越过三域边界。
