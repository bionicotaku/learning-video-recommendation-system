# 视频召回模块设计文档

## 1. 文档目标

本文档定义 Recommendation 模块中的**视频召回模块**。
该模块的职责是在 scheduler 已经选出“当前应学习的 coarse unit 目标集合”之后，从切片视频库中召回最适合作为这些学习目标承载内容的候选视频，并将召回结果交给后续 task 层进行学习任务组装。

本模块不是通用内容推荐系统，也不是全文搜索系统。
它的核心目标是：

**对于当前一批最值得学习的 coarse unit，找到最适合承载这些学习目标的切片视频集合。**

这里“最适合”不是简单指“出现过这些知识点”，而是指：

- 覆盖目标知识点
- 覆盖质量高
- 学习承载度高
- 近期不过度重复
- 同源内容不过于集中
- 对当前 session 的学习体验更合适

---

## 2. 模块定位

视频召回是 Recommendation 内部的第二层子模块，位于 scheduler 和 task 之间。

系统整体链路如下：

```text
Learning engine
    ↓
scheduler
    ↓
    输出：目标 coarse unit 列表
    ↓
video recall
    ↓
    输出：候选视频列表
    ↓
task
    ↓
    输出：最终学习任务列表
```

这一定义意味着三个边界必须清楚。

第一，scheduler 决定“现在该学什么”。
第二，recall 决定“这些学习目标适合由哪些视频承载”。
第三，task 决定“这些视频如何被组织成最终学习任务”。

因此 recall 不应回头修改 scheduler 的目标决策，也不应越权直接输出最终 UI 任务对象。

---

## 3. 与当前入库设计的对齐前提

为了避免设计打架，这里明确列出本模块与当前视频入库模块共享的前提。

### 3.1 视频主对象是切片

`catalog.videos.video_id` 代表的就是最终切片视频对象。
recall 的选择对象始终是 `video_id`，而不是原始视频，也不是“视频下面的 segment”。

### 3.2 原始视频只作为来源标识存在

原始视频不入库，不作为实体，不作为父表存在。
`parent_video_name` 和 `parent_video_slug` 仅用于：

- 来源追踪
- 同源软去重
- 多样性控制
- 调试解释

### 3.3 主索引入口不是 transcript 全文，而是 `video_unit_index`

recall 的主读路径应是：

`target coarse_unit_ids -> catalog.video_unit_index -> candidate video_ids`

而不是：

`target words -> 扫 transcript_text`

全文 transcript 主要用于解释、展示和辅助诊断，不是主召回入口。

### 3.4 cue 与 mention 用于证据，不用于主对象建模

系统中已存在：

- `catalog.video_transcript_cues`
- `catalog.video_unit_mentions`

这些表的价值在于：

- 定位知识点出现位置
- 生成解释
- 计算覆盖密度与证据质量

它们不是新的推荐对象层。

### 3.5 recall 只写 Recommendation 自己的表

recall 不回写：

- `catalog.videos`
- `catalog.video_transcripts`
- `catalog.video_transcript_cues`
- `catalog.video_unit_index`
- `catalog.video_unit_mentions`

这些内容索引表由内容分析与入库流水线维护。
recall 只写自己的审计表，以及 Recommendation 持有的 serving 状态表。

---

## 4. 模块职责与非职责

## 4.1 recall 负责什么

视频召回模块负责以下事情：

接收 scheduler 输出的目标 coarse unit 列表。
从内容索引中扩展候选视频池。
对候选视频做资格过滤、视频级聚合、评分、排序与多样性选择。
输出一组带解释信息的视频候选结果。
维护 recall 运行审计记录。

## 4.2 recall 不负责什么

视频召回模块不负责：

决定学什么，这是 scheduler 的职责。
维护学习状态，这是 Learning engine 的职责。
写内容索引表，这是入库与内容分析流水线的职责。
构造最终学习任务，这是 task 层的职责。
直接对外暴露独立 API，Recommendation 作为整体模块统一对外。
修改用户真实观看状态，真实行为由下游曝光与播放链路维护。

---

## 5. 设计目标

本模块的设计目标有五个。

第一，**目标一致性**。
recall 的核心排序逻辑必须与 scheduler 的目标优先级保持一致，不能丢掉 coarse unit 的重要性权重。

第二，**学习适配性优先**。
recall 的判断标准必须是“这个视频是否适合承载当前学习目标”，而不是“它是否简单包含这些知识点”。

第三，**结果去冗余**。
不能简单按分数取 top-N，否则极容易出现多个视频反复覆盖同一批目标 unit，学习收益冗余。

第四，**过程可解释**。
必须能说明某个视频为什么被选中，覆盖了哪些目标，证据在哪里，被哪些质量信号加分或降权。

第五，**工程可落地**。
应采用规则驱动和结构化索引驱动的方案，避免在 MVP 阶段引入黑盒排序模型。

---

## 6. 核心读路径

视频召回模块的数据库设计和实现流程，应从以下读路径倒推。

### 6.1 给定目标 coarse unit 集合，找到覆盖这些目标的视频

这是 recall 的起点。
系统需要快速回答：

- 哪些视频覆盖这些目标 coarse unit
- 覆盖了几个
- 覆盖质量如何

### 6.2 给定某个视频和某个目标 unit，拿到覆盖详情和质量特征

系统需要知道：

- 这个 unit 在视频里出现了几次
- 它是 primary 还是 supporting
- 它的语境等级如何
- `salience_score / teachability_score / confidence_score` 如何
- evidence cue 是哪些

### 6.3 给定候选视频集合，读取用户状态与推荐曝光状态

系统需要区分：

- 用户是否真正看过
- 最近看过没有
- 看过多少次
- 系统最近是否刚推荐过
- 近 7 天是否已经反复推过

### 6.4 给定候选视频，做来源与覆盖冗余控制

系统需要知道：

- 这些视频是否来自同一 `parent_video_slug`
- 某些高权重目标是否已经被前面候选充分覆盖
- 后续视频的边际价值是否正在下降

---

## 7. 输入与输出

## 7.1 模块输入

视频召回模块直接消费 scheduler 的输出。
不要求 scheduler 改接口，但在 recall 语义里，会把 scheduler 输出的 `Score` 理解为目标权重。

建议沿用如下输入结构：

```go
type SchedulerBatchItem struct {
    CoarseUnitID   int64
    RecommendType  string   // "review" | "new"
    Score          float64  // recall 中将其解释为 target_weight
    Rank           int
    ReasonCodes    []string
}
```

recall 额外需要如下上下文：

```go
type RecallRequest struct {
    UserID              uuid.UUID
    SchedulerRunID      uuid.UUID
    SchedulerBatch      []SchedulerBatchItem

    // 可选 session 约束
    ExpectedTotalDurationMs *int64

    // 可选参数覆盖
    Params *RecallParams
}
```

其中 `SchedulerBatch` 是最核心输入。
它定义了当前这次推荐真正要解决的学习目标集合。

---

## 7.2 模块输出

视频召回模块输出的是**视频级候选结果**，而不是最终任务对象。

```go
type RecallResult struct {
    RunID        uuid.UUID
    GeneratedAt  time.Time
    Videos       []RecallVideoItem
}

type RecallVideoItem struct {
    VideoID        uuid.UUID
    RecallScore    float64
    Rank           int

    CoveredUnits   []RecallCoveredUnit
    ReasonCodes    []string
}

type RecallCoveredUnit struct {
    CoarseUnitID       int64
    RecommendType      string   // review | new
    SchedulerScore     float64  // 继承自 scheduler
    Role               string   // primary | supporting | incidental
    ContextRelevance   string   // primary | contextual | normal | incidental

    EvidenceCueNos     []int
    MentionCount       int
    CoverageMicros     int64
}
```

这里有一个关键变化：

**不再使用 `SegmentIndexes`。**

因为在当前数据模型里，没有 `video_segments` 作为主事实对象。
更自然、更一致的证据表达方式是：

- `EvidenceCueNos`
- `MentionCount`
- `CoverageMicros`

## 8. 数据依赖

## 8.1 只读表

### `catalog.videos`

用于读取视频资产元数据与可用性状态。
recall 只应从满足以下条件的视频中选择候选：

- `status = 'active'`
- `media_status = 'ready'`
- `analysis_status = 'succeeded'`
- `deleted_at is null`

并读取如下字段：

- `video_id`
- `duration_micros`
- `title`
- `thumbnail_url`
- `parent_video_name`
- `parent_video_slug`
- `language`

### `catalog.video_unit_index`

这是 recall 的核心索引入口。
它提供：

- `video_id`
- `coarse_unit_id`
- `match_type`
- `role`
- `context_relevance`
- `mention_count`
- `coverage_micros`
- `coverage_ratio`
- `salience_score`
- `teachability_score`
- `confidence_score`
- `evidence_text`
- `evidence_cue_nos`

### `catalog.video_unit_mentions`

用于在需要时回溯更细的 evidence。
MVP 阶段可不作为主评分输入，但应可用于解释和调试。

### `catalog.video_transcript_cues`

用于 cue 级证据定位和后续任务层展示。

### `catalog.video_unit_coverage`

如果前面已经从 `video_unit_index` 派生出聚合视图，则这张视图可以用作**粗筛入口**。
但 recall 的主事实来源仍然应是 `video_unit_index`，因为它保留了更多质量细节。

### `catalog.video_user_states`

用于读取用户的真实消费状态。
如果已扩展字段，应优先读取：

- `has_watched`
- `last_watched_at`
- `watch_count`
- `completed_count`
- `avg_watch_ratio`
- `last_watch_ratio`

### `recommendation.user_video_serving_states`

用于读取推荐系统侧的曝光状态。
这是和 `video_user_states` 必须区分的另一层状态：

- `last_served_at`
- `serve_count_7d`
- `serve_count_total`

## 8.2 可写表

### `recommendation.recall_runs`

记录一次完整 recall 运行。

### `recommendation.recall_run_targets`

记录本次 recall 的输入目标。

### `recommendation.recall_run_candidates`

记录进入候选池并完成评分的视频。

### `recommendation.recall_run_candidate_units`

记录候选视频在本次 recall 中对每个目标 unit 的贡献分解。

如果未来需要记录最终曝光状态，则由下游 serving 模块或 exposure log 维护，不属于 recall 主流程本身。

## 9. 总体流程

视频召回模块建议拆为七个步骤。

```text
GenerateVideoRecall(userID, schedulerBatch)
    ↓
Step 1: 目标标准化
    ↓
Step 2: 候选扩展
    ↓
Step 3: 资格过滤
    ↓
Step 4: 视频级聚合
    ↓
Step 5: 视频评分
    ↓
Step 6: 贪心多样性选择
    ↓
Step 7: 结果组装与审计写入
```

下面逐步展开。

---

## 10. Step 1：目标标准化

目标标准化阶段负责把 scheduler 输出转成 recall 可直接消费的目标集合。

需要做的事情包括：

- 去重同一 `coarse_unit_id`
- 继承 `RecommendType` 和 `Score`
- 按 `Score` 和 `Rank` 确定目标优先级
- 校验 `coarse_unit_id` 合法性
- 计算 `total_target_score`

在 recall 语义里，可以把每个目标建模成：

```go
type RecallTarget struct {
    CoarseUnitID   int64
    RecommendType  string   // review | new
    TargetWeight   float64  // 来自 scheduler score
    TargetRank     int
    ReasonCodes    []string
}
```

这里的 `TargetWeight` 是 recall 最核心的上游信号。
它确保 recall 不会把所有目标当成同等重要。

---

## 11. Step 2：候选扩展

候选扩展的目标是：
从 `catalog.video_unit_index` 中找出所有覆盖任意目标 unit 的视频，并形成第一批候选命中行。

### 11.1 主策略

粗筛阶段可以优先从 `catalog.video_unit_coverage` 或直接从 `catalog.video_unit_index` 读取。
推荐策略是：

- 如果 `video_unit_coverage` 已存在且字段足够，可作为 first-pass 粗筛
- 精排阶段必须回到 `video_unit_index` 读取细粒度质量信号

也就是说：

**粗筛可以用聚合视图，主评分不能只靠聚合视图。**

### 11.2 候选扩展门槛

候选扩展阶段建议加入最低质量门槛，避免把低质量命中行全部拉入内存。

默认建议：

- `confidence_score >= min_confidence`
- `teachability_score >= min_teachability`
- `salience_score >= min_salience`，或者允许较低但只作为后备
- `role != incidental`，除非显式允许 incidental 进入候选池
- 视频状态合法

### 11.3 候选上限

为了控制规模，每个目标 unit 建议设置候选扩展上限：

- `max_candidates_per_unit`

MVP 默认可以是 30 到 50 之间。
这不是最终选出数量，而是 unit 级初筛规模。

### 11.4 候选扩展 SQL 思路

粗筛 SQL 可以类似这样：

```sql
select
  vui.video_id,
  vui.coarse_unit_id,
  vui.role,
  vui.context_relevance,
  vui.mention_count,
  vui.coverage_micros,
  vui.coverage_ratio,
  vui.salience_score,
  vui.teachability_score,
  vui.confidence_score,
  vui.evidence_cue_nos
from catalog.video_unit_index vui
join catalog.videos v on v.video_id = vui.video_id
where vui.coarse_unit_id = any(@target_unit_ids::bigint[])
  and v.status = 'active'
  and v.media_status = 'ready'
  and v.analysis_status = 'succeeded'
  and v.deleted_at is null
  and vui.confidence_score >= @min_confidence
  and vui.teachability_score >= @min_teachability
  and (@allow_incidental = true or vui.role <> 'incidental');
```

这一步的输出不是视频列表，而是**target-unit 到 video 的命中行集合**。

---

## 12. Step 3：资格过滤

候选扩展完成后，需要在视频级做资格过滤。

### 12.1 必须过滤的条件

视频不可用，直接排除。
如果语言不匹配，可排除。
如果时长远超模块允许范围，可排除。
如果 transcript 或分析状态不完整，也应排除。

建议的时长过滤默认值：

- `min_duration_micros = 30s`
- `max_duration_micros = 600s`

### 12.2 用户状态相关过滤

用户状态不建议一刀切。
更合适的策略是：

- 对明显不适合的新内容重复观看场景，可考虑排除最近刚完整看过的视频
- 对 review 场景，允许保留看过的视频，但应降权
- 对系统近期刚推过但用户未消费的视频，不直接过滤，而是加入 exposure penalty

也就是说：

**“是否看过”更像评分和惩罚信号，而不是简单布尔过滤条件。**

只有极端情况下，才启用硬过滤。

---

## 13. Step 4：视频级聚合

这一阶段是整个 recall 最关键的步骤之一。

前面拿到的是“目标 unit 命中了哪些视频”的明细行。
但 recall 真正的排序对象是视频，所以必须把这些命中行聚合成视频级候选对象。

### 13.1 聚合后的核心结构

```go
type AggregatedVideoCandidate struct {
    VideoID                 uuid.UUID
    ParentVideoSlug         string
    DurationMicros          int64

    MatchedUnitIDs          []int64
    CoverageCount           int
    WeightedCoverageRaw     float64

    ReviewCoveredCount      int
    NewCoveredCount         int

    PrimaryMatchCount       int
    SupportingMatchCount    int
    IncidentalMatchCount    int

    WeightedRoleScore       float64
    WeightedContextScore    float64
    WeightedSalienceScore   float64
    WeightedTeachability    float64
    WeightedConfidence      float64

    CoveredDurationMicros   int64

    HasWatched              bool
    LastWatchedAt           *time.Time
    WatchCount              int

    LastServedAt            *time.Time
    ServeCount7d            int

    CoveredUnits            []AggregatedCoveredUnit
}
```

### 13.2 聚合原则

同一个 `video_id` 下，命中的多个目标 unit 应当统一聚合。
聚合时不能简单算平均值，而应按 `target_weight` 做加权。

举例：

- 一个视频覆盖 1 个高权重 review unit 和 2 个低权重新词
- 另一个视频覆盖 3 个低权重新词

这两者在简单 count 上很接近，但在聚合权重后，应明显偏向前者。

### 13.3 每个目标 unit 的贡献信息

建议保留 unit 级贡献明细，用于后续审计与解释：

```go
type AggregatedCoveredUnit struct {
    CoarseUnitID        int64
    RecommendType       string
    SchedulerScore      float64

    Role                string
    ContextRelevance    string

    MentionCount        int
    CoverageMicros      int64

    SalienceScore       float64
    TeachabilityScore   float64
    ConfidenceScore     float64

    EvidenceCueNos      []int
}
```

## 14. Step 5：视频评分

视频评分的目标不是算出“用户喜不喜欢”，而是算出：

**该视频作为当前目标学习内容承载体的综合价值。**

### 14.1 评分设计原则

评分应以**加权覆盖**为主，以**学习质量**为核心，以**体验和重复控制**为辅助。
MVP 阶段应使用确定性规则，而不是黑盒排序模型。

---

## 14.2 核心信号一：加权覆盖分

这是最核心的信号。

### 定义

```text
weighted_coverage_score =
  sum(该视频覆盖的目标 unit 的 scheduler_score)
  / sum(所有目标 unit 的 scheduler_score)
```

范围为 `[0, 1]`。

### 含义

它直接回答：

**这个视频承载了当前这批学习目标中，多少真正重要的部分。**

这比单纯 `covered_count / total_targets` 更合理，因为它保留了 scheduler 的优先级。

---

## 14.3 核心信号二：学习质量分

这部分是当前 schema 相比旧版设计最大的优势。
因为你已经在 `video_unit_index` 中有了以下字段：

- `role`
- `context_relevance`
- `salience_score`
- `teachability_score`
- `confidence_score`

这些字段应该真正进入 recall 主评分，而不是只用于解释。

### 角色映射

```text
role_score:
  primary    -> 1.0
  supporting -> 0.7
  incidental -> 0.25
```

### 语境映射

```text
context_score:
  primary     -> 1.0
  contextual  -> 0.8
  normal      -> 0.6
  incidental  -> 0.3
```

### 单个 unit 的质量分

```text
unit_quality_score =
  0.20 * role_score
+ 0.20 * context_score
+ 0.20 * salience_score
+ 0.25 * teachability_score
+ 0.15 * confidence_score
```

然后再按该 unit 的 `scheduler_score` 做加权平均，得到视频级质量分：

```text
quality_score =
  weighted_avg(unit_quality_score, scheduler_score)
```

### 解释

这里特意提高了 `teachability_score` 权重。
因为 recall 不只是判断“出现质量”，而是判断“这个视频适不适合学”。

---

## 14.4 核心信号三：覆盖密度分

旧版方案里使用 segment 时长定义密度，这里要改成与当前入库设计一致的方式。

### 定义

```text
density_ratio = covered_duration_micros / duration_micros
```

其中 `covered_duration_micros` 来自本视频命中的目标 unit 对应的 `coverage_micros` 聚合结果。
为了避免多 unit 覆盖时重复叠加过高，可以采用：

- 上限裁剪
- 或基于时间区间合并后的实际覆盖时长

MVP 阶段可先采用近似版：

```text
covered_duration_micros = min(video.duration_micros, sum(unit.coverage_micros))
```

### 评分映射

```text
density_score:
  density_ratio >= 0.50 -> 1.0
  density_ratio >= 0.30 -> 0.8
  density_ratio >= 0.15 -> 0.5
  else                  -> 0.2
```

### 含义

密度太低，说明用户大部分时间都在看与目标学习内容无关的部分。
学习承载价值就会下降。

---

## 14.5 核心信号四：时长适配分

视频时长会直接影响学习负担。

### 默认理想时长

MVP 阶段可设：

- `ideal_duration_micros = 180s`

### 评分映射

```text
deviation = abs(duration - ideal_duration) / ideal_duration

duration_fit_score:
  deviation <= 0.3 -> 1.0
  deviation <= 0.6 -> 0.7
  deviation <= 1.0 -> 0.4
  else             -> 0.2
```

### 含义

太短，可能语境不足。
太长，学习负担过大。
3 分钟左右通常是比较平衡的学习片段长度。

---

## 14.6 核心信号五：review 优先分

这部分与 scheduler 的“先保 review，再补 new”策略对齐。

### 定义

```text
if total_review_targets > 0:
  review_priority_score =
    review_covered_weight / total_review_target_weight
else:
  review_priority_score = 0
```

这里推荐使用 **review target 的权重覆盖比例**，而不是只算数量。

### 含义

如果一个视频恰好覆盖了多个高优先级复习目标，它的召回价值应明显提升。

---

## 14.7 新鲜度与重复惩罚

这部分是旧版参考方案里比较弱的地方，这一版需要补强。

### 1. recently_watched_penalty

如果用户刚完整看过该视频，应降权。
但对 review 场景，不建议一刀切排除。

### 2. recently_served_penalty

如果系统近期多次把该视频推给用户但用户没有明显消费，应明显降权。

### 3. same_source_penalty

如果某个候选与已经选中的视频来自同一 `parent_video_slug`，则在多样性阶段给予惩罚。

### 4. duplicate_coverage_penalty

如果某个候选视频主要覆盖的都是已经被前面视频很好覆盖过的目标 unit，则其边际价值应下降。

---

## 14.8 最终评分公式

推荐的 MVP 初始公式如下：

```text
base_recall_score =
    0.40 * weighted_coverage_score
  + 0.30 * quality_score
  + 0.10 * review_priority_score
  + 0.10 * density_score
  + 0.10 * duration_fit_score
```

这是**多样性选择前的基础分**。

说明如下：

- 加权覆盖 40%，仍然是第一主信号
- 学习质量 30%，这是当前 schema 最大的价值所在
- review 优先 10%，与 scheduler 对齐
- 密度与时长各 10%，作为体验层优化

真正最终用于选择的视频边际分，应在此基础上再减去惩罚项：

```text
marginal_score =
  base_recall_score
  - recently_watched_penalty
  - recently_served_penalty
  - same_source_penalty
  - duplicate_coverage_penalty
```

---

## 15. reason_codes 设计

每个候选视频都应生成 reason codes，用于解释和调试。

推荐保留以下语义：

```text
high_weighted_coverage
high_quality_match
primary_context
review_heavy
good_density
ideal_duration
recently_watched_but_allowed
recently_served
same_source_penalty
duplicate_coverage_penalty
low_density
low_quality_match
```

注意这里的 reason codes 应是**解释信号**，不是主逻辑本身。
主逻辑仍由数值评分控制。

---

## 16. Step 6：贪心多样性选择

不能简单按 `base_recall_score` 取 top-N。
因为多个高分视频可能覆盖同一批目标 unit，或者来自同一来源视频，结果会很冗余。

### 16.1 目标

贪心选择阶段的目标是：

- 尽可能覆盖更多高权重目标
- 避免重复满足同一个目标
- 避免同源视频过度集中
- 在保证质量前提下，控制重复曝光

### 16.2 基本算法

维护：

- `uncovered_units`
- `selected_videos`
- `selected_source_counts`

每轮从剩余候选中计算边际分：

```text
marginal_score =
  candidate.base_recall_score
  * marginal_coverage_ratio
  - same_source_penalty
  - recently_served_penalty
  - duplicate_coverage_penalty
```

其中：

```text
marginal_coverage_ratio =
  sum(新增覆盖的 unit 的 scheduler_score)
  / total_target_score
```

### 16.3 贪心流程

1. 初始化 `uncovered_units = all target units`
2. 初始化 `selected_videos = []`
3. 按 `base_recall_score` 得到初始候选集
4. 循环直到：
   - `selected_videos` 达到上限，或
   - 没有候选能带来有效新增覆盖

5. 每轮计算所有剩余候选的 `marginal_score`
6. 选择边际价值最高的视频
7. 更新 `uncovered_units`
8. 更新 `selected_source_counts`
9. 继续下一轮

### 16.4 默认上限

MVP 默认可以设：

- `max_videos = 5`

如果 5 个视频仍未覆盖全部目标，不是问题。
不要为了凑覆盖率而明显降低质量门槛。

---

## 17. Step 7：结果组装与审计写入

完成选择后，需要做两件事：

第一，组装 recall 输出结果。
第二，把全过程写入 Recommendation 审计表。

### 17.1 结果组装

每个选中视频需要带出：

- `VideoID`
- `RecallScore`
- `Rank`
- `CoveredUnits`
- `ReasonCodes`

如果后续 Recommendation 整体结果需要更完整的视频字段，也可以在上层再 join：

- `title`
- `thumbnail_url`
- `duration_micros`

### 17.2 审计写入原则

审计至少应能回答：

- 这次 recall 的目标是什么
- 初始候选有哪些
- 为什么某些候选被选中
- 为什么某些候选没被选中
- 每个候选覆盖了哪些目标
- 每个目标对候选贡献了多少分

## 19. 与 scheduler 的串联

### 19.1 调用关系

当前 Recommendation 的内部流程可以定义为：

```text
GenerateRecommendations(userID, sessionContext)
    ↓
[scheduler] -> SchedulerBatch
    ↓
[recall] -> RecallResult
    ↓
[task] -> TaskResult
```

对外入口不必拆成多个 API。
Recommendation 模块内部串联 scheduler 和 recall 即可。

### 19.2 scheduler 输出不需要修改

recall 直接消费 scheduler 现有输出结构。
这点应保留，以降低模块耦合成本。

### 19.3 Recommendation 的中间结果应保留两层信息

即使 recall 已经输出视频结果，Recommendation 内部仍建议保留：

- scheduler batch
- recall result

因为 task 层可能仍需要学习内容级信息，而不只是视频级结果。

## 20. 边界情况处理

### 20.1 没有视频覆盖任何目标 unit

处理方式：

- recall 正常返回空列表
- `recall_runs.warning_codes` 记录 `no_candidate_found`
- Recommendation 可选择 fallback，只返回 scheduler batch 给后续层

### 20.2 候选视频都已看过

处理方式：

- 若存在 review 目标，允许保留看过的视频，但施加 `recently_watched_penalty`
- 若只有 new 目标，且候选全部刚看过，可返回空列表

### 20.3 覆盖率很低

即使选完所有合理候选，整体覆盖率仍然很低，这也是正常情况。

处理方式：

- 正常返回已选视频
- 写入 `low_coverage_warning`
- 不为了凑覆盖率而显著降低质量门槛

### 20.4 同一 unit 在同一视频中多次出现

这是正常且通常有利的信号。

处理方式：

- 覆盖数计算时按 unit 去重
- `mention_count` 和 `coverage_micros` 用于提高质量分和密度分
- evidence cue 记录所有关键位置

### 20.5 同源视频过多

如果高分候选主要来自同一 `parent_video_slug`：

- 不做硬排除
- 在贪心选择阶段加软惩罚
- 若最终仍不可避免同源集中，也允许返回，但保留 warning code

## 21. 性能设计

## 21.1 主读路径索引支撑

关键查询主要依赖以下索引：

- `catalog.video_unit_index(coarse_unit_id, role, salience_score desc, teachability_score desc)`
- `catalog.video_unit_index(coarse_unit_id, confidence_score desc)`
- `catalog.video_unit_index(video_id)`
- `catalog.video_transcript_cues(video_id, start_micros)`
- `catalog.video_user_states(user_id, video_id)`
- `recommendation.user_video_serving_states(user_id, video_id)`

## 21.2 MVP 性能判断

在当前预期规模下：

- 目标 unit 数通常在 10 到 20 左右
- 候选扩展规模通常在几十到几百条命中行
- 聚合和打分完全可以在应用内存中完成

因此：

- 不需要 embedding 检索
- 不需要复杂缓存
- 不需要黑盒排序服务
- 不需要分布式召回架构

只要读路径设计正确，Postgres 足以支撑 MVP。

## 21.3 后续扩展方向

如果后续规模明显增长，可以考虑：

- 将 `video_unit_coverage` 物化
- 对高频 recall 请求做缓存
- 将部分聚合前移到离线计算
- 引入更细粒度的用户偏好特征

但这些都不属于当前版本的必要能力。

---

## 22. MVP 明确不做的事

为了控制范围，以下能力在 MVP 阶段明确不做：

不做 embedding 语义召回。
不做复杂视频相似度去重。
不做个性化视频偏好模型。
不做 recall 层用户级配置界面。
不做 A/B 实验框架。
不做片段级推荐对象，只推荐整个切片视频。
不做黑盒学习排序模型。

当前 recall 的核心依然是：

**结构化索引 + 规则打分 + 贪心边际选择。**

---

## 23. 实现顺序建议

### 第一阶段：基础召回读链路

先打通：

- scheduler 输出读取
- `video_unit_index` 候选扩展
- `catalog.videos` 状态过滤
- 视频级聚合

这一阶段先不追求复杂评分。

### 第二阶段：评分模型

加入：

- `weighted_coverage_score`
- `quality_score`
- `density_score`
- `duration_fit_score`
- `review_priority_score`

先完成 `base_recall_score`。

### 第三阶段：多样性选择

实现：

- `duplicate_coverage_penalty`
- `same_source_penalty`
- `recently_served_penalty`
- greedy diversification

### 第四阶段：审计闭环

补全：

- `recall_runs`
- `recall_run_targets`
- `recall_run_candidates`
- `recall_run_candidate_units`

确保结果可追溯。

### 第五阶段：串联 Recommendation 输出

将 recall 正式串进 Recommendation 的内部流程，形成：

- scheduler batch
- recall result
- 后续 task 输入

---

## 24. 最终结论

这一版视频召回模块设计可以概括为一句话：

**以 scheduler 产出的 coarse unit 目标集合作为输入，以 `catalog.video_unit_index` 为核心倒排索引入口，以 `catalog.videos.video_id` 作为唯一召回对象，经过候选扩展、资格过滤、视频级聚合、规则化质量评分与贪心多样性选择，输出一组适合承载当前学习目标的切片视频候选，并将全过程写入 Recommendation 自己的审计表。**

它与当前视频入库模块完全一致，体现在几个关键点上：

- 都坚持“切片即视频主对象”
- 都不引入 `video_segments` 作为核心对象层
- 都把 transcript 与 cue 视为 side data，而不是新主实体
- 都围绕 `coarse_unit -> video` 的显式关系建模
- 都强调审计、幂等、可解释、工程可落地

如果你愿意，下一步最自然的是把这份文档继续落成两份更工程化的材料：
第一份是 **Postgres / Supabase DDL 草案**，第二份是 **Go 接口与实现任务拆解文档**。
