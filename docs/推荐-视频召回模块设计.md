# 视频召回模块设计文档

## 1. 文档目标

本文档定义 Recommendation 模块中的**视频召回模块**。

该模块的职责是：在 scheduler 已经选出“当前应学习的 coarse unit 目标集合”之后，从 Catalog 中召回最适合作为这些学习目标承载内容的切片视频候选，并把结果交给后续 task 层进一步组装成最终学习任务。

这个模块不是通用内容推荐系统，也不是全文搜索系统。
它的核心问题不是：

“用户可能喜欢看什么视频？”

而是：

**“针对当前这批最重要的学习目标，哪些切片视频最适合拿来承载这些学习内容？”**

因此，本模块的设计原则始终是：

- 以学习目标为中心
- 以 `coarse_unit_id -> video_id` 的结构化索引为核心入口
- 以确定性规则和可解释逻辑完成候选生成与排序
- 在 MVP 阶段避免依赖黑盒模型

---

## 2. 模块定位

视频召回模块位于 Recommendation 内部，处于 scheduler 与 task 之间。

整体链路如下：

```text
Learning engine
    ↓
scheduler
    ↓
    输出：目标 coarse unit 列表
    ↓
video recall
    ↓
    输出：候选视频列表
    ↓
task
    ↓
    输出：最终学习任务列表
```

这里有三个边界必须严格成立。

第一，scheduler 决定“现在应该学什么”。
第二，recall 决定“这些学习目标适合由哪些视频来承载”。
第三，task 决定“这些视频和学习目标如何被组织成最终任务”。

因此 recall 不应：

- 反过来修改 scheduler 的目标结果
- 越权构造最终任务对象
- 写回 Catalog 内容索引层

---

## 3. 与当前 Catalog 设计的对齐前提

为了避免后续设计打架，这里先把 Recall 所依赖的 Catalog 前提明确下来。

### 3.1 切片视频是唯一召回对象

`catalog.videos.video_id` 本身就是切片视频对象。
Recall 的候选、评分、排序、去重、多样性控制，全部围绕 `video_id` 展开。

当前不再存在：

- 原始视频作为推荐对象
- 视频下再拆一层 `video_segments`
- segment 级单独推荐对象

### 3.2 原始视频只保留来源信息

原始长视频不入库，不是实体表。
`parent_video_name` 和 `parent_video_slug` 仅表示来源，用于：

- 来源追踪
- 同源软去重
- 多样性控制
- 调试解释

### 3.3 Recall 主入口是 `video_unit_index`

Recall 的主读路径是：

```text
target coarse_unit_ids
    ↓
catalog.video_unit_index
    ↓
candidate video_ids
```

而不是：

- 扫 transcript 全文
- 扫 sentence 明细
- 扫 semantic span 明细

后两者主要用于解释与证据回溯，不是主召回入口。

### 3.4 当前 Catalog 的高层语义特征有限

当前 `catalog.video_unit_index` 中，真正稳定存在、可直接依赖的字段是：

- `mention_count`
- `sentence_count`
- `first_start_ms`
- `last_end_ms`
- `coverage_ms`
- `coverage_ratio`
- `sentence_indexes`
- `evidence_sentence_indexes`
- `evidence_span_indexes`
- `sample_surface_forms`

当前并**不存在**这些旧版假设中的高层字段：

- `role`
- `context_relevance`
- `salience_score`
- `teachability_score`
- `confidence_score`

因此 Recall 本版评分必须建立在当前真实存在的数据上，而不是假定 Catalog 已经完成高层语义 enrichment。

### 3.5 transcript 相关表的用途

当前 transcript 读模型是：

- `catalog.video_transcripts`
- `catalog.video_transcript_sentences`
- `catalog.video_semantic_spans`

它们的用途分别是：

- transcript 摘要
- 句子级时间轴
- span 级语义事实

它们主要服务于：

- 证据解释
- 任务层展示
- 必要时的细粒度回溯

不作为 Recall 第一阶段主入口。

### 3.6 Recall 只写 Recommendation 自己的表

Recall 不回写：

- `catalog.videos`
- `catalog.video_transcripts`
- `catalog.video_transcript_sentences`
- `catalog.video_semantic_spans`
- `catalog.video_unit_index`
- `catalog.video_user_states`

这些都属于 Catalog 的只读内容与投影层。
Recall 只写：

- Recommendation 自己的运行审计表
- Recommendation 自己的 serving 状态表

---

## 4. 模块职责与非职责

## 4.1 Recall 负责什么

视频召回模块负责以下事情：

接收 scheduler 输出的 coarse unit 目标集合。
从 `catalog.video_unit_index` 扩展候选视频池。
读取候选视频的内容资产、用户观看状态、推荐曝光状态。
在视频级进行聚合、评分、排序和多样性选择。
输出一组带解释信息的视频候选结果。
把本次召回的全过程写入 Recommendation 审计表。

## 4.2 Recall 不负责什么

Recall 不负责以下事情：

决定学什么，这是 scheduler 的职责。
维护学习状态，这是 Learning engine 的职责。
构造最终学习任务，这是 task 层的职责。
修改 Catalog 内容索引。
修改用户真实观看状态。
直接对外暴露独立 API。

---

## 5. 设计目标

本模块的设计目标有六个。

### 5.1 目标一致性

Recall 的排序逻辑必须继承 scheduler 的目标优先级，而不是把所有目标 coarse unit 当成等权。

### 5.2 学习承载优先

Recall 的判断标准不是“是否提到过这个知识点”，而是“这个视频是否足够适合承载当前学习目标”。

### 5.3 结果去冗余

不能简单按分数取 top-N，否则多个视频可能重复覆盖同一批目标，学习收益会非常冗余。

### 5.4 可解释

必须能说明：

- 这个视频为什么被召回
- 它覆盖了哪些目标
- 每个目标的贡献是什么
- 证据位置在哪里
- 哪些惩罚影响了它的排序

### 5.5 可落地

应尽量基于：

- `catalog.video_unit_index`
- 规则打分
- 简单贪心选择
- PostgreSQL 查询 + 应用层聚合

而不是复杂模型。

### 5.6 与现有 Catalog 强一致

Recall 的字段、流程、评分，必须和当前 Catalog 的真实表结构一致，不能再依赖旧版假设。

### 6.2 当前没有的数据

当前 Catalog 中没有以下高层字段，因此 Recall 本版不应假装它们已经存在：

- `role`
- `context_relevance`
- `salience_score`
- `teachability_score`
- `confidence_score`

### 6.3 这意味着什么

这意味着当前 Recall 的“质量判断”必须建立在更朴素但更真实的信号上，例如：

- 是否覆盖目标
- 覆盖了多少高权重目标
- 出现频次够不够
- 涉及句子数够不够
- 时间覆盖比例够不够
- transcript 整体映射密度如何
- 视频时长是否合适
- 用户最近是否已经消费过
- 系统最近是否已经重复推送过
- 是否与已选结果来自同一来源

这比硬造出不存在的 `teachability_score` 更稳。

## 7. 核心读路径

视频召回模块应围绕以下读路径设计。

### 7.1 目标 coarse unit → 候选视频

给定一批目标 `coarse_unit_id`，从 `catalog.video_unit_index` 中找到覆盖这些目标的视频。

### 7.2 候选视频 → 视频级覆盖特征

对于某个候选视频，需要知道它对每个目标 unit 的覆盖细节：

- `mention_count`
- `sentence_count`
- `coverage_ms`
- `coverage_ratio`
- 证据句子
- 证据 span

### 7.3 候选视频 → 视频内容属性

对于评分和任务体验，需要知道：

- `duration_ms`
- `language`
- `parent_video_slug`
- `thumbnail_url`
- `title`

### 7.4 候选视频 → 用户互动与系统曝光状态

对于惩罚项，需要知道：

- 最近看过没有
- 看过几次
- 最近完整看过没有
- 最近推过没有
- 近 7 天推过几次

### 7.5 候选视频集合 → 多样性与边际价值

对于最终选择，需要知道：

- 哪些目标已经被前面候选覆盖
- 同一来源是否已经出现太多
- 后续候选还能带来多少新增收益

## 8. 输入与输出定义

## 8.1 输入

Recall 直接消费 scheduler 输出，不要求 scheduler 改结构。

### Scheduler 输出结构

```go
type SchedulerBatchItem struct {
    CoarseUnitID   int64
    RecommendType  string   // "review" | "new"
    Score          float64  // recall 中将其解释为 target_weight
    Rank           int
    ReasonCodes    []string
}
```

### Recall 请求结构

```go
type RecallRequest struct {
    UserID                 uuid.UUID
    SchedulerRunID         uuid.UUID
    SchedulerBatch         []SchedulerBatchItem

    // 可选 session 约束
    ExpectedTotalDurationMs *int64

    // 可选参数覆盖
    Params *RecallParams
}
```

### Recall 目标内部结构

```go
type RecallTarget struct {
    CoarseUnitID   int64
    RecommendType  string   // review | new
    TargetWeight   float64  // 来自 scheduler score
    TargetRank     int
    ReasonCodes    []string
}
```

这里的 `TargetWeight` 是 Recall 最重要的上游输入。

## 8.2 输出

Recall 输出的是视频级候选结果，不是最终任务对象。

type RecallResult struct {

## 10. 总体流程

Recall 模块建议拆为七步：

```text
GenerateVideoRecall(userID, schedulerBatch)
    ↓
Step 1: 目标标准化
    ↓
Step 2: 候选扩展
    ↓
Step 3: 资格过滤
    ↓
Step 4: 视频级聚合
    ↓
Step 5: 视频评分
    ↓
Step 6: 贪心多样性选择
    ↓
Step 7: 结果组装与审计写入
```

---

## 11. Step 1：目标标准化

目标标准化负责把 scheduler 输出清洗为 Recall 内部目标集合。

### 11.1 处理内容

- 去重同一 `coarse_unit_id`
- 继承 `RecommendType`
- 将 `Score` 解释为 `TargetWeight`
- 按 `TargetWeight` 和 `Rank` 保留优先级
- 计算本次 recall 的 `total_target_score`
- 按 review / new 分组统计目标数量与总权重

### 11.2 为什么必须做这一步

因为 Recall 后面所有排序都依赖 `TargetWeight`。
如果这一步不做清洗，后面“覆盖高优先级目标”的逻辑就不稳定。

---

## 12. Step 2：候选扩展

候选扩展的目标是：

从 `catalog.video_unit_index` 中找出覆盖任意目标 unit 的视频，生成初始候选命中行集合。

### 12.1 主策略

当前主入口只应使用：

- `catalog.video_unit_index`
- `catalog.videos`

必要时可 join：

- `catalog.video_transcripts`
- `catalog.video_user_states`
- `recommendation.user_video_serving_states`

### 12.2 候选扩展门槛

由于当前没有高层质量分字段，所以候选扩展门槛应建立在现有真实字段之上。建议使用：

- 视频状态可用
- `duration_ms` 落在允许范围
- `mention_count >= min_mention_count`
- `coverage_ratio >= min_coverage_ratio` 或 `coverage_ms >= min_coverage_ms`
- 语言匹配
- transcript 映射密度满足最低要求，例如 `mapped_span_ratio >= min_mapped_span_ratio`

### 12.3 建议默认参数

MVP 默认值可以是：

- `max_candidates_per_unit = 30`
- `min_duration_ms = 30_000`
- `max_duration_ms = 600_000`
- `min_mention_count = 1`
- `min_coverage_ratio = 0.01`
- `min_mapped_span_ratio = 0.02`

这些值不是最终最优，只是合理起点。

### 12.4 粗筛 SQL 思路

```sql
select
  vui.video_id,
  vui.coarse_unit_id,
  vui.mention_count,
  vui.sentence_count,
  vui.first_start_ms,
  vui.last_end_ms,
  vui.coverage_ms,
  vui.coverage_ratio,
  vui.evidence_sentence_indexes,
  vui.evidence_span_indexes,
  vui.sample_surface_forms,

  v.duration_ms,
  v.language,
  v.parent_video_slug,
  v.parent_video_name,
  v.title,
  v.thumbnail_url,

  vt.sentence_count as transcript_sentence_count,
  vt.mapped_span_ratio
from catalog.video_unit_index vui
join catalog.videos v
  on v.video_id = vui.video_id
join catalog.video_transcripts vt
  on vt.video_id = vui.video_id
where vui.coarse_unit_id = any(@target_unit_ids::bigint[])
  and v.status = 'active'
  and v.visibility_status in ('public', 'unlisted')
  and v.duration_ms between @min_duration_ms and @max_duration_ms
  and vui.mention_count >= @min_mention_count
  and vui.coverage_ratio >= @min_coverage_ratio
  and vt.mapped_span_ratio >= @min_mapped_span_ratio;
```

注意：
这一步输出的还不是最终视频列表，而是“目标 unit 命中了哪些视频”的明细行。

---

## 13. Step 3：资格过滤

候选扩展后，需要做第二层视频级过滤。

### 13.1 必须过滤的条件

- 视频状态不可用
- 时长极端不合理
- 语言不匹配
- transcript 学习密度过低
- 该视频在当前 session 明显不适合作为学习材料

### 13.2 用户状态相关过滤

用户状态不建议一刀切。
更合理的策略是：

- 对 new 目标，近期完整看过的视频可以考虑直接排除
- 对 review 目标，即使看过，也可以保留，但需要降权
- 对最近被推过但没消费的视频，不直接过滤，而是交给惩罚项处理

所以总体原则是：

**用户状态优先作为惩罚信号，而不是硬过滤信号。**

---

## 14. Step 4：视频级聚合

前面拿到的是 unit-video 命中行。
Recall 真正的排序对象是视频，所以必须聚合成视频级候选对象。

### 14.1 聚合后的候选结构

```go
type AggregatedVideoCandidate struct {
    VideoID               uuid.UUID
    ParentVideoSlug       string
    DurationMs            int64
    Title                 string
    ThumbnailURL          string

    MatchedUnitIDs        []int64
    CoverageCount         int
    MatchedWeightSum      float64

    ReviewCoveredWeight   float64
    NewCoveredWeight      float64

    TotalMentionCount     int
    TotalSentenceCount    int
    CoveredDurationMs     int64
    AvgCoverageRatio      float64
    WeightedCoverageRatio float64
    WeightedUnitStrength  float64

    TranscriptSentenceCount int
    MappedSpanRatio         float64

    HasWatched            bool
    LastWatchedAt         *time.Time
    WatchCount            int
    CompletedCount        int
    LastWatchRatio        *float64
    MaxWatchRatio         *float64

    LastServedAt          *time.Time
    ServeCount7d          int

    CoveredUnits          []AggregatedCoveredUnit
}
```

### 14.2 每个目标 unit 的贡献信息

```go
type AggregatedCoveredUnit struct {
    CoarseUnitID            int64
    RecommendType           string
    SchedulerScore          float64

    MentionCount            int
    SentenceCount           int
    CoverageMs              int64
    CoverageRatio           float64

    EvidenceSentenceIndexes []int
    EvidenceSpanIndexes     []int
    SampleSurfaceForms      []string

    UnitStrengthScore       float64
}
```

### 14.3 聚合原则

聚合时不能简单按平均值处理，而应按目标权重加权。
原因很简单：

- 一个视频覆盖 1 个高权重 review unit
- 和覆盖 3 个低权重新词

它们在学习价值上不应该被同等对待。

---

## 15. Step 5：视频评分

当前版本的评分必须基于当前 Catalog 实际可用字段来设计。

## 15.1 评分原则

本版评分不再依赖不存在的字段：

- `role`
- `context_relevance`
- `teachability_score`
- `confidence_score`

而是用当前真实存在的四类信号来做：

第一，覆盖了多少高权重目标。
第二，对每个目标的支持强度如何。
第三，视频整体学习内容密度如何。
第四，用户近期是否已经过度消费或被过度曝光。

---

## 15.2 核心信号一：加权覆盖分

### 定义

```text
weighted_coverage_score =
  sum(该视频覆盖的目标 unit 的 target_weight)
  / sum(所有目标 unit 的 target_weight)
```

范围 `[0, 1]`。

### 含义

这是 Recall 的第一主信号。
它直接回答：

**这个视频承载了当前这批学习目标中多少真正重要的部分。**

---

## 15.3 核心信号二：单位目标支持强度分

由于当前没有 `role / teachability`，我们需要从现有事实构造一个更朴素但可靠的 `unit_strength_score`。

### 单个目标的支持强度

对于某个 `(video_id, coarse_unit_id)`，可以定义：

```text
mention_score =
  min(1.0, log(1 + mention_count) / log(1 + 5))

sentence_score =
  case
    when sentence_count >= 4 then 1.0
    when sentence_count = 3 then 0.8
    when sentence_count = 2 then 0.6
    when sentence_count = 1 then 0.35
    else 0
  end

coverage_score =
  case
    when coverage_ratio >= 0.50 then 1.0
    when coverage_ratio >= 0.30 then 0.8
    when coverage_ratio >= 0.15 then 0.6
    when coverage_ratio >= 0.05 then 0.4
    else 0.2
  end
```

然后组合成：

```text
unit_strength_score =
    0.35 * mention_score
  + 0.25 * sentence_score
  + 0.40 * coverage_score
```

### 视频级质量分

再按目标权重加权平均：

```text
quality_score =
  weighted_avg(unit_strength_score, target_weight)
```

### 含义

它不是在问“这个视频质量高不高”，而是在问：

**这个视频对它命中的目标 coarse unit，承载强度够不够。**

---

## 15.4 核心信号三：整体学习密度分

除了单个 unit 的支持强度，视频整体是否像一个“适合学习的片段”也很重要。

### 可用信号

当前 Catalog 中可直接使用：

- `mapped_span_ratio`
- `semantic_span_count`
- `sentence_count`

### 建议定义

```text
transcript_density_score =
  case
    when mapped_span_ratio >= 0.30 then 1.0
    when mapped_span_ratio >= 0.20 then 0.8
    when mapped_span_ratio >= 0.10 then 0.6
    when mapped_span_ratio >= 0.05 then 0.4
    else 0.2
  end
```

### 含义

如果一个视频 transcript 中几乎没有 coarse unit 映射，那么它即使命中了目标 unit，也很可能不是一个高价值学习材料。

---

## 15.5 核心信号四：时长适配分

### 默认理想时长

MVP 默认：

- `ideal_duration_ms = 180000`

### 评分映射

```text
deviation = abs(duration_ms - ideal_duration_ms) / ideal_duration_ms

duration_fit_score =
  case
    when deviation <= 0.3 then 1.0
    when deviation <= 0.6 then 0.7
    when deviation <= 1.0 then 0.4
    else 0.2
  end
```

### 含义

太短可能语境不足，太长学习负担过大。
3 分钟左右通常是比较平衡的学习片段长度。

---

## 15.6 核心信号五：review 优先分

Recall 应与 scheduler 的“先保 review，再补 new”保持一致。

### 定义

```text
if total_review_target_weight > 0:
  review_priority_score =
    review_covered_weight / total_review_target_weight
else:
  review_priority_score = 0
```

### 含义

如果一个视频恰好覆盖了多个高优先级 review 目标，它应获得显著加分。

---

## 15.7 新鲜度与惩罚项

### 1. recently_watched_penalty

如果用户最近刚完整看过该视频，应降权。
但对于 review 目标，不建议一刀切排除。

### 2. weak_watch_penalty

如果用户其实只浅看过，例如：

- `last_watch_ratio` 很低
- `completed_count = 0`

则惩罚应低于“完整消费过”的情况。

### 3. recently_served_penalty

如果系统近期已经多次推过该视频但用户没有明显消费，应明显降权。

### 4. same_source_penalty

如果候选与已经选中的视频来自同一 `parent_video_slug`，应加同源惩罚。

### 5. duplicate_coverage_penalty

如果某个候选覆盖的目标，已经被前面选中的候选很好覆盖，则它的边际价值应下降。

---

## 15.8 最终评分公式

推荐的 MVP 初始公式如下：

```text
base_recall_score =
    0.45 * weighted_coverage_score
  + 0.25 * quality_score
  + 0.10 * review_priority_score
  + 0.10 * transcript_density_score
  + 0.10 * duration_fit_score
```

解释如下：

- 加权覆盖 45%，仍然是第一主信号
- 目标承载强度 25%，反映单个目标被支持得够不够
- review 优先 10%，与 scheduler 对齐
- transcript 整体学习密度 10%
- 时长适配 10%

然后在多样性选择阶段再减惩罚项：

```text
marginal_score =
  base_recall_score
  - recently_watched_penalty
  - recently_served_penalty
  - same_source_penalty
  - duplicate_coverage_penalty
```

---

## 16. reason_codes 设计

每个候选视频都应生成 reason codes，用于解释和调试。

推荐语义如下：

```text
high_weighted_coverage
strong_unit_support
review_heavy
high_transcript_density
ideal_duration
recently_watched_but_allowed
recently_served
same_source_penalty
duplicate_coverage_penalty
low_density
weak_unit_support
```

注意：
reason codes 是解释层，不是主评分逻辑本身。

---

## 17. Step 6：贪心多样性选择

不能简单按 `base_recall_score` 取 top-N。
否则多个高分视频可能：

- 覆盖同一批目标
- 来自同一原视频来源
- 在学习收益上高度重复

### 17.1 目标

- 尽可能覆盖更多高权重目标
- 避免重复满足同一批目标
- 避免同源视频过于集中
- 在保证质量前提下减少重复曝光

### 17.2 维护状态

- `uncovered_units`
- `selected_videos`
- `selected_source_counts`

### 17.3 边际覆盖比例

```text
marginal_coverage_ratio =
  sum(新增覆盖的 target_weight)
  / total_target_weight
```

### 17.4 边际得分

```text
marginal_score =
  candidate.base_recall_score * marginal_coverage_ratio
  - same_source_penalty
  - recently_served_penalty
  - duplicate_coverage_penalty
```

### 17.5 选择流程

1. 初始化 `uncovered_units = 所有目标 unit`
2. 初始化 `selected_videos = []`
3. 得到全部候选的 `base_recall_score`
4. 循环直到：
   - 选中数量达到上限
   - 或者没有候选还能提供有效新增覆盖

5. 每轮重新计算所有剩余候选的 `marginal_score`
6. 选出当前边际价值最高的视频
7. 更新 `uncovered_units`
8. 更新同源计数
9. 继续下一轮

### 17.6 默认上限

MVP 可设：

- `max_videos = 5`

如果 5 个视频仍不能覆盖全部目标，不是问题。
不要为了追求覆盖率而明显降低质量门槛。

- 哪些惩罚影响了最终结果

## 20. 与 scheduler 的串联

### 20.1 调用关系

```text
GenerateRecommendations(userID, sessionContext)
    ↓
[scheduler] -> SchedulerBatch
    ↓
[recall] -> RecallResult
    ↓
[task] -> TaskResult
```

### 20.2 scheduler 输出不需要修改

Recall 直接消费 scheduler 输出结构即可。
`Score` 在 Recall 内解释为 `TargetWeight`。

### 20.3 Recommendation 内部应保留两层结果

即使 Recall 已经输出视频结果，Recommendation 内部仍建议同时保留：

- scheduler batch
- recall result

因为 task 层通常仍需要学习内容级信息，而不只是视频级结果。

## 21. 边界情况处理

### 21.1 没有视频覆盖任何目标 unit

处理方式：

- Recall 正常返回空列表
- `warning_codes` 写 `no_candidate_found`
- 上层可选择 fallback 到只保留 scheduler 结果

### 21.2 候选视频都已看过

处理方式：

- 若存在 review 目标，允许保留看过的视频，但施加惩罚
- 若只有 new 目标，且候选全部刚完整看过，可返回空列表

### 21.3 覆盖率很低

即使选完所有合理候选，整体覆盖率仍然很低，也不应为了凑覆盖而明显降低门槛。

处理方式：

- 正常返回已有候选
- 写入 `low_coverage_warning`

### 21.4 同一 unit 在同一视频中多次出现

这是正常且通常是正面信号。

处理方式：

- 目标覆盖按 unit 去重
- 但 `mention_count`、`sentence_count`、`coverage_ms` 用于提高支持强度分

### 21.5 同源视频过多

处理方式：

- 不做硬排除
- 在贪心阶段做软惩罚
- 若最终仍然同源集中，允许返回，但写 warning

## 22. 性能设计

## 22.1 关键索引支撑

当前主读路径依赖的关键索引应是：

- `catalog.video_unit_index(coarse_unit_id, mention_count desc, coverage_ratio desc)`
- `catalog.video_unit_index(video_id)`
- `catalog.videos(status, visibility_status)`
- `catalog.video_user_states(user_id, video_id)`
- `recommendation.user_video_serving_states(user_id, video_id)`

`video_semantic_spans` 的索引主要用于解释和调试，而不是第一阶段召回主入口。

## 22.2 MVP 性能判断

在当前规模下：

- scheduler 目标数一般在 10 到 20
- 候选扩展规模一般在几十到几百行
- 视频级聚合和打分完全可以在应用内存中完成

因此当前不需要：

- embedding 检索
- 搜索引擎
- 分布式召回
- 黑盒排序服务

只要索引正确，Postgres 足以支撑 MVP。

## 22.3 后续扩展方向

若后续规模增大，可以考虑：

- 把高频 candidate aggregation 前移成物化读模型
- 对高频 Recall 做缓存
- 增加视频级 enrichment 特征表
- 引入个性化偏好特征

但这些都不是当前版本的必需项。

---

## 23. MVP 明确不做的事

为了控制范围，当前版本明确不做：

- embedding 语义召回
- 复杂相似视频去重
- 个性化视频偏好排序
- Recall 层用户级配置 UI
- A/B 实验框架
- 黑盒排序模型
- 片段级推荐对象

当前 Recall 的核心仍然是：

**结构化索引 + 规则打分 + 贪心边际选择**

---

## 24. 实现顺序建议

### 第一阶段：基础读链路

打通：

- scheduler 输出读取
- `video_unit_index` 候选扩展
- `videos` 状态过滤
- 视频级聚合

### 第二阶段：评分模型

加入：

- `weighted_coverage_score`
- `unit_strength_score`
- `transcript_density_score`
- `duration_fit_score`
- `review_priority_score`

### 第三阶段：多样性选择

实现：

- `duplicate_coverage_penalty`
- `same_source_penalty`
- `recently_served_penalty`
- greedy diversification

### 第四阶段：审计闭环

补全：

- `recall_runs`
- `recall_run_targets`
- `recall_run_candidates`
- `recall_run_candidate_units`

### 第五阶段：串联 Recommendation 输出

把 Recall 正式串进 Recommendation 内部流程，交给 task 层使用。

---

## 25. 最终结论

这一版视频召回模块设计可以概括为一句话：

**以 scheduler 产出的 coarse unit 目标集合作为输入，以 `catalog.video_unit_index` 为核心倒排索引入口，以 `catalog.videos.video_id` 作为唯一召回对象，结合 `video_transcripts`、`video_user_states` 和 serving 状态，经过候选扩展、资格过滤、视频级聚合、基于当前真实数据模型的规则评分与贪心多样性选择，输出一组适合承载当前学习目标的切片视频候选，并将全过程写入 Recommendation 自己的审计表。**

这版与当前 Catalog 设计完全一致，体现在：

- 都坚持“切片即视频主对象”
- 都不引入 `video_segments`
- 都不再依赖 `cues / mentions / role / teachability` 这类当前不存在的数据
- 都把 transcript 拆成标准读模型
- 都围绕 `coarse_unit -> video` 的结构化关系建模
- 都强调审计、可解释、工程可落地

如果你愿意，下一步最合适的是把这份 Recall 文档也继续落成两份工程材料：
第一份是 **Recommendation schema 的 DDL 草案**，第二份是 **Go 接口与实现任务拆解文档**。
