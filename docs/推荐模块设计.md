# Recommendation 详细设计文档

## 0. 文档信息

文档名称：Recommendation 详细设计文档
适用阶段：MVP
目标读者：后端、数据、测试、Learning engine 对接方、Catalog 对接方
文档目标：定义 Recommendation 的职责边界、内部模块划分、共享读模型、核心规则、接口、事务、审计、测试与实施约束，使其可以独立实施，并与 Learning engine、Catalog 并行开发。当前 MVP 的唯一核心目标是：**输入一个用户，稳定输出一组当前足够合理的视频 feed plan**。Recommendation 与 Learning engine 是平级模块；Learning engine 负责维护学习状态，Recommendation 负责基于学习状态与内容索引输出视频列表，并为学习推荐视频附带本轮预期学习的 `learning_units`；video-level 补全视频允许 `learning_units=[]`。当前 Catalog 的内容事实链为 `videos -> video_transcripts -> video_transcript_sentences -> video_semantic_spans -> video_unit_index`，Recommendation 应建立在这条链之上，而不是要求独立的 `video_segments` / `segment_unit_mappings` 物理表。

## 1. 模块定位

Recommendation 是视频优先的教育推荐模块。它回答的问题只有一类：在当前时刻，应该给这个用户返回哪些视频；这些视频为什么适合当前学习阶段；每个视频在本轮预期承载哪些学习单元；在供给稀疏时，如何仍然返回一组不空且不失焦的视频学习计划。换句话说，Recommendation 负责“这次推什么视频，以及每个视频预期学什么”，不负责“学得怎么样”。Learning engine 输出 `learning.user_unit_states`，Recommendation 读取这些状态，但不写 `learning.unit_learning_events`，也不写 `learning.user_unit_states`。Recommendation 自己拥有 serving state、video recommendation 审计、需求规划、候选生成、证据解析、视频聚合、排序与选择逻辑。

Recommendation 不是 Learning engine，不是事件归约系统，不是学习状态真相层，不是课程编排系统，不是非视频任务系统，也不是通用娱乐 feed。对上层业务 API 而言，Recommendation 应作为一个整体模块对外提供能力，而不是把“调度器、候选生成、排序器、选择器”分别暴露出去，否则 Recommendation 的内部组织会被泄露到外部边界，造成耦合。

## 2. 设计目标与非目标

Recommendation 当前只追求三件事。第一，结果稳定，不能因为 exact target 稀疏就塌掉。第二，目标不失焦，不能为了补量把推荐做成泛视频推荐。第三，工程可落地，规则化、可解释、先不引入 ML，也不把 task 层耦进主流程。当前 Recommendation 的核心问题不是 unit 调度本身，而是视频候选生成与排序：如果候选池只由当前 exact target unit 反查得到，数量和多样性都会不足，后续 ranking 空间也会很小。Recommendation 的设计必须正面解决这个结构性问题。

本期明确不做以下事情：task layer 进入主流程、非视频 fallback task、embedding / vector recall、ML ranking、复杂个性化兴趣模型、完整 curriculum / frontier 平台、Recommendation 回写 Learning engine。这些都不属于当前 MVP 要解决的问题。当前阶段只把“输入用户 -> 输出视频学习计划”的闭环做稳。

## 3. 设计原则

第一，最终输出始终以视频为主。Recommendation 对外返回的是排序后的视频列表，以及学习推荐视频的本轮预期学习单元 `learning_units`。补全视频返回空数组。它不是独立 unit list，不是 task list，也不是学习任务卡片；前端可以基于非空 `learning_units` 组织字幕高亮、lookup、小测和学习模式，但 task 层先后置，不进入当前主流程。

第二，Learning engine 与 Recommendation 的边界保持清晰。Learning engine 负责学习事件、学习状态、due/new/mastery 等信号；Recommendation 负责根据学习状态与内容供给生成视频候选池、对视频做教育场景排序、输出最终视频列表，并维护自己的审计与 serving state。`last_recommended_at` 之类的字段属于 Recommendation 的投放状态，不能进入学习状态表。

第三，planner 输出的是“学习需求画像”，而不是一个硬 unit 名单。Recommendation 的主问题不是“当前 unit 调度不够精确”，而是“视频候选生成与排序空间不够”。因此上游必须输出可执行的需求分桶和预算，而不是只输出 exact unit 列表。

第四，planner 必须具备轻量供给感知。它不能只看“该学什么”，还要知道“视频库接不接得住”。这一步不要求复杂库存优化，但必须知道某个 unit 的视频供给是否为 `none / weak / ok / strong`，并在 exact 候选不足时扩大 bundle / soft-future 的预算。

第五，内部证据单位是 span / sentence-window，外部输出单位是 video。当前 Catalog 已明确以 `video_semantic_spans` 作为最细事实层，以 `video_unit_index` 作为视频级 Recall 主入口，并不建立 `catalog.video_segments` / `catalog.segment_unit_mappings`。因此 Recommendation 的内部证据建模必须适配这条内容链：先在视频-学习单元层做粗候选，再通过 `best_evidence_*` 精确回查 span 与 sentence 形成细粒度证据。

第六，召回不能只有 exact lane。必须允许当前核心需求、多 unit 组合承载、近未来内容以及质量兜底一起进入候选池，但权重必须受控。

## 4. 与 Learning engine、Catalog 的数据契约

Learning engine 对 Recommendation 暴露的核心输入是 `learning.user_unit_states`。Recommendation 当前只读取 target 过滤和 planner 实际需要的字段：`is_target`、`target_priority`、`status`、`mastery_score`、`last_progress_quality`、`next_review_at`。Learning engine 不提供 `last_recommended_at`，该字段属于 Recommendation 自己的 serving state。

Catalog 当前提供 Recommendation 所需的内容事实链：`catalog.videos`、`catalog.video_transcripts`、`catalog.video_transcript_sentences`、`catalog.video_semantic_spans`、`catalog.video_unit_index`，并可选提供 `catalog.video_user_states` 作为消费状态投影。Catalog 当前明确不建立 `catalog.video_segments` 与 `catalog.segment_unit_mappings`，因为数据库中的 `video_id` 本身就是最终可播放、可推荐的切片视频对象，而语义跨度映射已经由 `video_semantic_spans` 与 `video_unit_index` 共同承担。

Recommendation 自己拥有的读模型与状态包括：`recommendation.v_video_unit_recall_index`、`recommendation.v_unit_video_inventory`、`recommendation.user_unit_recall_queue`、`recommendation.user_unit_recall_queue_states`、`recommendation.recall_projection_metadata`、`recommendation.user_unit_serving_states`、`recommendation.user_video_serving_states`、`recommendation.video_recommendation_runs`、`recommendation.video_recommendation_items`。其中 `v_video_unit_recall_index` 和 `v_unit_video_inventory` 是内容召回读模型；`user_unit_recall_queue` 是用户级召回 projection，用来避免每次 feed 都从全量 target units 进入 recall。

Recommendation 对以下 Catalog / Recommendation 配套能力存在显式前置条件依赖：

- `catalog.video_unit_index.best_evidence_*` 已可用；
- `recommendation.v_video_unit_recall_index` 已落地；
- `catalog.video_unit_index.best_evidence_start_ms/end_ms` 已可用。

若上述条件未满足，则 Evidence Resolver、unit-level evidence explanation、以及 `video_recommendation_items.learning_units` 中的 evidence 写入路径不得正式启用，仅可作为 shadow 流程或开发态能力存在。

## 5. 总体架构

Recommendation 的总体数据流如下：

```text
RecommendationRequest
    +
learning.user_unit_states
    +
recommendation.v_video_unit_recall_index
    +
recommendation.v_unit_video_inventory
    +
recommendation.user_unit_serving_states
    +
recommendation.user_video_serving_states
    +
(可选) catalog.video_user_states
    +
catalog.video_semantic_spans
    +
catalog.video_transcript_sentences
        ↓
[1] Context Assembler
        ↓
[2] Demand Planner
        ↓
Demand Bundle
        ↓
[3] Candidate Generator
        ↓
VideoUnitCandidates
        ↓
[4] Evidence Resolver
        ↓
ResolvedEvidenceWindows
        ↓
[5] Video Evidence Aggregator
        ↓
Video Candidates + Expected Learning Units
        ↓
[6] Video Ranker
        ↓
Ranked Videos
        ↓
[7] Video Selector
        ↓
Selected Videos
        ↓
[8] Video Fill Service
        ↓
Selected Videos + Fill Videos
        ↓
[9] Explanation Builder
        ↓
Final Video Learning Plan
        ↓
[10] Audit Writer + Serving State Manager
```

这里的关键点有四个。第一，planner 不再输出一个硬 unit 名单，而是输出一份带预算的 Demand Bundle。第二，候选生成不再只有 exact lane，而是多 lane 并行扩张。第三，内部按细粒度证据建模，外部按 video 输出结果，并且学习推荐 video 附带本轮预期学习的 `learning_units`。第四，selector 之后才做 video-level fill，补全 item 的 `learning_units=[]`，只用于填满 feed，不伪装成学习任务。主审计对象是 video recommendation run/item，而不再只围绕 scheduler item。

## 6. 核心领域对象

`RecommendationRequest` 表示一次推荐请求，至少包含 `user_id`、`target_video_count`、`request_context`。默认 `target_video_count = 8`。时长偏好不是外部请求参数，而是 Recommendation 内部固定策略，当前使用 `[45, 200]` 秒，用于 duration fit / educational fit / overload penalty。内部一旦涉及与 Catalog 内容时间字段比较，应统一转换为毫秒，以与 Catalog 的 `duration_ms / start_ms / end_ms` 对齐。

`RecommendationContext` 是 Context Assembler 的输出，至少包含：规范化后的请求、内部时长偏好、当前时间、active unit states、unit inventory、unit serving states、video serving states，以及可选的 video user states。

`DemandUnit` 表示某个学习单元在本轮推荐中的需求项，至少包含 `unit_id`、`bucket`、`weight`、`supply_grade`。

`DemandBundle` 表示本轮视频推荐的学习需求画像，包含四个 bucket：`hard_review`、`new_now`、`soft_review`、`near_future`，并包含 `session_mode`、`target_video_count`、`preferred_duration_sec`、`lane_budget`、`mix_quota`、`flags`。当前的最终方案采用 `hard_review / new_now / soft_review / near_future` 这四个 bucket，以及 `exact_core / bundle / soft_future / quality_fallback` 四条 lane，这是当前最适合直接定稿的收敛方案。

`EvidenceRef` 表示一个 best evidence 引用，来源于 `video_unit_index.best_evidence_sentence_index` 与 `video_unit_index.best_evidence_span_index`，包含 `sentence_index` 与 `span_index`。它是 Catalog 对该 `(video_id, coarse_unit_id)` 已选定的单一 unit-level 定位事实；时间边界来自 `best_evidence_start_ms/end_ms`。

`VideoUnitCandidate` 是 Candidate Generator 的统一输出，它表示某个视频对某个学习单元、在某条 lane 下的候选命中，至少包含：`video_id`、`coarse_unit_id`、`lane`、`bucket`、`unit_weight`、`mention_count`、`sentence_count`、`coverage_ms`、`coverage_ratio`、`sentence_indexes`、`best_evidence_ref`、`best_evidence_start_ms/end_ms`、`duration_ms`、`mapped_span_ratio`、`content_quality_score`、`candidate_score`。

`ResolvedEvidenceWindow` 是 Evidence Resolver 的输出，表示某个 `VideoUnitCandidate` 在本轮推荐语义下被解析成的更细粒度证据窗口，至少包含：`best_evidence_ref`、`best_evidence_start_ms`、`best_evidence_end_ms`、`window_sentence_indexes`、`window_start_ms`、`window_end_ms` 等。

`ExpectedLearningUnit` 表示某个视频在本轮推荐中预期承载的学习单元，至少包含：`coarse_unit_id`、`role`、`is_primary`、`evidence`。`role` 固定为 `hard_review / new_now / soft_review / near_future`，来自对应 Demand bucket；`evidence` 是 unit 级定位信息，不是 video 级全局 best。

`VideoCandidate` 是 Aggregator 聚合后的 video-level 候选对象，至少包含：`video_id`、`lane_sources`、`dominant_role`、`dominant_unit_id`、`learning_units`、`hard_review_cover`、`new_now_cover`、`soft_review_cover`、`near_future_cover`、`coverage_strength_score`、`bundle_value_score`、`educational_fit_score`、`future_value_score`。`learning_units` 是唯一的 unit plan 事实来源；role 维度统计只能从它派生。

`FinalRecommendationItem` 是最终 selection 的内部 plan/audit 输入，至少包含：`video_id`、`duration_ms`、`score`、`reason_codes`、`learning_units`。对外 DTO 只映射 `video_id`、`duration_ms`、`learning_units`；`score` 与 `reason_codes` 只进入 audit。

## 7. 内部模块设计

### 7.1 Recommendation Facade / Usecase Orchestrator

这是 Recommendation 对外暴露的唯一主入口。上层业务 API 只调用 `GenerateVideoRecommendations`，不直接感知内部的 planner、candidates、resolver、ranker 或 selector。

它负责组织整体时序、调用各内部模块、组织最终写事务、返回统一结果。它不负责写 bucket 规则、不负责算视频分、不负责 selection quota，也不直接做 SQL 查询。

推荐伪代码如下：

```text
GenerateVideoRecommendations(req):
  ctx         = ContextAssembler.Assemble(req)
  bundle      = DemandPlanner.Plan(ctx)
  candidates  = CandidateGenerator.Generate(ctx, bundle)
  resolved    = EvidenceResolver.Resolve(candidates, ctx, bundle)  // no DB hydration on hot path
  videos      = VideoEvidenceAggregator.Aggregate(resolved, ctx, bundle)
  ranked      = VideoRanker.Rank(videos, ctx, bundle)
  selected    = VideoSelector.Select(ranked, ctx, bundle)
  response    = ExplanationBuilder.Build(selected, ctx, bundle)

  tx:
    AuditWriter.WriteRunAndItems(...)
    ServingStateManager.ApplySelection(...)

  return response
```

### 7.2 Context Assembler

Context Assembler 的目标是把 Recommendation 所需输入一次性装配成统一上下文。在线路径先检查 `user_unit_recall_queue_states` 与 `recall_projection_metadata`，当用户 queue 缺失、active target 数量变化、落后于 `learning.user_unit_states`，或落后于 recall projection 时，同步 rebuild 当前用户的 `user_unit_recall_queue`。随后它从 queue 中选出两层 scope：`planner_scope` 和 `recall_fetch_scope`。`planner_scope` 表示本轮进入 Demand Planner 的学习需求，可少量包含 `supply_grade='none'` 的 unit 用于表达低供给学习目标；`recall_fetch_scope` 只包含 `supply_grade <> 'none'` 的 planner units，用于读取 `v_video_unit_recall_index` top-N rows。它不读取 spans / sentences，也不对全量未 mastered target units 读取 recall rows。

### 7.3 Demand Planner

Demand Planner 是 Recommendation 的起点。它把用户当前学习状态转成一份 Demand Bundle，而不是一个硬 unit 名单。它的输入包括：active unit states、unit inventory、unit serving states、request 中的 `target_video_count`，以及 Recommendation context 中的内部 `preferred_duration_sec`。它的输出是带四个 bucket、lane budget、mix quota 与 flags 的 Demand Bundle。

若 planner 内部使用 `fragility_score`、`instability_score` 之类的术语，它们都属于 Recommendation 派生信号，不要求 Learning engine 预先持久化为单独列。当前实现只基于 `status`、`next_review_at`、`mastery_score`、`last_progress_quality` 与供给信息推导 hard / soft review，不读取 Learning Engine 内部的 recent window 或 progress 计数字段。

当前阶段固定四个 bucket：`hard_review`、`new_now`、`soft_review`、`near_future`。当前 Bucket precedence 必须写死为：

```text
hard_review > new_now > soft_review > near_future
```

一个 unit 在一次 planner run 中只能进入一个主 bucket。后续 lane 预算、coverage 统计、selector quota 都以这个唯一 bucket 为准。这是当前方案里最重要的约束之一。

`hard_review` 用于当前必须优先服务的学习内容，例如 overdue、due now、最近失败、最近变脆弱的 unit。`new_now` 用于当前允许引入的新内容，但必须 supply-aware，供给极差的新内容不应成为主推。`soft_review` 用于今天不是硬 due，但近期弱、容易忘、值得顺带覆盖的内容。`near_future` 用于未来很快进入学习/复习窗口的内容，是扩召回的合法来源，但不能压过当前主线。

Demand Planner 必须具备轻量供给感知。它使用 `v_unit_video_inventory` 中的 `distinct_video_count`、`avg_mention_count`、`avg_sentence_count`、`avg_coverage_ms`、`avg_coverage_ratio`、`strong_video_count`、`supply_grade` 这些确定性聚合事实。关键规则是：`hard_review` 永不因供给弱而消失；`new_now` 必须 supply-aware；当 `hard_review` 供给弱时，提高 `bundle / soft_future` 预算；`quality_fallback` 只作为保险丝。

### 7.4 Candidate Generator

Candidate Generator 根据 Demand Bundle，通过四条学习候选 lane 生成足够健康的 `VideoUnitCandidate` 候选池，而不再假设底层有独立物理 segment 表。它的第一跳输入是 Context Assembler 已按 scoped units 预取的 `recommendation.v_video_unit_recall_index` rows，即按 `coarse_unit_id` 预排序的“视频 × coarse unit”召回索引。

当前 MVP 的学习候选 lane 固定为四条：`exact_core`、`bundle`、`soft_future`、`quality_fallback`。selector 之后还有两条 video-level fill lane：`mastered_target_fill`、`popular_fill`。fill lane 不参与 Candidate Generator，不生成 `VideoUnitCandidate`，也不进入 Evidence Resolver / Aggregator。

`exact_core` 的目标是保证当前最核心学习需求一定进入候选池。它的输入是 `hard_review` 与 top-ranked `new_now`。在当前 Catalog 能力下，它的强度不看 `primary/contextual/incidental`，而看 `mention_count`、`sentence_count`、`coverage_ms / coverage_ratio`、`mapped_span_ratio` 这些确定性事实。它提供高精度的当前主线候选，但不能承担候选扩张职责。

`bundle` 是当前方案里最关键的 lane。它的目标是找那些整体上能承载多个高价值 unit 的视频，而不是单点命中。它的输入是 `hard_review`、`new_now`、`soft_review` 与少量高权重 `near_future`。实现上应先从 `v_video_unit_recall_index` 按这些 unit 拉出 top-N video-unit rows，再在内存按 `video_id` 聚合，找出满足 bundle 条件的视频。默认要求至少覆盖两个有价值 unit，且至少包含一个 `hard_review` 或 `new_now`。当 `hard_review_low_supply = true` 时，允许动态放宽为 `1 个 hard/new + 1 个 soft_review`，或受控的 `2 个 soft_review`。这条 lane 直接解决“视频目标太单一、候选池太窄”的问题。

`soft_future` 的目标是扩大候选池，但不让系统跑偏。它输入 `soft_review` 与 `near_future`，只做补充，不承担主排序目标。尽管这条 lane 内部可以把 `soft_review` 与 `near_future` 放在一起扩召回，但后续 Aggregator、Ranker、Selector 里必须继续把两者分开统计。当前方案明确认为：“当前主线优先，边缘不稳优先于未来内容”，因此 `soft_review` 应高于 `near_future`。

`quality_fallback` 的目标是在前三条 lane 合起来仍不足时，兜底保证列表不空、不塌。它只在 distinct candidate videos 明显不足，且预测无法满足 `target_video_count` 时触发。Fallback 候选只能来自教育适配较高、时长合适、transcript 清晰、近期未高频重复推荐、且至少与当前需求弱相关的视频。它只能补位，不能反客为主，最终列表中通常最多 1 条。若仍无足够合理候选，宁可 under-fill，也不要乱推。

`quality_fallback` 和 video-level fill 的区别必须保持清晰：`quality_fallback` 仍是学习候选 fallback，至少弱相关于当前未掌握学习需求，后续会解析 evidence 并生成 `learning_units`；`mastered_target_fill` / `popular_fill` 是 feed 视频补全，只在 selector 后发现数量不足时触发，返回 `learning_units=[]`。

### 7.5 Evidence Resolver

Evidence Resolver 把 Candidate Generator 输出的 `VideoUnitCandidate` 转成 unit-level best evidence window。它不再回查 `catalog.video_semantic_spans` 或 `catalog.video_transcript_sentences`；`sentence_index`、`span_index`、`start_ms`、`end_ms` 都直接来自 `v_video_unit_recall_index`。因此在线主链路没有 per-candidate evidence IO，只做结构转换、上下文取消检查和缺失边界的稳定处理。

Catalog 回查句子文本、span surface text、解释、词典等富展示字段不属于 Recommendation plan 热路径；Feed detail / subtitle / lookup 类 API 需要时由对应 facade 或 Catalog API 批量读取。若 recall row 缺少 best evidence 时间边界，该 unit 仍可保留 sentence/span index，但 `window_completeness` 会按缺失证据降权。Recommendation 主链路禁止按 candidate 逐条读 span 或 sentence。

### 7.6 Video Evidence Aggregator

Video Evidence Aggregator 的目标是把多个 `ResolvedEvidenceWindow` 聚合成 video-level candidate，并构造学习候选视频本轮对外返回的 `learning_units`。它必须保证：同 `video_id + unit_id` 只保留最强证据，以防同一视频因反复提到同一个 unit 被高估；可选地保留第二条强证据作为弱增量，但第三条以后忽略。它必须为每个 expected unit 保存 unit-level evidence，用于 UI 高亮、reason code、调试与审计。

Aggregator 输出的 `learning_units` 只来自本轮 demand/candidate 命中的 unit，不遍历视频里所有词；学习候选视频 MVP 最多输出 8 个 learning units；排序按 role priority、evidence strength、start time、unit id 稳定执行。`hard_review` 与 `new_now` 默认是 primary；如果没有 core role，则从 `soft_review / near_future` 中选前 1 到 2 个作为 primary，避免学习候选视频没有主学习目标。补全视频不经过 Aggregator，`learning_units=[]`。

Aggregator 至少输出这些 video-level 特征：`hard_review_cover`、`new_now_cover`、`soft_review_cover`、`near_future_cover`、`coverage_strength_score`、`bundle_value_score`、`educational_fit_score`、`future_value_score`、`lane_sources`、`distinct_matched_unit_count`。这些统计必须从 `learning_units` 派生，不得重新维护一组与 `learning_units` 并行的 covered arrays。`lane_sources` 是该视频在本轮命中过的所有候选 lane 的集合，不能只记录 per-unit winning candidate 的 lane；`primary_lane` 再从完整 `lane_sources` 按 lane priority 派生。

其中 `coverage_strength_score` 当前不依赖上游 `relevance_type`，而依赖确定性内容强度，例如 `mention_count`、`sentence_count`、`coverage_ratio`、`evidence_window_focus`、`mapped_span_ratio`。`educational_fit_score` 主要看 `duration_fit`、`coverage_density`、`window_completeness` 与 `noise_penalty`。

### 7.7 Video Ranker

Video Ranker 的职责是给每个 video 计算基础分 `base_score`。它的排序目标不是“命中当前 target unit 最多”，而是“这个视频作为当前一轮学习内容载体的总体价值有多高”。

建议的核心评分维度包括：`demand_coverage`、`coverage_strength_score`、`bundle_value_score`、`educational_fit_score`、`future_value_score`，以及轻量的 `freshness_score`，再减去 `recent_served_penalty`、`recent_watched_penalty`（如有）和 `overload_penalty`。当前建议的初版公式为：

```text
base_score =
  0.40 * demand_coverage
+ 0.18 * coverage_strength_score
+ 0.15 * bundle_value_score
+ 0.15 * educational_fit_score
+ 0.05 * future_value_score
+ 0.05 * freshness_score
- 0.03 * recent_served_penalty
- 0.02 * overload_penalty
```

`demand_coverage` 建议定义为：

```text
demand_coverage =
  0.50 * hard_review_cover
+ 0.20 * new_now_cover
+ 0.20 * soft_review_cover
+ 0.10 * near_future_cover
```

这里的原则是：当前主线优先，组合承载能力第二，未来价值只能是弱加分项，重复与过载必须惩罚。

### 7.8 Video Selector

Selector 决定最终返回给用户的列表，不能只按基础分取前 N 条，否则列表会很难看。它的职责是保证当前主线覆盖、去重、冷却、future 占比受控、fallback 占比受控，并在极端稀疏库存下优雅降级。

当前固定三种 selector mode：`normal`、`low_supply`、`extreme_sparse`。`normal` 表示 `hard_review` 供给健康且 exact/bundle 候选足够；`low_supply` 表示 `hard_review_low_supply = true` 或 exact_core distinct videos 明显不足；`extreme_sparse` 表示加上 fallback 后仍不足，或在合理约束下无法凑满 N 条。

配额规则建议比例化而不是写死绝对数。令 `N = target_video_count`。在 `normal` 模式下，`core_dominant_min = ceil(N * 0.50)`，`future_dominant_max = floor(N * 0.25)`，`fallback_max = 1`，`same_primary_unit_max = 2`。在 `low_supply` 模式下，`core_dominant_min = ceil(N * 0.375)`，`future_like_max = ceil(N * 0.375)`，`fallback_max = 1`。在 `extreme_sparse` 模式下，允许 under-fill，不强行凑满 N 条。same-unit 硬上限只作用于 `learning_units` 中 `is_primary=true` 的 unit；非 primary support units 只参与边际覆盖和解释，不作为硬排除条件。

选择算法建议使用轻量贪心。Phase A 先保主线，从高分到低分遍历，优先选那些能覆盖尚未覆盖的 `hard_review / new_now` learning units 的视频，直到满足 `core_dominant_min` 或无候选。Phase B 再按边际收益填充：`marginal_score = base_score + uncovered_core_gain + uncovered_soft_gain - redundancy_penalty - recent_repeat_penalty`。Phase C 最后才允许 fallback 进入。硬原则是：**宁可 under-fill，也不要为了凑满条数而推明显弱相关视频。**

### 7.8.1 Video Fill Service

Video Fill Service 是 selector 之后、Explanation Builder / Audit Writer 之前的轻量补全组件。它只在 selected videos 少于 `target_video_count` 时触发，最终顺序固定为：

```text
normal learning recommendations
-> mastered_target_fill
-> popular_fill
```

`mastered_target_fill` 从当前用户 `is_target=true AND status='mastered'` 的 units 对应视频里补，保持和用户当前目标集合相关。`popular_fill` 从全局 active/public/published 视频里按 `view_count / like_count / favorite_count` 粗排补；`catalog.video_engagement_stats` 是可选统计行，缺失时按 0 计数处理，不能因此排除可用视频。两者都会排除本轮已选视频，并读取 `recommendation.user_video_serving_states` 与 `catalog.video_user_states` 做 video-level freshness/watch penalty。

补全每类最多做一次小查询，查询 limit 固定为 `min(gap * 3, 30)`。它不走 demand planning、evidence resolver、learning unit aggregate、subtitle/span/sentence evidence 查询，也不生成 end quiz 目标。构造出的 `VideoCandidate` 允许 `DominantRole=""`、`DominantUnitID=nil`、`LearningUnits=[]`。

补全后仍不足时，`underfilled=true` 表示完整策略链路后最终仍少于目标数量。

### 7.9 Explanation Builder

Explanation Builder 是独立支撑模块，不参与选择，只负责把 selected video 转成最终 plan item，并生成 audit 用 `reason_codes`。当前对外 response 不暴露 explanation 文本。reason codes 包括：`hard_review_covered`、`new_unit_introduced`、`soft_review_supported`、`near_future_warmup`、`bundle_coverage_high`、`strong_evidence_window`、`good_learning_fit`、`recently_not_served`、`low_supply_core_preserved`、`fallback_quality_fill`、`mastered_target_fill`、`popular_feed_fill`。这里不使用 `strong_primary_match` 这类依赖不存在上游字段的 reason code。

### 7.10 Serving State Manager

Serving State Manager 维护 Recommendation 自己的投放状态，包括 `user_unit_serving_states` 和 `user_video_serving_states`。前者用于 unit 级冷却和统计哪些 unit 最近已被视频承载，后者用于 video 级冷却和统计视频最近被推荐次数。unit 级 serving state 只能从最终 `FinalRecommendationItem.learning_units` 收集，不能从独立 covered arrays 或视频内全量词表收集。补全 item 的 `learning_units=[]`，因此只进入 video serving state，不进入 unit serving state。若系统已有 `catalog.video_user_states`，Recommendation 可将其作为可选消费状态输入，用于 penalty，但这些消费真相不应写回 Recommendation 自己的表。

Serving state 写入必须使用数据库侧原子递增：每次最终 selection 对每个 distinct unit / video 贡献一次 `served_count + 1`，由 `INSERT ... ON CONFLICT DO UPDATE SET served_count = existing.served_count + 1` 完成。应用层不允许先读取旧 `served_count`、加一、再覆盖写回，避免并发推荐请求造成 lost update。`last_served_at` 与 `last_run_id` 写入本次 selection 的时间和 run id。

#### 7.10.1 `catalog.video_user_states` 允许读取字段清单

`catalog.video_user_states` 在当前 MVP 中为可选消费状态输入，不是 Recommendation 的硬依赖。若启用该输入，Recommendation 当前只允许读取以下字段用于轻量 penalty：

- `last_watched_at`
- `watch_count`
- `completed_count`
- `last_position_ms`
- `max_position_ms`

Recommendation 不得要求 Catalog 为其增加推荐曝光相关字段；推荐曝光与冷却状态继续由 `recommendation.user_video_serving_states` 维护。`last_watch_ratio` 与 `max_watch_ratio` 不再作为持久化字段读取；如果 Ranker 需要观看比例，应使用 `catalog.video_user_states.max_position_ms / VideoCandidate.duration_ms` 在 Recommendation 内部派生，且 `duration_ms` 必须来自视频候选内容属性而不是用户状态 reader。

### 7.11 Audit Writer

Audit Writer 记录每次 video recommendation run 及其 items。run 级至少记录：`run_id`、`user_id`、`request_context`、`session_mode`、`selector_mode`、`planner_snapshot`、`lane_budget_snapshot`、`candidate_summary`、`underfilled`、`result_count`。`candidate_summary` 额外记录 queue/scope 指标、recall row count、video state lookup count、`learning_selected_count`、`mastered_target_fill_count`、`popular_fill_count`、`fill_triggered` 与 `pipeline_timing_ms`。其中 scope 指标使用 `planner_scope_unit_count`、`planner_scope_unit_count_by_bucket`、`no_supply_scope_unit_count`、`recall_fetch_unit_count` 表达双 scope 语义；`pipeline_timing_ms` 只统计 audit 持久化前的推荐主链路，至少包含 `context_assemble`、`plan`、`candidate_generate`、`evidence_resolve`、`aggregate`、`video_state_enrich`、`rank`、`select`、`fill`、`final_item_build`、`total`，不统计 audit / serving state 写入耗时。item 级至少记录：`video_id`、`rank`、`score`、`primary_lane`、`dominant_role`、`dominant_unit_id`、`reason_codes`、`learning_units`。补全 item 的 `primary_lane` 是 `mastered_target_fill` 或 `popular_fill`，`dominant_unit_id=null`，`learning_units=[]`。`learning_units` 以 JSONB 保存完整前端学习计划快照，covered counts 与 video-level best evidence 均从该字段派生，不作为持久化字段双写。

## 8. Recommendation 自有数据层

`recommendation.v_video_unit_recall_index` 是 Recommendation own 的预计算反向索引，也是当前视频-学习单元级候选生成的唯一权威读模型。其 owner 为 Recommendation；其上游事实来源为：

- `catalog.video_unit_index`
- `catalog.videos`
- `catalog.video_transcripts`

最小字段集合固定为：

- `video_id`
- `coarse_unit_id`
- `mention_count`
- `sentence_count`
- `coverage_ms`
- `coverage_ratio`
- `sentence_indexes`
- `best_evidence_sentence_index`
- `best_evidence_span_index`
- `best_evidence_start_ms`
- `best_evidence_end_ms`
- `best_evidence_scores`
- `best_evidence_question_reject_reason`
- `best_evidence_selection_reason`
- `best_evidence_candidate_score`
- `best_evidence_target_text`
- `duration_ms`
- `mapped_span_ratio`
- `content_quality_score`
- `rank_within_unit`
- `status`
- `visibility_status`
- `publish_at`

过滤规则固定为：

- `status = 'active'`
- `visibility_status = 'public'`
- `publish_at is null or publish_at <= now()`

刷新策略固定为：

- 默认每日离线刷新一次；
- 内容导入成功后可按需触发增量刷新；
- Recommendation 不依赖该视图的实时强一致更新。

`recommendation.v_unit_video_inventory` 是 Demand Planner 的库存感知视图，建议字段包括：`coarse_unit_id`、`distinct_video_count`、`avg_mention_count`、`avg_sentence_count`、`avg_coverage_ms`、`avg_coverage_ratio`、`strong_video_count`、`supply_grade`、`updated_at`。这层视图让 planner 知道“该学什么”和“视频库接不接得住”之间的关系。`strong_video_count` 为 Recommendation 在当前 MVP 中基于确定性内容事实计算得到的库存评估指标，不是 Catalog 的原始事实字段；后续即使替换其判定规则，也不改变该视图作为 Recommendation own 读模型的 owner 归属。

`recommendation.user_unit_serving_states` 维护 unit 级 serving state。`recommendation.user_video_serving_states` 维护 video 级 serving state。`recommendation.video_recommendation_runs` 与 `recommendation.video_recommendation_items` 维护主审计对象。由于最终输出是 video learning plan，因此 Recommendation 的审计中心也必须是 video recommendation 维度，并保存每个 item 的 `learning_units` 快照，而不是只围绕 scheduler item。

## 9. 外部接口与内部接口

Recommendation 对外只暴露一个主用例：`GenerateVideoRecommendations`。请求结构至少包含 `user_id`、`target_video_count` 与 `request_context`。返回结构是精简 plan：`run_id` 与 `items[]`；每个 item 只包含 `video_id`、`duration_ms`、`learning_units`。`learning_units[]` 的单项包含 `coarse_unit_id`、`role`、`is_primary`、`evidence`。`selector_mode`、`underfilled`、`rank`、`score`、`reason_codes`、`explanation` 不进入 response，只保留在 audit。

内部接口建议分为读取接口与模块接口两类。读取接口包括：`LearningStateReader`、`RecommendableVideoUnitReader`、`VideoFillCandidateReader`、`UnitInventoryReader`、`UnitServingStateRepository`、`VideoServingStateRepository`、`RecommendationAuditRepository`。模块接口包括：`DemandPlanner`、`CandidateGenerator`、`EvidenceResolver`、`VideoEvidenceAggregator`、`VideoRanker`、`VideoSelector`、`VideoFillService`、`ExplanationBuilder`。只有 Context Assembler、Candidate Generator、Video Fill Service、Serving State Manager、Audit Writer 直接接触 repository，其余模块只消费领域对象，避免业务规则与 SQL 混杂。

## 10. 事务、一致性与并发

Recommendation 的只读计算过程不应包在长事务里。推荐做法是：先完成所有只读装配与纯计算，最后只在写 `video_recommendation_runs`、`video_recommendation_items`、`user_unit_serving_states`、`user_video_serving_states` 时开启短事务。这样既保证写入一致性，又避免长时间锁用户或锁表。

同一用户短时间内多次请求 Recommendation，MVP 不强制做全局串行，但建议每次请求以独立 `run_id` 区分，serving state 使用原子 upsert 递增计数。若某些场景需要更强一致性，可以对单用户的 serving state 写入增加轻量互斥。Recommendation 与 Learning engine 之间不做跨模块分布式事务，接受边界时间点的短暂最终一致。

## 11. 性能与预计算

`v_video_unit_recall_index`、`v_unit_video_inventory` 与 `user_unit_recall_queue` 都应预计算或 projection 化，避免在热路径中频繁临时 join / group by。在线推荐先把本轮 `planner_scope` 限制为 `min(max(target_video_count * 12, 64), 200)`，再按每个 `recall_fetch_scope` unit `min(max(target_video_count * 4, 20), 50)` 读取 recall rows。默认 bucket 配额是 `hard_review 40% / new_now 30% / soft_review 20% / near_future 10%`；hard backlog 大时调整为 `50% / 25% / 15% / 10%`，bucket 不足时按全局 dynamic priority 回填。`supply_grade='none'` 的 unit 只允许少量进入 `planner_scope` 保留低供给信号，当前 cap 为 `min(8, max(1, planner_scope_limit / 10))`，且不进入 `recall_fetch_scope`。SQL 层必须分开 rank supplied / no-supply candidates，避免 no-supply 在截断阶段挤出有供给 units；Go 层 cap 只是第二道保险。Evidence Resolver 不再回查 spans/sentences，只把 recall row 中的 best evidence index 和 start/end 透传给 Aggregator。Video fill 每类最多一次小查询，limit 为 `min(gap * 3, 30)`。

## 13. 测试设计

单测必须覆盖：Demand Planner 的 bucket precedence、hard/new/soft/future 分桶、supply-aware `new_now`、lane budget 动态调整；Candidate Generator 的 `exact_core`、`bundle` 动态放宽、`soft_future`、fallback 触发条件；Evidence Resolver 直接透传 recall row 的 `best_evidence_ref/start/end` 与缺失边界处理；Aggregator 的去重、unit-level evidence 选择、四类 role 覆盖统计；Ranker 的 `demand_coverage`、`bundle_value_score` 不重复计分、`coverage_strength_score`、overload penalty；Selector 的三种模式、主线覆盖约束、future/fallback 比例与 under-fill；Explanation Builder 的 reason code 与模板拼接。

集成测试至少覆盖以下场景：正常库存 + 正常复习；`hard_review` 供给弱；极度稀疏库存；长视频反复命中同一 unit；future 候选很多但 hard_review 也很多；`best_evidence_*` 与 span 表不一致。Golden tests 应固定若干输入输出快照，包括 `planner_snapshot`、`candidate_summary`、最终 `video_ids` 顺序、`reason_codes`、`learning_units` 与 unit-level evidence。

## 14. 非目标与后续扩展口

当前 Recommendation 明确不做 task layer 进入主流程、非视频 fallback task、embedding / semantic recall、ML ranking、完整 curriculum / backlog 平台化、复杂个性化兴趣模型、Catalog 层的高层语义 enrichment。当前只保留轻量能力：frontier/near-future 信号、bundle lane、inventory-aware planner、简单 diversity selector、quality fallback。未来如果 Catalog 增加更高层的内容 enrichment pipeline，Recommendation 只需要扩展 Planner / Candidate Generator / Aggregator 使用的新特征，而不需要推翻这版模块边界。

## 15. 总结

Recommendation 的最终职责可以概括成一句话：

> 基于 Learning engine 提供的学习状态、Catalog 提供的 `video_unit_index + best_evidence + video_semantic_spans + video_transcript_sentences` 内容索引，以及 Recommendation 自己的 serving state，先生成供给感知的学习需求画像，再通过多路 video-unit 候选生成构造健康的视频候选池，随后精确解析 best evidence、在 video 级完成证据聚合、规则化排序与选择，最终稳定输出一组当前足够合理的视频列表，并以 video recommendation run/item 为主对象完成审计与冷却状态更新。

因此，在这份设计里：

- Learning engine 只负责学习事件与学习状态；
- Catalog 只负责内容事实与 Recall-ready 索引；
- Recommendation 只负责视频推荐主链路；
- Recommendation 内部被拆成边界明确、可独立测试、可独立替换的小模块；
- 对外始终只输出 video list；
- 内部通过 Demand Planner、Multi-lane Candidate Generator、Evidence Resolver、Video Evidence Aggregator、Video Ranker、Video Selector、Video Fill Service 构成一条完整、可解释、可审计的教育视频推荐流水线。
