# 练习题绑定与生成设计

## 0. 文档信息

文档状态：MVP 设计草案  
适用范围：视频学习场景中的练习题绑定、题源选择、上下文溯源与学习反馈回写  
当前约束：本文件暂不详细确定数据库表结构、具体题型集合、AI 生成 prompt、审核流或批处理实现方式。

本文档的目标是先把练习题应该和什么绑定说清楚，避免过早把题目建模成“每个视频的所有词都预生成题”，也避免完全脱离视频上下文做成纯通用词库。

## 1. 背景

当前视频学习产品形态已经确定的部分是：

- 前端会给用户分发学习视频；
- 视频包含可点击字幕；
- 点击 token 后会弹出上下文释义或字典义；
- Recommendation 返回的每个视频会携带本轮预期学习的 `learning_units`；
- Learning engine 通过 `RecordLearningEvents` 接收学习事件，`quiz` 属于强事件，最终仍以 `coarse_unit_id` 更新用户对某个学习单元的状态。

因此练习题设计必须同时满足两件事：

第一，题目要能利用视频上下文，尤其是刚看过的字幕句子，否则视频学习闭环会弱。第二，题目不能因为视频里出现大量可学习词而在视频 AI pipeline 中全量预生成，否则 AI 成本、存储成本和后续维护成本都会失控。

## 2. 核心结论

练习题分为两类题源：

1. **视频上下文题**：主要查找维度是 `video_id + coarse_unit_id`，题目内容记录一个被选中的上下文锚点。
2. **通用单元题**：主要查找维度是 `coarse_unit_id`，不依赖具体视频上下文。

查询时优先使用视频上下文题；如果没有可用题，再 fallback 到通用单元题。

关键边界是：上下文锚点不是题目的主查询 key。也就是说，不应该为同一个 unit 在同一个视频里的每一次出现都生成题目。`evidence_ref` 只用于记录“这道题基于视频里的哪一句、哪个 span 生成”，用于溯源、定位、回放和失效判断。

```text
学习状态绑定：coarse_unit_id
上下文题主查询：video_id + coarse_unit_id
上下文题溯源：selected evidence_ref
通用题主查询：coarse_unit_id
```

## 3. 两类题源

### 3.1 视频上下文题

视频上下文题用于回答：“这个用户刚看的这个视频里，这个学习单元在当前语境下应该怎么理解或回忆？”

它的主要绑定关系是：

```text
video_id + coarse_unit_id
```

题目内部还应记录一个上下文锚点，例如：

```text
selected evidence_ref:
- sentence_index
- span_index
- start_ms
- end_ms
- context text snapshot
```

这里的 `selected evidence_ref` 是题目生成和展示的上下文来源，不是默认的题目唯一维度。对于同一个 `video_id + coarse_unit_id`，MVP 默认最多准备一题；如果后续需要多题，也应该通过题型、难度或语义差异来区分，而不是通过每个 evidence 出现位置来扩张。

视频上下文题适合的场景包括：

- 视频结尾小测；
- 学习模式下的即时练习；
- 用户 lookup 某个 expected unit 后的轻量练习；
- 需要回跳到视频字幕位置的题目。

### 3.2 通用单元题

通用单元题用于回答：“这个学习单元本身应该如何被练习或复习？”

它的主要绑定关系是：

```text
coarse_unit_id
```

通用题不依赖具体视频，因此复用率高，适合作为 fallback 题源，也适合作为独立复习、错题复习或未来非视频学习形态的基础。

通用单元题适合的场景包括：

- 视频上下文题缺失时的 fallback；
- 独立复习；
- 长期记忆复习；
- 多视频共享的基础释义、拼写、听写或选择题。

## 4. 查询与组题流程

练习模块不应直接遍历视频里的所有词，而应从 Recommendation 返回的 `learning_units` 出发。

推荐流程如下：

```text
Recommendation 返回 video + learning_units
        ↓
Practice Planner 选择本轮要练的 units
        ↓
优先查询 video_id + coarse_unit_id 的视频上下文题
        ↓
不足时 fallback 查询 coarse_unit_id 的通用单元题
        ↓
仍不足时按策略临时生成、延迟生成或跳过
        ↓
前端展示练习题
        ↓
答题结果通过 RecordLearningEvents 写入 Learning engine
```

选择本轮要练的 units 时，MVP 可以按以下优先级：

1. `learning_units.is_primary = true` 的 unit；
2. 用户在本视频中 lookup 过的 expected unit；
3. role 为 `hard_review` 或 `new_now` 的 unit；
4. role 为 `soft_review` 的 unit；
5. role 为 `near_future` 的 unit。

这个优先级只定义练习选择，不改变 Recommendation 的排序和选择逻辑。

## 5. 上下文锚点选择原则

对于一个 `video_id + coarse_unit_id`，视频上下文题只需要选择一个代表性上下文锚点。这个选择可以发生在视频 AI pipeline、推荐后异步生成任务，或首次请求练习题时。

上下文锚点选择规则建议保持简单：

- 句子文本完整，用户不需要额外上下文也能理解；
- 目标 token 或短语边界清楚；
- start/end 时间可用于前端跳转或高亮；
- 语义足够典型，适合作为该 unit 在本视频中的代表用法；
- 避免过短、过长、代词过多、信息缺失或噪声高的句子；
- 多个候选差不多时，选择证据强度高或出现更早的候选。

这与 Recommendation 的 `learning_units[].evidence` 可以复用思路，但二者不必完全相同。Recommendation 的 evidence 是本轮推荐解释与前端学习计划的证据；练习题的 context anchor 是题目生成所采用的上下文来源。MVP 可以先直接复用 Recommendation 提供的 unit-level evidence，后续再引入独立的题目上下文选择器。

## 6. 生成策略

### 6.1 不建议全量预生成

视频 AI pipeline 不应为视频中所有可能学习的词都预生成上下文题。原因是：

- 大量题目不会被当前用户或当前推荐策略使用；
- 同一 unit 在多个视频里会产生大量重复或低价值题；
- 字幕、span、mapping 或 evidence 修正后，题目失效面会扩大；
- AI 生成与存储成本会随视频词量线性膨胀。

### 6.2 推荐的 MVP 生成方式

MVP 可以采用混合策略：

```text
通用单元题：按 coarse_unit_id 预生成或人工维护
视频上下文题：只对高价值 video + unit 预生成，或按需生成后缓存
```

如果视频 AI pipeline 需要提前生成视频上下文题，也应限制范围：

- 只针对高价值 unit；
- 每个视频只取 top N 个候选 unit；
- 每个 `video_id + coarse_unit_id` 默认最多生成一题；
- 只为有稳定 context anchor 的 unit 生成；
- 对 `learning_units.is_primary` 和高频被推荐的 unit 提高优先级。

如果采用按需生成，则触发时机可以是：

- 用户打开视频后，后台为本视频 `learning_units` 预取题目；
- 用户 lookup 某个 expected unit 后异步生成；
- 视频播放接近结束时，为结尾小测补齐题目；
- 服务端批处理根据最近推荐 run 的 `learning_units` 补题。

## 7. 题目选择接口的概念边界

后续可以设计一个独立的 Practice / Exercise 模块，但它不应改变当前三域边界：

- Catalog 负责内容事实、字幕、span、视频索引；
- Recommendation 负责返回本轮视频学习计划 `learning_units`；
- Learning engine 负责接收学习事件并维护学习状态；
- Practice / Exercise 负责选择题源、生成题目、组题和记录题目尝试。

Practice / Exercise 的核心输入可以是：

```text
video_id
recommendation_run_id
learning_units[]
session signals:
- lookup units
- playback progress
- desired question count
- practice mode
```

核心输出可以是：

```text
practice_items[]
```

每个 practice item 至少应能表达：

- 考察哪个 `coarse_unit_id`；
- 题源是视频上下文题还是通用单元题；
- 如果是视频上下文题，使用了哪个 context anchor；
- 前端如何展示题干、选项、答案验证信息；
- 答题后如何构造 Learning engine 事件。

这里不在本文档中确定具体 DTO 和数据库字段。

## 8. 学习反馈回写

无论题目来自视频上下文题还是通用单元题，答题结果写回 Learning engine 时，核心绑定仍然是：

```text
user_id + coarse_unit_id
```

答题行为应写为 `quiz` 事件。`source_type` 可以使用 `quiz_session` 或后续更明确的练习来源类型；题目 id、视频 id、推荐 run id、context anchor、题型等信息进入 `source_ref_id` 或 `metadata`。

示例语义：

```json
{
  "coarse_unit_id": 101,
  "event_type": "quiz",
  "source_type": "quiz_session",
  "source_ref_id": "exercise_attempt_xxx",
  "quality": 4,
  "is_correct": true,
  "metadata": {
    "question_source": "video_context",
    "video_id": "video_xxx",
    "recommendation_run_id": "run_xxx",
    "sentence_index": 12,
    "span_index": 3,
    "exercise_type": "meaning_in_context"
  }
}
```

Learning engine 不需要知道题目是如何生成的，也不需要直接读取题库。它只关心标准化学习事件。

## 9. 暂不确定的设计点

以下内容本轮不定稿：

- 题目表是否一张表还是多张表；
- 题目 attempt 是否独立持久化，以及如何和 Learning engine event 对应；
- 视频上下文题具体有哪些题型；
- 通用单元题具体有哪些题型；
- 题目生成是否完全 AI、模板优先还是人工审核优先；
- 题目质量评分、审核状态和失效重建策略；
- context anchor 是否直接复用 Recommendation evidence，还是由 Practice 模块重新选择；
- 是否需要题目版本、transcript version、semantic mapping version 等细粒度失效控制。

当前只确定绑定原则：

```text
视频上下文题：主查 video_id + coarse_unit_id，记录 selected evidence_ref 作为上下文溯源
通用单元题：主查 coarse_unit_id
答题反馈：统一写 Learning engine 的 quiz/review 类强事件，核心绑定 coarse_unit_id
```

## 10. MVP 决策摘要

1. 练习题不应该只绑定 video，也不应该只绑定词。
2. 视频上下文题的主查询维度是 `video_id + coarse_unit_id`。
3. `evidence_ref` 是上下文锚点和溯源信息，不是默认生成粒度。
4. 通用单元题的主查询维度是 `coarse_unit_id`。
5. 查询时优先使用视频上下文题，缺失时 fallback 到通用单元题。
6. 视频 AI pipeline 不应为全量 token 预生成题；最多只做受控 top N 或按需生成缓存。
7. 题目形态、生成方式和具体存储方案后置，不在当前文档定死。
8. 答题结果最终仍通过 Learning engine 标准事件进入学习状态归约。
