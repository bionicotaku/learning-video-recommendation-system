# 模块统一文件结构规范

## 1. 文档目的

本文档定义仓库内业务模块和子模块的统一文件结构标准。

目标是：

> 任何新实现都先服从统一结构，再落具体代码。

这是一份**先于实现存在**的规范文档。  
具体实现应依赖本规范；本规范不追随某一版现有目录做事后解释。

## 2. 适用范围

本文档适用于：

- `internal/` 下的顶层业务模块
- 顶层业务模块下的子模块
- 模块内的分层目录
- 新增模块、新增子模块、结构重构、目录扩展

本文档不要求为了整齐补空目录。  
要求是：

- 目录命名必须统一
- 职责落点必须统一
- 一旦某类职责出现，必须落到标准目录

## 3. 结构原则

### 3.1 先统一骨架，再写实现

新增模块时，先确定：

- 这是顶层模块还是子模块
- 它是否会继续横向拆分
- 它的实现应该落在哪一层

不要先把文件写出来，再回头解释为什么这样摆。

### 3.2 同层模块使用同一套结构

同一层级的模块，应使用同一套骨架。

例如：

- 顶层模块和顶层模块之间
- 子模块和子模块之间

都应尽量保持同样的目录组织方式。

### 3.3 模块根负责边界，子模块负责实现

如果一个模块未来会承载多块独立能力，则模块根应主要负责：

- 模块边界
- 子模块组织
- 模块级说明

而具体实现应下沉到子模块。

### 3.4 同一职责只允许一个标准落点

同一种职责，在整个仓库中应只有一个标准目录。

例如：

- 用例编排固定放 `application/usecase`
- 领域模型固定放 `domain/model`
- SQL 固定放 `infrastructure/persistence/query`
- `sqlc` 生成代码固定放 `infrastructure/persistence/sqlcgen`

不要在不同模块里反复发明：

- `handlers`
- `managers`
- `impl`
- `sql`
- `dbsql`
- `misc`
- `helper2`

这类非标准目录。

### 3.5 目录启用按需，不强制占位

本规范定义的是完整骨架，不是要求所有目录都立即存在。

规则是：

- 某类职责当前不存在，就不要为了形式补空目录
- 某类职责一旦出现，就必须落到标准目录

## 4. 统一层级模型

仓库内的业务结构统一按三层理解：

1. 顶层模块
2. 子模块
3. 分层目录

### 4.1 顶层模块

顶层模块负责：

- 定义业务边界
- 定义 owner
- 定义对外协作关系
- 决定是否继续拆子模块

### 4.2 子模块

子模块负责：

- 承载一块可独立理解的实现能力
- 按统一分层组织代码

### 4.3 分层目录

真正承载实现的模块或子模块，统一优先按以下四层组织：

- `application`
- `domain`
- `infrastructure`
- `test`

## 5. 顶层模块标准骨架

### 5.1 模块根型

如果顶层模块主要负责组织子模块，标准骨架如下：

```text
internal/<module>/
  README.md
  doc.go
  <submodule-a>/
  <submodule-b>/
```

适用条件：

- 该模块未来会继续横向扩展
- 不止一块独立能力会落在该模块下

### 5.2 直接实现型

如果顶层模块本身就是一块稳定能力，且短期内不需要继续拆多个子模块，标准骨架如下：

```text
internal/<module>/
  README.md
  doc.go
  application/
  domain/
  infrastructure/
  test/
```

适用条件：

- 模块边界单一
- 实现集中
- 没有明显的多子能力拆分需求

### 5.3 顶层模块必须有的文件

#### `README.md`

必须有。

用途：

- 解释模块职责
- 解释模块边界
- 解释 owner
- 指向实现入口

#### `doc.go`

建议有。

用途：

- 提供稳定的包说明
- 明确根包语义

## 6. 子模块标准骨架

任何承载具体实现的子模块，统一使用以下骨架：

```text
internal/<module>/<submodule>/
  README.md
  doc.go
  application/
  domain/
  infrastructure/
  test/
```

后续所有新子模块，默认先用这一套，不重新设计第二套结构。

## 7. 分层目录标准

### 7.1 `application/`

职责：

- 用例编排
- 输入命令
- 输出 DTO
- 依赖 repository port
- 不承载纯领域规则

标准骨架：

```text
application/
  doc.go
  command/
  dto/
  query/
  repository/
  service/
  usecase/
```

#### `command/`

放 use case 输入对象。

#### `dto/`

放 use case 输出对象。

#### `query/`

放 application 层读模型或候选结构。

只有当模块存在“偏用例输入视图”的结构时才启用。

#### `repository/`

放 application 依赖的 port interface。

规则：

- 这里定义接口
- `infrastructure` 负责实现

#### `service/`

放 application 层辅助服务。

适用场景：

- 需要辅助用例编排
- 但不属于纯领域规则

#### `usecase/`

放主业务入口。

### 7.2 `domain/`

职责：

- 纯业务规则
- 不依赖数据库实现
- 不依赖 `sqlc` 生成代码
- 不依赖具体 repository 实现

标准骨架：

```text
domain/
  doc.go
  aggregate/
  enum/
  model/
  policy/
  rule/
  service/
```

#### `aggregate/`

放聚合根或 reducer。

只有当模块存在明确的聚合中心时启用。

#### `enum/`

放稳定枚举。

#### `model/`

放领域模型。

#### `policy/`

放固定策略、默认参数、阈值定义。

#### `rule/`

放原子规则处理器。

适用场景：

- 单步状态处理
- 原子业务约束
- 事件分类处理

#### `service/`

放可复用的领域计算器或纯规则服务。

例如：

- scorer
- allocator
- assembler
- transitioner
- updater

### 7.3 `infrastructure/`

职责：

- 技术实现
- 配置
- 数据库连接
- migration
- SQL
- repository 实现
- 事务实现

标准骨架：

```text
infrastructure/
  doc.go
  config.go
  db.go
  migration/
  persistence/
    mapper/
    query/
    queryctx/
    repository/
    schema/
    sqlcgen/
    tx/
```

#### `config.go`

放模块基础配置读取和校验。

#### `db.go`

放数据库连接池初始化。

#### `migration/`

放该模块自己的 migration。

规则：

- 只定义本模块 owner 的数据库对象
- 不跨模块定义别人的表

#### `persistence/mapper/`

放持久化类型和领域对象之间的转换。

#### `persistence/query/`

放 SQL 文件。

#### `persistence/queryctx/`

放事务上下文和 querier 解析。

#### `persistence/repository/`

放 repository interface 的具体实现。

#### `persistence/schema/`

放 `sqlc` 需要的外部 schema 补充定义。

#### `persistence/sqlcgen/`

放 `sqlc` 生成代码。

规则：

- 只生成，不手改

#### `persistence/tx/`

放事务管理器实现。

### 7.4 `test/`

职责：

- 单元测试
- 模块级测试辅助
- 集成测试
- 场景测试

标准骨架：

```text
test/
  doc.go
  unit/
  integration/
```

扩展骨架：

```text
test/
  doc.go
  unit/
  integration/
  scenario/
  fixture/
```

是否启用扩展目录，按真实需要决定。

#### `unit/`

放单元测试。

规则：

- `unit/` 下的目录结构应尽量镜像源码路径
- 例如 `domain/service/review_scorer.go` 的单测应落到 `test/unit/domain/service/review_scorer_test.go`
- 单元测试默认测试导出行为和稳定边界
- 如果未导出逻辑复杂到必须单测，应优先提炼成可命名的领域对象或服务，而不是把测试放回业务目录

#### `integration/`

放真实数据库、真实事务、真实跨层编排相关测试。

#### `scenario/`

放跨多个用例或更长链路的业务场景测试。

#### `fixture/`

放测试夹具、测试数据构造器、共享测试辅助资源。

## 8. 目录启用规则

### 8.1 先统一命名和落点

一旦某类职责出现，必须优先落到标准目录。

例如：

- 出现用例输入对象，就放 `application/command`
- 出现 SQL，就放 `infrastructure/persistence/query`
- 出现事务实现，就放 `infrastructure/persistence/tx`

### 8.2 不要求提前补空目录

如果当前没有：

- `application/service`
- `domain/aggregate`
- `domain/policy`
- `test/scenario`

那就不要为了形式统一补空目录。

### 8.3 不允许出现第二套落点

例如已经有：

- `domain/service`

就不要再发明：

- `domain/services`
- `domain/helper`
- `domain/manager`

## 9. 文件命名规则

### 9.1 文件名按职责命名

优先采用：

- `generate_xxx.go`
- `record_xxx.go`
- `quota_allocator.go`
- `review_scorer.go`

不要优先采用：

- `utils.go`
- `common.go`
- `manager.go`
- `helper.go`

除非文件内容确实只承担极少量共享辅助职责。

### 9.2 测试命名规则

- 业务代码目录下禁止出现 `*_test.go`
- `application/`、`domain/`、`infrastructure/` 只放业务实现，不混放测试文件
- 单元测试统一放 `test/unit`
- 集成测试统一放 `test/integration`
- 场景测试统一放 `test/scenario`
- `test/` 下的目录结构应尽量镜像源码路径
- 只有极少数必须偏离本规则的情况才允许例外，且必须在对应模块 `README.md` 中说明原因、边界和维护约束

### 9.3 文档文件规则

- 模块根必须优先有 `README.md`
- 模块根建议有 `doc.go`
- 承载实现的关键子模块建议有自己的 `README.md`
- 关键层目录建议保留 `doc.go`

## 10. 新增模块时的标准流程

以后新增模块或子模块，统一按下面顺序执行：

1. 先确认它是顶层模块还是子模块
2. 再确定使用“模块根型”还是“直接实现型”
3. 按本规范创建当前真实需要的目录
4. 同时补 `README.md`
5. 如涉及数据库，再补 `migration / query / sqlcgen`
6. 最后把入口加入文档导航

## 11. 不该做什么

以后继续扩展时，不要做这些事：

- 不要每个模块自己发明一套目录命名
- 不要同类职责出现多个落点
- 不要把模块根和子模块实现混在一起
- 不要为了整齐大量创建空目录
- 不要在没有标准说明的情况下临时堆 `utils`、`misc`、`impl`、`helper` 目录

## 12. 文档定位

这份文档只负责定义：

- 模块骨架
- 子模块骨架
- 分层目录标准
- 目录启用规则

它不负责：

- 解释具体业务规则
- 解释某个现有模块当前实现细节
- 为当前代码现状做事后背书

业务设计、工程实现和目录级说明，应由其他文档分别承担。
