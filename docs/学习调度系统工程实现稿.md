# 学习内容推荐模块工程实现稿（Go / PostgreSQL / sqlc）

## 1. 工程目标

本文档只讨论 **Recommendation 模块** 的工程实现。

当前工程目标已经落地为：

- Recommendation 只保留“读状态、算推荐、写审计、写 serving state”
- Recommendation 不再写 Learning engine 的业务表

Learning engine 的工程设计已单独抽离到：

- [学习引擎设计文档](/Users/evan/Downloads/learning-video-recommendation-system/docs/学习引擎设计文档.md)

## 2. 唯一采用的技术方案

- Go
- PostgreSQL 直连
- `pgx/v5`
- `sqlc`
- 纯 application 编排 + domain 规则 + infrastructure 持久化

Recommendation 仍然坚持：

- 不手写动态字符串 SQL
- 不把评分公式塞进 SQL
- 不让 repository 承担业务规则

## 4. Recommendation 分层职责

## 4.1 `domain/`

Recommendation 领域层只保留推荐逻辑：

- `BacklogCalculator`
- `QuotaAllocator`
- `ReviewScorer`
- `NewScorer`
- `PriorityZeroExtractor`
- `RecommendationAssembler`

## 4.2 `application/`

Recommendation application 层只保留：

- `GenerateLearningUnitRecommendations`

它负责：

- 读取候选
- 计算 quota
- 打分
- 组装 batch
- 写审计
- 更新 serving state

## 4.3 `infrastructure/`

这里只负责：

- 读 `learning.user_unit_states`
- 读 `semantic.coarse_unit`
- 读写 `recommendation.*`

不再负责：

- 读写 `learning.unit_learning_events`
- 读写 Learning engine replay 相关内容

## 6. 持久化接口设计

## 6.1 读取 Learning engine 的 repository

Recommendation 只需要面向读取的接口。

### `UserUnitStateReadRepository`

建议接口：

```go
type UserUnitStateReadRepository interface {
    FindDueReviewCandidates(ctx context.Context, userID uuid.UUID, now time.Time) ([]query.ReviewCandidate, error)
    FindNewCandidates(ctx context.Context, userID uuid.UUID) ([]query.NewCandidate, error)
}
```

注意：

- Recommendation 不再拥有 `Upsert` / `BatchUpsert` / `Delete` 这类写 Learning engine 状态的能力

## 6.2 MVP 不支持用户级调度配置

当前工程实现明确收敛为：

- 不创建用户级调度配置表
- `GenerateLearningUnitRecommendations` 不读取用户级配置
- quota / limit 统一来自 Recommendation 模块内默认值

当前默认值集中在：

- [recommendation_defaults.go](/Users/evan/Downloads/learning-video-recommendation-system/internal/recommendation/domain/model/recommendation_defaults.go)

当前 use case 只使用默认配置：

- [generate_recommendations.go](/Users/evan/Downloads/learning-video-recommendation-system/internal/recommendation/application/usecase/generate_recommendations.go)

这不是实现缺失，而是 MVP 设计约束。这样可以先把 Recommendation 稳定在：

- 读 Learning engine 状态
- 计算候选与 quota
- 生成推荐批次
- 写 serving state
- 写推荐审计

而不在当前阶段引入额外的用户级配置 owner 和配置管理链路。

MVP 之后如果需要扩展，建议仍由 Recommendation 自己维护配置。最小方案是新增一张 Recommendation 自有配置表，并在 use case 中做 `GetOrDefault`，只影响：

- `session_default_limit`
- `daily_new_unit_quota`
- `daily_review_soft_limit`
- `daily_review_hard_limit`
- `timezone`

不改变当前评分器、配额器和推荐审计的主体结构。

## 7. SQL 设计

## 7.1 Candidate query

candidate query 的关键变化是：

- `last_recommended_at` 不再从 `learning.user_unit_states` 取
- 应从 `recommendation.user_unit_serving_states` 左连接

### Review candidate SQL 原则

- 主表：`learning.user_unit_states`
- 联表：`semantic.coarse_unit`
- 左连接：`recommendation.user_unit_serving_states`
- 条件：
  - `is_target = true`
  - `status in ('learning', 'reviewing', 'mastered')`
  - `next_review_at <= now`

### New candidate SQL 原则

- 主表：`learning.user_unit_states`
- 联表：`semantic.coarse_unit`
- 左连接：`recommendation.user_unit_serving_states`
- 条件：
  - `is_target = true`
  - `status = 'new'`

## 7.2 Recommendation 自有写入 SQL

需要三类写入：

1. 插入 `recommendation.scheduler_runs`
2. 插入 `recommendation.scheduler_run_items`
3. upsert `recommendation.user_unit_serving_states`

第三类很关键，因为 Recommendation 的 `last_recommended_at` 闭环依赖它。

## 8. Recommendation 领域服务设计

## 8.1 `BacklogCalculator`

逻辑保持不变：

```text
backlog = due review 候选数
```

## 8.2 `QuotaAllocator`

仍然按当前 MVP 区间规则：

- `R = 0`
- `1..5`
- `6..20`
- `soft < R <= hard`
- `R > hard`

这部分逻辑不需要因为模块拆分而改变。

## 8.3 `ReviewScorer`

评分公式保留：

```text
review_score =
  0.45 * overdue_score
  + 0.25 * target_priority
  + 0.20 * weak_memory_score
  + 0.10 * recency_adjustment
```

唯一变化是：

- `recency_adjustment` 的输入来源改为 `recommendation.user_unit_serving_states.last_recommended_at`

## 8.4 `NewScorer`

评分公式保留：

```text
new_score =
  0.75 * target_priority
  + 0.15 * freshness_score
  + 0.10 * not_recently_recommended
```

同样：

- `not_recently_recommended` 的输入来源改为 `recommendation.user_unit_serving_states.last_recommended_at`

## 8.5 `PriorityZeroExtractor`

逻辑保留：

- `status=learning` 的 due review
- `last_quality <= 2` 的最近失败内容

## 8.6 `RecommendationAssembler`

组装顺序保留：

1. P0 review
2. scored reviews
3. scored news

但 assembler 本身不应该负责更新 serving state。

serving state 的写入应放在 usecase 编排层。

## 9. Recommendation 应用层用例设计

## 9.1 `GenerateLearningUnitRecommendations`

这是 Recommendation 模块唯一主用例。

### 输入 DTO

建议保留：

```go
type GenerateRecommendationsCommand struct {
    UserID         uuid.UUID
    RequestedLimit int
    Now            time.Time
    RequestContext map[string]any
}
```

### 输出 DTO

保留：

```go
type GenerateRecommendationsResult struct {
    Batch model.RecommendationBatch
}
```

### 流程

```text
1. 读取用户推荐设置或默认值
2. 读取 due review candidates
3. 读取 new candidates
4. 计算 backlog
5. 分配 quota
6. 对 review/new 进行评分
7. 提取 priority-0
8. 组装 RecommendationBatch
9. 保存 scheduler_runs
10. 保存 scheduler_run_items
11. 批量更新 user_unit_serving_states.last_recommended_at
12. 返回 batch
```

### 关键要求

- 不写 `learning.unit_learning_events`
- 不写 `learning.user_unit_states`
- 审计和 serving state 更新应与 run 结果保持一致
- 如果 run 持久化失败，不应只更新 serving state

## 10. 事务边界

Recommendation 自己需要一个事务边界，但这个事务只覆盖 Recommendation 自己的表：

- `recommendation.scheduler_runs`
- `recommendation.scheduler_run_items`
- `recommendation.user_unit_serving_states`

它不再与 Learning engine 的事务绑定。

这意味着：

- Learning engine 的“写事件 + 写状态”事务是一套
- Recommendation 的“写 run + 写 run items + 写 serving state”事务是另一套

两者平级，互不混用。

## 11. migration 与 schema 策略

Recommendation 需要自己的 schema：

```sql
create schema if not exists recommendation;
```

MVP migration 至少包括：

1. `recommendation.user_unit_serving_states`
2. `recommendation.scheduler_runs`
3. `recommendation.scheduler_run_items`
4. 必要索引

建议索引：

- `recommendation.user_unit_serving_states (user_id, last_recommended_at)`
- `recommendation.scheduler_runs (user_id, generated_at desc)`
- `recommendation.scheduler_run_items (run_id, rank)`

## 12. 测试与验收设计

## 12.1 领域单测

继续保留：

- quota
- review scorer
- new scorer
- priority-0
- assembler

## 12.2 集成测试

Recommendation 集成测试应覆盖：

### 场景 A：新用户初次推荐

- 有多个 `status='new'` 的目标单元
- review 候选为空
- 结果应全部来自 new

### 场景 B：review 优先

- due review backlog 明显高于 0
- quota 应偏向 review

### 场景 C：recently recommended 抑制

- `user_unit_serving_states.last_recommended_at` 很近
- 分数应受 recency 信号影响

### 场景 D：审计落库闭环

- 生成推荐后：
  - `scheduler_runs` 已写入
  - `scheduler_run_items` 已写入
  - `user_unit_serving_states` 已更新

### 场景 E：只读 Learning engine

- Recommendation 执行后：
  - `learning.unit_learning_events` 不变
  - `learning.user_unit_states` 不变

## 14. 最终结论

Recommendation 模块的最终工程定义可以概括为：

> Recommendation 是一个纯读取 Learning engine 业务输入、生成推荐批次并维护 Recommendation 自有投放状态与审计数据的模块。

因此在最终版本里：

- Recommendation 不再维护学习事件
- Recommendation 不再维护学习状态
- Recommendation 只维护：
  - `recommendation.user_unit_serving_states`
  - `recommendation.scheduler_runs`
  - `recommendation.scheduler_run_items`
