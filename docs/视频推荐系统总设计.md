## 1. 文档目标与结论

本文档定义一套**由学习状态驱动、以视频为最终输出、以规则化推荐为核心**的教育推荐系统 MVP。系统当前唯一主目标是：

> **给定一个用户，稳定输出一组当前足够合理的视频列表。**

这里的“合理”有五层含义：

1. 结果不能因为当前 exact target unit 稀疏而塌掉。
2. 结果不能为了补量而跑偏成泛视频推荐。
3. 结果必须可解释、可审计、可调参。
4. 工程上必须可落地，不引入 ML 排序、embedding 召回、重型 curriculum 平台或 task 主流程。
5. 模块边界必须稳定，Learning engine、Catalog、Recommendation 各自 owner 清晰，且彼此依赖方向单向、语义明确。

这套系统的最终整体结构是：

```text
离线内容资产 + transcript JSON
        ↓
      Catalog
        ↓
Learning engine states
        ↓
Recommendation
        ↓
Final Video List
```

更细一点，系统内部的推荐主链路是：

```text
learning.user_unit_states
+ recommendation.v_recommendable_video_units
+ recommendation.v_unit_video_inventory
+ recommendation.user_unit_serving_states
+ recommendation.user_video_serving_states
+ (可选) catalog.video_user_states
+ catalog.video_semantic_spans
+ catalog.video_transcript_sentences
        ↓
Context Assembler
        ↓
Demand Planner
        ↓
Candidate Generator
  - exact_core
  - bundle
  - soft_future
  - quality_fallback
        ↓
Evidence Resolver
        ↓
Video Evidence Aggregator
        ↓
Video Ranker
        ↓
Video Selector
        ↓
Explanation Builder
        ↓
Final Video List
        ↓
video_recommendation_runs / items
```

系统内部使用细粒度内容证据，当前物理落点是 **semantic span / sentence-window evidence**；系统对外始终输出 **video list**。Learning engine 只负责“学得怎么样”，Catalog 只负责“内容事实与可检索索引”，Recommendation 只负责“这次推什么视频”。

## 2. 系统定位

### 2.1 这是一个什么系统

这不是传统信息流推荐系统，也不是单纯的单词记忆工具。
它是一个面向“**通过视频学习学习内容**”场景的教育系统，核心不是最大化停留时长，而是围绕用户当前学习状态，选择最适合当前阶段的视频内容作为学习载体。系统用统一学习单元 `semantic.coarse_unit` 建模学习目标，当前至少支持：

- `word`
- `phrase`
- `grammar`

因此，系统的核心问题不是“用户可能想看什么视频”，而是：

- 用户当前应该学哪些内容
- 这些内容当前处于什么学习阶段
- 视频内容如何承载这些学习需求
- 如何在内容供给有限时仍然稳定输出一组不空且不失焦的视频列表。

### 2.2 当前阶段的产品表达

当前阶段对外表达是**视频列表**，不是任务列表。
系统的长期产品表达可以继续往“学习任务”演进，但在当前 MVP 中，task layer 不进入主流程，Recommendation 的统一输出就是排序后的视频列表，每条视频带推荐原因、覆盖信息和可跳转的证据位置。这样做是为了先把最核心的闭环做稳：输入用户，输出视频；用户观看 / 学习；反馈回流；状态更新；下一轮推荐变化。

## 3. 设计原则

### 3.1 对外输出始终是视频列表

当前 MVP 对外只返回：

- 排序后的视频列表
- 每条视频附带原因码、覆盖信息和证据位置

当前明确不做：

- task layer 主流程
- unit list 主输出
- 非视频 fallback task

这条边界必须定死，否则系统会在第一版同时背负“内容推荐 + 任务编排 + UI 表达”的复杂度。

### 3.2 Learning engine、Catalog、Recommendation 三域边界必须硬隔离

Learning engine 继续负责：

- 记录学习事件
- 维护学习状态
- 提供 replay 重建

Catalog 继续负责：

- 切片视频内容资产
- transcript 标准读模型
- Recall-ready 视频级 coarse unit 索引
- 入库审计
- 用户视频互动投影

Recommendation 继续负责：

- 读取学习状态与内容索引
- 生成候选视频池
- 对视频排序与选择
- 写 Recommendation 自己的 serving state 与审计表

Recommendation 不得反写 Learning engine；Recommendation 的投放状态不得塞回 Catalog；Catalog 也不提前固化 Recommendation 的高层语义判断。

### 3.3 事件是真相，状态是归约结果，投放状态是 Recommendation own state

Learning engine 的根原则是：

- `learning.unit_learning_events` 是事实真相层
- `learning.user_unit_states` 是归约后的当前学习状态层

Recommendation 的根原则是：

- `recommendation.user_unit_serving_states` / `user_video_serving_states` 是投放与冷却状态
- `video_recommendation_runs / items` 是推荐审计

不要把这三层揉到一起。
特别是 `last_recommended_at` 这类字段，必须属于 Recommendation，不属于学习状态，也不属于 Catalog 内容状态。

### 3.4 planner 输出需求画像，不输出死的 exact unit 名单

Recommendation 的上游不再输出：

- unit A
- unit B
- unit C

而必须输出：

> **本轮视频推荐要满足哪些学习需求、这些需求的优先级和预算是什么、召回和选择时应如何分配容量。**

这就是从原始 exact-only 调度升级为可执行视频需求包的关键。

### 3.5 供给感知是 MVP 生死线

planner 不能只看“应该学什么”，还必须知道“内容库接不接得住”。
供给感知至少要回答：

1. 这个 unit 当前有没有足够可用视频
2. 如果供给很弱，是否要降低 exact 预期并抬高 bundle / soft_future 预算
3. 如果整体库存极差，系统是否进入降级模式

这是当前 MVP 真正解决“候选池过窄”的关键。

### 3.6 内部按细粒度证据建模，外部按 video 输出

系统内部必须有细粒度证据层，但在当前 schema 下，这层不是独立 `video_segments` 物理表，而是：

- `catalog.video_semantic_spans`
- `catalog.video_transcript_sentences`
- `catalog.video_unit_index.evidence_span_refs`

Recommendation 可以把这些内容事实解析成 span / sentence-window evidence，再在 video 级聚合与排序。对外输出仍然是视频。

### 3.7 selector 是必选项，不是可选项

不能只做 rank 后直接 top N 截断。
最终视频列表必须经过一层轻量 selector，控制：

- 核心覆盖
- 去重
- 冷却
- future 占比
- fallback 占比
- 极端 case 降级

否则高分不等于结果好看，也不等于结果合理。

## 4. 系统总体架构

### 4.1 三个核心业务域

当前系统由三个核心业务域组成：

1. **Catalog**
2. **Learning engine**
3. **Recommendation**

系统外围还有两个支撑层：

- 上层业务 API / 事件结算服务
- 离线内容处理与导入流程

### 4.2 模块级视角

```text
用户学习行为
        ↓
[上层业务 API / 事件结算服务]
        ├──> [Learning engine]
        │       ├── learning.unit_learning_events
        │       └── learning.user_unit_states
        │
        └──> [Catalog.video_user_states]   // 可选视频消费投影
```

```text
离线切片视频资产 + transcript JSON
        ↓
[内容导入逻辑]
        ↓
[Catalog]
  ├── catalog.videos
  ├── catalog.video_transcripts
  ├── catalog.video_transcript_sentences
  ├── catalog.video_semantic_spans
  ├── catalog.video_unit_index
  └── catalog.video_ingestion_records
```

```text
推荐请求
        ↓
[上层业务 API]
        ↓
[Recommendation]
  ├── 读取 learning.user_unit_states
  ├── 读取 recommendation.v_recommendable_video_units
  ├── 读取 recommendation.v_unit_video_inventory
  ├── 读取 serving states
  ├── Candidate Generator / Resolver / Ranker / Selector
  ├── 写 recommendation.user_unit_serving_states
  ├── 写 recommendation.user_video_serving_states
  └── 写 recommendation.video_recommendation_runs / items
        ↓
Final Video List
```

Learning engine 与 Recommendation 是平级模块，Catalog 作为内容事实与内容索引域，也是独立 owner 的平级模块。上层业务 API 只调用 Learning engine 和 Recommendation 的对外用例，不感知 Recommendation 内部 planner / candidate / selector 的拆分。

## 5. 系统边界与职责

### 5.1 Catalog 的职责

Catalog 只负责以下五类内容：

1. 切片视频内容资产主记录
2. transcript 的标准化读模型
3. 从 semantic span 聚合而来的 Recall-ready 视频级 coarse unit 索引
4. 入库审计
5. 用户对视频互动状态的聚合投影

Catalog 不负责：

- Learning state
- Recommendation serving state
- Recommendation audit
- 原始视频实体
- 媒体 / AI 流水线状态机
- 复杂高层语义评分字段

当前 Catalog 的核心判断是：数据库中的 `catalog.videos` 只表示最终可播放、可推荐、可学习的切片视频；最细粒度内容事实落在 `catalog.video_semantic_spans`；`catalog.video_unit_index` 是 Recall 主入口；`video_unit_index` 中的 evidence 应以 `evidence_span_refs` 这类可回查引用表达，而不是提前固化 Recommendation 解释层的 “best”。

### 5.2 Learning engine 的职责

Learning engine 负责：

- 记录学习行为事件
- 维护用户目标学习单元状态
- 提供状态重建能力
- 向 Recommendation 暴露稳定的读取输入

Learning engine 的核心聚合是：

- `user_id + coarse_unit_id`

核心原则是：

- 事件是真相，状态是归约结果
- 弱事件不推进记忆状态，强事件才推进
- 在线写入与 replay 共用同一个 reducer
- MVP 只支持 full replay

Recommendation 只消费 `learning.user_unit_states`，不拥有这张表。

### 5.3 Recommendation 的职责

Recommendation 负责：

1. 读取学习状态和内容索引
2. 生成本轮视频需求画像
3. 通过多路候选生成构造视频候选池
4. 解析细粒度 evidence
5. 在 video 级完成排序与选择
6. 输出最终视频列表
7. 记录 video recommendation 审计与投放状态

其中 `catalog.video_user_states` 仅作为可选消费状态输入，当前允许依赖的字段范围以《推荐模块设计.md》中的明确字段清单为准。

Recommendation 不负责：

- 维护 Learning engine 状态
- 直接做 task layer 编排
- 引入 ML 排序
- 维护 Catalog 内容事实

## 6. 核心概念定义

### 6.1 coarse unit

统一学习单元，当前至少支持：

- `word`
- `phrase`
- `grammar`

Learning engine 的状态管理、Recommendation 的需求规划与覆盖统计，统一以 coarse unit 为最小学习对象。

### 6.2 视频内容对象

数据库中的 `catalog.videos.video_id` 就是最终内容对象。
当前系统不单独为原始长视频建实体，不再维护一个“原视频 -> clip”父子表体系。数据库里每一行 `catalog.videos` 都表示一个最终可播放、可推荐、可学习的切片视频。

### 6.3 transcript 顶层摘要

`catalog.video_transcripts` 保存：

- transcript 原始对象路径
- checksum
- format version
- full_text
- sentence / span 统计信息

原始 transcript JSON 保留在对象存储中，数据库不把整坨 JSON 当主查询模型，而是拆成摘要、sentence、semantic span 三层读模型。

### 6.4 sentence

`catalog.video_transcript_sentences` 表示句子级时间轴。
它服务于：

- 句子展示
- 句子级跳转
- 解释生成
- sentence-window evidence 组装。

### 6.5 semantic span

`catalog.video_semantic_spans` 是 transcript 中最细粒度语义事实。
它可能对应：

- 单个单词
- 多词短语
- 可映射 coarse unit 的知识片段

它不是普通 tokenizer token，而是带时间戳的语义跨度单元。Recommendation 当前的最细粒度 evidence 物理落点就在这里。

### 6.6 video-unit index

`catalog.video_unit_index` 是 Recall 主入口索引。
它表示：

- 某个视频覆盖了哪些 coarse unit
- 覆盖强度与聚合事实是什么
- 少量代表性 evidence 引用是什么

它提供的确定性信号包括：

- `mention_count`
- `sentence_count`
- `coverage_ms`
- `coverage_ratio`
- `sentence_indexes`
- `sample_surface_forms`
- `evidence_span_refs`

它当前不提供高层语义评分。

### 6.7 evidence ref

`evidence_span_refs` 表示一组轻量、可回查、低语义承诺的 span 引用，例如：

```json
[
  { "sentence_index": 3, "span_index": 1 },
  { "sentence_index": 4, "span_index": 0 }
]
```

Catalog 只负责保存这种可逆引用；Recommendation 在具体 run 中再动态选择本轮使用的 `best_evidence_ref`，并将对应的 `best_evidence_*` 写入 `video_recommendation_items`。Catalog 不提前固化全局 `best_evidence_*` 语义。

### 6.8 bucket

Demand Planner 输出的学习需求分桶。当前固定为四类：

- `hard_review`
- `new_now`
- `soft_review`
- `near_future`

它们之间有固定优先级：

```text
hard_review > new_now > soft_review > near_future
```

这意味着一个 unit 在一次 planner run 中只能进入一个主 bucket。

### 6.9 lane

Candidate Generator 的召回通道。当前固定为四条：

- `exact_core`
- `bundle`
- `soft_future`
- `quality_fallback`

这四条 lane 的角色分别是：

- 保主线
- 做组合承载
- 做受控扩张
- 做兜底补位。

### 6.10 serving state

Serving state 是 Recommendation 自己的投放状态，用于：

- 冷却
- 重复控制
- 统计最近哪些 unit / video 已被推荐过

它包括：

- `recommendation.user_unit_serving_states`
- `recommendation.user_video_serving_states`

不属于 Learning engine，也不属于 Catalog。

### 6.11 recommendation run

一次 Recommendation 请求的完整审计对象。
run 级记录保存：

- 请求上下文
- planner snapshot
- lane budget
- candidate summary
- selector mode
- underfilled 标志

item 级记录保存：

- 最终选中的 video
- rank 与 score
- dominant bucket / unit
- 覆盖计数
- best evidence 位置
- 原因码与 explanation。

## 7. 数据所有权与表级边界

下面给出当前最重要的 owner 划分。

### 7.1 Catalog owner

- `catalog.videos`
- `catalog.video_transcripts`
- `catalog.video_transcript_sentences`
- `catalog.video_semantic_spans`
- `catalog.video_unit_index`
- `catalog.video_ingestion_records`
- `catalog.video_user_states`

### 7.2 Learning engine owner

- `learning.unit_learning_events`
- `learning.user_unit_states`

### 7.3 Recommendation owner

- `recommendation.v_recommendable_video_units`
- `recommendation.v_unit_video_inventory`
- `recommendation.user_unit_serving_states`
- `recommendation.user_video_serving_states`
- `recommendation.video_recommendation_runs`
- `recommendation.video_recommendation_items`

### 7.4 为什么这样划分

这样划分有三个直接好处：

第一，内容事实、学习状态、推荐投放状态彻底分家。
第二，Learning engine 的 replay 只重建学习域，不误伤推荐投放域。
第三，Catalog 可以继续坚持“先存确定事实，再做轻聚合，不提前固化高层推荐语义”的原则。

## 8. Catalog 设计摘要

### 8.1 目标

Catalog 负责把离线批处理完成的切片内容资产，落成一套适合业务查询与 Recommendation 读取的内容事实层和内容索引层。

### 8.2 数据链

```text
catalog.videos
    ↓
catalog.video_transcripts
    ↓
catalog.video_transcript_sentences
    ↓
catalog.video_semantic_spans
    ↓
catalog.video_unit_index
```

### 8.3 设计关键点

#### 只存切片视频，不存原始视频实体

数据库中每一行 `catalog.videos` 都是最终内容对象。
原始长视频不入库，不建立父表。

#### transcript JSON 保留在对象存储中，数据库存读模型

数据库不以大 JSON blob 为主读模型，而是拆成：

- 顶层摘要
- sentence 级时间轴
- semantic span 明细

#### semantic span 是最细事实层

Recommendation 的细粒度 evidence 当前物理落点就在 span 与 sentence-window 级别，而不是单独 segment 表。

#### `video_unit_index` 是视频级 Recall 主入口

Recommendation 先用 `video_unit_index` 做视频级粗召回，再用 `evidence_span_refs` 回查更细粒度证据。

#### `video_user_states` 只承接互动投影，不承接推荐曝光

推荐曝光状态继续留在 Recommendation。

## 9. Learning engine 设计摘要

### 9.1 目标

Learning engine 基于统一学习事件流，维护用户对目标学习单元的稳定状态总表，并通过 full replay 保证状态可重建、可验证、可修复。

### 9.2 核心表

#### `learning.unit_learning_events`

事实表，append-only。
记录：

- `event_type`
- `source_type`
- `quality`
- `occurred_at`
- 以及与 coarse unit 的关系

#### `learning.user_unit_states`

状态总表。
至少包含：

- `is_target`
- `target_priority`
- `status`
- `progress_percent`
- `mastery_score`
- `last_quality`
- `next_review_at`
- `recent_quality_window`
- `recent_correctness_window`
- `strong_event_count`
- `review_count`

### 9.3 事件模型

事件分两类：

#### 弱事件

- `exposure`
- `lookup`

作用：

- 更新 seen / exposure 信息
- 不推进记忆状态

#### 强事件

- `new_learn`
- `review`
- `quiz`

作用：

- 更新记忆状态
- 更新复习节奏
- 更新掌握程度
- 推动状态迁移

### 9.4 状态模型

统一使用：

- `new`
- `learning`
- `reviewing`
- `mastered`
- `suspended`

### 9.5 复习调度模型

MVP 使用简化 SM-2：

- 成功：推进 repetition、增长 interval_days、更新 next_review_at
- 失败：repetition 重置、interval_days 回到 1 天、next_review_at 快速回拉

`ease_factor` 保留下限，Replay 只做 full replay。

### 9.6 Recommendation 读取契约

Recommendation 当前最重要的输入字段是：

- `is_target`
- `target_priority`
- `status`
- `progress_percent`
- `mastery_score`
- `last_quality`
- `next_review_at`
- `recent_quality_window`
- `recent_correctness_window`
- `strong_event_count`
- `review_count`

这批字段必须保持语义稳定。

## 10. Recommendation 设计摘要

### 10.1 目标

Recommendation 的当前目标是：

> **基于 Learning engine 的学习状态和 Catalog 的内容索引，稳定输出一组当前足够合理的视频列表。**

### 10.2 主链路

```text
RecommendationRequest
    +
learning.user_unit_states
    +
recommendation.v_recommendable_video_units
    +
recommendation.v_unit_video_inventory
    +
recommendation.user_unit_serving_states
    +
recommendation.user_video_serving_states
    +
(可选) catalog.video_user_states
    +
catalog.video_semantic_spans
    +
catalog.video_transcript_sentences
        ↓
Context Assembler
        ↓
Demand Planner
        ↓
Candidate Generator
        ↓
Evidence Resolver
        ↓
Video Evidence Aggregator
        ↓
Video Ranker
        ↓
Video Selector
        ↓
Explanation Builder
        ↓
Final Video List
        ↓
Audit Writer + Serving State Manager
```

### 10.3 需求规划

Demand Planner 当前输出四类学习需求：

- `hard_review`
- `new_now`
- `soft_review`
- `near_future`

并给出：

- session mode
- lane budget
- mix quota
- supply flags

它的核心判断不是“单纯该学什么”，而是“这一轮视频推荐需要满足什么样的学习需求画像”。

### 10.4 多路候选生成

Candidate Generator 当前固定四条 lane：

#### `exact_core`

服务当前最核心学习需求。

#### `bundle`

找能同时承载多个高价值 unit 的视频，是解决候选空间狭窄问题的关键 lane。

#### `soft_future`

引入 `soft_review + near_future`，做受控扩召回。

#### `quality_fallback`

只在候选不足时触发，做质量兜底。

### 10.5 细粒度证据解析

Evidence Resolver 基于：

- `evidence_span_refs`
- `video_semantic_spans`
- `video_transcript_sentences`

把 video-unit candidate 转成更具体的 span / sentence-window evidence，用于：

- best evidence 选择
- explanation
- context completeness
- evidence density

### 10.6 视频级聚合与排序

Video Evidence Aggregator 负责：

- video-unit 去重
- bucket 覆盖统计
- best evidence 选择
- coverage strength / bundle value / educational fit 等 video 级特征构造

Video Ranker 负责：

- 计算 `demand_coverage`
- 计算 `coverage_strength_score`
- 计算 `bundle_value_score`
- 计算 `educational_fit_score`
- 加上 `future_value_score`
- 扣除 `recent_served_penalty` / `overload_penalty`

推荐的 MVP 基础公式为：

```text
base_score =
  0.40 * demand_coverage
+ 0.18 * coverage_strength_score
+ 0.15 * bundle_value_score
+ 0.15 * educational_fit_score
+ 0.05 * future_value_score
+ 0.05 * freshness_score
- 0.03 * recent_served_penalty
- 0.02 * overload_penalty
```

### 10.7 最终选择

Video Selector 负责：

- 核心覆盖
- 多样性
- 去重
- future/fallback 占比控制
- low-supply / extreme-sparse 降级模式

最终选择不等于简单取 top N，而是一个带边际收益与配额控制的轻量贪心选择过程。

## 11. Recommendation 依赖的共享读模型

### 11.1 `recommendation.v_recommendable_video_units`

这是一张 Recommendation own 的读视图，用来把常用视频-单元联合字段预拼出来，避免每条 lane 重复 join。该视图的唯一权威定义位于《推荐模块设计.md》；其 owner、最小字段集合、过滤规则与刷新策略，以该文档中的权威定义为准。

### 11.2 `recommendation.v_unit_video_inventory`

这是 planner 的供给感知基础视图。
当前最小字段包括：

- `distinct_video_count`
- `avg_mention_count`
- `avg_sentence_count`
- `avg_coverage_ms`
- `avg_coverage_ratio`
- `strong_video_count`
- `supply_grade`

### 11.3 为什么 Recommendation 不直接依赖 `video_segments / segment_unit_mappings`

因为当前 Catalog 已经明确不建立这两张表。
Recommendation 当前应接受的物理事实是：

- `video_semantic_spans` 是最细语义事实
- `video_unit_index` 是视频级粗召回入口
- `evidence_span_refs` 是可回查证据引用集合

这套结构已经足够支撑当前视频主链路。

## 12. 三条主链路

### 12.1 内容入库链路

```text
离线切片视频资产 + transcript JSON
        ↓
内容导入逻辑
        ↓
校验 metadata 与 transcript 结构
        ↓
写 catalog.videos
写 catalog.video_transcripts
写 catalog.video_transcript_sentences
写 catalog.video_semantic_spans
聚合写 catalog.video_unit_index
        ↓
写 catalog.video_ingestion_records
```

关键点：

- 单 clip 单事务 replace 写入
- `source_clip_key` 作为幂等锚点
- 原始 transcript JSON 保留在对象存储，数据库存标准读模型
- `video_unit_index` 由 `video_semantic_spans` 聚合生成。

### 12.2 学习反馈链路

```text
前端行为
        ↓
上层业务 API / 事件结算服务
        ↓
标准化学习事件
        ↓
Learning engine
  ├── append learning.unit_learning_events
  └── reduce + upsert learning.user_unit_states
```

可选地，视频观看 / 点赞 / 收藏等行为还会更新：

- `catalog.video_user_states`

但这不等于学习状态，也不等于推荐曝光状态。

### 12.3 推荐生成链路

```text
推荐请求
        ↓
上层业务 API
        ↓
Recommendation
  ├── 读取 learning.user_unit_states
  ├── 读取 v_recommendable_video_units / v_unit_video_inventory
  ├── 读取 serving states
  ├── 读取可选的 video_user_states
  ├── Demand Planner
  ├── Candidate Generator
  ├── Evidence Resolver
  ├── Aggregator
  ├── Ranker
  ├── Selector
  ├── Explanation Builder
  ├── 写 video_recommendation_runs / items
  ├── 写 user_unit_serving_states
  └── 写 user_video_serving_states
        ↓
Final Video List
```

## 13. 跨模块数据契约

### 13.1 Learning engine -> Recommendation

Learning engine 对 Recommendation 暴露的是：

- 目标状态
- due / mastery 信号，以及供 Recommendation 派生 `fragility` / `instability` 判断的底层状态输入
- 基础窗口统计

Recommendation 不直接依赖 Learning engine 的内部 reducer 或 SM-2 公式，只依赖其状态语义。
本文中的 `fragility`、`instability` 等术语均指 Recommendation 基于 Learning engine 稳定输出字段推导得到的派生信号，而不是 Learning engine 持久化表中的独立字段。Learning engine 对外暴露的稳定输入仍以 `status`、`next_review_at`、`mastery_score`、`last_quality`、`recent_quality_window`、`recent_correctness_window` 等字段为准。

### 13.2 Catalog -> Recommendation

Catalog 对 Recommendation 暴露的是：

- 视频内容资产可推荐性
- transcript 结构化读模型
- 最细语义事实
- 视频级 coarse unit 覆盖索引
- 轻量证据引用

Recommendation 不要求 Catalog 直接告诉它：

- 哪个 evidence 是全局 best
- 哪条视频教学性最高
- 哪个命中是 primary / contextual / incidental

这些属于 Recommendation 自己的推导层。

### 13.3 Recommendation -> 上层业务 API

Recommendation 对外只暴露一个主用例：

> `GenerateVideoRecommendations`

上层业务 API 不分别调用：

- planner
- candidate generator
- ranker
- selector

这样可以保持内部流水线不泄漏到模块边界之外。

## 14. 审计与可观测性

### 14.1 Catalog 审计

- `video_ingestion_records`

## 16. 实施顺序

### Phase 1：Catalog 与 Recommendation 读模型基建

- 落实 `video_unit_index.evidence_span_refs`

- 建 `v_unit_video_inventory`

### Phase 2：Learning engine 与 Recommendation 基础边界收稳

- 确认 `learning.user_unit_states` 字段契约稳定
- 确认 Recommendation 不再依赖学习引擎内部实现
- 确认 `last_recommended_at` 等 serving state 不混入学习域

### Phase 3：Recommendation 主链路落地

- Context Assembler
- Demand Planner
- exact_core + bundle
- soft_future
- quality_fallback

### Phase 4：Evidence Resolver + Aggregator + Ranker

- 基于 `evidence_span_refs` 打通细粒度证据
- 构造 video-level 特征
- 落基础排序

### Phase 5：Selector + Explanation + Audit

- low-supply / extreme-sparse 模式
- 贪心选择
- 原因码与 explanation
- run/item 审计闭环

### Phase 6：灰度与调参

- 候选池规模
- planner 预算
- selector quota
- penalty 系数
- bad-case 审计修复

### 系统级实施前置顺序

当前 MVP 的系统级实施前置顺序如下：

1. Learning engine 先完成 `learning.user_unit_states` 契约稳定；
2. Catalog 再完成 `evidence_span_refs` 增量迁移、新索引落地与历史回填；
3. Recommendation 再完成 `v_recommendable_video_units`、`v_unit_video_inventory`、serving state 与 video recommendation audit；
4. Recommendation 的 Evidence Resolver、best-evidence explanation 与 `video_recommendation_items.best_evidence_*` 写入路径，必须在步骤 2 完成后方可正式启用；
5. 全链路切流前应完成双读兼容期与 shadow 验证，并满足下文“Recommendation 新读路径的正式启用前提”。

### Recommendation 新读路径的正式启用前提

以下能力必须在 Catalog 增量改造完成后方可正式启用：

- `video_unit_index.evidence_span_refs`
- `recommendation.v_recommendable_video_units`
- 基于 `evidence_span_refs` 的 Evidence Resolver
- `video_recommendation_items.best_evidence_*` 写入链路

正式启用前提如下：

1. Catalog 已完成 `evidence_span_refs` 字段增量迁移；
2. 新写入路径已支持 `evidence_span_refs`；
3. 历史数据已基于 `video_semantic_spans` 完成回填；
4. 新读路径已在双读兼容阶段验证通过；
5. Recommendation explanation / jump-to / audit item 构建已切换到新读路径。

## 17. 一个完整示例

假设某用户当前状态如下：

- `hard_review`：
  - unit 101（供给弱）
  - unit 102（供给强）

- `new_now`：
  - unit 201（供给 ok）

- `soft_review`：
  - unit 301（供给强）

- `near_future`：
  - unit 401（供给强）

- `target_video_count = 8`

### 17.1 planner 输出

- `hard_review_low_supply = true`
- `session_mode = review_heavy`
- lane budget：
  - exact_core 0.35
  - bundle 0.35
  - soft_future 0.20
  - quality_fallback 0.10

### 17.2 候选生成

- `exact_core`：
  - 102 命中较多
  - 101 命中稀疏，但不会被 planner 丢掉

- `bundle`：
  - 找到同时承载 `102 + 301` 的视频
  - 找到同时承载 `201 + 301` 的视频

- `soft_future`：
  - 用 `301 + 401` 扩张

- `fallback`：
  - 暂不触发

### 17.3 evidence 解析

对候选视频中 `video_unit_index` 提供的 `evidence_span_refs` 回查：

- 具体 `video_semantic_spans`
- 对应 `video_transcript_sentences`

形成本轮实际使用的 best evidence 与 sentence-window。

### 17.4 selector 结果

- 进入 `low_supply mode`
- 先保 102 / 201 的核心覆盖
- 再用 bundle 承载 301
- 允许少量 401 进入
- 若 101 没有足够高质量视频，不强行用多个弱视频凑数

最终结果：

- 列表仍然围绕当前主线
- 但不会被 101 的供给稀疏完全锁死
- 同时保持对 301 和 401 的受控扩张

## 18. 最终结论

这套系统的整体设计可以压缩成一句话：

> **Catalog 负责把离线处理完成的切片内容资产落成可查询的内容事实层与 Recall-ready 视频级索引；Learning engine 负责基于统一学习事件流维护用户对目标学习单元的稳定状态；Recommendation 负责基于学习状态与内容索引，先生成供给感知的学习需求画像，再通过多路候选生成、细粒度 evidence 解析、视频级聚合、规则化排序与轻量选择，最终稳定输出一组当前足够合理的视频列表，并维护 Recommendation 自己的投放状态与视频级推荐审计。**

因此，在当前最终版本里：

- `learning.unit_learning_events` 与 `learning.user_unit_states` 属于 Learning engine
- `catalog.videos / transcripts / sentences / semantic_spans / video_unit_index` 属于 Catalog
- `recommendation.v_recommendable_video_units / v_unit_video_inventory / serving states / video_recommendation_runs / items` 属于 Recommendation
- Recommendation 只消费 Learning engine 与 Catalog 的输出，不回写它们的业务表
- 当前系统的主结果是视频列表，不是任务列表
- 当前系统的关键目标是把“输入一个用户，输出一个视频列表”的闭环做稳，而不是在第一版同时解决所有长期产品形态。
