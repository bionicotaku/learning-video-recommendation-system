## 1. 总体结论

当前仓库已经不再只是“项目骨架与基础设施”。更准确地说，当前状态是：

- `catalog` 已从 migration-only 扩展出观看进度上报用例，维护 `analytics.video_watch_events` session ledger 与 `catalog.video_user_states` / `catalog.video_engagement_stats` 消费投影
- `analytics` 已从 migration-only 进展到 raw fact batch write 模块，并已包含答题原始事实表 `analytics.quiz_events`、观看 session 摘要表 `analytics.video_watch_events` 与学习互动原始事实表 `analytics.learning_interaction_events`；三张 raw fact 表已统一包含 `client_context`
- `recommendation` 已经完成第一轮完整主链路实现，并完成第二轮契约收口
- `learningengine` 已经完成第一轮完整业务实现，包括 domain 规则、主 usecase、normalizer 子模块和模块内数据库测试
- `internal/api` 已落地 HTTP 基座，并已实现学习事件上报三条 endpoint 与 `POST /api/video-watch-progress`；当前移动端 MVP 不实现 CORS

更准确地说，当前仓库已经具备以下能力：

1. 仓库级 Go 工程基线已建立
2. 仓库内置 DB 工具 `dbtool` 已建立
3. `catalog / analytics / learningengine / recommendation` 四个顶层业务模块的基础目录结构已建立
4. `catalog / analytics / learningengine / recommendation` 的 owner migration 已写入仓库
5. `analytics`、`catalog` watch-progress、`learningengine`、Learning Engine normalizer 与 `recommendation` 的 query / `sqlc` / mapper / repository 基础层已建立
6. DTO、repository interface、usecase interface 已建立
7. `make check` 与 `make sqlc-generate` 在当前仓库内已验证通过
8. `make e2e-test` 已建立并可运行跨模块真实迁移驱动 E2E

但当前仓库**还没有**以下内容：

1. Catalog 导入流程、内容读取 repository、内容查询 API、worker、CLI 以外的业务入口
2. Watch-progress 以外的 Catalog 业务代码

因此，当前项目最准确的定位是：

> **设计文档已经完整；Analytics raw fact 写入、Catalog watch-progress、Learning Engine 和 Recommendation 都已进入可执行业务逻辑阶段；系统已经能在“API -> Analytics raw fact -> Learning Engine normalizer / reducer”和“API -> Catalog watch-progress -> Catalog projections”条件下跑通跨模块闭环测试。**

跨模块时间与客户端上下文的当前实现口径：

- API 设计统一要求 `*_at` 时间点字段使用带 `Z` 或 offset 的 RFC3339 字符串；后端内部统一按 UTC instant 处理。
- Analytics、Learning Engine reducer、Learning Engine normalizer、Recommendation 的 application 边界已把业务输入时间归一化为 `.UTC()`。
- Postgres 基础类型映射已收敛到 `internal/platform/postgres/*`：`pgtime` 统一 `time.Time <-> pgtype.Timestamptz` 的 UTC 归一化，`pguuid` / `pgtext` / `pgnumeric` 统一 UUID、nullable text、numeric 的纯技术转换。
- Postgres integration 测试底座已收敛到 `internal/platform/postgres/pgtest`：它只负责 embedded Postgres、template database、per-test clone 和 schema plan 执行；模块 fixture 仍拥有各自 schema plan 与 seed helper。
- 各业务模块仍保留本地 persistence mapper 函数名，用于维持模块语义边界；这些函数只委托平台层做通用 Postgres 类型转换。
- 从 Postgres `timestamptz` 映射回 domain / application model 的路径统一返回 UTC `time.Time`。
- DB 继续使用 `timestamptz` 表示业务时间点，`*_ms` 表示相对毫秒 / 时长 / 播放位置；本轮没有新增 schema migration。
- `client_context` 当前保持宽松 JSON object 语义：后端只要求是 object，不固定字段集合，不拒绝未来新增字段；API 样例继续推荐 `platform`、`app_version`、`os_version`、`device_model` 四个基础字段。

### 2.3 `internal/learningengine`

当前已经建立：

`reducer` 是 `learning.*` 表 owner，已经真正有内容的部分包括：

- domain policy / aggregate：
  - 事件校验
  - `reducer_effect` 分发
  - `set_mastered` 终态掌握分支

  - `progress_percent`
  - `mastery_score`
  - 状态迁移
  - 统一 reducer
- SQL：事件表与状态表的 CRUD / list / replay 支撑查询。
- `sqlc` 生成代码。
- mapper：`pgtype` 与领域模型之间的转换。
- repository：事件写入、状态读写、target 控制命令。
- tx manager：事务内 repository 装配入口。
- test：`test/unit`、`test/integration`、`test/fixture`。

#### `internal/learningengine/normalizer`

`normalizer` 是 raw fact 到 normalized ledger 的解释层，已经真正有内容的部分包括：

- DTO：`NormalizeLearningInteractionsByIDsRequest/Response`、`NormalizeQuizAttemptByIDRequest/Response`、`NormalizeSelfMarkMasteredByIDRequest/Response`、`NormalizePendingEventsRequest/Response`。
- domain model：`RawQuizEvent`、`RawLearningInteraction`、`NormalizedLearningEvent`、`NormalizationResult`。
- domain policy：quiz `5000ms` 快慢阈值和四档 quality 规则。
- domain rule：quiz / exposure / lookup / self mark 映射规则。
- application service：by-IDs 读取本批 raw facts、pending repair/backfill 读取 raw facts，分组调用 reducer `RecordLearningEvents`。
- persistence：normalizer 专属 SQLC、read-only repository、by-IDs query、pending anti-join query。
- test：unit 与 real Postgres integration。

当前明确还没有：

- HTTP/API 层
- partial/scoped replay
- 复杂乱序历史修复工具

因此，Learning engine 当前处于“**根模块作为边界容器，reducer 与 normalizer 两个平级子模块均已实现，并且已经通过跨模块链路接入 Recommendation 与 Unit Progress 读取 API；后续主要缺的是 worker 入口与更复杂的历史修复能力**”。

补充说明：

- `learning.unit_learning_events` 当前是 reducer-owned normalized Learning Engine event ledger，不是 analytics raw log；raw fact 到 `reducer_effect` / `progress_quality` 的解释层已在 `internal/learningengine/normalizer` 中实现。
- `normalizer` 当前 read-only 读取 `analytics.quiz_events` 与 `analytics.learning_interaction_events`，不写 Analytics，不直接写 `learning.*`，所有写入统一调用 reducer `RecordLearningEvents`。
- `normalizer` 当前没有 checkpoint / rollup / `normalized_at` 状态字段；API 主路径依赖 by-IDs 输入和 reducer `RecordLearningEvents` 幂等 append，repair/backfill 依赖 pending anti-join 和 `learning.unit_learning_events` 的唯一约束。
- `RecordLearningEvents` 当前已经支持批量幂等 append；重复 `(user_id, source_type, source_ref_id, coarse_unit_id)` 不报错，不重复 reduce，并在 response 中返回 duplicate 计数。状态读取/写入在批内按 affected units 批量锁定和批量 upsert，reducer 业务规则仍在 Go domain 中执行。
- quiz normalizer 规则固定为：首次答对且 `total_elapsed_ms <= 5000` 为 quality `5`，首次答对且 `> 5000` 为 `4`，首次答错且 `<= 5000` 为 `2`，首次答错且 `> 5000` 为 `1`。答错一次和多次不影响 quality 档位。
- exposure / lookup normalizer 当前只生成 `observe_only`，不推进 progress；`self_mark_mastered` 生成 `set_mastered`。
- `event_type` 当前支持 `exposure`、`lookup`、`quiz`、`self_mark_mastered`；reducer 由 `reducer_effect = observe_only / affects_progress / set_mastered` 分发，其中 `self_mark_mastered` 必须使用 `set_mastered`。
- `learning.user_unit_states` 已改为 observation / progress / schedule 字段分组：例如 `observation_count`、`progress_event_count`、`last_progress_quality`、`schedule_repetition`、`schedule_interval_days`。
- schedule path 当前使用短间隔 MVP 规则：通过后间隔为 `0.25d -> 1d -> 2d -> 4d`，后续按 ease factor 增长并封顶 14 天；失败回拉到 `0.25d`。
- `set_mastered` 和自然进入 `mastered` 都会把状态收敛到 completed mastered：`status = mastered`、`progress_percent = 100`、`mastery_score = 1`、`next_review_at = null`、`is_target = false`。其中 `set_mastered` 会把 `last_progress_at` 更新为 self-mark 事件时间，但不设置 `last_progress_quality`，也不增加 `progress_event_count`；自然 mastered 当前按连续通过 3 次、最近 3 次无失败、`mastery_score >= 0.75` 判定，不再等待 21 天间隔。
- `learningengine` 当前已经通过跨模块 E2E 验证了：
  - target/control 字段对 Recommendation 的过滤语义
  - replay 前后 Recommendation 的可观察行为一致性
  - 非 target 的学习事件行不会误进入 Recommendation 输入
- `RecordLearningEvents` 在空状态上初始化时，默认 `is_target = false`，target 只能来自显式 control 命令，而不是学习事件隐式创建
- reducer 的模块内测试结构已经对齐统一规范：
  - 纯单测统一收口到 `internal/learningengine/reducer/test/unit`
  - 真实数据库测试统一收口到 `internal/learningengine/reducer/test/integration`
  - 共享测试基座统一收口到 `internal/learningengine/reducer/test/fixture`
- reducer 的真实数据库测试基座已经从“每个测试 case 单独起 embedded Postgres”调整为“每个 integration 测试包共享 embedded Postgres server + 每个测试 case 独立 database”，底层生命周期复用 `internal/platform/postgres/pgtest`

## 4.3 Recommendation

仓库中已经新增 `recommendation` owner migration，包含：

- `recommendation.user_unit_serving_states`
- `recommendation.user_video_serving_states`
- `recommendation.video_recommendation_runs`

  - item 审计以 `dominant_role`、`dominant_unit_id` 和 `learning_units jsonb` 保存本轮学习计划快照
- `recommendation.v_video_unit_recall_index`
- `recommendation.v_unit_video_inventory`
- `recommendation.user_unit_recall_queue`
- `recommendation.user_unit_recall_queue_states`
- `recommendation.recall_projection_metadata`
- 对应索引

当前 SQL/query 侧已支持：

- 读取 `learning.user_unit_states`
- lazy rebuild 当前用户 recall queue，并从 queue 选择本轮 `planner_scope` 与 `recall_fetch_scope`；queue freshness 同时比较 active target 数量、learning max updated_at 与 recall projection metadata
- 用户级 queue rebuild 已做并发幂等处理，避免同用户并发 feed 请求重复插入 queue rows
- SQL 层分开返回 supplied / no-supply queue candidates，避免 no-supply 在截断阶段挤出有供给 units
- 对 `recall_fetch_scope` units 按 `coarse_unit_id + per_unit_limit` 读取 `v_video_unit_recall_index` top-N rows；`supply_grade='none'` 的 units 可少量保留在 planner scope，但不触发 recall fetch
- 读取 `v_unit_video_inventory`
- 读取 `catalog.video_user_states`
- `catalog.video_user_states` 的观看比例由 `max_position_ms / VideoCandidate.duration_ms` 派生，不再读取持久化 ratio 字段
- 原子递增 upsert unit/video serving state
- insert run
- insert item
- refresh 物化视图并更新 `recall_projection_metadata`

当前 `v_unit_video_inventory` 已经采用一版仓库内固定的默认规则：

- `strong_video_count` 按 `mention_count >= 2 AND coverage_ratio >= 0.05 AND mapped_span_ratio >= 0.50` 统计
- `supply_grade` 分为 `strong / ok / weak / none`

这一版规则已经写入 migration，因此当前仓库里**确实已经存在 Recommendation 供给感知基础规则的 SQL 表达**，但后续仍允许按 Recommendation owner 边界继续调整。

当前 Recommendation 已经真正落地的业务链路包括：

- `Context Assembler`
- `Video State Enricher`
- `Demand Planner`
- `Candidate Generator`
- `Evidence Resolver`
- `Video Evidence Aggregator`
- `Video Ranker`
- `Video Selector`
- `Video Fill Service`
- `Explanation Builder`
- `Audit Writer`
- `Serving State Manager`
- `GenerateVideoRecommendations` 完整 pipeline
  - 对外响应按 video learning plan 返回 `items[].learning_units[]`
  - 每个 `learning_unit` 带 `coarse_unit_id`、`role`、`is_primary` 和可选 unit-level evidence
  - selector 后若数量不足，会按 `mastered_target_fill -> popular_fill` 追加 video-level 补全 item；补全 item 的 `learning_units=[]`
  - 不再返回 `Covered*` 或 video-level `best_evidence`

当前 Recommendation 相关测试也已落地：

- unit / scenario / golden
- repository integration
- Learning engine × Recommendation 真实迁移驱动跨模块 E2E

当前 Recommendation 的实现边界还需要特别注意 4 点：

1. `GenerateVideoRecommendations` 现在只保留完整 pipeline constructor，不再存在 shell constructor 降级路径
2. 对外响应已经切换为 video learning plan：学习推荐视频通过 `learning_units` 暴露预期学习 unit 与 unit-level evidence，补全视频返回空数组；不再暴露 `Covered*` 或 video-level `best_evidence`
3. Unit recall 读取边界是 `planner_scope` / `recall_fetch_scope` + per-unit top-N；默认最多 `min(max(target_video_count * 12, 64), 200)` 个 planner units，实际 recall fetch 只覆盖 `supply_grade <> 'none'` 的 units
4. `underfilled` 现在表示经过正常学习推荐、mastered target fill、popular fill 后，最终 item 数仍少于目标数量
5. Learning engine 与 Recommendation integration 都已进入默认 `make check`
6. Recommendation integration 基座已从“每测单独起 embedded Postgres + 重跑 migration”迁到 shared suite：每包共享一个 embedded Postgres server，template database 初始化一次，每测 clone 独立数据库
