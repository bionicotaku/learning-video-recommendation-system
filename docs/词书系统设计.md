# 词书系统设计

## 0. 文档信息

文档状态：MVP 设计稿，已实现。
目标读者：后端、前端、Semantic、Learning Engine、Recommendation、API 维护者。
当前范围：定义词书集合的数据模型、模块归属、用户激活目标集合的状态投影、切换词书的事务语义，以及与 Recommendation / Unit Progress / Feed 的关系。

## 1. 一句话结论

词书系统只解决两个问题：

```text
1. 系统有哪些学习目标合集，每个合集包含哪些 coarse unit。
2. 某个用户当前激活了哪个合集，以及该合集如何投影到 learning.user_unit_states.is_target。
```

推荐模块不感知词书。Recommendation 继续只读取 `learning.user_unit_states`：

```sql
is_target = true
and status in ('new', 'learning', 'reviewing')
```

## 2. 核心原则

### 2.1 词书集合和学习状态分离

`semantic.unit_collections` 和 `semantic.unit_collection_members` 是静态内容定义。它们回答：

```text
TOEFL 1000 Essential 是什么？
它包含哪些 coarse_unit？
每个 coarse_unit 在这本词书里的顺序是什么？
```

`learning.user_learning_profiles` 和 `learning.user_unit_states` 是用户学习状态。它们回答：

```text
这个用户当前激活哪本词书？
哪些 coarse_unit 属于当前目标？
这个用户对每个 coarse_unit 的状态是什么？
```

### 2.2 `is_target` 和 `status` 分离

`is_target` 表示某个 unit 是否属于用户当前激活学习目标范围。

`status` 表示用户对这个 unit 的学习状态：

```text
new / learning / reviewing / mastered
```

因此同一行可以合法出现：

```text
is_target=true, status=mastered
```

它表示这个 unit 属于当前词书，但用户已经掌握。Recommendation 不推荐它，但当前词书进度统计需要它。

### 2.3 词书切换不重置学习进度

从雅思切到托福时，只改变目标范围，不清空学习历史：

```text
is_target
target_source
target_source_ref_id
target_priority
```

不修改：

```text
status
progress_percent
mastery_score
next_review_at
schedule_*
observation/progress counters
```

## 3. 模块归属

| 模块 | 拥有对象 | 职责 |
|---|---|---|
| `internal/semantic` | `semantic.unit_collections`、`semantic.unit_collection_members` | 词书定义、词书成员读取。 |
| `internal/learningengine/reducer` | `learning.user_learning_profiles`、`learning.user_unit_states` | 用户当前 active collection、target projection、学习状态。 |
| `internal/api` | HTTP handler / facade | 调用 Semantic 读接口和 Learning Engine control usecase，做 validation 和错误映射。 |
| `internal/recommendation` | 无新增对象 | 只读 `learning.user_unit_states`，不直接理解词书。 |

不新增 `internal/user` 来保存 active collection。当前激活词书会直接驱动 `learning.user_unit_states` 批量变化，所以它属于 Learning Engine control state。

## 4. 数据模型

### 4.1 `semantic.unit_collections`

一行表示一个可选学习目标合集，例如 TOEFL、IELTS、TOEFL 1000 Essential。

```sql
create table semantic.unit_collections (
  collection_id uuid primary key default gen_random_uuid(),
  slug text not null unique
    constraint unit_collections_slug_canonical_check
    check (slug = lower(slug) and slug ~ '^[a-z0-9][a-z0-9-]{0,80}$'),
  name text not null,
  description text,
  category text not null default 'wordbook',
  status text not null default 'active'
    check (status in ('active', 'inactive')),
  coarse_unit_count integer not null default 0
    check (coarse_unit_count >= 0),
  word_unit_count integer not null default 0
    check (word_unit_count >= 0),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

字段说明：

| 字段 | 含义 |
|---|---|
| `collection_id` | 词书稳定 ID，用于外键关联。 |
| `slug` | API 使用的可读唯一标识，例如 `toefl`、`ielts-core`；必须是 lowercase slug，匹配 `^[a-z0-9][a-z0-9-]{0,80}$`。 |
| `name` | 展示名称。 |
| `description` | 词书说明，可为空。 |
| `category` | 集合类型。MVP 固定或主要使用 `wordbook`。 |
| `status` | `active` 可展示和激活；`inactive` 不对用户开放。 |
| `coarse_unit_count` | 集合包含的全部 coarse unit 数量。 |
| `word_unit_count` | 集合中 `semantic.coarse_unit.kind = 'word'` 的数量。 |
| `created_at` / `updated_at` | 创建和更新时间。 |

`coarse_unit_count` 和 `word_unit_count` 是缓存字段，权威来源是 `semantic.unit_collection_members`。

### 4.2 `semantic.unit_collection_members`

一行表示某个 coarse unit 属于某个 collection。

```sql
create table semantic.unit_collection_members (
  collection_id uuid not null references semantic.unit_collections(collection_id) on delete cascade,
  coarse_unit_id bigint not null references semantic.coarse_unit(id) on delete cascade,
  sort_order integer not null,
  target_priority numeric(8,4) not null default 0,
  created_at timestamptz not null default now(),

  primary key (collection_id, coarse_unit_id)
);

create index idx_unit_collection_members_collection_order
on semantic.unit_collection_members (collection_id, sort_order, coarse_unit_id);

create index idx_unit_collection_members_unit
on semantic.unit_collection_members (coarse_unit_id, collection_id);
```

字段说明：

| 字段 | 含义 |
|---|---|
| `collection_id` | 所属词书。 |
| `coarse_unit_id` | 词书中的学习单元。 |
| `sort_order` | 单元在词书中的顺序，用于详情页、导入校验和后续分批学习。 |
| `target_priority` | 激活词书时投影到 `learning.user_unit_states.target_priority` 的优先级。MVP 固定为 `0`。 |
| `created_at` | membership 创建时间。 |

同一个 `coarse_unit_id` 可以属于多个 collection，所以 membership 必须独立建表，不能用 `semantic.coarse_unit.tags[]` 替代。

### 4.3 `learning.user_learning_profiles`

一行表示 Learning Engine 视角下某个用户的当前学习控制快照。

```sql
create table learning.user_learning_profiles (
  user_id uuid primary key references auth.users(id) on delete cascade,
  active_collection_id uuid not null references semantic.unit_collections(collection_id),
  active_collection_slug text not null
    constraint user_learning_profiles_active_collection_slug_canonical_check
    check (active_collection_slug = lower(active_collection_slug) and active_collection_slug ~ '^[a-z0-9][a-z0-9-]{0,80}$'),
  active_collection_activated_at timestamptz not null default now(),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

字段说明：

| 字段 | 含义 |
|---|---|
| `user_id` | 用户 ID，来自 `auth.users.id`。 |
| `active_collection_id` | 当前激活的词书 ID。 |
| `active_collection_slug` | 当前激活词书 slug 的冗余快照，方便排查和前端返回；必须保持 canonical lowercase slug。 |
| `active_collection_activated_at` | 当前词书最近一次被激活的时间。 |
| `created_at` / `updated_at` | profile 创建和更新时间。 |

这张表不放：

```text
一共学习了多少单词
还有多少未学完
一共答题多少次
连续学习了多少天
```

这些属于聚合统计或 activity summary，不是 active collection control state。

## 5. Count 字段维护

`semantic.unit_collections.coarse_unit_count` 和 `word_unit_count` 是缓存字段。

MVP 不使用 trigger。每次导入或更新词书 members 后，由导入脚本或维护命令执行 refresh SQL：

```sql
update semantic.unit_collections c
set
  coarse_unit_count = stats.coarse_unit_count,
  word_unit_count = stats.word_unit_count,
  updated_at = now()
from (
  select
    m.collection_id,
    count(*)::integer as coarse_unit_count,
    count(*) filter (where cu.kind = 'word')::integer as word_unit_count
  from semantic.unit_collection_members m
  join semantic.coarse_unit cu on cu.id = m.coarse_unit_id
  group by m.collection_id
) stats
where c.collection_id = stats.collection_id;

update semantic.unit_collections c
set
  coarse_unit_count = 0,
  word_unit_count = 0,
  updated_at = now()
where not exists (
  select 1
  from semantic.unit_collection_members m
  where m.collection_id = c.collection_id
);
```

列表 API 直接读取缓存字段，不在每次请求时 join membership 统计。

## 6. 激活和切换词书

### 6.1 Usecase

API facade 提供组合 usecase：

```text
ActivateUnitCollectionTarget(user_id, collection_slug)
  -> collection_id
  -> collection_slug
  -> target_count
```

它是唯一允许从词书切换用户 target projection 的入口。Learning Engine 仍拥有 `learning.*` target projection 的 SQL；User 模块仍拥有 onboarding 状态；API facade 只负责把这两个模块的 repository port 放进同一个用户级事务里提交。

### 6.2 事务流程

事务内执行：

```text
1. 查询 active collection。
2. 查询该 collection 的 members。
3. upsert learning.user_learning_profiles。
4. deactivate 当前不属于新 collection 的旧 unit_collection targets。
5. upsert 新 collection members 到 learning.user_unit_states。
6. 如 app_user.user_profiles 缺失，则从 auth.users lazy repair。
7. 更新 app_user.user_profiles.onboarding_status = collection_selected。
8. 返回 collection_id / slug / target_count。
```

首次激活时不要求用户已经有 Learning Engine profile。`learning.user_learning_profiles` 通过 upsert 创建，collection members 通过 bulk upsert 创建缺失的 `learning.user_unit_states` 行。

### 6.3 同步提交语义

`PUT /api/learning-targets/active-collection` 是同步接口。`200 OK` 表示以下状态已经一起提交：

- `learning.user_learning_profiles.active_collection_*`
- `learning.user_unit_states.is_target / target_source / target_source_ref_id / target_priority`
- `app_user.user_profiles.onboarding_status = collection_selected`

前端应在请求处理中展示 loading 或 disabled 提交态，并在 `200 OK` 后再刷新 feed、me 或 unit progress。MVP 不做“提交成功后后台慢慢切换”的异步语义，因为 Recommendation 直接读取 `learning.user_unit_states`，异步切换会让前端很难判断 feed 使用的是旧目标、切换中目标还是新目标。

如果词书规模导致同步切换持续超过可接受时长，优先优化批量 SQL、索引和 no-op update。只有当这些优化仍不够时，再升级为 activation job：

```text
PUT /api/learning-targets/active-collection -> 202 Accepted
GET /api/learning-targets/active-collection/status -> pending / processing / succeeded / failed
```

引入异步 job 时必须同时定义 feed 在 switching 状态下的行为，以及 onboarding 只在 job succeeded 后更新。

### 6.4 重合 unit 处理

切换时不能简单地先把旧词书全部 `is_target=false`，再把新词书全部 `is_target=true`。这样会对重合 unit 做无意义的双写。

正确语义：

```sql
update learning.user_unit_states s
set is_target = false
where s.user_id = $1
  and s.target_source = 'unit_collection'
  and s.is_target = true
  and not exists (
    select 1
    from new_members nm
    where nm.coarse_unit_id = s.coarse_unit_id
  );
```

然后批量 upsert 新 collection members。重合 unit 最终仍为 `is_target=true`，学习进度不被重置。

### 6.5 `target_priority`

MVP 阶段统一为 `0`：

```sql
target_priority = 0
```

`semantic.unit_collection_members.target_priority` 保留字段，但第一版不做排序策略。后续如果需要词书内优先级，只要把激活 SQL 从 `0::numeric` 改成 `m.target_priority`。

## 7. 与 Learning Engine 的关系

`learning.user_unit_states` 继续是 per user + coarse unit 的学习状态投影。

词书激活只写 control 字段：

```text
is_target
target_source = 'unit_collection'
target_source_ref_id = collection_id::text
target_priority = 0
updated_at
```

词书激活不改变 progress 字段：

```text
status
progress_percent
mastery_score
first_observed_at
last_observed_at
observation_count
progress_event_count
last_progress_at
last_progress_quality
recent_progress_qualities
recent_progress_passes
progress_success_count
progress_failure_count
consecutive_success_count
consecutive_failure_count
schedule_repetition
schedule_interval_days
schedule_ease_factor
next_review_at
```

当前 reducer 语义为：

```text
mastered 不自动把 is_target 改成 false。
```

这样当前 active collection 可以统计 mastered units。Recommendation 普通学习候选通过 `status in ('new','learning','reviewing')` 排除 mastered；feed 数量不足时，`mastered_target_fill` 仍可以使用 `is_target=true AND status='mastered'` 的视频级补全候选。

## 8. 与 Recommendation 的关系

Recommendation 不读取：

```text
semantic.unit_collections
semantic.unit_collection_members
learning.user_learning_profiles
```

Recommendation 只读取：

```sql
learning.user_unit_states
where is_target = true
  and status in ('new', 'learning', 'reviewing')
```

因此：

- 用户没有 active collection：Recommendation 仍可走无 target / fill 视频逻辑。
- 用户激活 collection：Recommendation 更偏向当前词书中未掌握 units。
- 当前词书中已掌握 units：保留 `is_target=true`，不进入普通学习候选；仅可进入 `mastered_target_fill` 这类 video-level 补全。

## 9. 与 Unit Progress 的关系

现有 Unit Progress：

```text
unmastered = is_target=true and status in ('new','learning','reviewing')
mastered = status='mastered'
```

词书系统落地后，当前词书未掌握列表仍可复用 unmastered endpoint。

如果前端需要“当前词书总进度”，建议新增 summary API，而不是修改已有 mastered list：

```http
GET /api/learning-targets/active-collection/progress
```

但该接口不属于本 MVP 第一阶段。

MVP 需要的当前词书统计可以先在 Learning Engine 内部聚合：

```sql
select
  count(*) filter (where is_target = true) as total_count,
  count(*) filter (where is_target = true and status = 'mastered') as mastered_count,
  count(*) filter (where is_target = true and status in ('new', 'learning', 'reviewing')) as unmastered_count
from learning.user_unit_states
where user_id = $1;
```

第一版不新增 `learning.user_learning_stats`。

## 10. 与 Feed 的关系

Feed 不因为没有 active collection 而失败。

```text
没有 active collection:
  Feed 仍调用 Recommendation。

有 active collection:
  Feed 调用 Recommendation，Recommendation 从 user_unit_states 看到 target units。
```

因此不设计 `target_required` 错误。

## 11. API 范围

第一阶段只实现：

```http
GET /api/unit-collections
PUT /api/learning-targets/active-collection
```

`GET /api/unit-collections` 返回 active 词书列表，并按 trusted principal 读取当前用户已激活词书的
`active_collection` slug。新用户没有 Learning profile，或历史 profile 指向 inactive / missing collection
时，`active_collection` 返回 `null`，不阻断列表读取。

接口细节见 [API/Unit-Collections-API-MVP设计.md](API/Unit-Collections-API-MVP设计.md)。

## 12. 数据导入

MVP 可以通过离线 SQL、脚本或后续管理端导入 collection 和 members。

导入流程应保证：

1. 先插入或更新 `semantic.unit_collections`。
2. 批量插入 `semantic.unit_collection_members`。
3. refresh `coarse_unit_count` 和 `word_unit_count`。
4. 校验所有 `coarse_unit_id` 都存在于 `semantic.coarse_unit`。

导入不是本次 API 范围。

## 13. 错误和边界

### 13.1 空词书

空 collection 可以存在，也可以被激活。

激活空词书时：

- `learning.user_learning_profiles` 指向该 collection。
- 旧 `unit_collection` targets 被关闭。
- `target_count = 0`。
- Recommendation 走无 target / fill 逻辑。

### 13.2 inactive 词书

inactive collection：

- 不出现在 `GET /api/unit-collections`。
- 不能被 `PUT /api/learning-targets/active-collection` 激活。

### 13.3 非词书 target

`PUT /api/learning-targets/active-collection` 只关闭：

```sql
target_source = 'unit_collection'
```

不关闭未来可能存在的：

```text
manual
wrong_answer_review
custom_list
```

MVP 如果暂时没有这些 target_source，也保持这个条件，避免后续扩展时误伤。

## 14. 测试计划

Semantic：

- active collections 列表按稳定顺序返回。
- inactive collections 不返回。
- count 字段可被 refresh。
- 同一 coarse unit 可属于多个 collections。

Learning Engine：

- 首次激活 collection 创建 `user_learning_profiles`。
- 首次激活 collection 批量创建缺失的 `user_unit_states`。
- 已存在的 `user_unit_states` 被设为 target，但学习进度不变。
- 切换 collection 后旧独有 units `is_target=false`。
- 切换 collection 后新 units `is_target=true`。
- 重合 units 不丢状态，最终 `target_source_ref_id` 指向新 collection。
- 空 collection 激活后关闭旧词书 targets。
- `target_source != 'unit_collection'` 的 targets 不被关闭。

API：

- `GET /api/unit-collections` 需要 principal。
- `GET /api/unit-collections` 返回字段完整，并包含当前用户 `active_collection` slug / null。
- `PUT /api/learning-targets/active-collection` 需要 principal。
- invalid slug 返回 `400`。
- inactive / missing collection 返回 `404`。
- 重复 PUT 幂等。

Recommendation / E2E：

- 激活词书后，Recommendation 能看到未掌握 target units。
- mastered target 不进入 Recommendation。
- 没有 active collection 时，Feed 不返回 target required。
