# 学习内容调度系统设计文档（MVP 最终版）

## 1. 文档目标

本文档定义推荐系统模块内部的**学习内容调度系统**。
它位于整个推荐系统链路的最上游，负责根据用户当前目标学习内容集合、用户-内容状态、学习事件与复习节奏，生成一轮最值得学习的目标学习内容列表，供推荐系统模块内部后续的“视频召回层”和“学习任务层”继续处理。整体链路仍然是：

> **先决定现在该学什么学习内容，再决定通过什么视频来学，最后包装成最终学习任务。**

本文档主要解决八个问题：

第一，如何把 `semantic.coarse_unit` 作为统一学习内容层来调度，而不是只处理单词。
第二，如何维护每个“用户-学习内容”的当前状态。
第三，如何定义学习事件，并区分弱事件与强事件。
第四，如何使用一套轻量但可靠的间隔重复算法生成 due review 与 new item。
第五，如何在 review backlog 存在时控制新学习内容引入，保证系统可收敛。
第六，如何输出一组结构化、可解释、可追踪的目标学习内容列表。
第七，如何让调度系统作为推荐系统模块内部能力运行，而不是直接向前端暴露接口。
第八，如何为后续 `word / phrase / grammar` 的差异化策略预留扩展位，但不在 MVP 阶段拆三套系统。

---

## 2. 系统定位与边界

## 2.1 系统在整体架构中的位置

学习内容调度系统不是一个独立对外服务，而是推荐系统模块内部的一个子系统。
从调用关系上看：

```text id="g38d4r"
前端
  ↓
[上层业务 API]
  ↓
[推荐系统模块]
  ├── 学习内容调度系统
  ├── 视频召回层
  └── 学习任务层
  ↓
最终学习任务结果
```

也就是说：

- 前端不直接调用调度系统
- 上层业务 API 也不单独调用调度器并自己拼接结果
- 上层业务 API 只调用“推荐系统模块”这一整体能力
- 推荐系统模块内部先运行调度系统，再运行视频召回层和学习任务层

前文已经明确了它在整体架构中的位置：它是**推荐系统模块内部**的上游子系统，不直接面向前端，也不单独对上层业务 API 暴露一组独立拼装接口；上层业务 API 调用的是推荐系统模块整体，调度系统只是模块内部第一层。这个边界与你更新后的整体设计文档是一致的。

---

## 2.2 这不是视频推荐系统

学习内容调度系统本质上是**学习节奏调度系统**，不是内容兴趣推荐系统。
它首先回答的是“用户现在最值得学什么”，而不是“用户现在最想看什么视频”。如果直接从视频出发，系统会退化成内容消费系统，而不是学习系统。你在整体设计文档里已经把这个判断讲得很清楚：学习产品首先要确定“学什么”，再决定“通过什么内容学”。

## 2.3 统一处理所有学习内容

这版文档不再把 `semantic.coarse_unit(kind='word')` 单独视作唯一调度对象，而是明确采用统一学习内容抽象。当前统一学习内容至少包括：

- `word`
- `phrase`
- `grammar`

它们在 MVP 阶段共享同一套主架构：

- 统一目标集合
- 统一事件日志
- 统一状态快照
- 统一调度主流程
- 统一 review/new 配额与 backlog 保护

同时，系统在状态更新与排序时允许读取 `kind`，为未来差异化规则预留扩展位，但 **MVP 阶段不拆成三套独立调度器**。这是因为当前数据库本身已经把 `semantic.coarse_unit` 作为统一学习内容主表来承载具体学习单元了。

## 2.4 MVP 当前明确不做的事情

为了保证闭环尽快成立，当前阶段明确不做以下能力：

- 词频排序
- 难度分层建模
- `phrase / grammar` 的专属复杂参数学习
- 协同过滤
- embedding 召回
- 大模型生成式长期路径优化
- FSRS、HLR 等更现代但更重的记忆模型

当前阶段只做一件事：**让统一学习内容调度闭环成立，并且可解释、可维护、可回放。** 参考设计稿中虽然讨论过更复杂的路径，但它们都不属于 MVP 必需项。

## 3. 与现有数据库的关系

### 3.1 `auth.users`

`auth.users` 是用户主体表。调度系统所有用户态数据都以它作为外键源，不复制账号信息。它提供的是“谁在学”的主体身份，而不是学习逻辑本身。

### 3.2 `semantic.coarse_unit`

这是整个调度系统最关键的基础表。
按当前数据库说明，它是“粗粒度语义单元”主表，`kind` 当前受约束为 `word / phrase / grammar`。这意味着：

- 学习对象本质上已经统一
- 单词只是其中一种最常见内容类型
- phrase 与 grammar 不是未来想象，而是当前模型已能表达的学习内容

因此，调度系统应该直接建立在 `semantic.coarse_unit` 上，而不是再造一套 `word / phrase / grammar` 分裂主表。

### 3.3 `catalog.videos`

`catalog.videos` 是视频主表。
调度系统本身不直接排序视频，但学习事件会记录 `video_id`，这样后面才有可能分析“某种学习内容在什么视频语境里学得更有效”，并把这些经验反馈到后续召回层与任务层。

### 3.4 `catalog.video_user_states`

`catalog.video_user_states` 是视频级用户状态投影表。
它不替代学习内容状态表，但在后续视频召回层与任务层里，可以用来做重复曝光控制、最近已看视频抑制、视频级去重等。对调度系统来说，它不是主输入表，但与整个推荐系统有协作关系。

## 4. 核心设计原则

### 4.1 统一抽象，统一调度，差异通过策略位表达

既然底层知识层已经统一成 `semantic.coarse_unit`，调度层就不应该再人为拆出三套主链路。MVP 阶段采用：

- 一张统一状态表
- 一张统一事件表
- 一套统一调度主循环

同时保留按 `kind` 做轻量策略分支的能力。也就是说，**架构统一，策略可扩展**。

### 4.2 只补真正缺的数据层

因为你已经有用户表、学习内容表、视频表、视频用户状态表，所以调度系统真正缺的只有：

第一，用户-学习内容状态快照。
第二，学习内容学习事件日志。
第三，可选的调度参数表与推荐快照表。

这一点和前面几轮分析结论一致：MVP 最怕的是过度造表，而不是表太少。

### 4.3 事件流是真相，状态快照是投影

如果系统后续要支持规则修复、批量重算、局部修复和回放一致性，那么必须明确：

- `learning.unit_learning_events` 是事实真相层
- `learning.user_unit_states` 是当前投影层

也就是说，状态表不是历史来源，只是事件流在当前规则版本下的一次投影结果。
在线学习时，系统是在“追加事件并同步刷新投影”；
离线 replay 时，系统是在“重新跑一遍投影”。

这条原则会直接决定 replay 的设计边界：

- 不允许另写一套只服务 replay 的状态规则
- MVP 阶段只做全量 replay，不提前设计复杂 replay scope

### 4.4 弱事件不推进调度，强事件才推进调度

这是这套系统最重要的约束之一。
“在视频里见过某个内容”不等于“已经掌握这个内容”。

因此系统必须严格区分：

- **弱事件**：曝光、注意到、点开解释
- **强事件**：带质量判定的学习行为

只有强事件才允许推进：

- `repetition`
- `interval_days`
- `next_review_at`
- `status`

这条原则对 `word / phrase / grammar` 三类学习内容全部成立。参考稿里反复强调了“视频行为不能直接等价于学会”的问题，这里正式收敛成系统级原则。

### 4.5 进度百分比必须和调度状态一致

当前阶段最稳的方案是：

- `progress_percent`：用户可见进度，来源于 `interval_days` 的对数映射
- `mastery_score`：内部排序分，不直接给用户看

这样可以避免前端显示进度与真实复习状态完全脱钩的情况。几份参考材料都指出了这一点的重要性。

### 4.6 先保复习，再引入新内容

新学习内容的引入必须服从当前 review backlog。
只要 due review 已经堆积，就应该抑制甚至完全停止新内容引入。否则系统很容易进入“复习雪崩”状态。这个判断在两份参考材料里都被强调过，属于 MVP 非常重要的工程约束。

## 5. 统一概念定义

### 5.1 学习内容

学习内容就是 `semantic.coarse_unit` 中的一条记录。
在当前系统中，它可能是：

- 一个单词
- 一个短语
- 一个语法点

它是整个系统真正稳定的调度对象。

### 5.2 目标学习内容

某个用户当前被纳入学习范围的内容。
它不等于系统里所有 coarse unit，而是当前学习计划或当前目标集合里的子集。

### 5.3 学习事件

用户一次真实学习行为在系统中的事实记录。
它是 append-only 的，不直接修改过去。

### 5.4 用户-学习内容状态

系统聚合后的当前快照，包括：

- 是否为当前目标
- 当前阶段
- 当前掌握度
- 下次复习时间
- 最近表现

### 5.5 目标学习内容列表

调度系统输出给推荐系统模块后续子层的一组中间层目标项。
它不是最终 UI 任务，更不是纯视频列表。

## 6. 数据架构设计

建议新增 schema：

```sql
create schema if not exists learning;
```

### 6.1 `learning.user_unit_states`

这是调度系统的核心状态表。
一行表示某个用户对某个学习内容的当前状态快照。

```sql
create table if not exists learning.user_unit_states (
  user_id uuid not null references auth.users(id) on delete cascade,
  coarse_unit_id bigint not null references semantic.coarse_unit(id) on delete cascade,

  is_target boolean not null default true,
  target_source text,
  target_source_ref_id text,
  target_priority numeric(5,4) not null default 0.5,

  status text not null default 'new'
    check (status in ('new', 'learning', 'reviewing', 'mastered', 'suspended')),

  progress_percent numeric(5,2) not null default 0
    check (progress_percent between 0 and 100),

  mastery_score numeric(5,4) not null default 0
    check (mastery_score between 0 and 1),

  first_seen_at timestamptz,
  last_seen_at timestamptz,
  last_reviewed_at timestamptz,
  last_recommended_at timestamptz,

  seen_count int not null default 0,
  strong_event_count int not null default 0,
  review_count int not null default 0,
  correct_count int not null default 0,
  wrong_count int not null default 0,
  consecutive_correct int not null default 0,
  consecutive_wrong int not null default 0,

  last_quality smallint check (last_quality between 0 and 5),

  repetition int not null default 0,
  interval_days numeric(8,2) not null default 0,
  ease_factor numeric(6,4) not null default 2.5,
  next_review_at timestamptz,

  suspended_reason text,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  primary key (user_id, coarse_unit_id)
);
```

#### 字段解释

`is_target`
表示该学习内容是否属于用户当前目标集合。

`target_source / target_source_ref_id`
记录该目标内容如何进入系统，例如手动加入、课程导入、视频任务带入、批量导入等。这里把 `ref_id` 设计为 `text`，避免过早绑定某种主键类型。

`target_priority`
当前目标优先级。因为你明确不做 `frequency_rank / difficulty_score`，所以它会是 MVP 阶段新内容排序最重要的显式输入信号。

`status`
当前阶段。统一使用：

- `new`
- `learning`
- `reviewing`
- `mastered`
- `suspended`

`progress_percent`
用户可见掌握度百分比。

`mastery_score`
内部排序分，不直接面向用户。

`repetition / interval_days / ease_factor / next_review_at`
构成简化 SM-2 的核心调度状态。

---

### 6.2 `learning.unit_learning_events`

这是事实日志表。
所有学习内容的学习行为都写入这张表。

```sql
create table if not exists learning.unit_learning_events (
  event_id bigserial primary key,

  user_id uuid not null references auth.users(id) on delete cascade,
  coarse_unit_id bigint not null references semantic.coarse_unit(id) on delete cascade,
  video_id uuid references catalog.videos(video_id) on delete set null,

  event_type text not null
    check (event_type in ('exposure', 'lookup', 'new_learn', 'review', 'quiz')),

  source_type text not null,
  source_ref_id text,

  is_correct boolean,
  quality smallint check (quality between 0 and 5),

  response_time_ms int,
  metadata jsonb not null default '{}'::jsonb,

  occurred_at timestamptz not null,
  created_at timestamptz not null default now()
);
```

#### 为什么它必须 append-only

因为这张表是**事实层**。
后续无论你怎么调整参数、修复 bug、重算状态，都必须能基于事件表回放重建 `user_unit_states`。状态表只是当前快照，不是历史真相。参考设计稿都强调了事件日志与状态快照分层的重要性。

---

### 6.3 `learning.user_scheduler_settings`

这张表不是第一版闭环的绝对前提，但我建议保留。
它把关键调度参数从硬编码变成显式配置。

```sql
create table if not exists learning.user_scheduler_settings (
  user_id uuid primary key references auth.users(id) on delete cascade,

  session_default_limit int not null default 20,
  daily_new_unit_quota int not null default 8,
  daily_review_soft_limit int not null default 30,
  daily_review_hard_limit int not null default 60,

  timezone text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

### 6.4 `learning.scheduler_runs` 与 `learning.scheduler_run_items`

这两张表也不是第一天就必须上线，但很推荐保留。
它们的意义是：

- 可解释
- 可审计
- 可排障
- 可重放“当时为什么推了这些内容”

```sql
create table if not exists learning.scheduler_runs (
  run_id uuid primary key,
  user_id uuid not null references auth.users(id) on delete cascade,

  requested_limit int not null,
  generated_at timestamptz not null default now(),

  due_review_count int not null,
  selected_review_count int not null,
  selected_new_count int not null,

  context jsonb not null default '{}'::jsonb
);
```

```sql
create table if not exists learning.scheduler_run_items (
  run_id uuid not null references learning.scheduler_runs(run_id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  coarse_unit_id bigint not null references semantic.coarse_unit(id) on delete cascade,

  recommend_type text not null check (recommend_type in ('review', 'new')),
  rank int not null,
  score numeric(8,4) not null,
  reason_codes text[] not null default '{}',

  primary key (run_id, coarse_unit_id)
);
```

## 7. 索引设计

```sql
create index if not exists idx_user_unit_states_target_status
on learning.user_unit_states (user_id, is_target, status);

create index if not exists idx_user_unit_states_next_review
on learning.user_unit_states (user_id, next_review_at);

create index if not exists idx_user_unit_states_last_recommended
on learning.user_unit_states (user_id, last_recommended_at);

create index if not exists idx_unit_learning_events_user_unit_time
on learning.unit_learning_events (user_id, coarse_unit_id, occurred_at desc);

create index if not exists idx_unit_learning_events_user_video_time
on learning.unit_learning_events (user_id, video_id, occurred_at desc);
```

最核心的两条查询路径始终是：

- 找当前 due review
- 找当前 new item

所以 `(user_id, status, next_review_at)` 相关索引必须优先保证。

## 8. 事件模型设计

### 8.1 统一事件类型

所有学习内容统一使用以下事件类型：

- `exposure`
- `lookup`
- `new_learn`
- `review`
- `quiz`

### 8.2 弱事件

#### `exposure`

表示用户在视频语境中看到了这个学习内容。
例如字幕里出现了某个单词、短语或语法点，并且用户有有效停留。它只表示**接触过**，不表示学会。

#### `lookup`

表示用户主动点开某个学习内容，查看释义、例句或解释。
它比 exposure 强，但仍不足以推进正式复习节奏。

### 8.3 强事件

#### `new_learn`

表示用户对该学习内容完成了一次正式学习动作。
它通常用于第一次把某个目标单元从 `new` 推到 `learning`。

#### `review`

表示用户对一个已进入学习循环的内容做了一次复习，并可提供质量反馈。

#### `quiz`

表示用户在测验或回忆任务中对该内容完成了一次可判定的作答。

### 8.4 核心原则

- 弱事件更新接触痕迹，但不推进调度
- 强事件才允许推进间隔与状态
- 这条原则对 `word / phrase / grammar` 完全一致

## 9. 视频行为到学习事件的映射

这是你这个系统区别于普通背词系统的关键。

| 用户行为                               | 事件类型    | 是否推进调度 |
| -------------------------------------- | ----------- | ------------ |
| 视频播放中出现学习内容，用户有效停留   | `exposure`  | 否           |
| 用户点开内容解释/释义/解析             | `lookup`    | 否           |
| 用户第一次正式学习该内容               | `new_learn` | 是           |
| 用户在视频任务中对该内容完成回忆或判断 | `review`    | 是           |
| 用户在题目里对该内容作答               | `quiz`      | 是           |

最重要的产品判断是：

> **不能把“视频里出现过这个内容”误判成“用户已经掌握这个内容”。**

否则整个系统会天然高估掌握度，后续调度全部失真。

## 10. 质量评分体系

底层 `quality` 保留 `0..5` 的完整取值域。
当前前端可以只暴露 4 个按钮，但统一映射到这个区间。

建议映射如下：

- 忘记：`1`
- 很难才想起：`3`
- 正常想起：`4`
- 很轻松：`5`

如果用户是“看了提示、看了答案、看了字幕后才答对”，统一下调为 `3`。这样不同学习页面与不同题型最终都能回到同一套调度尺度。

## 11. 状态模型设计

### 11.1 状态定义

#### `new`

目标学习内容已存在，但尚未发生正式学习。

#### `learning`

已经开始学习，但仍处于短周期强化阶段，记忆非常不稳定。

#### `reviewing`

已经通过初始学习，进入标准复习循环。

#### `mastered`

当前记忆稳定，退出高频日常复习，但仍可低频维护性复习。

#### `suspended`

暂时停用，不参与调度。

### 11.2 状态迁移

#### `new -> learning`

发生第一次强事件时进入 `learning`。

#### `learning -> reviewing`

满足以下条件时进入 `reviewing`：

- `strong_event_count >= 2`
- 最近两次强事件 `quality >= 3`

#### `reviewing -> mastered`

满足以下条件时进入 `mastered`：

- `interval_days >= 21`
- 最近表现稳定
- `consecutive_wrong = 0`

#### `mastered -> reviewing`

如果近期再次发生连续失败或明显不稳定，则降回 `reviewing`。

### 11.3 为什么统一用 21 天作为 MVP 掌握阈值

参考材料里同时出现过 21 天和 90 天两类目标间隔定义。为了避免文档里一套“掌握”有两种含义，MVP 阶段统一收敛为：

> **当一个学习内容的调度间隔达到 21 天，且近期表现稳定时，视为当前阶段已掌握。**

21 天的含义是：它已经足够退出高频日常复习。
90 天可以留给以后做长期维护层或长期留存分析，但不作为当前 MVP 的主状态分界。

## 12. 调度核心算法

### 12.1 总体选择

MVP 采用**简化 SM-2** 作为核心调度引擎。
原因是：

- 比 Leitner 更细粒度
- 比 FSRS、HLR 更轻、更容易冷启动
- 工程上更适合当前阶段
- 参考材料也都把 SM-2 视为 MVP 的合理基线。

### 12.2 成功分支

当 `quality >= 3`：

- `repetition += 1`
- `consecutive_correct += 1`
- `consecutive_wrong = 0`

间隔更新规则：

- `repetition = 1` 时，`interval_days = 1`
- `repetition = 2` 时，`interval_days = 3`
- `repetition = 3` 时，`interval_days = 6`
- `repetition >= 4` 时，`interval_days = round(interval_days * ease_factor)`

这里采用 **1 / 3 / 6**，而不是经典 SM-2 的 **1 / 6**。
原因是你的学习信号主要来自视频学习环境，早期学习信号相对更弱，用更保守的前几步间隔更稳。

`ease_factor` 更新公式：

[
EF_{new} = EF_{old} + (0.1 - (5-q)\times(0.08 + (5-q)\times0.02))
]

并强制：

[
EF_{new} \ge 1.3
]

### 12.3 失败分支

当 `quality < 3`：

- `repetition = 0`
- `interval_days = 1`
- `next_review_at = occurred_at + 1 day`
- `consecutive_wrong += 1`
- `consecutive_correct = 0`

这版 MVP 明确选择：

> **失败时先不衰减 EF。**

原因是：

- 行为更稳定、可解释
- 避免早期把内容打进 ease hell
- 对视频学习场景更稳妥，因为一次失败未必说明内容本身难度永久升高

### 12.4 next_review_at 计算

默认：

[
next_review_at = occurred_at + interval_days
]

当 `interval_days >= 3` 时，推荐加入轻微 jitter，用来打散同批次聚集回流。这不是第一天必须上线的功能，但建议保留为 MVP+1 增强。

## 13. progress_percent 与 mastery_score

### 13.1 progress_percent

`progress_percent` 是前端可见掌握度百分比。
它不再采用经验加减分，而是由 `interval_days` 的对数映射直接得到。

MVP 统一设定：

[
I_{target} = 21
]

进度计算：

[
progress_percent =
\min\left(
100,
\frac{\ln(interval_days + 1)}{\ln(I_{target}+1)} \times 100
\right)
]

这样做有三个好处：

- 与间隔重复的指数增长一致
- 用户在早期也能看到明显进展
- 不会因为纯曝光或重复点击而虚增百分比

### 13.2 mastery_score

`mastery_score` 是内部排序分，范围 `0..1`。
它综合反映：

- 当前 progress
- 近期强事件正确率
- 当前稳定程度

建议公式：

[
stability_score = \min(1, interval_days / 21)
]

[
mastery_score =
0.45 \times (progress_percent / 100)
+ 0.35 \times recent_accuracy
+ 0.20 \times stability_score
]

其中 `recent_accuracy` 取最近 5 次强事件的正确率。
由于三项输入都落在 `0..1`，且权重和为 `1`，所以该分数天然落在 `0..1` 区间内。

## 14. 调度主流程

一次调度执行链路如下：

第一步，读取用户调度参数。
第二步，读取该用户所有 `is_target = true` 的学习内容状态，并 join `semantic.coarse_unit`。
第三步，生成 review 候选集。
第四步，生成 new 候选集。
第五步，根据 review backlog 计算动态配额。
第六步，分别对 review/new 打分和排序。
第七步，按配额组装最终目标学习内容列表。
第八步，可选写入 `scheduler_runs / scheduler_run_items`。
第九步，把目标学习内容列表交给推荐系统模块内部后续的视频召回层。

## 15. 候选集生成

### 15.1 review 候选集

条件：

- `is_target = true`
- `status in ('learning', 'reviewing', 'mastered')`
- `next_review_at is not null`
- `next_review_at <= now`

查询示意：

```sql
select
  s.user_id,
  s.coarse_unit_id,
  c.kind,
  c.label,
  c.pos,
  c.english_def,
  c.chinese_def,
  s.status,
  s.target_priority,
  s.progress_percent,
  s.mastery_score,
  s.repetition,
  s.interval_days,
  s.ease_factor,
  s.next_review_at,
  s.consecutive_correct,
  s.consecutive_wrong,
  s.last_quality,
  s.last_recommended_at
from learning.user_unit_states s
join semantic.coarse_unit c
  on c.id = s.coarse_unit_id
where s.user_id = $1
  and s.is_target = true
  and s.status in ('learning', 'reviewing', 'mastered')
  and s.next_review_at <= $2;
```

### 15.2 new 候选集

条件：

- `is_target = true`
- `status = 'new'`

查询示意：

```sql
select
  s.user_id,
  s.coarse_unit_id,
  c.kind,
  c.label,
  c.pos,
  c.english_def,
  c.chinese_def,
  s.target_priority,
  s.progress_percent,
  s.created_at
from learning.user_unit_states s
join semantic.coarse_unit c
  on c.id = s.coarse_unit_id
where s.user_id = $1
  and s.is_target = true
  and s.status = 'new';
```

因为你明确不做词频与难度，新内容排序主要依赖：

- `target_priority`
- 是否最近被推荐过但未消费
- 进入目标集合的时间

## 16. 动态配额与 backlog 保护

### 16.1 backlog 定义

统一定义：

[
review_backlog = 当前所有 due review 候选数量
]

### 16.2 配额规则

设本轮请求上限为 `N`，当前 due backlog 为 `R`。

当 `R = 0`：

- `review_quota = 0`
- `new_quota = min(N, daily_new_unit_quota)`

当 `1 <= R <= 5`：

- `review_quota = ceil(0.5N)`
- `new_quota = min(N - review_quota, daily_new_unit_quota)`

当 `6 <= R <= 20`：

- `review_quota = ceil(0.7N)`
- `new_quota = min(N - review_quota, daily_new_unit_quota)`

当 `21 <= R <= daily_review_soft_limit`：

- `review_quota = ceil(0.85N)`
- `new_quota = min(3, N - review_quota)`

当 `R > daily_review_soft_limit`：

- `review_quota = N`
- `new_quota = 0`

当 `R > daily_review_hard_limit`：

- `review_quota = min(N, daily_review_hard_limit)`
- `new_quota = 0`
- 开启 `backlog_protection = true`

### 16.3 为什么需要 soft / hard 双阈值

- `soft_limit` 用于抑制新内容引入
- `hard_limit` 用于防止当天 session 本身过载

这样用户断更几天回来，不会被几百个 due 直接砸死。

## 17. 评分与排序

### 17.1 review 排序

review 项最终不能只按 `next_review_at` 排。
更稳的评分公式是：

[
review_score =
0.45 \times overdue_score

+ 0.25 \times target_priority
+ 0.20 \times weak_memory_score
+ 0.10 \times recency_adjustment
  ]

其中：

[
overdue_score = \min\left(1,\frac{now-next_review_at}{72h}\right)
]

[
weak_memory_score =
\min\left(
1,
0.5 \times (1-mastery_score)

+ 0.3 \times \frac{consecutive_wrong}{3}
+ 0.2 \times quality_penalty
  \right)
  ]

如果最近一次 `last_quality <= 2`，则 `quality_penalty = 1`，否则为 `0`。

`recency_adjustment` 用于避免某个项目刚被推荐又立刻反复上榜。这里它不是“负号扣分项”，而是一个二值奖励项：

- 最近 6 小时内未被推荐过，`recency_adjustment = 1`
- 最近 6 小时内已被推荐过，`recency_adjustment = 0`

也就是说，最近刚推过的项目不会被额外加分，从而达到轻微抑制重复霸榜的目的。

### 17.2 new 排序

因为不做词频与难度，MVP 新内容排序采用朴素稳定策略：

[
new_score =
0.75 \times target_priority

+ 0.15 \times freshness_score
+ 0.10 \times not_recently_recommended
  ]

其中：

- `freshness_score = 1`，当从未发生任何事件
- `freshness_score = 0.5`，当曾被推荐但未发生强事件
- 最近 24 小时没被推荐过，则 `not_recently_recommended = 1`，否则为 `0`

这两个量都是正向信号：

- 越新鲜的内容越值得进入 new 列表
- 越久没被推荐过的内容越值得重新获得曝光

### 17.3 priority-0 队列

在最终 review 排序前，再额外提一层优先级：

- `status='learning'` 且 due 的内容
- 最近一次强事件 `quality <= 2` 的失败内容

这些内容优先于普通 `reviewing` due，因为它们要么仍在短周期强化期，要么刚刚明确失败。

## 18. 最终列表组装与会话内部顺序

### 18.1 列表组装

第一步，取 priority-0 项。
第二步，再按 `review_score` 取 review。
第三步，若还有额度，再按 `new_score` 取 new。
第四步，形成目标学习内容列表。

### 18.2 会话内部顺序

调度系统输出的是“选哪些内容”，而不是完整 UI 编排，但建议在调度系统末端或任务层做轻量交错，而不要块状输出：

- 不建议：先 all review，再 all new
- 建议：`RRNRRNRRN...` 这类交错模式

这样 session 体验更好，也更符合 interleaving 的学习效果。

## 19. 状态更新规则

### 19.1 弱事件更新

#### `exposure`

更新：

- `seen_count += 1`
- `last_seen_at = occurred_at`

不更新：

- `repetition`
- `interval_days`
- `ease_factor`
- `next_review_at`
- `status`

#### `lookup`

更新：

- `seen_count += 1`
- `last_seen_at = occurred_at`

仍不推进正式调度。

### 19.2 强事件更新

所有强事件统一更新：

- `seen_count += 1`
- `strong_event_count += 1`
- `last_seen_at = occurred_at`

若 `event_type in ('review','quiz')`：

- `review_count += 1`
- `last_reviewed_at = occurred_at`

若 `is_correct = true`：

- `correct_count += 1`
- `consecutive_correct += 1`
- `consecutive_wrong = 0`

若 `is_correct = false`：

- `wrong_count += 1`
- `consecutive_wrong += 1`
- `consecutive_correct = 0`

更新 `last_quality` 后，再进入 SM-2 状态更新、progress 重算、mastery_score 重算和状态迁移。

### 19.3 状态更新的事务边界

事件写入与状态投影更新必须放在同一个事务里。
不要把“先写事件、状态异步慢慢补”作为默认实现，否则推荐时会读到不一致状态。

更准确地说，这里的同事务应包含三层：

1. 追加事件到 `learning.unit_learning_events`
2. 更新 `learning.user_unit_states`

这样在线链路和离线 replay 才能在同一套一致性口径下运行。

## 20. 冷启动与目标集合建立

### 20.1 新用户冷启动

如果新用户已经有目标学习内容集合，则为每个目标内容插入一条 `user_unit_states`：

- `status = 'new'`
- `progress_percent = 0`
- `mastery_score = 0`
- `interval_days = 0`
- `next_review_at = null`

### 20.2 目标集合如何进入系统

调度系统不负责生成目标集合，它只消费目标集合。
目标学习内容可以通过以下方式进入：

- 用户手动加入
- 课程或词书导入
- 视频学习过程中加入
- 某个学习计划批量写入

最终都落到：

`learning.user_unit_states (is_target = true)`

### 20.3 不做“已有百分比反推 interval”的功能

虽然参考文档讨论过从已有进度反推 interval 的可能性，但这会引入另一套初始化语义。当前 MVP 先不做，保持状态初始化简单统一。

## 22. 对外接口边界说明

这里必须明确：

**学习内容调度系统本身不直接提供面向前端或上层业务 API 的独立 HTTP API。**

它的输入输出应理解为推荐系统模块内部的调用契约，而不是对外 REST 接口。也就是说，前面写的推荐列表结构、本轮限制、事件写入结构，都是**内部命令对象 / 查询对象**的语义，而不是对前端暴露的网络协议。

前端与上层业务 API 的交互，由上层业务 API 自己定义。
上层业务 API 调用推荐系统模块时，再由推荐系统模块内部调用调度系统。

## 23. 状态重算与回放机制

这是工程上必须有的一层。

- `learning.unit_learning_events` 是事实层
- `learning.user_unit_states` 是状态快照层

当出现以下情况时，允许基于事件回放重算状态：

- 调度参数调整
- 质量映射规则修改
- 某段时间写入 bug
- 想重新计算 progress
- 想修复一批错误状态

### 23.1 replay 的正确语义

replay 本质上不是“把旧状态修补一下”，而是：

> **基于事件流重新构建某个用户-学习内容聚合，再把聚合投影回状态快照。**

因此更准确的口径应是：

- 事件流是真相
- 快照是投影
- replay 是重跑投影

### 23.2 replay scope

MVP 阶段，replay 只支持一种 scope：

- `FullUserReplay`

也就是重建某个用户的全部学习内容状态。

按用户全量重建有两个直接好处：

- 语义最简单，不容易误解
- 不会引入局部回放的历史上下文丢失问题

### 23.3 replay 原则

回放原则应改为：

1. 指定用户
2. 清空该用户的状态快照
3. 按 `occurred_at asc, event_id asc` 读取该用户的完整事件历史
4. 重建聚合
5. 重新投影 `user_unit_states`

更复杂的 replay scope，例如：

- 单 unit replay
- 按时间范围筛受影响 unit
- 增量 replay

都属于 MVP 之后再考虑的增强能力。

这也是为什么事件表必须 append-only。

## 24. 边界场景处理

### 24.1 用户断更很多天

如果 due backlog 很大：

- 新内容立即停供
- 只推最急的 due 项
- 进入 `backlog_protection`

### 24.2 用户只看视频，不做强学习行为

会累计很多 `exposure`，但不会显著推进 `interval_days` 与 `progress_percent`。
这不是 bug，而是系统应该坚持的口径。

### 24.3 某项刚推荐过，又再次出现在推荐列表

通过 `last_recommended_at` 和 `recency_adjustment` 降低重复霸榜概率。
如果仍然入榜，通常说明它真的 overdue 很严重，或者刚刚失败过。

### 24.4 同一天学了很多新项目导致后续扎堆回流

可通过 interval jitter 轻度打散，但不作为 MVP 第一阶段硬依赖。

## 27. 总结

这版最终 MVP 设计的核心判断可以概括成一句话：

> **围绕 `auth.users × semantic.coarse_unit` 维护统一的用户-学习内容状态，使用统一事件流水记录 `word / phrase / grammar` 的学习行为，严格区分弱曝光与强学习行为，用简化 SM-2 管理记忆节奏，用对数映射生成可见进度，再输出一组可解释的目标学习内容列表交给推荐系统模块内部后续子层。**

它的优点在于：

第一，它真正尊重了你系统底层已经存在的统一学习内容抽象，而不是又退回“只做单词”的局部方案。
第二，它不依赖当前并不存在的词频、难度字段，完全建立在你现在真实可落地的数据库现实上。
第三，它解决了视频学习场景里最危险的问题：不会把“刷过视频”误判成“已经掌握某个学习内容”。
