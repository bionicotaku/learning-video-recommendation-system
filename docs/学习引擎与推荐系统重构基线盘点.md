# 学习引擎与推荐系统重构基线盘点

## 1. 盘点目的

本文档用于冻结当前仓库在重构开始前的真实状态，作为后续拆分：

- `internal/learningengine`
- `internal/recommendation`

的对照基线。

本盘点对应重构任务列表中的：

- 子任务 01：冻结当前边界并建立重构基线

## 2. 当前代码结构现状

当前 `internal` 目录下只有一个实际模块：

```text
internal/
  recommendation/
    scheduler/
```

也就是说，Learning engine 与 Recommendation 当前仍然混合在：

- `internal/recommendation/scheduler`

之下。

当前并不存在：

- `internal/learningengine`
- `internal/recommendation` 下的独立 recommendation 代码骨架

## 4. 当前表与字段 owner 盘点

## 4.1 当前表 owner 现状

当前代码与 migration 实际定义的表有：

- `learning.user_unit_states`
- `learning.unit_learning_events`
- `learning.user_scheduler_settings`
- `learning.scheduler_runs`
- `learning.scheduler_run_items`

按最终方案，它们的正确 owner 应为：

| 当前对象 | 当前 schema | 最终 owner | 最终处理 |
| --- | --- | --- | --- |
| `user_unit_states` | `learning` | Learning engine | 保留，迁入 Learning engine migration |
| `unit_learning_events` | `learning` | Learning engine | 保留，迁入 Learning engine migration |
| `user_scheduler_settings` | `learning` | Recommendation | 暂按 Recommendation 配置输入保留，后续与 Recommendation 一起重组 |
| `scheduler_runs` | `learning` | Recommendation | 迁到 `recommendation.scheduler_runs` |
| `scheduler_run_items` | `learning` | Recommendation | 迁到 `recommendation.scheduler_run_items` |

## 4.2 当前关键字段 owner 现状

| 字段 | 当前所在表 | 最终 owner | 最终处理 |
| --- | --- | --- | --- |
| `is_target` | `learning.user_unit_states` | Learning engine | 保留 |
| `target_source` | `learning.user_unit_states` | Learning engine | 保留 |
| `target_source_ref_id` | `learning.user_unit_states` | Learning engine | 保留 |
| `target_priority` | `learning.user_unit_states` | Learning engine | 保留 |
| `last_recommended_at` | `learning.user_unit_states` | Recommendation | 迁出并删除旧列 |

## 5. 当前 migration 现状

当前仓库只有一套 migration 根：

- [scheduler migration](/Users/evan/Downloads/learning-video-recommendation-system/internal/recommendation/scheduler/infrastructure/migration)

这违反最终重构要求，因为最终必须只保留两套干净 migration：

1. Learning engine migration
2. Recommendation migration

因此当前 migration 基线结论是：

- 旧 `scheduler` migration 必须整体退役
- 新模块必须各自建立从空库独立可重放的 migration 根

执行结果：

- 当前旧结构是可运行的
- 后续重构每一步都必须至少不低于这个基线

## 7. 重构起点结论

当前仓库处于以下状态：

1. Learning engine 与 Recommendation 仍混在 `internal/recommendation/scheduler`
2. 数据表 owner 混杂
3. `last_recommended_at` 仍在错误表里
4. migration 只有一套旧根目录
5. 旧结构当前是可运行、可测试的

因此后续重构的正确顺序应是：

1. 先冻结现状
2. 先拆 migration 根
3. 再拆代码 owner
4. 最后删除旧 `scheduler`
