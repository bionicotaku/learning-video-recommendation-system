# Recommendation 详细设计文档

## 0. 文档信息

文档名称：Recommendation 详细设计文档
适用阶段：MVP
目标读者：后端、数据、测试、Learning engine 对接方、Catalog 对接方
文档目标：定义 Recommendation 的职责边界、内部模块划分、共享读模型、核心规则、接口、事务、审计、测试与实施约束，使其可以独立实施，并与 Learning engine、Catalog 并行开发。当前 MVP 的唯一核心目标是：**输入一个用户，稳定输出一组当前足够合理的视频列表**。Recommendation 与 Learning engine 是平级模块；Learning engine 负责维护学习状态，Recommendation 负责基于学习状态与内容索引输出视频列表。当前 Catalog 的内容事实链为 `videos -> video_transcripts -> video_transcript_sentences -> video_semantic_spans -> video_unit_index`，Recommendation 应建立在这条链之上，而不是要求独立的 `video_segments` / `segment_unit_mappings` 物理表。

## 1. 模块定位

Recommendation 是视频优先的教育推荐模块。它回答的问题只有一类：在当前时刻，应该给这个用户返回哪些视频；这些视频为什么适合当前学习阶段；它们分别覆盖了哪些当前应学内容、顺带承载了哪些近期内容；在供给稀疏时，如何仍然返回一组不空且不失焦的视频列表。换句话说，Recommendation 负责“这次推什么视频”，不负责“学得怎么样”。Learning engine 输出 `learning.user_unit_states`，Recommendation 读取这些状态，但不写 `learning.unit_learning_events`，也不写 `learning.user_unit_states`。Recommendation 自己拥有 serving state、video recommendation 审计、需求规划、候选生成、证据解析、视频聚合、排序与选择逻辑。

Recommendation 不是 Learning engine，不是事件归约系统，不是学习状态真相层，不是课程编排系统，不是非视频任务系统，也不是通用娱乐 feed。对上层业务 API 而言，Recommendation 应作为一个整体模块对外提供能力，而不是把“调度器、候选生成、排序器、选择器”分别暴露出去，否则 Recommendation 的内部组织会被泄露到外部边界，造成耦合。

## 2. 设计目标与非目标

Recommendation 当前只追求三件事。第一，结果稳定，不能因为 exact target 稀疏就塌掉。第二，目标不失焦，不能为了补量把推荐做成泛视频推荐。第三，工程可落地，规则化、可解释、先不引入 ML，也不把 task 层耦进主流程。当前 Recommendation 的核心问题不是 unit 调度本身，而是视频候选生成与排序：如果候选池只由当前 exact target unit 反查得到，数量和多样性都会不足，后续 ranking 空间也会很小。Recommendation 的设计必须正面解决这个结构性问题。

本期明确不做以下事情：task layer 进入主流程、非视频 fallback task、embedding / vector recall、ML ranking、复杂个性化兴趣模型、完整 curriculum / frontier 平台、Recommendation 回写 Learning engine。这些都不属于当前 MVP 要解决的问题。当前阶段只把“输入用户 -> 输出视频列表”的闭环做稳。

## 3. 设计原则

第一，最终输出始终是视频列表。Recommendation 对外返回的是排序后的视频列表，以及每条视频的推荐原因与覆盖信息，不是 unit list，不是 task list，也不是学习任务卡片。task 层先后置，不进入当前主流程。

第二，Learning engine 与 Recommendation 的边界保持清晰。Learning engine 负责学习事件、学习状态、due/new/mastery 等信号；Recommendation 负责根据学习状态与内容供给生成视频候选池、对视频做教育场景排序、输出最终视频列表，并维护自己的审计与 serving state。`last_recommended_at` 之类的字段属于 Recommendation 的投放状态，不能进入学习状态表。

第三，planner 输出的是“学习需求画像”，而不是一个硬 unit 名单。Recommendation 的主问题不是“当前 unit 调度不够精确”，而是“视频候选生成与排序空间不够”。因此上游必须输出可执行的需求分桶和预算，而不是只输出 exact unit 列表。

第四，planner 必须具备轻量供给感知。它不能只看“该学什么”，还要知道“视频库接不接得住”。这一步不要求复杂库存优化，但必须知道某个 unit 的视频供给是否为 `none / weak / ok / strong`，并在 exact 候选不足时扩大 bundle / soft-future 的预算。

第五，内部证据单位是 span / sentence-window，外部输出单位是 video。当前 Catalog 已明确以 `video_semantic_spans` 作为最细事实层，以 `video_unit_index` 作为视频级 Recall 主入口，并不建立 `catalog.video_segments` / `catalog.segment_unit_mappings`。因此 Recommendation 的内部证据建模必须适配这条内容链：先在视频-学习单元层做粗候选，再通过 `evidence_span_refs` 回查 spans 与 sentences 形成细粒度证据。

第六，召回不能只有 exact lane。必须允许当前核心需求、多 unit 组合承载、近未来内容以及质量兜底一起进入候选池，但权重必须受控。

## 4. 与 Learning engine、Catalog 的数据契约

Learning engine 对 Recommendation 暴露的核心输入是 `learning.user_unit_states`。Recommendation 至少使用这些字段：`is_target`、`target_priority`、`status`、`progress_percent`、`mastery_score`、`last_quality`、`next_review_at`、`recent_quality_window`、`recent_correctness_window`、`strong_event_count`、`review_count`。Learning engine 不提供 `last_recommended_at`，该字段属于 Recommendation 自己的 serving state。

Catalog 当前提供 Recommendation 所需的内容事实链：`catalog.videos`、`catalog.video_transcripts`、`catalog.video_transcript_sentences`、`catalog.video_semantic_spans`、`catalog.video_unit_index`，并可选提供 `catalog.video_user_states` 作为消费状态投影。Catalog 当前明确不建立 `catalog.video_segments` 与 `catalog.segment_unit_mappings`，因为数据库中的 `video_id` 本身就是最终可播放、可推荐的切片视频对象，而语义跨度映射已经由 `video_semantic_spans` 与 `video_unit_index` 共同承担。

Recommendation 自己拥有的读模型与状态包括：`recommendation.v_recommendable_video_units`、`recommendation.v_unit_video_inventory`、`recommendation.user_unit_serving_states`、`recommendation.user_video_serving_states`、`recommendation.video_recommendation_runs`、`recommendation.video_recommendation_items`。其中 `v_recommendable_video_units` 和 `v_unit_video_inventory` 是 Recommendation 的预计算读视图，不改变 Catalog 的 owner，只是为 Recommendation 的热路径提供轻量读模型。

## 5. 总体架构

Recommendation 的总体数据流如下：

```text
RecommendationRequest
    +
learning.user_unit_states
    +
recommendation.v_recommendable_video_units
    +
recommendation.v_unit_video_inventory
    +
recommendation.user_unit_serving_states
    +
recommendation.user_video_serving_states
    +
(可选) catalog.video_user_states
    +
catalog.video_semantic_spans
    +
catalog.video_transcript_sentences
        ↓
[1] Context Assembler
        ↓
[2] Demand Planner
        ↓
Demand Bundle
        ↓
[3] Candidate Generator
        ↓
VideoUnitCandidates
        ↓
[4] Evidence Resolver
        ↓
ResolvedEvidenceWindows
        ↓
[5] Video Evidence Aggregator
        ↓
Video Candidates
        ↓
[6] Video Ranker
        ↓
Ranked Videos
        ↓
[7] Video Selector
        ↓
Selected Videos
        ↓
[8] Explanation Builder
        ↓
Final Video List
        ↓
[9] Audit Writer + Serving State Manager
```

这里的关键点有三个。第一，planner 不再输出一个硬 unit 名单，而是输出一份带预算的 Demand Bundle。第二，候选生成不再只有 exact lane，而是多 lane 并行扩张。第三，内部按细粒度证据建模，外部按 video 输出结果。主审计对象是 video recommendation run/item，而不再只围绕 scheduler item。

## 6. 核心领域对象

`RecommendationRequest` 表示一次推荐请求，至少包含 `user_id`、`target_video_count`、`preferred_duration_sec`、`session_hint`、`request_context`。默认 `target_video_count = 8`，默认时长偏好可取 `[45, 180]` 或 `[60, 240]` 秒。内部一旦涉及与 Catalog 内容时间字段比较，应统一转换为毫秒，以与 Catalog 的 `duration_ms / start_ms / end_ms` 对齐。

`RecommendationContext` 是 Context Assembler 的输出，至少包含：规范化后的请求、当前时间、active unit states、unit inventory、unit serving states、video serving states，以及可选的 video user states。

`DemandUnit` 表示某个学习单元在本轮推荐中的需求项，至少包含 `unit_id`、`bucket`、`weight`、`supply_grade`。

`DemandBundle` 表示本轮视频推荐的学习需求画像，包含四个 bucket：`hard_review`、`new_now`、`soft_review`、`near_future`，并包含 `session_mode`、`target_video_count`、`preferred_duration_sec`、`lane_budget`、`mix_quota`、`flags`。当前的最终方案采用 `hard_review / new_now / soft_review / near_future` 这四个 bucket，以及 `exact_core / bundle / soft_future / quality_fallback` 四条 lane，这是当前最适合直接定稿的收敛方案。

`EvidenceRef` 表示一个轻量证据引用，来源于 `video_unit_index.evidence_span_refs`，至少包含 `sentence_index` 与 `span_index`。它不是“全局 best”，只是一个无歧义、可回查的证据引用。

`VideoUnitCandidate` 是 Candidate Generator 的统一输出，它表示某个视频对某个学习单元、在某条 lane 下的候选命中，至少包含：`video_id`、`coarse_unit_id`、`lane`、`bucket`、`unit_weight`、`mention_count`、`sentence_count`、`coverage_ms`、`coverage_ratio`、`sentence_indexes`、`evidence_span_refs`、`sample_surface_forms`、`duration_ms`、`mapped_span_ratio`、`candidate_score`。

`ResolvedEvidenceWindow` 是 Evidence Resolver 的输出，表示某个 `VideoUnitCandidate` 在本轮推荐语义下被解析成的更细粒度证据窗口，至少包含：`best_evidence_ref`、`best_evidence_start_ms`、`best_evidence_end_ms`、`window_sentence_indexes`、`window_start_ms`、`window_end_ms` 等。

`VideoCandidate` 是 Aggregator 聚合后的 video-level 候选对象，至少包含：`video_id`、`lane_sources`、`dominant_bucket`、`dominant_unit_id`、`covered_hard_review_units`、`covered_new_now_units`、`covered_soft_review_units`、`covered_near_future_units`、`hard_review_cover`、`new_now_cover`、`soft_review_cover`、`near_future_cover`、`coverage_strength_score`、`bundle_value_score`、`educational_fit_score`、`future_value_score` 以及最终需要落审计的 `best_evidence_*`。

`FinalRecommendationItem` 是对外返回的最终条目，至少包含：`video_id`、`rank`、`score`、`reason_codes`、`covered_units`、`covered_hard_review_units`、`covered_new_now_units`、`covered_soft_review_units`、`covered_near_future_units`、`best_evidence`、`explanation`。

## 7. 内部模块设计

### 7.1 Recommendation Facade / Usecase Orchestrator

这是 Recommendation 对外暴露的唯一主入口。上层业务 API 只调用 `GenerateVideoRecommendations`，不直接感知内部的 planner、candidates、resolver、ranker 或 selector。

它负责组织整体时序、调用各内部模块、组织最终写事务、返回统一结果。它不负责写 bucket 规则、不负责算视频分、不负责 selection quota，也不直接做 SQL 查询。

推荐伪代码如下：

```text
GenerateVideoRecommendations(req):
  ctx         = ContextAssembler.Assemble(req)
  bundle      = DemandPlanner.Plan(ctx)
  candidates  = CandidateGenerator.Generate(ctx, bundle)
  resolved    = EvidenceResolver.Resolve(candidates, ctx, bundle)
  videos      = VideoEvidenceAggregator.Aggregate(resolved, ctx, bundle)
  ranked      = VideoRanker.Rank(videos, ctx, bundle)
  selected    = VideoSelector.Select(ranked, ctx, bundle)
  response    = ExplanationBuilder.Build(selected, ctx, bundle)

  tx:
    AuditWriter.WriteRunAndItems(...)
    ServingStateManager.ApplySelection(...)

  return response
```

### 7.2 Context Assembler

Context Assembler 的目标是把 Recommendation 所需输入一次性装配成统一上下文。它从 Learning engine 读取 `learning.user_unit_states`，推荐过滤条件是 `user_id = $1 AND is_target = true AND status <> 'suspended'`；从 Recommendation 自己的读模型读取 `v_recommendable_video_units`、`v_unit_video_inventory`、`user_unit_serving_states`、`user_video_serving_states`；可选读取 `catalog.video_user_states`。它不应在一开始把所有 spans / sentences 全部读进来，`catalog.video_semantic_spans` 与 `catalog.video_transcript_sentences` 应由后续模块按需延迟读取。它只负责上下文装配，不做需求规划。

### 7.3 Demand Planner

Demand Planner 是 Recommendation 的起点。它把用户当前学习状态转成一份 Demand Bundle，而不是一个硬 unit 名单。它的输入包括：active unit states、unit inventory、unit serving states、request 中的 `target_video_count`、`preferred_duration_sec` 与 `session_hint`。它的输出是带四个 bucket、lane budget、mix quota 与 flags 的 Demand Bundle。

当前阶段固定四个 bucket：`hard_review`、`new_now`、`soft_review`、`near_future`。当前 Bucket precedence 必须写死为：

```text
hard_review > new_now > soft_review > near_future
```

一个 unit 在一次 planner run 中只能进入一个主 bucket。后续 lane 预算、coverage 统计、selector quota 都以这个唯一 bucket 为准。这是当前方案里最重要的约束之一。

`hard_review` 用于当前必须优先服务的学习内容，例如 overdue、due now、最近失败、最近变脆弱的 unit。`new_now` 用于当前允许引入的新内容，但必须 supply-aware，供给极差的新内容不应成为主推。`soft_review` 用于今天不是硬 due，但近期弱、容易忘、值得顺带覆盖的内容。`near_future` 用于未来很快进入学习/复习窗口的内容，是扩召回的合法来源，但不能压过当前主线。

Demand Planner 必须具备轻量供给感知。它使用 `v_unit_video_inventory` 中的 `distinct_video_count`、`avg_mention_count`、`avg_sentence_count`、`avg_coverage_ms`、`avg_coverage_ratio`、`strong_video_count`、`supply_grade` 这些确定性聚合事实。关键规则是：`hard_review` 永不因供给弱而消失；`new_now` 必须 supply-aware；当 `hard_review` 供给弱时，提高 `bundle / soft_future` 预算；`quality_fallback` 只作为保险丝。

### 7.4 Candidate Generator

Candidate Generator 是 Recommendation 当前最需要重构的模块。它根据 Demand Bundle，通过四条 lane 生成足够健康的 `VideoUnitCandidate` 候选池，而不再假设底层有独立物理 segment 表。它的第一跳输入是 `recommendation.v_recommendable_video_units`，即“视频 × coarse unit”的轻量联合读模型。

当前 MVP 固定四条 lane：`exact_core`、`bundle`、`soft_future`、`quality_fallback`。这是最终收敛方案。

`exact_core` 的目标是保证当前最核心学习需求一定进入候选池。它的输入是 `hard_review` 与 top-ranked `new_now`。在当前 Catalog 能力下，它的强度不看 `primary/contextual/incidental`，而看 `mention_count`、`sentence_count`、`coverage_ms / coverage_ratio`、`mapped_span_ratio` 这些确定性事实。它提供高精度的当前主线候选，但不能承担候选扩张职责。

`bundle` 是当前方案里最关键的 lane。它的目标是找那些整体上能承载多个高价值 unit 的视频，而不是单点命中。它的输入是 `hard_review`、`new_now`、`soft_review` 与少量高权重 `near_future`。实现上应先从 `v_recommendable_video_units` 拉出这些 unit 的 video-unit rows，再在内存按 `video_id` 聚合，找出满足 bundle 条件的视频。默认要求至少覆盖两个有价值 unit，且至少包含一个 `hard_review` 或 `new_now`。当 `hard_review_low_supply = true` 时，允许动态放宽为 `1 个 hard/new + 1 个 soft_review`，或受控的 `2 个 soft_review`。这条 lane 直接解决“视频目标太单一、候选池太窄”的问题。

`soft_future` 的目标是扩大候选池，但不让系统跑偏。它输入 `soft_review` 与 `near_future`，只做补充，不承担主排序目标。尽管这条 lane 内部可以把 `soft_review` 与 `near_future` 放在一起扩召回，但后续 Aggregator、Ranker、Selector 里必须继续把两者分开统计。当前方案明确认为：“当前主线优先，边缘不稳优先于未来内容”，因此 `soft_review` 应高于 `near_future`。

`quality_fallback` 的目标是在前三条 lane 合起来仍不足时，兜底保证列表不空、不塌。它只在 distinct candidate videos 明显不足，且预测无法满足 `target_video_count` 时触发。Fallback 候选只能来自教育适配较高、时长合适、transcript 清晰、近期未高频重复推荐、且至少与当前需求弱相关的视频。它只能补位，不能反客为主，最终列表中通常最多 1 条。若仍无足够合理候选，宁可 under-fill，也不要乱推。

### 7.5 Evidence Resolver

Evidence Resolver 把 Candidate Generator 输出的 `VideoUnitCandidate` 解析为更细粒度的证据窗口。它利用 `evidence_span_refs` 回查 `catalog.video_semantic_spans` 与 `catalog.video_transcript_sentences`，为每个候选生成一个本轮推荐语义下的“本地最佳证据窗口”，供 explanation、best evidence 定位、coverage density / window completeness 计算和最终审计使用。

Evidence Resolver 不需要把所有 spans 全量展开。它只解析 `evidence_span_refs` 引用到的少量 spans，并按需补上下文句子窗口即可。当前阶段的本地 best 选择规则可以是稳定但低语义承诺的，例如：优先最早出现的 ref；若多个 ref 在同一句中，则优先 `start_ms` 更早者。关键点是，这个 best 是 Recommendation 当前这轮解释语义下的 best，不是 Catalog 事实层中的全局 best。

### 7.6 Video Evidence Aggregator

Video Evidence Aggregator 的目标是把多个 `ResolvedEvidenceWindow` 聚合成 video-level candidate。它必须保证：同 `video_id + unit_id` 只保留最强证据，以防同一视频因反复提到同一个 unit 被高估；可选地保留第二条强证据作为 30% 增量，但第三条以后忽略。它还必须为每个视频选出一个 `best_evidence`，用于 UI 高亮、explanation、调试与审计。

Aggregator 输出的覆盖统计必须分别统计：`covered_hard_review_units`、`covered_new_now_units`、`covered_soft_review_units`、`covered_near_future_units`。它至少输出这些 video-level 特征：`hard_review_cover`、`new_now_cover`、`soft_review_cover`、`near_future_cover`、`coverage_strength_score`、`bundle_value_score`、`educational_fit_score`、`future_value_score`、`lane_sources`、`distinct_matched_unit_count`。

其中 `coverage_strength_score` 当前不依赖上游 `relevance_type`，而依赖确定性内容强度，例如 `mention_count`、`sentence_count`、`coverage_ratio`、`evidence_window_focus`、`mapped_span_ratio`。`educational_fit_score` 主要看 `duration_fit`、`coverage_density`、`window_completeness` 与 `noise_penalty`。

### 7.7 Video Ranker

Video Ranker 的职责是给每个 video 计算基础分 `base_score`。它的排序目标不是“命中当前 target unit 最多”，而是“这个视频作为当前一轮学习内容载体的总体价值有多高”。

建议的核心评分维度包括：`demand_coverage`、`coverage_strength_score`、`bundle_value_score`、`educational_fit_score`、`future_value_score`，以及轻量的 `freshness_score`，再减去 `recent_served_penalty`、`recent_watched_penalty`（如有）和 `overload_penalty`。当前建议的初版公式为：

```text
base_score =
  0.40 * demand_coverage
+ 0.18 * coverage_strength_score
+ 0.15 * bundle_value_score
+ 0.15 * educational_fit_score
+ 0.05 * future_value_score
+ 0.05 * freshness_score
- 0.03 * recent_served_penalty
- 0.02 * overload_penalty
```

`demand_coverage` 建议定义为：

```text
demand_coverage =
  0.50 * hard_review_cover
+ 0.20 * new_now_cover
+ 0.20 * soft_review_cover
+ 0.10 * near_future_cover
```

这里的原则是：当前主线优先，组合承载能力第二，未来价值只能是弱加分项，重复与过载必须惩罚。

### 7.8 Video Selector

Selector 决定最终返回给用户的列表，不能只按基础分取前 N 条，否则列表会很难看。它的职责是保证当前主线覆盖、去重、冷却、future 占比受控、fallback 占比受控，并在极端稀疏库存下优雅降级。

当前固定三种 selector mode：`normal`、`low_supply`、`extreme_sparse`。`normal` 表示 `hard_review` 供给健康且 exact/bundle 候选足够；`low_supply` 表示 `hard_review_low_supply = true` 或 exact_core distinct videos 明显不足；`extreme_sparse` 表示加上 fallback 后仍不足，或在合理约束下无法凑满 N 条。

配额规则建议比例化而不是写死绝对数。令 `N = target_video_count`。在 `normal` 模式下，`core_dominant_min = ceil(N * 0.50)`，`future_dominant_max = floor(N * 0.25)`，`fallback_max = 1`，`same_dominant_unit_max = 2`。在 `low_supply` 模式下，`core_dominant_min = ceil(N * 0.375)`，`future_like_max = ceil(N * 0.375)`，`fallback_max = 1`。在 `extreme_sparse` 模式下，允许 under-fill，不强行凑满 N 条。

选择算法建议使用轻量贪心。Phase A 先保主线，从高分到低分遍历，优先选那些能覆盖尚未覆盖的 `hard_review / new_now` 的视频，直到满足 `core_dominant_min` 或无候选。Phase B 再按边际收益填充：`marginal_score = base_score + uncovered_core_gain + uncovered_soft_gain - redundancy_penalty - recent_repeat_penalty`。Phase C 最后才允许 fallback 进入。硬原则是：**宁可 under-fill，也不要为了凑满条数而推明显弱相关视频。**

### 7.9 Explanation Builder

Explanation Builder 是独立支撑模块，不参与选择，只负责把事实型信号转成人可读结果。每条视频都应生成 `reason_codes` 与 `explanation`。建议的 reason codes 包括：`hard_review_covered`、`new_unit_introduced`、`soft_review_supported`、`near_future_warmup`、`bundle_coverage_high`、`strong_evidence_window`、`good_learning_fit`、`recently_not_served`、`low_supply_core_preserved`、`fallback_quality_fill`。这里不使用 `strong_primary_match` 这类依赖不存在上游字段的 reason code。 explanation 应采用模板化拼接，例如：“覆盖 2 个当前应复习内容，顺带覆盖 1 个当前可引入的新内容，并支持 1 个近期不稳内容，主要学习证据集中在 00:01.24–00:01.82，最近未重复推荐。”

### 7.10 Serving State Manager

Serving State Manager 维护 Recommendation 自己的投放状态，包括 `user_unit_serving_states` 和 `user_video_serving_states`。前者用于 unit 级冷却和统计哪些 unit 最近已被视频承载，后者用于 video 级冷却和统计视频最近被推荐次数。若系统已有 `catalog.video_user_states`，Recommendation 可将其作为可选消费状态输入，用于 penalty，但这些消费真相不应写回 Recommendation 自己的表。

### 7.11 Audit Writer

Audit Writer 记录每次 video recommendation run 及其 items。run 级至少记录：`run_id`、`user_id`、`request_context`、`session_mode`、`selector_mode`、`planner_snapshot`、`lane_budget_snapshot`、`candidate_summary`、`underfilled`、`result_count`。item 级至少记录：`video_id`、`rank`、`score`、`primary_lane`、`dominant_bucket`、`dominant_unit_id`、`reason_codes`、`covered_*_count`、`best_evidence_sentence_index`、`best_evidence_span_index`、`best_evidence_start_ms`、`best_evidence_end_ms`。主审计对象是 video recommendation run/item，而不是 scheduler-only 的中间产物。

## 8. Recommendation 自有数据层

`recommendation.v_recommendable_video_units` 是 Recommendation 的关键读视图，它把 Recommendation 常用的“视频 × coarse unit”联合字段预先拼出来。建议字段包括：`video_id`、`coarse_unit_id`、`mention_count`、`sentence_count`、`first_start_ms`、`last_end_ms`、`coverage_ms`、`coverage_ratio`、`sentence_indexes`、`evidence_span_refs`、`sample_surface_forms`、`duration_ms`、`mapped_span_ratio`、`status`、`visibility_status`、`publish_at`。

`recommendation.v_unit_video_inventory` 是 Demand Planner 的库存感知视图，建议字段包括：`coarse_unit_id`、`distinct_video_count`、`avg_mention_count`、`avg_sentence_count`、`avg_coverage_ms`、`avg_coverage_ratio`、`strong_video_count`、`supply_grade`、`updated_at`。这层视图让 planner 知道“该学什么”和“视频库接不接得住”之间的关系。

`recommendation.user_unit_serving_states` 维护 unit 级 serving state。`recommendation.user_video_serving_states` 维护 video 级 serving state。`recommendation.video_recommendation_runs` 与 `recommendation.video_recommendation_items` 维护主审计对象。由于最终输出是 video list，因此 Recommendation 的审计中心也必须是 video recommendation 维度，而不是只围绕 scheduler item。

## 9. 外部接口与内部接口

Recommendation 对外只暴露一个主用例：`GenerateVideoRecommendations`。请求结构至少包含 `user_id`、`target_video_count`、`preferred_duration_sec`、`session_hint` 与 `request_context`。返回结构至少包含 `run_id`、`selector_mode`、`underfilled`，以及一个 `videos[]` 数组；每个视频条目包含 `video_id`、`rank`、`score`、`reason_codes`、`covered_units`、`covered_hard_review_units`、`covered_new_now_units`、`covered_soft_review_units`、`covered_near_future_units`、`best_evidence` 和 `explanation`。

内部接口建议分为读取接口与模块接口两类。读取接口包括：`LearningStateReader`、`RecommendableVideoUnitReader`、`UnitInventoryReader`、`SemanticSpanReader`、`TranscriptSentenceReader`、`UnitServingStateRepository`、`VideoServingStateRepository`、`RecommendationAuditRepository`。模块接口包括：`DemandPlanner`、`CandidateGenerator`、`EvidenceResolver`、`VideoEvidenceAggregator`、`VideoRanker`、`VideoSelector`、`ExplanationBuilder`。只有 Context Assembler、Serving State Manager、Audit Writer 直接接触 repository，其余模块只消费领域对象，避免业务规则与 SQL 混杂。

## 10. 事务、一致性与并发

Recommendation 的只读计算过程不应包在长事务里。推荐做法是：先完成所有只读装配与纯计算，最后只在写 `video_recommendation_runs`、`video_recommendation_items`、`user_unit_serving_states`、`user_video_serving_states` 时开启短事务。这样既保证写入一致性，又避免长时间锁用户或锁表。

同一用户短时间内多次请求 Recommendation，MVP 不强制做全局串行，但建议每次请求以独立 `run_id` 区分，serving state 使用幂等 upsert。若某些场景需要更强一致性，可以对单用户的 serving state 写入增加轻量互斥。Recommendation 与 Learning engine 之间不做跨模块分布式事务，接受边界时间点的短暂最终一致。

## 11. 性能与预计算

`v_recommendable_video_units` 与 `v_unit_video_inventory` 都应预计算，避免在热路径中频繁临时 join / group by。`video_unit_index` 必须具备以 `coarse_unit_id -> candidate videos` 为核心的读索引，`video_semantic_spans` 必须支持按 `video_id + coarse_unit_id` 或按 `video_id + evidence ref` 的证据回查。Evidence Resolver 不能全量扫 spans，必须只按 candidate videos、candidate units 与 `evidence_span_refs` 做定向解析。每条 lane 都要有独立 cap，避免某一条 lane 候选爆炸。Explanation 一律采用模板化拼接，不走大模型生成。

## 13. 测试设计

单测必须覆盖：Demand Planner 的 bucket precedence、hard/new/soft/future 分桶、supply-aware `new_now`、lane budget 动态调整；Candidate Generator 的 `exact_core`、`bundle` 动态放宽、`soft_future`、fallback 触发条件；Evidence Resolver 的 `evidence_span_refs` 回查与本地 best 选择；Aggregator 的去重、best evidence 选择、四类 bucket 覆盖统计；Ranker 的 `demand_coverage`、`bundle_value_score` 不重复计分、`coverage_strength_score`、overload penalty；Selector 的三种模式、主线覆盖约束、future/fallback 比例与 under-fill；Explanation Builder 的 reason code 与模板拼接。

集成测试至少覆盖以下场景：正常库存 + 正常复习；`hard_review` 供给弱；极度稀疏库存；长视频反复命中同一 unit；future 候选很多但 hard_review 也很多；`evidence_span_refs` 与 span 表不一致。Golden tests 应固定若干输入输出快照，包括 `planner_snapshot`、`candidate_summary`、最终 `video_ids` 顺序、`reason_codes`、`best_evidence`。

## 14. 非目标与后续扩展口

当前 Recommendation 明确不做 task layer 进入主流程、非视频 fallback task、embedding / semantic recall、ML ranking、完整 curriculum / backlog 平台化、复杂个性化兴趣模型、Catalog 层的高层语义 enrichment。当前只保留轻量能力：frontier/near-future 信号、bundle lane、inventory-aware planner、简单 diversity selector、quality fallback。未来如果 Catalog 增加更高层的内容 enrichment pipeline，Recommendation 只需要扩展 Planner / Candidate Generator / Aggregator 使用的新特征，而不需要推翻这版模块边界。

## 15. 总结

Recommendation 的最终职责可以概括成一句话：

> 基于 Learning engine 提供的学习状态、Catalog 提供的 `video_unit_index + video_semantic_spans + video_transcript_sentences` 内容索引，以及 Recommendation 自己的 serving state，先生成供给感知的学习需求画像，再通过多路 video-unit 候选生成构造健康的视频候选池，随后解析 evidence refs、在 video 级完成证据聚合、规则化排序与选择，最终稳定输出一组当前足够合理的视频列表，并以 video recommendation run/item 为主对象完成审计与冷却状态更新。

因此，在这份设计里：

- Learning engine 只负责学习事件与学习状态；
- Catalog 只负责内容事实与 Recall-ready 索引；
- Recommendation 只负责视频推荐主链路；
- Recommendation 内部被拆成边界明确、可独立测试、可独立替换的小模块；
- 对外始终只输出 video list；
- 内部通过 Demand Planner、Multi-lane Candidate Generator、Evidence Resolver、Video Evidence Aggregator、Video Ranker、Video Selector 构成一条完整、可解释、可审计的教育视频推荐流水线。
