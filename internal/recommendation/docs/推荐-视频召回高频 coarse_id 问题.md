下面给你一份可直接纳入设计资料库的文档草稿，定位是**问题说明 + 未来优化方向备忘**。
它的立场不是“现在立刻实现”，而是明确记录：**当前可以先不处理，但这是一个迟早会遇到的结构性问题**。

# 高频 Coarse Unit 问题描述与可能解决方案文档

## 1. 文档目的

本文档用于描述推荐系统中“高频 coarse unit”带来的潜在问题，并整理未来可选的治理方案。

当前系统的数据规模仍然较小，相关问题在工程上**暂时不会构成阻塞**，因此本阶段不作为必须实现项。
但从数据分布和召回逻辑来看，这属于一个典型的结构性问题，随着视频数量、用户量和 coarse unit 覆盖规模增加，未来很可能逐渐放大。因此需要提前记录问题边界、影响范围和可选解法，避免后续系统扩张时临时应对。

---

## 2. 背景

当前视频内容索引链路大致为：

```text
transcript JSON
  → sentence / semantic span 结构化入库
  → video_unit_index 聚合
  → Recall 基于 coarse_unit_id 反查 candidate videos
```

在这个链路中，`catalog.video_unit_index` 实际承担了**视频级 coarse unit 倒排索引**的角色。
它的核心用途是支持如下查询：

- 给定一个或一组 `coarse_unit_id`，找到覆盖它们的视频候选
- 基于覆盖次数、覆盖比例、句子分布等特征，对视频进行排序和过滤

这个设计对于大多数中低频 coarse unit 是合理的。
但如果 coarse unit 本身是极高频功能词，或者是几乎每个视频中都会出现的常见表达，那么它们会在倒排索引中形成非常长的 posting list，从而带来一系列问题。

---

## 3. 什么是高频 coarse unit

这里的“高频 coarse unit”并不是一个绝对固定的词类，而是一个相对数据集分布而言的概念。

可以从以下角度理解：

### 3.1 视频覆盖频率很高

如果某个 coarse unit 出现在极高比例的视频中，例如：

- `video_df_ratio > 0.5`
- 或者更极端地，出现在 80% 以上的视频中

那么它在召回中的区分度就会非常弱。

### 3.2 自然语言中本身是高频功能成分

例如：

- 冠词：`the`, `a`
- 介词：`of`, `to`, `in`
- 系动词 / 助动词：`be`, `do`, `have`
- 高频代词或常用口语启动词：`you`, `it`, `okay`

这类 coarse unit 即使从语言学习角度仍然重要，也未必适合作为**视频级召回目标**。

### 3.3 在学习系统中常见但区分度低

有些 coarse unit 不是纯 stopword，但由于它们在内容中几乎普遍出现，也可能在 Recall 层失去筛选价值。

因此，“高频 coarse unit”在这里更准确的定义应是：

**在当前内容库中，对视频召回几乎不提供区分度、却会显著扩大候选集规模的 coarse unit。**

---

## 4. 当前为什么不是必须立刻处理的问题

当前阶段，这个问题可以先不做，原因有三点。

### 4.1 数据规模还小

目前视频规模、索引规模和 coarse unit 覆盖规模都还在可控范围内。
即使某些高频 coarse unit 会命中大量视频，数据库和应用层仍然能够处理。

### 4.2 系统仍处于主链路验证阶段

当前更重要的是先把以下主链路打通并稳定下来：

- 视频入库
- transcript 结构化
- coarse unit 聚合索引
- Recall 候选扩展与排序
- Task 组装

在这个阶段，过早引入复杂的高频 unit 治理逻辑，可能反而会分散实现精力。

### 4.3 高频问题通常在规模增大后才会真正放大

只有当以下条件逐渐出现时，这个问题才会明显：

- 视频总数显著增多
- coarse unit 覆盖更密
- Recall 请求频率升高
- 候选扩展成本变高
- 高频词开始主导目标集合或污染候选质量

所以当前更合适的处理方式不是立即实现，而是**明确记录，未来按需升级**。

---

## 5. 潜在问题描述

尽管当前可暂缓处理，但高频 coarse unit 在未来主要会带来以下问题。

## 5.1 候选集膨胀

如果某个 coarse unit 出现在大量视频中，那么基于它做倒排查询时，会一次性拉出很大的候选集。

例如：

- scheduler 输出了一个高频 coarse unit
- Recall 从 `video_unit_index` 中查询该 unit
- 得到的大部分视频都命中

这会使得：

- SQL 查询结果变大
- 候选扩展阶段负担增加
- 后续应用层排序与过滤成本增加

---

## 5.2 区分度极低

对于 Recall 来说，真正有价值的 coarse unit 应当能帮助回答：

**哪些视频更适合承载当前学习目标。**

但高频 coarse unit 往往只能回答：

**几乎所有视频都沾点关系。**

这会导致它们在召回中的信号价值很低，甚至会稀释真正有区分度的目标。

---

## 5.3 误导 Recall 排序

如果高频 coarse unit 被赋予了正常权重，它们可能会：

- 让大量视频都进入候选池
- 让“覆盖数”或“覆盖比例”看起来很好看
- 但这些覆盖其实不代表真正的教学承载价值

最终会造成一种错觉：

- 好像很多视频都很适合当前目标
- 实际上只是都碰巧含有高频功能词

---

## 5.4 影响 Recall 的稳定性和可解释性

高频 coarse unit 一旦进入 Recall 主信号，可能会让推荐结果变得不稳定。
因为它们覆盖太广，导致真正拉开差距的不是学习价值，而是一些偶然的次级排序因素。

这样会让系统更难解释：

- 为什么这个视频被选中
- 为什么那个视频没被选中
- 当前 coarse unit 到底有没有提供有意义的筛选能力

---

## 5.5 索引规模与查询热点问题

即使当前数据库还能承受，将来随着视频数增长，高频 unit 对应的 posting list 会越来越长。
这意味着：

- 某些 coarse unit 会成为高热点 key
- 相关索引页会更频繁被扫描
- 一旦 Recall 对这些 unit 高频查询，成本会逐渐显著

---

# 6. 当前系统里哪些地方会受到影响

高频 coarse unit 的问题并不只影响 Recall 本身，而是可能影响多个层面。

## 6.1 `catalog.video_unit_index`

这是最直接的影响点。
高频 coarse unit 会在这里形成极长的倒排结果集。

## 6.2 Scheduler → Recall 接口

如果 scheduler 把高频 coarse unit 和普通 coarse unit 一视同仁地作为 recall target 输出，那么 Recall 的输入质量就会下降。

## 6.3 Recall 候选扩展与排序

Recall 的第一阶段通常是：

- 拿到目标 coarse unit
- 从 `video_unit_index` 扩展候选视频

高频 unit 会在这一步直接放大候选规模，并降低候选的区分度。

## 6.4 Task 层的学习任务质量

如果 Recall 候选被高频 coarse unit 污染，那么最终 task 层拿到的视频集合，可能也会变得：

- 覆盖广但不聚焦
- 学习承载度弱
- 解释不够有说服力

---

# 7. 当前阶段建议采取的态度

当前阶段建议采取的不是“立刻实现完整治理”，而是以下策略：

### 第一，保持 awareness

明确知道这是一个真实问题，不假装不存在。

### 第二，先让主链路工作

当前先允许所有 coarse unit 正常进入索引和 Recall 链路，优先验证整体系统可用性。

### 第三，尽量保留未来治理所需的统计基础

即使现在不做，也建议后续能方便地补充：

- coarse unit 的全局视频覆盖统计
- 高频 unit 标记
- recall eligibility 配置

这样将来不需要重构主链路，只需在现有链路上加规则。

---

# 8. 未来可能的解决方案

下面列出几类未来可考虑的方案。
这些方案并不是互斥关系，更可能是分层组合使用。

---

## 8.1 方案一：在 Scheduler 层过滤或降权高频 coarse unit

这是最根本、也通常最有效的方案。

### 思路

不是所有 coarse unit 都适合作为**视频级召回目标**。
高频功能词可能更适合作为：

- 句子解释目标
- 局部学习点
- task 内部练习目标

而不是 Recall 驱动目标。

因此可以在 scheduler 输出 recall target 时：

- 直接排除某些高频 coarse unit
- 或显著降低其 recall 权重
- 或要求它们只能和其它高区分度 coarse unit 一起参与 Recall

### 优点

- 从源头控制污染
- 能有效减小 Recall 候选规模
- 不增加 Recall 查询复杂度

### 缺点

- 需要 scheduler 具备 coarse unit 的全局统计或额外规则
- 会引入“学习对象”和“召回对象”不完全一致的概念，需要在系统设计上讲清楚

### 适用场景

适合中后期，当系统已经明确区分：

- coarse unit 作为学习对象
- coarse unit 作为视频召回目标

这两个角色时使用。

---

## 8.2 方案二：在 Recall 扩展阶段对单个 coarse unit 限流

### 思路

即使 scheduler 没有过滤掉高频 coarse unit，Recall 在候选扩展时也不必无脑拉出所有视频。

可以对每个 coarse unit 设置候选扩展上限，例如：

- 每个 unit 最多取 top 50
- 或 top 100

排序可依据：

- `mention_count`
- `coverage_ratio`
- `sentence_count`
- 其它视频级质量特征

### 优点

- 实现简单
- 能立刻限制候选规模
- 不需要修改上游调度逻辑

### 缺点

- 只能控制成本，不能根治“区分度差”的问题
- 仍然会让高频 unit 占用部分候选池预算

### 适用场景

适合作为最早期、最便宜的缓冲措施。

---

## 8.3 方案三：引入 coarse unit 全局频率统计

### 思路

为每个 coarse unit 维护一张全局统计信息，例如：

- 出现在多少视频中
- 占全部视频的比例是多少
- 是否属于高频 coarse unit
- 是否建议参与 Recall

这类统计可以放在新的表中，例如：

- `semantic.coarse_unit_stats`
- 或未来某个 analytics / derived schema 下

可维护字段包括：

- `coarse_unit_id`
- `video_df`
- `video_df_ratio`
- `is_high_frequency`
- `is_recall_eligible`

### 优点

- 为 scheduler 和 Recall 都提供统一依据
- 可以支持更精细的规则
- 有利于长期治理

### 缺点

- 需要额外的离线统计或周期刷新
- 需要确定口径和更新机制

### 适用场景

适合未来系统规模变大后作为标准基础设施建设。

---

## 8.4 方案四：在 Recall 打分中加入稀有度/区分度修正

### 思路

Recall 对 unit 的贡献分，不只看 scheduler 给的权重，也看这个 unit 的全局区分度。

即使 scheduler 认为某个 unit 当前重要，如果它是一个几乎所有视频都包含的高频 coarse unit，它在 Recall 中的“有效权重”也应下降。

可以设计类似：

```text
effective_unit_weight = scheduler_score * discriminative_weight
```

其中 `discriminative_weight` 可以来源于：

- `video_df_ratio`
- 或某种简单的 IDF-like 权重

### 优点

- 不需要完全禁止高频 coarse unit
- 可以更柔和地调整它们在 Recall 中的作用
- 能和现有加权覆盖逻辑自然结合

### 缺点

- 需要 coarse unit 全局统计支持
- 参数设计需要调试

### 适用场景

适合在 Recall 已经比较稳定后，作为排序优化手段引入。

---

## 8.5 方案五：定义“Recall-eligible coarse unit”子集

### 思路

在系统概念层面明确：

**不是所有 coarse unit 都自动具有视频召回资格。**

也就是说：

- coarse unit 是统一的学习对象集合
- 但其中只有一部分适合作为视频级内容召回目标

可以新增一个概念，例如：

- `recall_eligible`
- `recall_priority_class`
- `recall_exclusion_reason`

### 优点

- 从概念层面彻底讲清楚问题
- 让 Recall 和 Learning 的职责边界更清晰
- 能避免后续不断在规则里打补丁

### 缺点

- 需要产品和系统层接受“学习对象”和“召回对象”不是完全同一个集合
- 需要额外维护这套标记

### 适用场景

适合中长期演进，是比较成熟的做法。

---

## 8.6 方案六：在任务层而不是 Recall 层处理高频 coarse unit

### 思路

对于某些高频 coarse unit，可以不依赖视频召回去专门找视频，而是：

- 它们随着其它更高区分度的视频一起被带出来
- 然后在 task 层中被作为“附带学习点”处理

也就是说：

- Recall 主要由更有区分度的目标驱动
- 高频 coarse unit 更多作为视频内部的次级学习点存在

### 优点

- 减少高频 unit 对 Recall 的干扰
- 更符合某些功能词的学习特点

### 缺点

- 需要 task 层更成熟
- 高频 coarse unit 的显式学习路径会变弱

### 适用场景

适合当 task 层逐渐丰富之后使用。

---

# 9. 各方案的取舍建议

如果未来真的要开始处理这个问题，我建议按以下顺序演进，而不是一口气做完所有方案。

## 第一阶段：最小控制成本

先做：

- Recall 候选扩展限流
- 高频 unit 观测统计

这两项成本低、收益直接。

## 第二阶段：建立全局统计基础

再补：

- coarse unit 全局频率统计
- `is_high_frequency`
- `is_recall_eligible`

这一步是后续所有治理方案的基础。

## 第三阶段：进入策略治理

在此基础上再决定：

- 是在 scheduler 过滤
- 还是在 Recall 打分降权
- 或两者结合

## 第四阶段：进入产品语义层拆分

当系统成熟后，再正式建立：

- coarse unit 学习对象
- coarse unit 视频召回对象

这两个概念的边界。

---

# 10. 当前阶段的建议结论

当前阶段的建议非常明确：

**先不实现高频 coarse unit 的专门治理逻辑。**

原因是：

- 数据量还小，数据库和 Recall 还能承受
- 当前更重要的是先跑通主链路
- 过早做复杂优化会分散实现精力

但同时需要明确：

**这不是“问题不存在”，而是“当前先接受这个问题，未来再治理”。**

所以当前阶段建议做两件事：

### 第一，文档层记录清楚

也就是本文档本身的目的。

### 第二，设计上保留未来治理的接口

例如未来可以很自然地增加：

- coarse unit 全局统计表
- high frequency 标记
- recall eligibility 标记

而不需要推翻现有 catalog 主链路。

---

# 11. 最终结论

高频 coarse unit 问题，本质上不是一个简单的数据库性能问题，而是一个**内容索引区分度问题**。

它会带来的核心影响是：

- 候选集膨胀
- Recall 区分度下降
- 排序信号被污染
- 解释性变弱

当前阶段可以先不处理，因为数据规模不大，系统也仍处于主链路建设期。
但未来一旦视频规模和 coarse unit 覆盖规模增长，这会成为一个必须面对的问题。

因此，当前最合理的策略是：

**暂时接受，提前记录，未来分层治理。**

其中未来最有潜力的治理方向包括：

- Scheduler 过滤或降权高频 coarse unit
- Recall 扩展限流
- coarse unit 全局频率统计
- Recall 中加入区分度修正
- 定义 recall-eligible coarse unit 子集
- 将某些高频 coarse unit 主要留给 task 层处理
