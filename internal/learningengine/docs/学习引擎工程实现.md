# 学习引擎工程实现（Go / PostgreSQL / sqlc）

## 1. 文档目标

本文档只讨论 **Learning engine 模块** 的工程实现。

它回答的问题是：

- 当前 Learning engine 的真实代码结构是什么
- 各层分别承担什么职责
- 事件写入与状态归约如何落到代码
- full replay 如何工程化实现
- persistence / migration / `sqlc` / 测试如何组织

本文档不重复定义完整业务规则细节。学习事件语义、状态迁移、SM-2、progress、mastery 等规则以：

- [学习引擎设计.md](/Users/evan/Downloads/learning-video-recommendation-system/docs/学习引擎设计.md)

为准。

## 2. 工程目标

当前 Learning engine 的工程目标已经明确收敛为：

- 只拥有 `learning.unit_learning_events`
- 只拥有 `learning.user_unit_states`
- 在线写入时先追加事实，再更新状态投影
- replay 只支持 full replay
- 在线写入和 replay 共享同一套 reducer
- 不依赖 Recommendation 的任何内部实现

这意味着 Learning engine 不是一个“推荐子系统”，而是一个独立的学习状态引擎。

## 3. 唯一采用的技术方案

- Go
- PostgreSQL 直连
- `pgx/v5`
- `sqlc`
- application 编排
- domain 纯规则
- infrastructure 持久化落地

Learning engine 坚持这些工程约束：

- 不把领域规则写进 SQL
- 不把 `sqlc` 生成类型直接暴露到 domain
- 不让 repository 负责状态机、SM-2、progress 或 mastery
- 不通过 Recommendation 反向驱动状态维护

### `usecase/`

当前只有两个主用例：

#### `RecordLearningEventsUseCase`

入口文件：

- [record_learning_events.go](/Users/evan/Downloads/learning-video-recommendation-system/internal/learningengine/application/usecase/record_learning_events.go)

职责：

- 接收标准化学习事件
- 在事务中读取当前状态
- 追加写入 `learning.unit_learning_events`
- 调 reducer 计算新状态
- upsert `learning.user_unit_states`

#### `ReplayUserStatesUseCase`

入口文件：

- 读取指定用户的完整事件历史
- 删除该用户所有现有状态

- 批量回写状态

MVP 只支持 full replay。

## 6. 两条主链路的工程实现

## 6.1 在线事件写入链路

真实调用链可以概括为：

```text
RecordLearningEventsUseCase
  -> TxManager.WithinTx
  -> UserUnitStateRepository.GetByUserAndUnit
  -> UnitLearningEventRepository.Append
  -> UserUnitReducer.Reduce
  -> UserUnitStateRepository.Upsert
```

关键工程约束：

- 事件先写，状态后写
- 两者在同一事务中完成
- 失败时不能留下半成品

## 6.2 full replay 链路

真实调用链可以概括为：

```text
ReplayUserStatesUseCase
  -> TxManager.WithinTx
  -> UnitLearningEventRepository.ListByUserOrdered
  -> UserUnitStateRepository.DeleteByUser
  -> UserStateRebuilder.Rebuild
  -> UserUnitStateRepository.BatchUpsert
```

关键工程约束：

- 只按用户全量重建
- 不支持 partial / scoped / incremental replay
- 与在线写入共享同一套 reducer

## 7. 持久化接口设计

当前核心 port 如下。

### `UnitLearningEventRepository`

```go
type UnitLearningEventRepository interface {
    Append(ctx context.Context, events []model.LearningEvent) error
    ListByUserOrdered(ctx context.Context, userID uuid.UUID) ([]model.LearningEvent, error)
}
```

### `UserUnitStateRepository`

```go
type UserUnitStateRepository interface {
    GetByUserAndUnit(ctx context.Context, userID uuid.UUID, coarseUnitID int64) (*model.UserUnitState, error)
    Upsert(ctx context.Context, state *model.UserUnitState) error
    BatchUpsert(ctx context.Context, states []*model.UserUnitState) error
    DeleteByUser(ctx context.Context, userID uuid.UUID) error
}
```

### `TxManager`

```go
type TxManager interface {
    WithinTx(ctx context.Context, fn func(ctx context.Context) error) error
}
```

这三个 port 就定义了 Learning engine application 层与 infrastructure 的最小耦合面。

## 8. `sqlc` 与 schema 约束

Learning engine 继续坚持：

- `sqlc`
- 主题化 SQL 文件
- 生成代码与业务代码隔离
- mapper 隔离 `sqlc` 类型与 domain 模型

当前 `sqlc` 相关目录包括：

- `infrastructure/persistence/query`
- `infrastructure/persistence/schema`
- `infrastructure/persistence/sqlcgen`

要求是：

- `query/` 按主题拆 SQL
- `schema/` 只放 `sqlc` 需要的外部 schema 补充
- `sqlcgen/` 不手改

## 10. 与 Recommendation 的工程边界

Learning engine 与 Recommendation 的工程边界必须保持清楚：

- Learning engine 不 import Recommendation
- Learning engine 不维护 `recommendation.*`
- Learning engine 不关心 `last_recommended_at`
- Learning engine 不关心 run 审计

Recommendation 对 Learning engine 的依赖只允许是：

- 读取 `learning.user_unit_states`

而不是：

- 调用 Learning engine 内部 use case
- 复用 Learning engine 内部 repository 实现

## 11. 当前实现结论

当前 Learning engine 已经是一个独立完整的工程模块。

它提供的稳定能力只有两条：

1. `RecordLearningEvents`
2. `ReplayUserStates`

背后的稳定 owner 只有两张业务表：

1. `learning.unit_learning_events`
2. `learning.user_unit_states`

这也是 Recommendation 能继续在外部稳定消费的前提。
